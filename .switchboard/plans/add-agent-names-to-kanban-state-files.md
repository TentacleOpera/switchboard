# Add Agent Names to Kanban-State File Headers

## Goal

The kanban-state files (`kanban-state-<column>.md`) are written by the extension on every board move. They currently contain the column name and plan list, but not the agent assigned to each column. The `/switchboard` workflow agent needs the actual agent name per role (e.g. `devin`, `claude`, `agy`, `qwen`) at startup — it's already available in `agents.startupCommands` in `state.json` / `GlobalIntegrationConfigService`, but requires a separate sqlite3 query to fetch. If the extension writes the agent name into each kanban-state file's header, the agent gets it in the same awk pass it's already doing — zero extra commands.

### Problem & Root Cause

The `/switchboard` workflow entry protocol reads kanban-state files for board counts in a single awk pass (`.agents/workflows/switchboard.md` lines 57-60). To get agent names per role, it would need a second command (sqlite3 query against kanban.db). This adds latency to startup for data the extension already has. The kanban-state files are the natural home for this — they're already written on every board move and already contain column metadata. The workflow's own "Future" note (switchboard.md lines 118-120) already documents this intent: "once the extension writes agent names into kanban-state file headers, read them from there instead of `/health`."

The per-column state files are generated by `KanbanDatabase._writeLocalBoardMirror()` (`src/services/KanbanDatabase.ts` lines 8073-8171), which runs on a debounced schedule after every `_persist` (board move, plan import, config change). The current per-column file format is:
```
## INTERN CODED

**Column:** INTERN CODED
- [plan-path](plan-path) — Plan Title <!-- planId:... -->
```
The `**Column:**` line is per-plan, not per-header. There is a blank line after the `## COL` heading.

## Complexity Audit

### Routine
- Adding a `**Agent:** <name>` line to the per-column file header in `_writeLocalBoardMirror()` — a few lines of string concatenation in an existing loop.
- The column→role mapping for built-in columns is already available via `DEFAULT_KANBAN_COLUMNS` (imported in `KanbanDatabase.ts` line 8 from `./agentConfig`).
- The agent-name parsing logic already exists in `KanbanProvider._getAgentNames()` (KanbanProvider.ts lines 5609-5617) — same logic can be reused.
- The workflow awk-pass follow-up is a small edit to one awk command in `.agents/workflows/switchboard.md`.

### Complex / Risky
- **Data-source gap:** `KanbanDatabase._writeLocalBoardMirror()` does NOT currently have access to `startupCommands` or `visibleAgents`. `getConfigSync` reads the `config` table of `kanban.db`, which does not contain agent config. Agent config lives in `state.json` and `GlobalIntegrationConfigService` (machine-global store). The writer must read it from a new source.
- **Content-hash staleness:** the content-hash skip (KanbanDatabase.ts lines 8087-8106) hashes `activeFilter` + `allPlans`. If `startupCommands` changes but no card moves, the hash is unchanged → no file rewrite → stale `**Agent:**` line. The agent config must be folded into the hash.
- **Custom columns:** custom columns (KanbanDatabase.ts lines 8127-8135) have no role in the writer's context. Custom agents live in `state.json` `customAgents`. Resolving custom-column→agent requires reading and parsing `customAgents`.

## Edge-Case & Dependency Audit

**Race Conditions:**
- `_writeLocalBoardMirror` is debounced (`_scheduleLocalBoardMirror`, line 8065-8071) and guarded by `_localMirrorInFlight` / `_localMirrorPending` (lines 8075-8078). Reading `state.json` inside the write is safe — it's a snapshot read, not a lock. No new race introduced.

**Security:**
- Agent names are derived from `startupCommands` config (user-controlled, local). No external input. The name is written into a markdown file in `.switchboard/`. No injection risk — markdown comment syntax is not used for the agent line.

**Side Effects:**
- The per-column file format changes. Any external consumer that parses these files (beyond the `/switchboard` awk pass) must tolerate the new `**Agent:**` line. The awk pass counts `planId:` lines — the new header line does not contain `planId:`, so counts are unaffected.
- The content-hash change means the first write after deployment will rewrite all per-column files once (hash input changed). This is a one-time cost.

**Dependencies & Conflicts:**
- The workflow awk-pass follow-up (`.agents/workflows/switchboard.md` entry protocol) depends on this extension change shipping first. The awk command at lines 57-60 must be extended to capture the `**Agent:**` line per column.
- `GlobalIntegrationConfigService` is the machine-global authoritative source for `startupCommands`. If the writer reads `state.json` directly, there is a staleness window when the global store has been updated but `state.json` has not been synced. This is bounded by the next board move and is acceptable for a display-only field.

## Dependencies

- None (no prior plan sessions required).

## Adversarial Synthesis

Key risks: (1) the data-source gap — `KanbanDatabase` has no native access to `startupCommands`/`visibleAgents`, so the writer must read `state.json` directly; (2) content-hash staleness — agent config must be folded into the hash or config-only changes won't trigger a rewrite; (3) custom columns have no role in the writer's context and need `customAgents` parsing. Mitigations: read `state.json` in the writer (Option 1), fold agent config into the content hash, and resolve custom-column roles via `customAgents` + `toCustomAgentRole`. The configured-vs-live distinction is accepted as a limitation — the header carries configured names; liveness stays with `/health`.

## Proposed Changes

### `src/services/KanbanDatabase.ts` — `_writeLocalBoardMirror()` (lines 8073-8171)

**Context:** This method writes per-column `kanban-state-<slug>.md` files and the `kanban-board.md` index on every debounced board-mirror schedule. It currently has no access to agent config.

**Logic:**

1. **Read agent config at the top of `_writeLocalBoardMirror`** (after the `workspaceId` guard, before the content-hash computation). Read `state.json` from `path.join(this._workspaceRoot, '.switchboard', 'state.json')`. Parse `startupCommands` (role→command string map) and `visibleAgents` (role→boolean map) and `customAgents` (array of custom agent configs). Wrap in try/catch — on failure, default to empty maps (no `**Agent:**` lines written). This is Option 1 (direct `state.json` read).

   > **Superseded:** The original plan said "Read `agents.startupCommands` from config" without specifying the source.
   > **Reason:** `KanbanDatabase.getConfigSync` reads the `config` table of `kanban.db`, which does NOT contain `startupCommands`. The agent config lives in `state.json` / `GlobalIntegrationConfigService`. The plan omitted the access path entirely.
   > **Replaced with:** Read `state.json` directly in `_writeLocalBoardMirror` via `fs.promises.readFile`. This is the simplest path that doesn't require a new import or a config-sync layer. The staleness window versus `GlobalIntegrationConfigService` is bounded by the next board move and acceptable for a display-only field.

2. **Build a column→role→agentName resolver.** For each column ID in `allColumns`:
   - Look up the role: check `DEFAULT_KANBAN_COLUMNS` (already imported, line 8) for built-in columns. For custom columns, attempt to match the column ID against `customAgents` entries (each custom agent has a `role`; use `toCustomAgentRole` from `agentConfig.ts` to derive the role from the column ID if the custom agent's column matches). If no role resolves, skip the `**Agent:**` line for that column.
   - Check visibility: if `visibleAgents[role] === false`, skip the `**Agent:**` line. (Default visibility is `true` for most roles; `tester`, `researcher`, `ticket_updater`, `jules` default to `false` — see KanbanProvider.ts lines 5642-5652.)
   - Parse the agent name from `startupCommands[role]`: take the command string, split on whitespace, take the first token, apply `path.basename`, strip `.(exe|cmd|bat)$`, uppercase, append ` CLI`. This mirrors `_getAgentNames` (KanbanProvider.ts lines 5612-5614) exactly. If no command is configured, skip the `**Agent:**` line (do not write "No agent assigned" — the absence of the line is cleaner for the awk consumer).

   > **Superseded:** The original plan said to strip flags and write the raw binary name (e.g. `devin`).
   > **Reason:** The codebase's existing display convention (used in the kanban webview and setup panel) is `DEVIN CLI` — uppercase, basename, ` CLI` suffix. Writing a different format into the state files creates inconsistency.
   > **Replaced with:** Use the same parsing as `_getAgentNames`: `path.basename(cmd.split(/\s+/)[0]).replace(/\.(exe|cmd|bat)$/i, '').toUpperCase() + ' CLI'`. The workflow awk pass can transform this if a shorter form is needed for display.

3. **Write the `**Agent:**` line into the per-column file header.** In the per-column loop (line 8142-8171), after `let colMd = \`## ${col}\n\n\`;` (line 8144), insert:
   ```typescript
   const agentName = this._resolveAgentForColumn(col, startupCommands, visibleAgents, customAgents);
   if (agentName) {
       colMd += `**Agent:** ${agentName}\n\n`;
   }
   ```
   The resulting format for a column with a configured, visible agent:
   ```
   ## INTERN CODED

   **Agent:** AGY CLI

   **Column:** INTERN CODED
   - [plan-path](plan-path) — Plan Title <!-- planId:... -->
   ```
   For a column with no agent configured or not visible, the format is unchanged (no `**Agent:**` line).

   > **Superseded:** The original plan proposed the format `## COL\n**Agent:** agy\n**Column:** COL\n- [...]`.
   > **Reason:** The actual code format has a blank line after `## COL`, and `**Column:**` is a per-plan line, not a per-header line. The proposed format did not match the code being modified.
   > **Replaced with:** `## COL\n\n**Agent:** <name>\n\n` header, followed by the existing per-plan `**Column:**` / `- [...]` lines.

4. **Fold agent config into the content hash.** In the content-hash computation (lines 8092-8103), add the agent config to the serialized object:
   ```typescript
   const serialized = JSON.stringify({
       activeFilter,
       agentConfig: { startupCommands, visibleAgents, customAgents },
       allPlans: allPlans.map(p => ({ ... }))
   });
   ```
   This ensures that changing `startupCommands` (without a card move) still triggers a rewrite, so the `**Agent:**` line stays current. Without this, the plan's own verification criterion ("agent name should update if startupCommands is changed") would fail.

5. **Add a private helper `_resolveAgentForColumn`.** Add a new private method to `KanbanDatabase`:
   ```typescript
   private _resolveAgentForColumn(
       columnId: string,
       startupCommands: Record<string, string>,
       visibleAgents: Record<string, boolean>,
       customAgents: any[]
   ): string | null {
       // 1. Find role for this column
       const builtIn = DEFAULT_KANBAN_COLUMNS.find(c => c.id === columnId);
       let role: string | undefined = builtIn?.role;
       if (!role) {
           // Custom column: try to match via customAgents
           const customMatch = customAgents.find(a => a.column === columnId);
           if (customMatch) role = customMatch.role;
       }
       if (!role) return null;

       // 2. Check visibility (default true if not explicitly false)
       if (visibleAgents[role] === false) return null;

       // 3. Parse agent name from startup command
       const cmd = (startupCommands[role] || '').trim();
       if (!cmd) return null;
       const binary = cmd.split(/\s+/)[0];
       const name = path.basename(binary).replace(/\.(exe|cmd|bat)$/i, '').toUpperCase();
       return `${name} CLI`;
   }
   ```
   Note: the `customAgents` column-matching field name must be verified against the `CustomAgentConfig` shape in `agentConfig.ts` — the custom agent config may use a different field name for the column association. Check `parseCustomAgents` and `CustomAgentConfig` interface before implementing.

### `src/services/KanbanDatabase.ts` — `state.json` read helper

Add a private helper to read agent config from `state.json`:
```typescript
private async _readAgentConfig(): Promise<{
    startupCommands: Record<string, string>;
    visibleAgents: Record<string, boolean>;
    customAgents: any[];
}> {
    try {
        const statePath = path.join(this._workspaceRoot, '.switchboard', 'state.json');
        const content = await fs.promises.readFile(statePath, 'utf8');
        const state = JSON.parse(content);
        return {
            startupCommands: state.startupCommands || {},
            visibleAgents: state.visibleAgents || {},
            customAgents: state.customAgents || []
        };
    } catch {
        return { startupCommands: {}, visibleAgents: {}, customAgents: [] };
    }
}
```

### `.agents/workflows/switchboard.md` — entry protocol awk pass (lines 57-60) — FOLLOW-UP after extension change ships

**Context:** The entry protocol's Command B awk pass (lines 57-60) counts plans and features per column from the kanban-state files. It must be extended to also extract the `**Agent:**` line per column.

**Logic:** Extend the awk to capture the agent name:
```bash
awk 'FNR==1{col=FILENAME; sub(/.*kanban-state-/,"",col); sub(/\.md$/,"",col); plans[col]+=0; feats[col]+=0; agent[col]=""}
     /^\*\*Agent:\*\*/{ agent[col]=$0; sub(/^\*\*Agent:\*\* /,"",agent[col]) }
     /planId:/{ if (/ feature -->/) feats[col]++; else plans[col]++ }
     END{for (c in plans) printf "%s: %d plans, %d features, agent=%s\n", c, plans[c], feats[c], agent[c]}' \
  "$ROOT"/.switchboard/kanban-state-*.md
```
Then update the terminals-line rendering (switchboard.md lines 114-120) to use the agent names from the awk output instead of `/health`, and remove the "Future" note at lines 118-120 (it becomes the present).

**Edge Cases:** Columns with no `**Agent:**` line produce `agent=` (empty). The workflow should render those roles as "no agent configured" rather than blank. The `/health` terminals field remains the source for *liveness* — the awk gives *configured* names, `/health` gives *running* names. Both are needed: configured names for the role labels, `/health` for the live indicator.
