# Fix: Runsheet JSON Race Condition

> [!IMPORTANT]
> This plan has been updated with findings from the `/challenge` adversarial review. See **Challenge Review Amendments** section before implementing.

When the agent edits the brain plan file (e.g. [implementation_plan.md](file:///C:/Users/patvu/.gemini/antigravity/brain/d48ab396-b6ef-48e7-86e5-5c465fba5785/implementation_plan.md)), the VS Code file watcher fires multiple change events within milliseconds. Each event triggers a separate [updateRunSheet](file:///c:/Users/patvu/Documents/GitHub/switchboard/src/services/SessionActionLog.ts#404-419) call, which does an unguarded read-modify-write on the same JSON file. Concurrent interleaved writes corrupt the JSON. A malformed JSON file is silently dropped by the sidebar parser → plan disappears from the dropdown.

**Two vulnerable code paths identified:**

1. **`SessionActionLog.updateRunSheet`** — read → `JSON.parse` → mutate → [writeFile](file:///c:/Users/patvu/Documents/GitHub/switchboard/src/services/TaskViewerProvider.ts#5551-5557), with no per-file lock. Called from 4 sites in [TaskViewerProvider](file:///c:/Users/patvu/Documents/GitHub/switchboard/src/services/TaskViewerProvider.ts#68-6411).
2. **`TaskViewerProvider._handlePlanTitleSync`** (line 3382) — does its own raw `readFile` → `JSON.parse` → patch → [writeFile](file:///c:/Users/patvu/Documents/GitHub/switchboard/src/services/TaskViewerProvider.ts#5551-5557) completely bypassing [updateRunSheet](file:///c:/Users/patvu/Documents/GitHub/switchboard/src/services/SessionActionLog.ts#404-419), so any mutex there wouldn't protect this path.

The correct pattern (store `next` first, then register `.finally()` as the map value):
```typescript
async updateRunSheet(sessionId: string, updater: (current: any) => any): Promise<void> {
    const tail = this._writeLocks.get(sessionId) ?? Promise.resolve();
    const next: Promise<void> = tail.then(() => this._doUpdateRunSheet(sessionId, updater));
    this._writeLocks.set(sessionId, next.catch(() => {}).finally(() => {
        if ((this._writeLocks.get(sessionId) as unknown) === next) {
            this._writeLocks.delete(sessionId);
        }

    return next;
}
```

### Fix 4 — Kanban Auto-Promotion: Route via IPC instead of direct file write

**Root cause**: [appendRunSheetEvent](file:///c:/Users/patvu/Documents/GitHub/switchboard/src/mcp-server/register-tools.js#373-396) in [register-tools.js](file:///c:/Users/patvu/Documents/GitHub/switchboard/src/mcp-server/register-tools.js) writes directly to the runsheet from the MCP server process. The extension's file watcher fires on the concurrent `state.json` update from `stop_workflow`, reads the runsheet, and writes it back without the just-appended challenge event — erasing it. The extension-side mutex doesn't protect cross-process writes.

**Fix**: Replace the direct [appendRunSheetEvent](file:///c:/Users/patvu/Documents/GitHub/switchboard/src/mcp-server/register-tools.js#373-396) call in `complete_workflow_phase` with an IPC message to the extension. The extension already has an `mcpServerProcess.on('message', ...)` handler — add a new message type [appendRunSheetEvent](file:///c:/Users/patvu/Documents/GitHub/switchboard/src/mcp-server/register-tools.js#373-396) that routes through `SessionActionLog.updateRunSheet` (and thus the mutex).
