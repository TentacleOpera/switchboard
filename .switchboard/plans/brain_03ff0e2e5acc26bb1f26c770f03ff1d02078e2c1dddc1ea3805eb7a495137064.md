# Feature Plan: Add Acceptance Tester Role

## Goal
Add a new `tester` built-in agent role and corresponding `ACCEPTANCE TESTED` Kanban column. This role executes after the `Reviewer` and is responsible for comparing the implemented code strictly against the original product requirements document (PRD) to identify any missed requirements. Because it relies heavily on the design document, this role is **disabled by default** and strictly requires a Design Doc / PRD to be attached in the Switchboard setup.

## Complexity Audit
### Routine
- Expand `BuiltInAgentRole`, `BUILT_IN_AGENT_LABELS`, and `DEFAULT_KANBAN_COLUMNS` in `agentConfig.ts`.
- Expand column mapping and dispatch dictionaries in `KanbanProvider.ts` and `TaskViewerProvider.ts`.
- Add `tester` to `PROMPT_ROLES` and the default visibility maps in `implementation.html`.
- Add a new `tester` branch in `buildKanbanBatchPrompt()` (`agentPromptBuilder.ts`).

### Complex / Risky
- **Conditional Kanban Pipeline Routing:** `TaskViewerProvider._getNextKanbanColumnForSession()` currently hardcodes `CODE REVIEWED` to return `null` (end of pipeline). We must change this to return `ACCEPTANCE TESTED`, but **only if the tester agent is enabled**. If we don't conditionally guard this, advancing a card when the tester is disabled will send it to a hidden column, trapping the plan.
- **Strict PRD Enforcement Gate:** The `tester` dispatch logic must explicitly check `_isDesignDocEnabled()`. If false, it must cleanly block the dispatch and surface an error message to the user, preventing a useless token-wasting prompt.

## Dependencies & Conflicts
- No known conflicts with active plans. Additive changes to prompt builders and switch statements.

## Adversarial Synthesis

### Grumpy Critique
> You think you can just wire up a new role and call it a day? Let's look at the traps you laid out:
> 
> 1. **The Phantom Column Trap:** In `TaskViewerProvider.ts`, if you change `CODE REVIEWED` to advance to `ACCEPTANCE TESTED`, what happens for the 99% of users who have this disabled by default? The card moves to a column with `hideWhenNoAgent: true`. The card will literally vanish from their board. You must check agent visibility before advancing!
> 
> 2. **`planStateUtils.ts` Data Loss:** If you don't add `ACCEPTANCE TESTED` to `VALID_COLUMNS` in `planStateUtils.ts`, any Kanban state imported from the clipboard or read from disk will fail validation. The plan will default back to `CREATED`, destroying its workflow progress.
> 
> 3. **The Empty Payload Trap:** The prompt builder needs the PRD content. But wait—`_getDesignDocContent()` in `TaskViewerProvider` is an async call. If you don't explicitly fetch and pass `designDocContent` in the `tester` dispatch branch, the agent gets a prompt telling it to review a PRD that isn't there. 

### Balanced Response
Grumpy is entirely correct about the Kanban routing and state validation risks. The implementation below mitigates these by:
1. **Dynamic Pipeline Routing:** `_getNextKanbanColumnForSession` will fetch `visibleAgents` and only return `ACCEPTANCE TESTED` if `visibleAgents.tester !== false`. Otherwise, it correctly returns `null` (Completed).
2. **State Validation Update:** Explicitly adding `ACCEPTANCE TESTED` to the `VALID_COLUMNS` set in `planStateUtils.ts`.
3. **PRD Guard & Enrichment:** Added a hard block in `_handleTriggerAgentActionInternal` if `_isDesignDocEnabled()` is false, and explicitly passing the asynchronously fetched `designDocContent` into the prompt builder.

## Verification Plan
1. **Disabled by Default:** Open Switchboard on a clean setup. Confirm the Acceptance Tester does not appear in the sidebar or the Kanban board.
2. **Hidden Column routing:** Move a card into `CODE REVIEWED`, then copy prompt/advance it. Ensure it moves to `COMPLETED` when Tester is disabled.
3. **Missing PRD Guard:** Enable the Acceptance Tester in settings. Move a card into `ACCEPTANCE TESTED` and attempt to dispatch. Ensure it blocks with an error toast asking for a PRD.
4. **Success Path:** Attach a Notion PRD. Click dispatch. Ensure the prompt is emitted correctly containing the `"check the code against the relevant sections of the prd..."` instruction alongside the PRD content.
