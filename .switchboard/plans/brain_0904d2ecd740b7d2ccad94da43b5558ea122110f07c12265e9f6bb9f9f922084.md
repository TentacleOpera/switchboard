# Structured Research Prompt Upgrades

This plan introduces structured, high-quality research prompt generation to the Switchboard webview's Research tab and the backend's `DEEP_RESEARCH_DIRECTIVE`. It translates simple topics entered by the user into a rigorous briefing prompt following the structure of professional security and privacy research reports, while leaving pre-structured user inputs untouched.

## Goal

Upgrade the Research tab to generate structured, professional-grade research prompts from simple topic inputs, and refine the backend `DEEP_RESEARCH_DIRECTIVE` to produce more rigorous synthesis output.

## Metadata

- **Tags:** [frontend, backend, UX, workflow]
- **Complexity:** 4

## User Review Required

> [!NOTE]
> The generated prompt template assumes a generic software engineering context by default. If the user has already input structured fields (like `ROLE`, `CONTEXT`, etc.), the generator will bypass the template to preserve their custom input.

## Open Questions

None.

## Complexity Audit

### Routine
- Adding a label and textarea to `planning.html` (single DOM insertion, existing CSS classes)
- Updating `generateResearchPrompt()` to read a second textarea value
- Appending clarifying-question instructions to `web_research.md` and `deep_planning.md` (pure text edits to markdown skill files)
- Updating the `DEEP_RESEARCH_DIRECTIVE` string constant in `agentPromptBuilder.ts` (single string edit)

### Complex / Risky
- Regex-based detection of pre-structured prompts in `generateResearchPrompt()` — must avoid false positives (e.g., `PROXY_ROLE` matching `ROLE`) and false negatives (e.g., `## Role` not matching). Requires a precise regex and clear matching criteria.

## Edge-Case & Dependency Audit

- **Race Conditions:** None. All UI interactions are synchronous (clipboard copy, message dispatch).
- **Security:** User input flows into a clipboard string and a `vscode.postMessage` payload. No HTML rendering of prompt content occurs — the prompt is plain text. Low risk in a local VS Code extension context.
- **Side Effects:** The `sendToAnalyst` flow (planning.js line 217) calls `generateResearchPrompt()` directly, so the structured template will automatically flow to the analyst terminal without additional wiring. No changes needed to `PlanningPanelProvider.ts` or `extension.ts`.
- **Dependencies & Conflicts:** The `DEEP_RESEARCH_DIRECTIVE` constant is consumed in two places — `buildKanbanBatchPrompt()` (line 1136) and the research-specific prompt builder (lines 842, 889). Both use `.replace()` to parameterize depth. The Phase 4 synthesis refinement only adds content to the string constant; the `.replace()` calls remain unaffected.

## Dependencies

None.

## Adversarial Synthesis

Key risks: (1) Regex for structured-prompt detection may produce false positives/negatives — mitigated by using a line-anchored regex `/^(ROLE|CONTEXT|OBJECTIVE|TASK|INSTRUCTIONS):/m` that requires a colon-delimited header at line start. (2) The generic sub-questions (moves 3, 5, 7) may feel too generic for domain-specific research — mitigated by the context textarea feeding move 2 and the user's topic always being sub-question 1; the generic follow-ups are intentionally broad to avoid overfitting. (3) The `DEEP_RESEARCH_DIRECTIVE` change appends SOURCE GUIDANCE and DECISION THIS FEEDS lines that duplicate the webview template's moves 5 and 6 — this is intentional so that agents receiving the directive directly (without the webview template) still follow the same discipline.

Enhance `generateResearchPrompt()` (lines 2377-2424):

- **Line ~2380:** Add `const contextInput = document.getElementById('research-context-input');` alongside the existing `promptInput` retrieval.
- **Line ~2384:** Add `const contextValue = contextInput ? contextInput.value.trim() : '';` alongside the existing `customPrompt` extraction.
- **After line 2384, before the prompt construction:** Add structured-prompt detection:
  ```js
  const STRUCTURED_PROMPT_RE = /^(ROLE|CONTEXT|OBJECTIVE|TASK|INSTRUCTIONS):/m;
  const isStructured = STRUCTURED_PROMPT_RE.test(customPrompt);
  ```
  This regex matches any line starting with one of the five common structured-prompt headers followed by a colon. It is line-anchored (`^` with `m` flag) to avoid false positives like `PROXY_ROLE`.

- **Replace the prompt construction block (lines 2404-2421):** If `isStructured` is true, emit `customPrompt` directly (no template wrapping). Otherwise, wrap in the 7-move structured template:

  **Move 1 — Role + quality bar:**
  ```
  ROLE: You are a research analyst. Prefer authoritative primary sources over blogs and marketing copy; where sources conflict, say so explicitly.
  ```

  **Move 2 — Context + who's reading:**
  ```
  CONTEXT: {contextValue, or "General software engineering and technology" as fallback}. The reader is a software engineer — explain domain-specific concepts; do not assume specialist expertise.
  ```
  The "who's reading this" line is always emitted. If the user provided context, it fills the first sentence; otherwise the static fallback is used. No codebase detection logic.

  **Move 3 — Central question + prioritized sub-questions:**
  ```
  CENTRAL QUESTION: {customPrompt}
  SUB-QUESTIONS (cover all, lead with the first three):
    1. {customPrompt} — core framing
    2. What are the current best practices and authoritative standards?
    3. What are the key trade-offs and failure modes?
    4. What is the current state of the art and recent developments?
  ```
  The user's input becomes both the central question and sub-question 1. Sub-questions 2-4 are generic follow-ups that apply to any research topic. If the user provided context, sub-question 1 can be refined to incorporate it (e.g., "How does {topic} apply to {context}?").

  **Move 4 — Output shape:**
  ```
  OUTPUT:
  1) Executive summary (≤ 1 page)
  2) Tiered findings: required vs recommended vs optional — clearly distinguish compliance levels
  3) Focused trade-off evaluation (e.g. searchability vs confidentiality, cost vs coverage)
  4) Defence-in-Depth checklist
  5) Plain-English glossary of domain-specific terms
  6) Full source list with direct links and retrieval dates
  ```

  **Move 5 — Decision this feeds:**
  ```
  DECISION THIS FEEDS: General engineering guidance — end with a recommended default for a platform of typical scale.
  ```
  This is the generic fallback. If the user provided context that implies a specific decision (e.g., mentions a technology choice or compliance question), the implementer may refine this line. But the static fallback ensures every prompt asks the agent to land on a recommendation, not just survey the field.

  **Move 6 — Honesty / anti-hallucination guardrails:**
  ```
  SOURCE GUIDANCE: Prefer official documentation, standards bodies, and peer-reviewed sources; distrust vendor marketing claims. Date-check all sources — flag anything older than 2 years. Separate "required" from "recommended" from "opinion" in every finding. Where law or standards are silent or ambiguous, say so rather than assuming applicability.
  ```

  **Move 7 — Scope:**
  ```
  SCOPE: Primary focus is the central question above. Related domains and alternative approaches as clearly-labelled benchmarks only. Out of scope: unrelated technology stacks, unrelated jurisdictions, and pure marketing comparisons.
  ```

  **Full assembled template** (what `generateResearchPrompt()` emits when `isStructured` is false):
  ```
  ROLE: You are a research analyst. Prefer authoritative primary sources over blogs and marketing copy; where sources conflict, say so explicitly.

  CONTEXT: {contextValue or "General software engineering and technology"}. The reader is a software engineer — explain domain-specific concepts; do not assume specialist expertise.

  CENTRAL QUESTION: {customPrompt}
  SUB-QUESTIONS (cover all, lead with the first three):
    1. {customPrompt} — core framing
    2. What are the current best practices and authoritative standards?
    3. What are the key trade-offs and failure modes?
    4. What is the current state of the art and recent developments?

  SOURCE GUIDANCE: Prefer official documentation, standards bodies, and peer-reviewed sources; distrust vendor marketing claims. Date-check all sources — flag anything older than 2 years. Separate "required" from "recommended" from "opinion" in every finding. Where law or standards are silent or ambiguous, say so rather than assuming applicability.

  SCOPE: Primary focus is the central question above. Related domains and alternative approaches as clearly-labelled benchmarks only. Out of scope: unrelated technology stacks, unrelated jurisdictions, and pure marketing comparisons.

  OUTPUT:
  1) Executive summary (≤ 1 page)
  2) Tiered findings: required vs recommended vs optional — clearly distinguish compliance levels
  3) Focused trade-off evaluation (e.g. searchability vs confidentiality, cost vs coverage)
  4) Defence-in-Depth checklist
  5) Plain-English glossary of domain-specific terms
  6) Full source list with direct links and retrieval dates

  DECISION THIS FEEDS: General engineering guidance — end with a recommended default for a platform of typical scale.

  DEPTH: {complexityLabels[complexity]}
  ```

- **Clarification:** The fallback context is a static string `"General software engineering and technology"` — no codebase detection logic. The "who's reading this" line always reads `"The reader is a software engineer — explain domain-specific concepts; do not assume specialist expertise."` This keeps the implementation simple and predictable while still capturing the highest-value line most prompts skip.

- **Note:** Both the COPY PROMPT button (line 195) and the SEND ANALYST REQUEST button (line 217) call `generateResearchPrompt()`, so the structured template automatically flows to both outputs without any changes to the message dispatch logic.
