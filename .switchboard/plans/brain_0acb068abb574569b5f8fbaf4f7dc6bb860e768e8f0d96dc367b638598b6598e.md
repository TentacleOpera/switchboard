# Audit Report: Autism 360 Technical Documentation

This report identifies, categorizes, and prioritizes documentation gaps across the Autism 360 platform.

## 1. Executive Summary
The Autism 360 platform has a robust technical foundation, but its documentation is currently fragmented and lacks depth in several critical areas, specifically regarding external integrations, infrastructure secrets, and specialized Cloud Functions. While most repositories have a basic `README.md`, they often provide only installation and basic usage instructions without explaining the underlying architecture or complex logic flows.

## 2. Identified Documentation Gaps

### A. Internal Subsystems (Cloud Functions)
| Subsystem | File(s) | Gap Description | Priority |
| :--- | :--- | :--- | :--- |
| **Zoom Recording** | `sessions/handleSessionRecordings.js` | Logic for Zoom webhook verification and file transfer to GCS is undocumented. | High |
| **Session Summarizer** | `sessions/summerizeSession.js` | The LangChain-based 3-stage summarization pipeline (Facts -> Dedupe -> Style) is complex and undocumented. | High |
| **Stripe/Kartra** | `user/handleStripeSubscription.js` | The logic for differentiating between standard Stripe and Kartra (Standard/PayPal) checkouts is undocumented. | High |
| **Revenue Cat** | `revenuecat/handleRevenueEvents.js` | Handler for Revenue Cat events and their impact on user state is undocumented. | Medium |

### B. Infrastructure & Secrets
| Area | Source | Gap Description | Priority |
| :--- | :--- | :--- | :--- |
| **Secrets Inventory** | `be/app/utils/secrets.js` | Over 60+ secrets are loaded from GCP Secret Manager without documentation of their purpose, allowed values, or rotation policies. | Critical |
| **GCP Infrastructure** | `.gitlab-ci.yml` | The GKE cluster setup, Helm chart configurations, and GCR registry patterns are only defined in code. | Medium |
| **Firebase Config** | `viaapp/functions/index.js` | Regional deployment constraints (e.g., Zoom handlers MUST be in AU) are only mentioned in comments. | Medium |

### C. External Service Integrations
| Service | Integration Point | Gap Description | Priority |
| :--- | :--- | :--- | :--- |
| **OpenAI Assistants** | `ai/app/controllers/` | The configuration of Assistant IDs and the specific prompts used for onboarding vs. daily chat are undocumented. | High |
| **Monday.com** | `be/app/utils/secrets.js` | Integration with Monday.com for tracking (indicated by secrets) is entirely undocumented. | Low |
| **Onesignal** | `be/app/utils/onesignal.js` | Push notification templates and trigger logic are undocumented. | Medium |

### D. Repository-Level Documentation
| Repository | State | Gap Description | Priority |
| :--- | :--- | :--- | :--- |
| **AI (`ai`)** | Very Sparse | Lacks architecture diagram, WebSocket message schema, and deployment details. | High |
| **Backend (`be`)** | Basic | Lacks detailed API documentation (Swagger exists but is external), database schema diagrams, and service layer logic. | Medium |
| **Frontend (`fe`)** | Basic | Lacks state management overview and component library usage. | Low |
| **Mobile (`viaapp`)** | Basic | Lacks release process documentation (Fastlane is used but the flow is undocumented). | High |

## 3. Prioritized Action Plan

### Phase 1: Critical & High Priority (Next 2-4 Weeks)
1.  **Secrets Master List**: Document every key in `be/app/utils/secrets.js` with its purpose and owner.
2.  **Cloud Functions Deep-Dive**: Create a `README.md` in `viaapp/functions/sessions` explaining the Zoom-to-LangChain-to-Backend pipeline.
3.  **AI Service Schema**: Document the WebSocket API for the `ai` service, including message formats for `/onboarding` and `/daily-chat`.
4.  **Mobile Release Guide**: Document the Fastlane lanes and certificate management for iOS/Android releases.

### Phase 2: Medium Priority (Next 1-2 Months)
1.  **Integration Guides**: Create a "Third-Party Integrations" section in the docs covering Stripe, Kartra, and Revenue Cat.
2.  **Infrastructure Overview**: Document the GKE cluster architecture and CI/CD flow.
3.  **Backend Service Layer**: Document core service logic (e.g., auth, tracking, subscription management).

### Phase 3: Low Priority (Continuous)
1.  **Frontend Component Library**: Document reusable UI components.
2.  **Monday.com Sync**: Document the legacy or background sync tasks for Monday.com.

## 4. Conclusion
Addressing these documentation gaps will significantly reduce onboarding time for new developers and decrease the "bus factor" for critical subsystems like AI and payments.
