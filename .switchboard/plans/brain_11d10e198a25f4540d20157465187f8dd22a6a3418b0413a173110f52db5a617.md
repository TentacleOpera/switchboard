# Prevent Cross-Workspace Brain Plan Leakage

The issue occurs because Antigravity brain plans are saved to a global directory (`~/.gemini/antigravity/brain/`) without encoded workspace metadata. 
Consequently, every open VS Code workspace running the Switchboard extension watches this folder and receives a file creation event when a new plan is generated. Because each workspace relies on its isolated, local `kanban.db` to check if it already claimed the plan, they all simultaneously assume it's untracked (`existingEntry = false`) and aggressively auto-claim it, thereby mirroring the plan into their respective Kanban boards.

## Goal

Gate the auto-claim decision in the brain watcher behind `vscode.window.state.focused` so that only the focused VS Code window auto-claims new brain plans from the shared global directory.

## Metadata
**Tags:** backend, bugfix
**Complexity:** 5

## User Review Required

> [!WARNING]
> After this change, if a user creates a brain plan while **no** VS Code window is focused (e.g., from an external terminal with all IDE windows minimized), the plan will not be auto-claimed by any workspace. The plan will still be mirrored on the next content-change event when the user focuses a window, but the initial auto-claim will be skipped. This is the safe default to prevent cross-workspace leakage.

## Complexity Audit
### Routine
- Reading `vscode.window.state.focused` is a synchronous, stable VS Code API — no new dependencies.
- The change is a simple boolean intersection (`allowAutoClaim && isFocused`) applied at two call sites within the same method.
- No database schema changes, no new files, no new configuration.

### Complex / Risky
- **Timing of focus capture:** The `isFocused` value must be captured at event-fire time (before the 300ms debounce `setTimeout`), not inside the debounce callback. If captured inside the callback, a rapid Alt-Tab could cause focus to shift between event and execution, leading to incorrect claim decisions.
- **`fs.watch` fallback path:** The native `fs.watch` handler (line 4313) also feeds into `_mirrorBrainPlan` with `allowAutoClaim = (_eventType === 'rename')`. This path must apply the same focus gate.

## Edge-Case & Dependency Audit
- **Race Conditions:** Two workspaces receive the same `onDidCreate` event near-simultaneously. Without the focus gate, both auto-claim. With it, only the focused window proceeds. If the user Alt-Tabs during the 300ms debounce window, the captured-at-fire-time snapshot ensures the originally-focused window still claims. The unfocused window logs a skip diagnostic.
- **Security:** No new attack surface. `vscode.window.state.focused` is a read-only VS Code API.
- **Side Effects:** Plans created while no window is focused will not be auto-claimed on creation. They will appear when the user next focuses a window and a `onDidChange` event fires (e.g., brain file is modified), but that event passes `allowAutoClaim = false` anyway. **Clarification:** this is acceptable because `onDidChange` already disables auto-claim — the plan will be mirrored but not registered until the user explicitly claims it via the Kanban UI.
- **Dependencies & Conflicts:** No conflicts with other Planned-column items. "Fix Clipboard Import Preamble Stub Issue" touches `_importMultiplePlansFromClipboard` (line 9142). "Switchboard Repository Exclusion System" touches `extension.ts`. Neither overlaps with `_setupBrainWatcher` (line 4256).

## Adversarial Synthesis
### Grumpy Critique
*"So you're betting the entire multi-workspace isolation strategy on a single boolean that reflects a transient UI state? Charming. Let me enumerate the ways this falls apart:*
1. *`vscode.window.state.focused` is `false` for ALL windows during certain OS transitions (screen lock, Mission Control, switching desktops on macOS). A brain plan created at that exact moment gets auto-claimed by NOBODY. The user comes back, opens their Kanban, and the plan is just... gone. Invisible. They'll file a bug saying 'Switchboard ate my plan.'*
2. *You capture `isFocused` before the debounce, fine. But what if two workspaces are both focused at the exact same instant? That shouldn't happen, but macOS Split View or multi-monitor setups with certain tiling managers can report focus on more than one window within the same tick. Now you're back to double-claiming.*
3. *The fs.watch fallback applies `allowAutoClaim = (_eventType === 'rename')`. So a rename event on an EXISTING file (e.g., Antigravity renames a sidecar) would set `allowAutoClaim = true` even though it's not a new plan. You're gating on focus but not on file-is-actually-new.*
4. *Where's the diagnostic logging for 'plan skipped because unfocused'? You mention it in verification but I don't see it in the implementation spec."*

### Balanced Response
All four points are addressed:
1. **No-focus scenario:** Acknowledged in User Review Required. The plan is not lost — it persists in `~/.gemini/antigravity/brain/`. On next focus + file change, `_mirrorBrainPlan` is called with `allowAutoClaim=false`, so the mirror .md is written but not auto-claimed. The user can claim it manually. This is the safer default vs. every-workspace-claims-it.
2. **Dual-focus edge case:** VS Code's `window.state.focused` is per-extension-host, not per-OS-window. In standard multi-window setups, exactly one extension host reports `focused=true` at a time. Split View on macOS still reports focus on the frontmost window. If an exotic tiling manager truly reports dual focus, the existing `_isPlanEligibleForWorkspace` / registry-ownership check provides a second layer of dedup.
3. **Rename events on existing files:** The `_isBrainMirrorCandidate` guard (line 4281/4318) plus the `_recentBrainWrites` TTL set already filter these. The focus gate is an additional layer, not the only layer.
4. **Diagnostic logging:** Added explicitly in the implementation below — a `console.log` with `[TaskViewerProvider] Brain auto-claim suppressed (window not focused)` in both watcher paths.

### Brain Watcher Focus Gate

#### [MODIFY] `src/services/TaskViewerProvider.ts`
- **Context:** `_setupBrainWatcher()` (line 4256) sets up two parallel watchers on `~/.gemini/antigravity/brain/`: a VS Code `FileSystemWatcher` (line 4277) and a native `fs.watch` fallback (line 4313). Both feed into `_mirrorBrainPlan(path, allowAutoClaim, workspaceRoot)`. The `allowAutoClaim` parameter is `true` for `onDidCreate` events and `(_eventType === 'rename')` for `fs.watch` events.
- **Logic:**
  1. In `handleBrainEvent` (line 4279): capture `const isFocused = vscode.window.state.focused;` **before** the debounce `setTimeout`. Inside the debounce callback, replace the raw `allowAutoClaim` argument with `allowAutoClaim && isFocused`. Add a diagnostic log when the gate suppresses auto-claim.
  2. In the `fs.watch` callback (line 4313): apply the identical pattern — capture `isFocused` before the debounce, intersect with the existing `_eventType === 'rename'` condition.
- **Implementation:**

**Change 1 — VS Code FileSystemWatcher handler (lines 4279–4300):**

Replace the `handleBrainEvent` closure with:

```typescript
            const handleBrainEvent = (uri: vscode.Uri, allowAutoClaim: boolean) => {
                const fullPath = uri.fsPath;
                if (!this._isBrainMirrorCandidate(brainDir, fullPath)) return;

                // Capture focus state NOW, before debounce timer fires
                const isFocused = vscode.window.state.focused;
                const effectiveAutoClaim = allowAutoClaim && isFocused;
                if (allowAutoClaim && !isFocused) {
                    console.log(`[TaskViewerProvider] Brain auto-claim suppressed (window not focused): ${path.basename(fullPath)}`);
                }

                const stablePath = this._getStablePath(fullPath);
                // Debounce: Windows fires multiple events per save (rename + change)
                const existing = this._brainDebounceTimers.get(stablePath);
                if (existing) clearTimeout(existing);
                this._brainDebounceTimers.set(stablePath, setTimeout(async () => {
                    try {
                        this._brainDebounceTimers.delete(stablePath);
                        // Skip if we wrote this brain file ourselves (mirror→brain direction)
                        if (this._recentBrainWrites.has(stablePath)) return;
                        if (fs.existsSync(fullPath)) {
                            await this._ensureTombstonesLoaded(workspaceRoot);
                            await this._mirrorBrainPlan(fullPath, effectiveAutoClaim, workspaceRoot);
                        }
                    } catch (e) {
                        console.error('[TaskViewerProvider] Brain watcher debounce callback failed:', e);
                    }
                }, 300));
            };
```

**Change 2 — Native `fs.watch` fallback handler (lines 4313–4341):**

Replace the `fs.watch` callback body with:

```typescript
            const brainFsWatcher = fs.watch(brainDir, { recursive: true }, (_eventType, filename) => {
                try {
                    if (!filename) return;
                    if (!/\.md(?:$|\.resolved(?:\.\d+)?$)/i.test(filename)) return;
                    const fullPath = path.join(brainDir, filename);
                    if (!this._isBrainMirrorCandidate(brainDir, fullPath)) return;

                    // Capture focus state NOW, before debounce timer fires
                    const isFocused = vscode.window.state.focused;
                    const rawAutoClaim = _eventType === 'rename';
                    const effectiveAutoClaim = rawAutoClaim && isFocused;
                    if (rawAutoClaim && !isFocused) {
                        console.log(`[TaskViewerProvider] Brain auto-claim suppressed via fs.watch (window not focused): ${filename}`);
                    }

                    const stablePath = this._getStablePath(fullPath);
                    const existing = this._brainDebounceTimers.get(stablePath);
                    if (existing) clearTimeout(existing);
                    this._brainDebounceTimers.set(stablePath, setTimeout(async () => {
                        try {
                            this._brainDebounceTimers.delete(stablePath);
                            if (this._recentBrainWrites.has(stablePath)) return;
                            if (fs.existsSync(fullPath)) {
                                await this._ensureTombstonesLoaded(workspaceRoot);
                                await this._mirrorBrainPlan(fullPath, effectiveAutoClaim, workspaceRoot);
                            }
                        } catch (e) {
                            console.error('[TaskViewerProvider] Brain fs.watch debounce callback failed:', e);
                        }
                    }, 300));
                } catch (e: any) {
                    if (e?.code !== 'ENOENT') {
                        console.error('[TaskViewerProvider] Brain fs.watch callback error:', e);
                    }
                }
            });
```

- **Edge Cases Handled:** Focus is captured at event-fire time (before 300ms debounce) to prevent Alt-Tab races. Diagnostic logging is emitted when the gate suppresses auto-claim, enabling debugging without user-facing noise.
