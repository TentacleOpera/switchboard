# Remove Deprecated Challenge Workflow and Cleanup Trigger Docs

## Goal

Remove the obsolete `challenge` workflow file and all references to the `/challenge` trigger and deprecated `/handoff` triggers from prompt configurations, templates, and documentation, preventing reviewer agents from wasting context/tokens on an unused workflow.

## Metadata

- **Tags:** [workflow, documentation, reliability]
- **Complexity:** 4

## User Review Required

> [!NOTE]
> This is a clean-up task that removes obsolete workflow files and configuration references. It has no impact on active Kanban features. The "Inline Adversarial Challenge" checkbox on the Kanban board is preserved — it controls the inline challenge *feature* (in-prompt adversarial review), which is unrelated to the deprecated `/challenge` *workflow*.

## Open Questions

None.

## Complexity Audit

### Routine
- Delete a single workflow file (`challenge.md`) from `.agent/workflows/`
- Remove `/challenge` text from prompt builder strings in `agentPromptBuilder.ts` and `TaskViewerProvider.ts`
- Remove `/challenge` and `/handoff` rows from trigger tables in `.github/` docs and `extension.ts` template strings
- Add `challenge.md` to legacy-file cleanup list and blocklist in `extension.ts`
- Remove `/challenge` sentence from README stub in `extension.ts`
- Remove `challenge` from `start_workflow` example list in `extension.ts`
- Bump patch version in `package.json`
- Update test fixture data referencing `challenge` workflow name

### Complex / Risky
- None

## Edge-Case & Dependency Audit

- **Race Conditions:** None. All changes are static text/string edits and a file deletion. No runtime state is affected.
- **Security:** No security implications. Removing a deprecated workflow reduces attack surface (fewer workflow entry points).
- **Side Effects:** The version bump (`1.7.1` → `1.7.2`) triggers `performSetup` on next extension activation, which re-copies `.agent/` files to user workspaces and applies the updated blocklist (deleting `challenge.md` from user workspaces). This is the intended migration mechanism.
- **Dependencies & Conflicts:** The `getDefaultTemplate` function in `extension.ts` already removed `/handoff` rows in a prior cleanup — only `/challenge` rows remain there. The `.github/` source files still contain both `/handoff` and `/challenge` rows; both must be removed. No active Kanban plans depend on the challenge workflow.

## Dependencies

None.

## Adversarial Synthesis

Key risks: (1) Test file `pipeline-orchestrator-regression.test.js` line 111 asserts a regex matching `lastWorkflow === 'challenge'` that doesn't exist in the current source — this test is already broken and must be updated. (2) The `buildReviewerExecutionModeLine` in both `agentPromptBuilder.ts` and `TaskViewerProvider.ts` contains a full line referencing "challenge stage" that needs rewording, not just parenthetical removal. Mitigations: Add test file updates to the plan; specify exact replacement text for the reworded lines.
