# Fix Antigravity Brain Watcher Stale Plan Auto-Claim and Pollution

## Goal

Resolve the issue where hundreds of past conversation files and non-plan artifacts (like research notes) are automatically imported as new plans in the Kanban "CREATED" column.

### Core Problems & Root Causes
1. **Broken Age Gate**: In a previous attempt to fix brain mirroring (`fix-antigravity-brain-plan-mirroring.md`), the age gate condition in `TaskViewerProvider.ts` was relaxed by adding `|| !claimMarkerExists`. Because older conversations (from before the claim marker feature was introduced) and other unclaimed files do not have claim markers, this bypass made *any* plan file, no matter how old, eligible for auto-claiming during the startup rescan.
2. **Missing Exclusions**: Non-plan files generated as artifacts (such as `analysis_results.md`, `research_notes.md`, and `experiment_results.md`) have H1 headings and are not excluded by name, leading them to be wrongly classified as plan files and mirrored.
3. **Pollution Cleanup**: The workspace has been polluted with ~104 stale plan entries in the "CREATED" column, although the on-disk mirror directory has 180 files (some of which are valid plans from past sessions in columns like `CODE REVIEWED` or `COMPLETED`).

### Complex / Risky

- Missing tombstoning step after cleanup could allow re-import on next rescan — must add tombstoning
- `_isLikelyPlanFile` (line 11361) accepts any H1-headed `.md` from antigravity sources without structural validation, making the exclusion list a fragile defense against future artifact types (architectural debt, not fixed in this plan)

## Edge-Case & Dependency Audit

- **Race Conditions**: Two-step cleanup (delete mirror files, then delete DB rows) creates a window where the watcher could re-register a plan between operations. Mitigated by the 5-second dedupe guard (`_recentMirrorProcessed` at line 11884) and by performing cleanup while the extension is not actively watching (e.g., during VS Code restart). Low risk.
- **Security**: Cleanup SQL uses `LIKE '%brain_%'` pattern which is safe (no user input interpolation). The DELETE is scoped to `kanban_column = 'CREATED' AND status = 'active'`, limiting blast radius.
- **Side Effects**: Tombstoning cleaned-up paths will prevent future re-import, which is the desired behavior. However, if a user later creates a plan with the same source path hash, the tombstone would block it. Tombstones can be cleared via the UI if needed.
- **Dependencies & Conflicts**: The age gate fix depends on the claim marker system (`_tryClaimBrainPlan`) working correctly for new plans. The 15-second `NEW_BRAIN_PLAN_AUTOCLAIM_WINDOW_MS` (line 358) is the real-time watcher window; the 30-minute `ANTIGRAVITY_RESCAN_WINDOW_MS` (line 359) is the rescan safety net. Plans between 15s and 30m old will be caught by rescan with `allowAutoClaim=true`; plans older than 30m without a claim marker will not be auto-claimed, which is the correct behavior for stale artifacts.

## Adversarial Synthesis

Key risks: (1) Cleanup SQL may delete legitimate new CREATED plans alongside polluted ones if not filtered by timestamp or manually reviewed, (2) Missing tombstoning step allows re-import of cleaned-up plans on next rescan, (3) Exclusion list is fragile against future artifact types due to `_isLikelyPlanFile` accepting any H1-headed brain-sourced `.md`. Mitigations: add manual review step before DELETE execution, add tombstoning step after cleanup, flag architectural debt in `_isLikelyPlanFile` for future improvement.
