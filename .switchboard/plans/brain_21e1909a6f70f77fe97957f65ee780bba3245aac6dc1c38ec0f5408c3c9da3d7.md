# Webview HTML Preview Rendering Security and Service Worker Fix

This implementation plan details the solution to resolve the issue where HTML previews in the `planning.html` webview do not display, resulting in a blank white screen. 

## Goal
The goal is to fix the HTML previews tab so that it correctly renders user-configured local HTML mockups and reports in the preview iframe, bypassing the VS Code Service Worker for HTML content delivery while maintaining adequate security posture.

### Core Problems & Root Cause Analysis
1. **VS Code Service Worker Registration Bug:** When loading local HTML files via `asWebviewUri` inside an iframe in a webview, VS Code's internal Service Worker is responsible for serving the resource. In many environments, this Service Worker fails to register or loads with a race condition, leading to a silent failure (a blank white screen) in the iframe.
2. **Restrictive Content Security Policy (CSP):** The current CSP meta tag in [planning.html](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/webview/planning.html) has a strict `frame-src` directive allowing only `{{WEBVIEW_CSP_SOURCE}}` and `https:`. If `webviewUri` is resolved to a `vscode-webview-resource:` or `vscode-resource:` scheme, it is blocked by the browser's CSP parser.
3. **Sandbox Restrictions:** The iframe currently has `sandbox="allow-scripts allow-same-origin"`. This combination is flagged by browsers and sometimes blocked inside Electron/webview contexts.

### Routine

### Complex / Risky
- CSP meta tag modification: adding VS Code webview resource schemes is safe, but adding `'unsafe-inline'` to `script-src` weakens nonce-based protection. This is justified because user-authored local HTML files commonly contain inline scripts, but it's a deliberate security trade-off.
- `srcdoc` approach partially bypasses the Service Worker for HTML content, but relative resources (CSS, images) loaded via `<base href>` still depend on the Service Worker. If the SW is broken, the HTML structure renders but assets may not load.

## Edge-Case & Dependency Audit

- **Race Conditions:** The existing `_latestRequestIds` race guard (line 2034) continues to apply. The `srcdoc` assignment is synchronous, so no new race condition is introduced.
- **Security:** The `srcdoc` content is user-authored local HTML. It runs inside a sandboxed iframe with `allow-scripts allow-same-origin`. The CSP must allow inline scripts for `srcdoc` content to execute. The `<base href>` value is constructed from `asWebviewUri` output (a VS Code-controlled URI), not user input, so injection via the base tag is not a concern.
- **Side Effects:** Changing `iframe.src` to `iframe.srcdoc` means the iframe no longer navigates to a URL. Browser history, `window.location` inside the iframe, and `postMessage` origin checks may behave differently. For a read-only preview, this is acceptable.
- **Dependencies & Conflicts:** The auto-refresh path (`_setupActiveDocWatcher` → `_handleFetchPreview`) already sends `webviewUri`; adding `htmlContent` to the same message is additive and non-breaking. The `handlePreviewError` function (line 1416) already uses `srcdoc` for error display, so the `srcdoc` pattern is precedent-setting, not novel.

## Dependencies
- None (self-contained fix)

## Adversarial Synthesis
Key risks: CSP relaxation with `'unsafe-inline'` weakens nonce-based script protection; `srcdoc` bypasses the Service Worker for HTML content but relative resources (CSS, images) still depend on it. Mitigations: `'unsafe-eval'` is excluded from the CSP (adversarial review finding); `allow-same-origin` is preserved in the sandbox to support `<base>` tag resource loading; the `srcdoc` approach is a net improvement over the current blank-screen failure mode.

```javascript
// Add after line 347 (after existing escapeAttr function):

// Inject <base> tag for relative asset resolution in srcdoc iframes
function injectBaseTag(html, baseUri) {
    if (!html || !baseUri) return html;
    if (html.includes('<base ')) return html;  // Don't duplicate
    
    const baseTag = `<base href="${escapeAttr(baseUri)}">`;
    const headMatch = html.match(/<head\b[^>]*>/i);
    if (headMatch) {
        const index = headMatch.index + headMatch[0].length;
        return html.slice(0, index) + baseTag + html.slice(index);
    }
    const htmlMatch = html.match(/<html\b[^>]*>/i);
    if (htmlMatch) {
        const index = htmlMatch.index + htmlMatch[0].length;
        return html.slice(0, index) + baseTag + html.slice(index);
    }
    return baseTag + html;  // Fallback: prepend for fragment HTML
}
```
