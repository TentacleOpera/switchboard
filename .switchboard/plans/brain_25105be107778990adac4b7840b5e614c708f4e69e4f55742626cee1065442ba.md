# Redesign and Refine Switchboard Research Tab

## Goal
The Switchboard Research tab has been redesigned but contains several usability and structural deficiencies:
1. It lacks an introductory explanation of what it does.
2. Step 1 has too many fields (redundant 'Context & Specifics' and 'Research Depth' controls).
3. The prompt template and research persona are hard-coded to technical/software engineering topics.
4. The document import folder selection is not synchronized with the user-defined folders from the local docs tab, does not persist user selection, lacks folder management, and doesn't handle empty folder lists gracefully.

This plan aims to simplify the UI, generalize prompt templates, align document import destination folders with user configuration, and integrate folder management directly into the Research tab.

**Root Cause Analysis:** The Research tab was initially built for a software-engineering-only audience, leading to hardcoded technical jargon in the prompt template and an assumption that users would want fine-grained control over research depth and context. In practice, the "Context & Specifics" field duplicates information users already put in the topic, and the 4-option depth selector adds decision fatigue for minimal value. The folder selection was an afterthought — it pulls from the same `localFoldersListed` message as the Local Docs sidebar but doesn't persist selection or handle the empty state, making it fragile when users haven't configured folders yet.

## User Review Required
- **Depth hardcoding:** This plan removes the Research Depth selector entirely and hardcodes `depth: 'deep'` (50-100+ sources). This removes user control over research scope. If you want users to still choose between a quick scan and deep research, consider a simpler Quick/Deep toggle instead of fully removing the option. Confirm this trade-off is acceptable.
- **Context removal:** The "Context & Specifics" field is removed. The Analyst agent will infer context from the topic alone. Confirm this is acceptable for your use cases where the topic alone may not convey sufficient domain context.

- Import button must be disabled when no destination folders are configured (currently no guard)
- Removing HTML elements requires simultaneous JS cleanup to prevent null-reference runtime errors in `generateResearchPrompt()` and `draftWithAnalyst` handler
- `draftResearchPrompt` message format change must be compatible with backend handler

## Edge-Case & Dependency Audit

- **Race Conditions:** The `localFoldersListed` message populates the folder select. If the Research tab is active before folders are loaded, the select will be empty. The warning element handles this, but the import button must also be disabled in this state.
- **Security:** No new security concerns. Folder paths come from the extension backend, not user input.
- **Side Effects:** Removing `#research-context-input` and `#research-depth` from the DOM means any code referencing these IDs will get `null` from `getElementById`. Must remove all JS references in the same change.
- **Dependencies & Conflicts:** The `draftResearchPrompt` backend message handler currently expects `{ topic, context, depth }`. The plan changes this to `{ topic, context: '', depth: 'deep' }` — must verify the backend handler tolerates empty context and always-deep depth. The `draft_research_prompt.md` skill must be updated to reflect that Depth is always 'deep' when invoked from the UI.

## Dependencies
- None

## Adversarial Synthesis
Key risks: (1) Persisted folder selection may point to a deleted path after folder removal — requires explicit staleness check and fallback. (2) Import button is not disabled when no folders exist, allowing a silent failure path. (3) Removing DOM elements without cleaning all JS references will cause runtime null-reference errors. Mitigations: Add staleness validation in `localFoldersListed` handler, disable import button on empty folder list, and audit all JS references to removed elements before making HTML changes.
