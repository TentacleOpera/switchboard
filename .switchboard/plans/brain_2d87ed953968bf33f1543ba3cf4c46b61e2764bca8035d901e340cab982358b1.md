# Fix Kanban Card Project and Workspace Reassignment Flow

## Goal
Fix the card assignment flow in the Kanban board so that when a user selects a plan card, changes the target workspace or project via the dropdown, and clicks the ASSIGN button, the reassignment succeeds and updates the view correctly. 
Currently, changing the workspace/project dropdown immediately triggers a board refresh (workspace switch or filter change). This re-render calls `renderBoard()`, which prunes any selected cards not in the new view, clearing `selectedCards` and leaving the ASSIGN button disabled before the user can click it.

## Complexity Audit

### Routine
- Adding `selectedCards.size > 0` guard to the dropdown `change` handler (line 6194) to defer view switch
- Moving delete-button state update into the early-return branch of the `change` handler
- Adding `postKanbanMessage({ type: 'selectWorkspace' })` after cross-workspace assignment
- Adding `postKanbanMessage({ type: 'setProjectFilter' })` after same-workspace project assignment
- Setting `currentWorkspaceRoot` and `activeProjectFilter` on the webview side before view-switch
- Resetting `lastBoardSignature = ''` before posting `selectWorkspace` to ensure refresh isn't short-circuited
- Optimistically clearing `selectedCards` and removing `.selected` class from cards after assignment

### Complex / Risky
- Async message ordering: assignment message + view-switch message sent in rapid succession; both trigger `_refreshBoard()` on the extension side, causing multiple `renderBoard()` calls on the webview. The `reassignPlansWorkspace` handler already refreshes both source and target boards; the additional `selectWorkspace` message causes a third refresh. The intermediate render (source workspace without moved cards) may cause a brief visual flash before the target workspace renders.
- State consistency: `currentWorkspaceRoot` and `activeProjectFilter` are updated on the webview side before the extension confirms the assignment succeeded. If the assignment fails (file move error, DB error), the webview state is inconsistent with the extension state. No rollback mechanism exists.

## Edge-Case & Dependency Audit

- **Race Conditions:** The `reassignPlansWorkspace` handler calls `_refreshBoard(sourceWorkspaceRoot)` then `_refreshBoard(targetWorkspaceRoot)`. The plan's additional `selectWorkspace` message triggers a third `_refreshBoard(targetWorkspaceRoot)`. These are processed sequentially (VS Code message queue is ordered), so no true race condition, but the redundant refresh may cause a visual flicker as the board re-renders multiple times. The `lastBoardSignature` reset ensures the third refresh is not short-circuited by a stale signature match.
- **Security:** No concern â€” all operations are local file moves and DB updates within the user's workspace.
- **Side Effects:** The `selectWorkspace` handler reinitializes session watchers (`_setupSessionWatcher()`), plan watchers (`_taskViewerProvider.reinitializePlanWatcher()`), and clears terminal dispatch maps (`_taskViewerProvider.clearRegisteredTerminalsMap()`). These side effects are appropriate and necessary when navigating to the target workspace post-assignment.
- **Dependencies & Conflicts:** The `renderBoard()` pruning logic (lines 4512-4519) is the root cause of the bug. The fix defers the view switch to prevent pruning from firing before assignment. No other features depend on the dropdown `change` handler firing immediately.

## Dependencies
- None

## Adversarial Synthesis
Key risks: redundant board refreshes may cause visual flicker during cross-workspace assignment; optimistic webview state update before extension confirmation creates a narrow window for state inconsistency on assignment failure. Mitigations: refreshes are sequential and fast (minimal user-visible delay); assignment failure is rare (atomic file moves with copy+delete fallback, DB transactions); extension shows error messages on failure allowing manual recovery.

### 2. In `src/webview/kanban.html` (around Line 6135)
Modify the `click` handler for `btn-assign-workspace-project` to post the view switch messages after posting the assignment:

```javascript
        document.getElementById('btn-assign-workspace-project')?.addEventListener('click', () => {
            const sessionIds = Array.from(selectedCards.keys());

            const select = document.getElementById('workspace-project-select');
            const selectedOption = select?.selectedOptions?.[0];
            if (!selectedOption || sessionIds.length === 0) return;

            const targetWorkspaceRoot = selectedOption.dataset.workspaceRoot || '';
            const targetProject = selectedOption.dataset.project || '';

            if (!targetWorkspaceRoot) return;

            // Build confirmation label
            const wsLabel = selectedOption.textContent || targetWorkspaceRoot;
            // Determine if this is a same-workspace or cross-workspace operation
            // by checking the source workspaces of the actually selected cards,
            // not currentWorkspaceRoot (which may have been updated by a dropdown switch).
            const sourceWorkspaces = new Set(selectedCards.values());
            const isSameWorkspace = sourceWorkspaces.size === 1 && sourceWorkspaces.has(targetWorkspaceRoot);

            if (isSameWorkspace && (!targetProject || targetProject === '__unassigned__')) {
                // No-op: same workspace, no project or unassigned
                vscode?.postMessage?.({ type: 'showInfo', message: 'Plans are already in this workspace with no project assignment.' });
                return;
            }

            if (!isSameWorkspace) {
                // Cross-workspace: reassign to target workspace first, optionally passing targetProject
                const sourceWorkspaceRoot = sourceWorkspaces.size === 1
                    ? Array.from(sourceWorkspaces)[0]
                    : '';
                postKanbanMessage({
                    type: 'reassignPlansWorkspace',
                    sessionIds: sessionIds,
                    sourceWorkspaceRoot: sourceWorkspaceRoot,
                    targetWorkspaceRoot: targetWorkspaceRoot,
                    targetProject: targetProject === '__unassigned__' ? '' : targetProject
                });

                // Post-assignment: switch workspace context to the target workspace so the user sees their moved cards
                currentWorkspaceRoot = targetWorkspaceRoot;
                activeProjectFilter = targetProject === '__unassigned__' ? null : targetProject;
                lastBoardSignature = '';  // Ensure the refresh triggered by selectWorkspace is not short-circuited
                postKanbanMessage({
                    type: 'selectWorkspace',
                    workspaceRoot: targetWorkspaceRoot,
                    project: targetProject === '__unassigned__' ? null : (targetProject || null)
                });
            } else {
                // Same workspace: project assignment only
                postKanbanMessage({
                    type: 'assignSelectedToProject',
                    projectName: targetProject === '__unassigned__' ? '' : targetProject,
                    planIds: sessionIds
                });

                // Post-assignment: switch project filter to the target project so the user sees their moved cards
                activeProjectFilter = targetProject === '__unassigned__' ? null : targetProject;
                postKanbanMessage({
                    type: 'setProjectFilter',
                    project: targetProject === '__unassigned__' ? null : (targetProject || null)
                });
            }

            // Optimistically clear the selection
            selectedCards.clear();
            updateReassignButtonVisibility();

            // Un-select visually on screen
            document.querySelectorAll('.kanban-card.selected').forEach(el => {
                el.classList.remove('selected');
            });
        });
```

**Clarification on sentinel handling**: The `__unassigned__` sentinel value is used only in the dropdown's `data-project` attribute to represent "no project". When posting messages to the extension, it must be converted to either `''` (empty string, for `projectName` in `assignSelectedToProject`) or `null` (for `project` in `selectWorkspace`/`setProjectFilter`). The extension's `setProjectFilter` handler checks `msg.project === null || typeof msg.project === 'string'`, and `selectWorkspace` uses `msg.project === null || msg.project === undefined` to decide whether to reset the filter. Passing `null` consistently for "no project" ensures the filter is properly cleared.

**Clarification on `lastBoardSignature` reset**: The existing `change` handler resets `lastBoardSignature = ''` before posting `selectWorkspace` (line 6206). The new `selectWorkspace` post in the ASSIGN handler must also reset this variable, otherwise the refresh triggered by the `selectWorkspace` message may be short-circuited if the target workspace's board signature happens to match the stale one.
