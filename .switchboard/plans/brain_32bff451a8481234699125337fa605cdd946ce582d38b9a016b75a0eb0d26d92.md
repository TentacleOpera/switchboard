# Sync Active Plan Complexity to Kanban Database on Mirror Change

Active plan files are synced back and forth from the Antigravity sandboxed brain directory to the local workspace directory (as `.switchboard/plans/brain_[hash].md` mirrors). Currently, when a planner agent (or a user) edits an active plan file, the brain watcher registers the update and copies it to the workspace. However, if the plan is already in the SQLite database, it returns early and does not update the plan's complexity, topic, or metadata in the database.

This plan modifies the early-return path of the brain-mirror watcher to parse the metadata from the updated plan content and update the database row.

## Goal

When an already-registered brain plan's mirror file changes, parse the updated metadata (complexity, topic, tags, dependencies) from the plan content and upsert it into the Kanban database so the UI reflects the change without requiring a column move or manual refresh.

## Edge-Case & Dependency Audit

- **Race Conditions:** The existing dedupe guard (`_recentMirrorProcessed` with `pathHash_mtimeMs` key, lines 11547–11550) prevents double-processing of the same file change. No additional locking is needed.
- **Security:** `parsePlanMetadata()` is a pure regex parser on file content — no injection risk. The `upsertPlans()` method uses parameterized SQL.
- **Side Effects:** If a plan file is malformed and loses its complexity header, `parsePlanMetadata()` returns `complexity: 'Unknown'`. To avoid overwriting a previously scored value with 'Unknown', the implementation should only update the DB complexity when the parsed value is not `'Unknown'`. Similarly, empty tags/dependencies should not overwrite existing values.
- **Dependencies & Conflicts:** `parsePlanMetadata` is currently not imported in `TaskViewerProvider.ts` — only `inferTopicFromPath` is (line 55). The import must be added. No other files are modified.

**Current code (lines 11615–11623):**
```typescript
if (db) {
    const alreadyInDb = await db.hasPlan(runSheetId);
    if (alreadyInDb) {
        console.log(`[TaskViewerProvider] Brain plan already in DB (session: ${runSheetId}), skipping runsheet creation`);
        if (!suppressFollowupSync) {
            await this._syncFilesAndRefreshRunSheets(resolvedWorkspaceRoot);
        }

```

**New code:**
```typescript
if (db) {
    const alreadyInDb = await db.hasPlan(runSheetId);
    if (alreadyInDb) {
        console.log(`[TaskViewerProvider] Brain plan already in DB (session: ${runSheetId}), updating metadata`);

        // Parse updated metadata from the plan content (already read at line 11560)
        const metadata = await parsePlanMetadata(content, path.relative(resolvedWorkspaceRoot, mirrorPath));
        const planFileRelative = path.relative(resolvedWorkspaceRoot, mirrorPath).replace(/\\/g, '/');
        const wsId = await this._getWorkspaceIdForRoot(resolvedWorkspaceRoot);
        const existingPlan = wsId ? await db.getPlanByPlanFile(planFileRelative, wsId) : null;

        if (existingPlan) {
            const updatedRecord: KanbanPlanRecord = {
                ...existingPlan,
                topic: metadata.topic || existingPlan.topic,
                complexity: metadata.complexity !== 'Unknown' ? metadata.complexity : existingPlan.complexity,
                tags: metadata.tags || existingPlan.tags,
                dependencies: metadata.dependencies || existingPlan.dependencies,
                updatedAt: new Date(mtimeMs).toISOString()
            };
            await db.upsertPlans([updatedRecord]);
            console.log(`[TaskViewerProvider] Updated brain plan metadata: topic="${metadata.topic}", complexity="${metadata.complexity}"`);
        }

        if (!suppressFollowupSync) {
            await this._syncFilesAndRefreshRunSheets(resolvedWorkspaceRoot);
        }
        return;
    }
}
```

**Logic walkthrough:**
1. `content` is already available in scope (read at line 11560).
2. `parsePlanMetadata(content, planFileRelative)` extracts topic, complexity, tags, dependencies from the updated plan content.
3. `db.getPlanByPlanFile(planFileRelative, wsId)` retrieves the existing DB record with all 24 fields preserved.
4. The spread pattern `{ ...existingPlan, ...overrides }` follows the `GlobalPlanWatcherService` convention (lines 456–463).
5. Guards prevent overwriting scored values with 'Unknown' (complexity) or empty strings (tags, dependencies).
6. The existing `_syncFilesAndRefreshRunSheets()` call pushes the updated DB state to the UI.
