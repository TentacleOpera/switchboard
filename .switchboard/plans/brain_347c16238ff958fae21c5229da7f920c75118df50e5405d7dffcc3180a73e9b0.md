# Implementation Plan - Add Dropdown Workspace to Parent Mappings via Setup UI

Introduce first-class support for child workspace folders that share a parent workspace database mapping but appear as separate choices in the Kanban workspace dropdown with their own distinct boards/identities.

## Goal

Add a `dropdownWorkspaces` field to the `WorkspaceDatabaseMapping` schema that lets child folders share a parent's SQLite database while still maintaining independent Kanban board identities (unique `workspace-id` files) and appearing as their own selectable entries in the Kanban workspace dropdown.

> **Critical ordering issue in `forWorkspace()`:**
> `_redirectToParentIfMapped()` runs before the `dropdownWorkspaces` DB-path lookup. If a `dropdownWorkspace` path is passed to `forWorkspace()`, it will be silently redirected to the parent workspace root before the `isDropdown` check can fire. The redirect function must explicitly skip paths that appear in any `dropdownWorkspaces` array.
>
> **workspace-id initialisation gap:**
> The plan requires `dropdownWorkspaces` to have independent `workspace-id` files. No step currently provisions them. `WorkspaceIdentityService.ensureWorkspaceIdentity()` must be called for each path when a mapping is saved or at extension activation.

## Complexity Audit

### Routine
- Extending `WorkspaceDatabaseMapping` interface with `dropdownWorkspaces?: string[]` (schema only)
- Adding `dropdownWorkspaces` textarea + browse button to `setup.html`
- Updating `captureMappingsFromDom()` and `btn-save-mappings` logic to include new field
- Adding `dropdownWorkspaces` paths to `seenFolders` set in `saveWorkspaceMappings` validation
- Adding `dropdownWorkspaces` to `newMapping` object in `initializeWorkspaceDatabase`
- Registering `browseWorkspaceMappingDropdownFolder` case in SetupPanelProvider message router
- Extending `_getAllMappedFolders()` in `GlobalPlanWatcherService` to append `dropdownWorkspaces` paths

### Complex / Risky
- **`_redirectToParentIfMapped()` bypass**: Must NOT redirect `dropdownWorkspace` paths to the parent. Failure breaks entire DB resolution chain for these paths.
- **`forWorkspace()` `isDropdown` check**: Must resolve to the parent mapping's `dbPath` without being intercepted by the redirect logic above.
- **`anyOpenFolderIsMapped` gate in `_getWorkspaceItems()`**: Must scan `dropdownWorkspaces` so the custom-mapping branch activates when only a dropdown child folder is open.
- **`_getAllowedRoots()` in `KanbanProvider.ts`**: Must add `dropdownWorkspaces` paths so `setCurrentWorkspaceRoot()` and `_resolveWorkspaceRoot()` accept them as valid roots.
- **workspace-id provisioning**: `ensureWorkspaceIdentity()` must be called per `dropdownWorkspace` path at save and activation time, or the boards will have no unique identity and plans from different dropdown folders will collide in the DB.
- **`handleGetAllDbPaths()` in `TaskViewerProvider.ts`**: Must include `dropdownWorkspaces` paths when building the DB map (they are not open VS Code workspace folders).

## Edge-Case & Dependency Audit

**Race Conditions**
- Two `dropdownWorkspace` folders opening simultaneously could both call `ensureWorkspaceIdentity()` concurrently for the same `dbPath`. Because the DB instance is cached by `dbPath`, both will share one `KanbanDatabase` instance — safe. But `ensureWorkspaceIdentity()` itself must be idempotent (it already is, it's a `CREATE IF NOT EXISTS` pattern).

**Security**
- Paths entered in the `dropdownWorkspaces` textarea are user-supplied strings. `expandHome` + `path.resolve` normalisation (already applied in `saveWorkspaceMappings`) must also be applied to `dropdownWorkspaces` entries before persisting.
- A `dropdownWorkspace` path that is equal to the `parentFolder` creates a silent identity loop. Validation must reject this.

**Side Effects**
- A `dropdownWorkspace` path that overlaps with a different mapping's `parentFolder` or `workspaceFolders` entry would route to two different DBs depending on order. The `seenFolders` deduplication must cover all three field types across all mappings.
- `GlobalPlanWatcherService` already de-duplicates folder paths before setting up watchers, so adding `dropdownWorkspaces` paths there is safe.

**Dependencies & Conflicts**
- `KanbanDatabase.forWorkspace()` depends on `_redirectToParentIfMapped()` running first. This ordering must be preserved for standard `workspaceFolders` paths. The bypass is conditional: only skip redirect if the path matches a `dropdownWorkspaces` entry.
- `WorkspaceIdentityService` must be importable from `SetupPanelProvider`. Currently it is only called from `KanbanDatabase.invalidateWorkspace()`. Check for circular dependency risk before adding a direct call from `SetupPanelProvider`.

## Adversarial Synthesis

Key risks are: (1) the redirect-before-lookup ordering in `forWorkspace()` silently swallowing `dropdownWorkspace` identities, (2) missing `workspace-id` provisioning causing board collisions, and (3) incorrect file attributions in the original plan leading a coder to modify the wrong files. Mitigations: patch `_redirectToParentIfMapped()` to skip `dropdownWorkspaces` paths before adding the `isDropdown` lookup, add explicit `ensureWorkspaceIdentity()` calls in `saveWorkspaceMappings` and at activation, and update all file references to their correct locations.
