# Integration Plan — Worktree Terminal Creation Fixes

## 1. Goal

Make worktree agent-terminal creation fast, correctly scoped, and free of cross-contamination with main-repo terminals.

## 2. Dependencies & Sequence

Both subtasks modify `ensureWorktreeTerminals` and `_createAutobanTerminal` in [TaskViewerProvider.ts](file:///Users/patrickvuleta/Documents/GitHub/worktrees/switchboard/remote-sync-2/src/services/TaskViewerProvider.ts) and `createAgentGrid` in [extension.ts](file:///Users/patrickvuleta/Documents/GitHub/worktrees/switchboard/remote-sync-2/src/extension.ts).
We will merge the fixes:
1. Land public `readTerminalRegistryState` wrapper in `TaskViewerProvider.ts`.
2. Land worktree-aware `matchesGridAgentName` in `extension.ts` using `readTerminalRegistryState`.
3. Parallelize `ensureWorktreeTerminals` in `TaskViewerProvider.ts`, moving pool-limit checks and terminal pool state persistence into the caller (`ensureWorktreeTerminals`) to avoid concurrency races when updating the state.
4. Update `_createAutobanTerminal` to defer PID resolution and use the VS Code shell execution start event instead of a fixed 1-second timeout.

## 3. Risks & Edge Cases

- **Bookkeeping State Mutation Race**: If multiple terminals are created in parallel, they cannot sequentially read and write `this._autobanState.terminalPools` in `_createAutobanTerminal` because they will overwrite each other.
  - *Mitigation*: We will strip out `terminalPools` and `managedTerminalPools` state updates from `_createAutobanTerminal`, returning the created `suffixedUniqueName` instead. Then, in `ensureWorktreeTerminals`, we will collect these names and perform a single atomic state write after all parallel promises resolve.
- **`_createAutobanTerminal` calls from other places**:
  - `addAutobanTerminalFromKanban` and message handlers call `_createAutobanTerminal` directly. They do NOT pass a worktree path in the same way, but they still expect the terminal to be added to the pool.
  - *Mitigation*: We will keep a boolean parameter or internal flag, or let `_createAutobanTerminal` handle its own state updates if called individually, or simply make it so that if `cwd` is not passed (or we are not in a parallel worktree build), we do the state update inline; whereas if we call a batch version (or return the name), we do it atomically. Even better: make `_createAutobanTerminal` accept an optional `skipStatePoolUpdate` parameter. If `true`, it skips updating `terminalPools` / `managedTerminalPools` and returns the name so the batch caller can update them atomically.
- **Compilation/Tests**: Skipped per user request.
