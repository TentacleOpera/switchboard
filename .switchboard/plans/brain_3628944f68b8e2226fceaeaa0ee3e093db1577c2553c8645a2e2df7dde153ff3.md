# Reset and Creation Logic Fix for Terminal Pools and Primary Agents

Currently, when terminal pools are created via the `kanban.html` automation tab, the primary agent terminals (e.g. Coder, Reviewer) cannot be created again using the "OPEN AGENT TERMINALS" button in `implementation.html`. Additionally, clicking "RESET ALL AGENTS" does not clean up the terminal pool states or dispose of orphaned pool terminals.

This plan addresses both issues by:
1. Refining the terminal matching logic in `createAgentGrid` to differentiate between primary agents (e.g., `Coder`, `Coder -Visual Studio Code`, `Coder (2)`) and pool agents (e.g., `Coder 2`).
2. Ensuring `RESET ALL AGENTS` triggers a reset of all configured autoban terminal pools and clears the associated runtime states.
3. Updating orphan terminal patterns to sweep and close any untracked/orphaned pool terminals during a reset.

## Goal

Fix the terminal matching logic in `createAgentGrid` so that pool terminals (e.g., "Coder 2") are not mistaken for primary agent terminals, and ensure that "RESET ALL AGENTS" fully cleans up autoban pool state and orphaned pool terminals.

## Complexity Audit

### Routine
- Adding `_resetAutobanPools()` call at the start of `_deregisterAllTerminals` — straightforward composition of two existing methods
- Adding pool terminal patterns to `ORPHAN_PATTERNS` — additive regex entries following existing pattern conventions
- Wrapping the `_resetAutobanPools()` call in a try/catch for error isolation

### Complex / Risky
- Refining `matchesGridAgentName` regex to distinguish primary vs. pool terminals — must correctly handle custom agent names that may contain regex metacharacters (e.g., "C++ Dev", "Agent (v2)"), requiring regex escaping of the `agentName` parameter

## Edge-Case & Dependency Audit

- **Race Conditions:** No race condition risk. `_resetAutobanPools()` and `_deregisterAllTerminals()` are both called sequentially from the same UI action handler. The state lock (`updateState`) serializes state mutations.
- **Security:** No security implications. Terminal names are not used as path segments or command arguments in this change.
- **Side Effects:** Changing `matchesGridAgentName` also affects `clearGridBlockers` (extension.ts:2176-2226), which uses the same function to find and dispose duplicate terminals. With the refined regex, `clearGridBlockers` will correctly skip pool terminals during cleanup — this is the intended behavior, as pool terminals should be managed by the autoban system, not by the grid cleanup logic.
- **Dependencies & Conflicts:** `_resetAutobanPools()` removes pool terminals from `state.terminals` via `_closeTerminal`. When `_deregisterAllTerminals` subsequently iterates `state.terminals`, those entries are already gone — no double-close occurs. The orphan sweep may encounter already-disposed pool terminals, but `t.exitStatus !== undefined` guards prevent any action.

## Dependencies

None.

## Adversarial Synthesis

Key risks: (1) Custom agent names with regex metacharacters could break the `matchesGridAgentName` regex if `agentName` is not escaped before interpolation. (2) Custom agent pool terminals (e.g., "CustomAgent 2") are not covered by the static ORPHAN_PATTERNS list. Mitigations: (1) Escape `agentName` with a standard `escapeRegex` replacement before constructing the regex. (2) `_resetAutobanPools()` (called first) already closes all managed pool terminals including custom ones; the orphan sweep is a safety net for unmanaged orphans only. Document the limitation for custom agents in ORPHAN_PATTERNS rather than over-engineering a dynamic pattern.

## Proposed Changes

### Extension Core

#### [MODIFY] [extension.ts](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/extension.ts)

Refine the `matchesGridAgentName` function inside `createAgentGrid` (line 2168-2175) to use a regex that matches primary agent terminal names but excludes pool terminals.

**Context:** Pool terminals are named `<RoleLabel> <Number>` (e.g., "Coder 2", "Coder 3") per `getNextAutobanTerminalName` (autobanState.ts:163-191). Primary agents with duplicates are named `<RoleLabel> (<Number>)` (e.g., "Coder (2)") — VS Code adds parentheses for duplicate terminal names. The current `startsWith(\`${agentName} \`)` check matches both patterns, causing pool terminals to be mistaken for primary agents.

**Current code (lines 2168-2175):**
```typescript
const matchesGridAgentName = (terminal: vscode.Terminal, agentName: string): boolean => {
    const creationName = (terminal.creationOptions as vscode.TerminalOptions | undefined)?.name;
    const terminalName = normalizeGridTerminalName(terminal.name);
    const createdName = normalizeGridTerminalName(creationName);
    const prefixedTerminalName = terminalName.startsWith(`${agentName} `);
    const prefixedCreatedName = createdName.startsWith(`${agentName} `);
    return terminalName === agentName || createdName === agentName || prefixedTerminalName || prefixedCreatedName;
};
```

**Implementation:**
1. Add an `escapeRegex` helper (inline or as a local function) to escape regex metacharacters in `agentName` — critical because custom agent names from `getCustomAgents` may contain characters like `+`, `(`, `)`, etc.
2. Replace the `prefixedTerminalName` / `prefixedCreatedName` checks with a regex that matches:
   - Exact name: `Coder`
   - VS Code duplicate suffix: `Coder (2)`, `Coder (3)`
   - But NOT pool terminal format: `Coder 2`, `Coder 3`

**Proposed replacement (lines 2168-2175):**
```typescript
const escapeRegex = (s: string) => s.replace(/[.*+?^${}()|[\]\\]/g, '\\$&');
const matchesGridAgentName = (terminal: vscode.Terminal, agentName: string): boolean => {
    const creationName = (terminal.creationOptions as vscode.TerminalOptions | undefined)?.name;
    const terminalName = normalizeGridTerminalName(terminal.name);
    const createdName = normalizeGridTerminalName(creationName);
    // Matches primary agent names (exact, or with VS Code duplicate suffix like " (2)")
    // but excludes pool terminals which use bare number suffix like " 2"
    const primaryPattern = new RegExp(`^${escapeRegex(agentName)}(?: \\(\\d+\\))?$`);
    return primaryPattern.test(terminalName) || primaryPattern.test(createdName);
};
```

**Edge Cases:**
- Custom agent names with regex metacharacters (e.g., "C++ Dev") — handled by `escapeRegex`
- Custom agent names with parentheses (e.g., "Agent (v2)") — `escapeRegex` escapes the parens, so the regex correctly matches the literal name
- Terminals with IDE suffix in their display name — not applicable; VS Code's `terminal.name` does not include the IDE suffix (that's only used in the `registeredTerminals` map keys)

**Change 1: Add `_resetAutobanPools()` call to `_deregisterAllTerminals`**

**Context:** `_deregisterAllTerminals` (line 14147) is the handler for "RESET ALL AGENTS". Currently it cleans up known terminals from state.json and does an orphan sweep, but does not reset autoban pool state. `_resetAutobanPools()` (line 5744) stops the autoban engine, clears pool/managed pool state, closes managed pool terminals, and reconciles.

**Implementation:** Add `await this._resetAutobanPools();` at the beginning of `_deregisterAllTerminals`, wrapped in a try/catch for error isolation so a partial autoban reset failure doesn't block the rest of the deregistration.

**Proposed insertion (after line 14147, before the PID pre-fetch block):**
```typescript
private async _deregisterAllTerminals(silent: boolean = false) {
    // Reset autoban pools first: stops engine, clears pool state, closes pool terminals.
    // Wrapped in try/catch so a partial autoban reset failure doesn't block the rest of deregistration.
    try {
        await this._resetAutobanPools();
    } catch (e) {
        console.error('[TaskViewerProvider] Failed to reset autoban pools during deregistration:', e);
    }

    // Pre-fetch PIDs outside the state lock...
    const activeTerminals = vscode.window.terminals;
    // ... (rest of existing code unchanged)
```

**Edge Cases:**
- If `_resetAutobanPools()` throws, the catch logs the error and `_deregisterAllTerminals` continues — some pool terminals may remain open, but the orphan sweep (step 2 of existing code) provides a safety net.
- Pool terminals closed by `_resetAutobanPools()` are removed from `state.terminals` by `_closeTerminal`, so the subsequent iteration of `state.terminals` won't encounter them — no double-close.

**Context:** Pool terminals follow the format `<RoleLabel> <Number>` (e.g., "Coder 2", "Reviewer 3"). The current `ORPHAN_PATTERNS` (line 14193-14205) only match exact role names (e.g., `/^coder$/i`) and won't catch orphaned pool terminals.

**Implementation:** Add regex patterns for each built-in role's pool terminal format. Role labels come from `_getAutobanRoleLabel` (line 5078-5087): Planner, Coder, Lead Coder, Reviewer, Intern.

**Current code (lines 14193-14205):**

```typescript
const ORPHAN_PATTERNS = [

    /^reviewer$/i,

    /^cortex/,

```

**Proposed replacement:**
```typescript
const ORPHAN_PATTERNS = [
    /^Switchboard -/,

    /^coder$/i,
    /^coder \d+$/i,
    /^reviewer$/i,
    /^reviewer \d+$/i,
    /^planner$/i,
    /^planner \d+$/i,
    /^analyst$/i,
    /^analyst \d+$/i,
    /^intern$/i,
    /^intern \d+$/i,
    /^Lead Coder$/i,
    /^Lead Coder \d+$/i,
    /^verification/,
    /^execution/,
    /^cortex/,
    // NOTE: Custom agent pool terminals (e.g., "CustomAgent 2") are not covered by these
    // static patterns. They are handled by _resetAutobanPools() (called above), which
    // closes all managed pool terminals. The orphan sweep is a safety net for truly
    // unmanaged orphans only.
];
```

**Edge Cases:**
- Custom agent pool terminals (e.g., "CustomAgent 2") — not matched by static patterns. Mitigated by `_resetAutobanPools()` which closes all managed pool terminals before the orphan sweep runs. Documented as a known limitation.
- The `\d+` pattern matches one or more digits, covering pool terminals at any index (e.g., "Coder 2", "Coder 10").
