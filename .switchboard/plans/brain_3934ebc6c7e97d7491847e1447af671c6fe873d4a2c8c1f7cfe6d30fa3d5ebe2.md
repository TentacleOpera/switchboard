# Fix Clipboard Research Doc Import Feedback and File Path Discrepancy

## Goal

Fix two bugs in the clipboard research doc import flow: (1) no status feedback is shown when importing from the Clipboard Import tab, and (2) the DB-registered file path omits the content hash, causing `healImports` to auto-delete the entry as orphaned.

## Metadata

- **Tags:** [bugfix, frontend, backend]
- **Complexity:** 4
- **Repo:** (single-repo workspace — omitted per directive)

## User Review Required

> [!NOTE]
> This fix does not introduce any breaking changes. Existing documents with non-hashed filenames are already resolved gracefully by `resolveImportedDocPath`'s directory-scan fallback. Old DB entries with wrong paths will be naturally cleaned up by `healImports` on its next hourly scan; no migration is needed.

## Complexity Audit

### Routine
- Adding a `#clipboard-import-status` div to the Clipboard Import tab HTML
- Updating `handleResearchImportClick` to clear both status elements
- Updating `importResearchDocResult` JS handler to write status to both elements
- Adding `filePath` to `registerImport` options type (backward-compatible optional field)
- Passing `filePath: result.savedPath` in all `registerImport` call sites within `PlannerPromptWriter`

### Complex / Risky
- None

## Edge-Case & Dependency Audit

- **Race Conditions:** `registerImport` already uses a per-slug write queue (`_registryWriteQueue`) inside `PlanningPanelCacheService`. The `filePath` addition does not change concurrency semantics.
- **Security:** No new attack surface — `filePath` is derived from `savedPath` returned by `_writeDocToDocsDir`, which already sanitizes filenames.
- **Side Effects:** `healImports` will clean up old entries with wrong paths on next scan. This is intended behavior, not a regression. The `resolveImportedDocPath` fallback continues to work for reads until entries are pruned.
- **Dependencies & Conflicts:** The `registerImportBatch` path (used for subpage imports in `_handleImportFullDoc`) already passes the correct `filePath` — no change needed there.

## Dependencies

None.

## Adversarial Synthesis

Key risks: (1) The original plan missed two `registerImport` call sites (`writeFromPlanningCache` line 229 and `_registerImport` helper line 2633), which would leave the path-mismatch bug partially unfixed. (2) The JS handler needs to write status to *both* status elements rather than trying to track which button triggered the import — the hidden element is invisible, so dual-write is simpler and more robust than threading origin state through the backend round-trip. Mitigations: audit all four call sites; use unconditional dual-write for status feedback.

- **`_registerImport` helper** (lines 2615–2640): This helper doesn't have access to `savedPath` because it's called after the write. It needs to either: (a) accept `filePath` as a parameter, or (b) be refactored. The simplest fix is to add an optional `filePath` parameter:
  ```typescript
  private async _registerImport(
      cacheService: PlanningPanelCacheService | undefined,
      sourceId: string,
      docId: string,
      docName: string,
      content: string,
      workspaceRoot: string,
      filePath?: string
  ): Promise<void> {
      if (!cacheService) return;
      try {
          const rawSlug = docName
              .toLowerCase()
              .replace(/[^a-z0-9]+/g, '_')
              .replace(/^_+|_+$/g, '')
              .slice(0, 60) || sourceId;
          const contentWithoutFrontMatter = content.replace(/^---\n[\s\S]*?\n---\n*/, '');
          const contentHash = crypto.createHash('sha256').update(contentWithoutFrontMatter).digest('hex');
          const workspaceId = await this._getWorkspaceId(workspaceRoot);
          await cacheService.registerImport(sourceId, docId, docName, rawSlug, {
              remoteContentHash: contentHash,
              workspaceId: workspaceId,
              filePath
          });
      } catch (regErr) {
          console.warn('[PlanningPanelProvider] Failed to register import:', regErr);
      }
  }
  ```
  Then update all callers of `_registerImport` to pass the `savedPath` from the write result.
