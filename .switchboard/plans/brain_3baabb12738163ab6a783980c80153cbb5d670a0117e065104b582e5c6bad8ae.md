# HTML Prototype and Architecture Diagram Preview Tab Support

This change adds a new **HTML PreVIEWS** tab to the Switchboard ARTIFACTS (`planning.html`) panel. This allows stakeholders and developers to inspect HTML-based prototypes and architecture diagrams (such as interactive Mermaid or D3 documents generated by AI agents) directly within VS Code, without requiring an external browser.

To support interactive elements and local style sheets, the files are loaded via Webview URIs into a sandboxed `<iframe>` with scripts allowed. Configured folder paths for HTML previews are kept separate from the text-based local planning docs.

## Goal

Add an HTML Previews tab to the ARTIFACTS panel that renders local `.html`/`.htm` files inside a sandboxed iframe via Webview URIs, supporting interactive content and relative assets, with folder configuration, file tree navigation, and auto-refresh on file changes.

## Metadata

## User Review Required

> [!IMPORTANT]
> **Dynamic localResourceRoots**: For VS Code to resolve relative assets (such as images, CSS, or JS scripts) loaded by local HTML documents, the directory containing the HTML file must be added to the Webview panel's `localResourceRoots`. This plan dynamically updates the Webview's options (`this._panel.webview.options.localResourceRoots`) when paths are configured or loaded. Only paths that pass `fs.existsSync()` verification are added.
> **CSP Modification**: The Content Security Policy in `planning.html` will be expanded to allow frames (`frame-src {{WEBVIEW_CSP_SOURCE}};`). Note: `data:` is intentionally excluded from `frame-src` to avoid unnecessary attack surface.
> **Sandbox Trade-off**: The iframe uses `sandbox="allow-scripts allow-same-origin"`. The `allow-same-origin` flag is required for the iframe to load its own local CSS/JS via webview URIs; without it, relative assets break. This combination weakens the sandbox (the iframe could programmatically remove its own sandbox attribute), but the risk is acceptable for local development tooling where the HTML content is user-controlled.

## Edge-Case & Dependency Audit

- **Race Conditions**: Rapid folder add/remove could trigger concurrent `_sendHtmlDocsReady()` calls. Mitigation: add a debounce (300ms, matching `_activeDocWatchDebounce` pattern) to batch rapid changes before updating `localResourceRoots` and sending `htmlDocsReady`.
- **Security**: CSP `frame-src` must not include `data:` to prevent arbitrary inline iframe content. The `sandbox` attribute on the iframe provides defense-in-depth but `allow-same-origin` weakens it — acceptable for local dev tooling where content is user-controlled.
- **Side Effects**: Updating `localResourceRoots` on a live panel could theoretically invalidate cached webview URIs for other tabs. In practice, `localResourceRoots` is additive (only adding new roots, never removing existing ones), so existing URIs remain valid.
- **Dependencies & Conflicts**: No conflicts with active kanban plans. The `switchboard.research.htmlFolderPaths` config key is new and independent of `switchboard.research.localFolderPaths`. The `_setupHtmlFolderWatchers()` method must be called alongside `_setupLocalFolderWatchers()` in `open()` (PlanningPanelProvider.ts:298) and re-called when folders change.

## Dependencies

- None — this is a self-contained feature addition.

## Adversarial Synthesis

Key risks: (1) CSP `frame-src` must exclude `data:` to avoid unnecessary attack surface; (2) `sandbox="allow-scripts allow-same-origin"` weakens iframe sandbox but is required for local asset loading — trade-off documented and acceptable for dev tooling; (3) `localResourceRoots` must be updated via `this._panel.webview.options` (not `this._panel.options`) with null-checks and existence verification. Mitigations: strip `data:` from CSP, add debounce to `_sendHtmlDocsReady()`, validate folder paths exist before adding to `localResourceRoots`, follow existing dispose-and-recreate pattern for watchers.

| ID | Severity | Description |

| CRITICAL-1 | CRITICAL | Path traversal vulnerability in `_handleFetchPreview` html-folder branch.
