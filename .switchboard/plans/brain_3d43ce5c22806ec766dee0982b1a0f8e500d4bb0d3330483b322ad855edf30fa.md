# Cyberpunk Theme Background and Glassmorphism Styling

## Goal
The optional "Cyberpunk" visual style in `planning.html` currently only styles text glow, headers, and scanlines, leaving the rest of the application on a flat pure black background. This does not deliver the immersive cyberpunk theme the user expected. We will replace the flat black background with a detailed retro-futuristic grid and soft magenta/cyan neon glow, and transform the various layout panels (sidebar, controls, cards, tabs, and modals) into semi-transparent glassmorphic containers.

**Root Cause:** The `.cyber-theme-enabled` CSS block (lines ~1453–1526 in `planning.html`) adds glow effects to text and selection states, but never overrides the solid `background: var(--panel-bg)` / `background: var(--panel-bg2)` applied to structural containers. The `body` background remains `#000000`, and all child panes paint solid dark backgrounds on top of it, making any body-level background art invisible.

## Metadata
**Tags:** frontend, ui, ux, feature
**Complexity:** 4

## User Review Required

> [!NOTE]
> The style changes are entirely scoped under the `.cyber-theme-enabled` CSS class selector. They will only take effect when the user enables the "Cyber Panel" theme option in the Switchboard extension configuration, meaning there is zero risk of altering the standard dark theme.

> [!WARNING]
> `backdrop-filter: blur()` requires the blurred element to have a non-opaque (semi-transparent) background AND must not be inside an ancestor with `overflow: hidden` that creates a clipping stacking context. Test each glassmorphic panel in DevTools to confirm blur renders correctly — if it fails silently, the element will still look acceptable (semi-transparent without blur), but the intended glass effect won't be present.

## Complexity Audit

### Routine
- Adding CSS-only rules under an existing `.cyber-theme-enabled` class selector — no JS changes required
- Grid and radial-glow backgrounds via `background-image` on `body` — standard CSS
- Glassmorphic pattern (semi-transparent bg + `backdrop-filter: blur`) is a well-established technique
- All changes isolated to `planning.html`; no other source files touched

### Complex / Risky
- `backdrop-filter` may be silently ignored in ancestors with `overflow: hidden` (common in VS Code webviews); each pane must be individually verified
- Several content containers (`#research-content`, `#clipboard-content`, `#notebook-content`, and per-tab `.active` wrappers) have solid `var(--panel-bg)` backgrounds and will visually block the body grid unless explicitly overridden — these are easy to miss
- The `.kanban-preview-pane` class referenced in existing CSS (line 1475) does not match the actual DOM element `#kanban-preview-pane` (id); the transparent-background override must use `#kanban-preview-pane` (id selector) to actually apply
- Z-index layering: `.cyber-scanlines` sits at `z-index: 5`; all glassmorphic panes must remain at `z-index: auto` (below scanlines) or the scanline overlay will be obscured

## Edge-Case & Dependency Audit

**Race Conditions**
- None. Pure CSS changes; no async/timing concerns.

**Security**
- None. No JS, no user input, no data handling.

**Side Effects**
- `backdrop-filter: blur()` is GPU-accelerated and can cause performance regressions on lower-end machines. The blur radius must remain modest (8–12px) to avoid jank. The effect is already gated behind a user opt-in toggle.
- Setting a large background gradient/image on `body` adds a small paint cost on scroll; this is negligible for a static background.
- Overriding `background` to `transparent` on multiple containers may expose unintended parent backgrounds in edge cases (e.g., nested `.planning-card` inside `.research-content`). Visual inspection across all tabs is required.

**Dependencies & Conflicts**
- The existing `--panel-bg` and `--panel-bg2` CSS custom properties are used by ~30+ selectors. This plan does NOT change those variables; instead it adds `background: rgba(...)` overrides under `.cyber-theme-enabled` context — no conflict.
- The `cyber-scanlines` element (already in DOM at `z-index: 5`) must remain the topmost decorative layer; glassmorphic panels must stay below it.

## Dependencies
- None (self-contained CSS-only change in a single file)

## Adversarial Synthesis
Key risks: (1) `backdrop-filter` may silently fail in parent `overflow: hidden` contexts — mitigation is to keep the blur value modest so the fallback (semi-transparent flat) is still attractive; (2) solid-background child containers will obscure the body grid unless explicitly overridden — the `Proposed Changes` section below enumerates all containers needing a transparent override; (3) using `.kanban-preview-pane` (class) instead of `#kanban-preview-pane` (id) would cause a silent selector miss — the id form must be used. Mitigations: thorough per-tab manual verification in DevTools after implementation.
