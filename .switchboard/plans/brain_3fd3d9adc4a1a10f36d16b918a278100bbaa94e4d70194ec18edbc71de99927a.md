# Enhancement of Planning Panel Document Cards and Antigravity Session Grouping

## Goal

Upgrade document listing cards in the Switchboard Planning webview from plain list items with emoji icons and bare filenames to rich cards with human-readable titles, metadata, and quick action controls (Set Context, Link Doc, Copy Path, Delete, Sync). Additionally, group Antigravity brain sessions by date subheaders and style them cleanly without brain icons.

### Background Context & Problem Analysis
The current document listings in the Planning webview use bare filenames with emoji file icons (`📄`, `📁`, `🖼️`, `🧠`), which makes it hard to identify documents at a glance — especially markdown files whose filenames don't match their H1 titles. The Antigravity sessions section shows UUID fragments with brain icons but no date grouping, making it difficult to find recent sessions. This plan addresses both issues: extracting titles from files for display, adding inline action buttons to each card, and organizing Antigravity sessions chronologically.

### Complex / Risky
- Title extraction in `LocalFolderService.listFiles()` and `listHtmlFiles()`: reading the first 1000 bytes of each file during the scan loop introduces async I/O inside a previously synchronous-feeling scan. Must ensure this doesn't significantly slow down the folder listing for large directories. The scan already uses `fs.promises.readdir`, so adding `fs.promises.readFile` for the first chunk is consistent but adds N file reads per scan. **Mitigation: add a file count threshold (e.g. 200 files) — if a folder exceeds this count, skip title extraction and fall back to filenames.**
- Card layout in `renderNode()`: changing from a single-line flex row to a multi-line card with title + subtitle + action buttons is a significant DOM structure change. All existing click handlers, selection logic, and delete buttons must continue to work. The `renderNode` function is shared across Local Docs, Online Docs, and HTML Previews tabs — the card layout must adapt per `sourceId`. **Future improvement: refactor to config-driven action button mapping per sourceId instead of hardcoded if/switch chains.**
- `handleImportedDocsReady()` creates doc items manually (not via `renderNode`). Adding card layout + action buttons here requires a **mandatory** shared `renderDocCard()` helper to prevent the two rendering paths from drifting out of sync.

## Edge-Case & Dependency Audit

- **Race Conditions:** Title extraction during `listFiles()` is async. If the file is being written while the scan runs, the extracted title may be partial or empty. This is acceptable — the fallback is the filename (current behavior). No new race conditions are introduced beyond what already exists.
- **Security:** Reading file content for title extraction is safe — only the first 1000 bytes are read, and the content is never executed or injected into the DOM at scan time. The title is rendered via `textContent` (not `innerHTML`), preventing XSS.
- **Side Effects:** Adding `title` and `absolutePath` to `TreeNode` increases the payload size of `localDocsReady` and `htmlDocsReady` messages. For typical folders (< 100 files), the increase is negligible. The `absolutePath` field exposes full filesystem paths to the webview, but this is already available via the `sourceFolder` + `relativePath` combination in the existing metadata.
- **Dependencies & Conflicts:** Plan 1 (Design System Browser Tab) also modifies `renderNode()` to support `sourceId === 'design-folder'`. Both plans must coordinate on the card layout structure. If Plan 1 is implemented first, the Design System tab's `renderNode` branch must also use the new card layout. If this plan is implemented first, the Design System tab must adopt the card layout from the start.

## Dependencies

- `sess_docCardsAndGrouping` — Document cards and Antigravity grouping (this plan)

## Adversarial Synthesis

Key risks: (1) Title extraction adds async I/O to the file scan loop, potentially slowing listing for large folders. Mitigation: cap at 1000 bytes per file and use `Buffer.byteLength` guard; typical Switchboard folders are small. (2) Card layout changes to `renderNode()` affect all tabs that use tree nodes. Mitigation: make card layout conditional on `sourceId` — only Local Docs, Antigravity, and HTML Previews get full cards with action buttons; Online Docs keep simpler layout. (3) `handleImportedDocsReady` duplicates rendering logic. Mitigation: extract a shared `renderDocCard()` helper used by both `renderNode` and `handleImportedDocsReady`.

  - Add a static `TITLE_EXTRACTION_FILE_LIMIT = 200` constant. If the number of file entries in `results` exceeds this limit at the time of extraction, skip title extraction for remaining files (fall back to filename).
  - After pushing a file result (around line 238-246), if the file is a text file and the file count is within the limit, read the first 1000 bytes and extract the title:
    ```typescript
    if (this._isTextFile(entry.name)) {
        try {
            const buf = Buffer.alloc(1000);
            const fd = await fs.promises.open(fullPath, 'r');
            const { bytesRead } = await fd.read(buf, 0, 1000, 0);
            await fd.close();
            const head = buf.toString('utf8', 0, bytesRead);
            // Extract first # Heading or frontmatter title/topic
            const fmMatch = head.match(/^---\s*\n[\s\S]*?^title:\s*["']?(.+?)["']?\s*$/m) ||
                            head.match(/^---\s*\n[\s\S]*?^topic:\s*["']?(.+?)["']?\s*$/m);
            const h1Match = head.match(/^#\s+(.+)$/m);
            const extractedTitle = fmMatch ? fmMatch[1].trim() : h1Match ? h1Match[1].trim() : undefined;
            if (extractedTitle) {
                results[results.length - 1].title = extractedTitle;
            }
        } catch { /* title extraction failure is non-critical */ }
    }
    ```
- Update `listHtmlFiles()` return type to include `title?: string` (around line 292-299).
- Update `_scanHtmlFolder()` to extract `<title>` from HTML files:
  - After pushing an HTML file result, read the first 1000 bytes and extract the `<title>` tag:
    ```typescript
    if (entry.isFile() && path.extname(entry.name).toLowerCase() === '.html') {
        try {
            const buf = Buffer.alloc(1000);
            const fd = await fs.promises.open(fullPath, 'r');
            const { bytesRead } = await fd.read(buf, 0, 1000, 0);
            await fd.close();
            const head = buf.toString('utf8', 0, bytesRead);
            const titleMatch = head.match(/<title[^>]*>([^<]+)<\/title>/i);
            if (titleMatch && titleMatch[1].trim()) {
                results[results.length - 1].title = titleMatch[1].trim();
            }
        } catch { /* title extraction failure is non-critical */ }
    }
    ```

  ```typescript
              ...(f.sourceFolder && f.relativePath ? { absolutePath: path.resolve(f.sourceFolder, f.relativePath) } : {})
          }
      }));
  }
  ```
