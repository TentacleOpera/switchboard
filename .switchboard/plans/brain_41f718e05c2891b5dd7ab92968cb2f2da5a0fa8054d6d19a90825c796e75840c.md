# Fix Delayed Kanban Card Movement by Supporting Plan ID Selectors

## Goal

Switch all Kanban DOM selectors and `currentCards` lookups to use `planId` as the primary key (with `sessionId` as legacy fallback), so optimistic card movements work instantly for all plans — including those without an active session.

**Core Problem:** Optimistic card movements on the Kanban board fail (causing a few seconds of lag while waiting for backend re-render) for any card without an active `sessionId`. The root cause is that DOM selectors like `.kanban-card[data-session="${id}"]` and `currentCards.find(c => c.sessionId === id)` only match by session, so sessionless cards are never found in the DOM or in-memory data, causing the optimistic path to silently skip.

**Background:** `planId` is the primary identifier; `sessionId` is legacy (explicitly documented as such at line 5379: "planId (primary), sessionId (legacy)"). The card rendering already sets `data-plan-id="${cardId}"` as the primary attribute on every card element (line 5237), with `data-session` as a secondary attribute that is often empty for new plans. Some handlers already use the correct primary-first pattern (`el.dataset.session || el.dataset.planId` at lines 4129, 4139, 4916, 4929, 4936), but the optimistic movement functions and drag-and-drop handlers still use `data-session` as the primary selector. The fix is to invert these to `data-plan-id`-first, matching the established identity model.

## Edge-Case & Dependency Audit

- **Race Conditions:** No new race conditions. The selector is synchronous and deterministic — `data-plan-id` is always set on every card (line 5237), so the primary match always succeeds.
- **Security:** No security implications. Selectors operate on pre-rendered DOM attributes set by the extension's own rendering code.
- **Side Effects:** The `promptSelected` backend message sends `sessionIds: [sessionId]` where `sessionId` may be `''` (or a plan ID with this fix) for plan-only cards. The backend already handles this by re-rendering. No new side effects are introduced.
- **Dependencies & Conflicts:** No dependencies on other plans or external changes. The `data-plan-id` attribute is already rendered on all card elements.

## Dependencies

None.

## Adversarial Synthesis

Key risks: (1) The `promptSelected` message sends empty/plan-only IDs in `sessionIds` — the backend handles this via re-render. (2) The `promptOnDropResult` handler uses `includes` which won't match plan-only cards without fallback, resolved by checking both plan and session IDs. Mitigations: Primary-first selectors ensure `data-plan-id` (always present) matches first; `data-session` is only consulted for legacy cards that still have a session.

#### 1. `moveCardsOptimistically` — DOM selector: planId-primary (line 4078)

**Context:** The function queries `.kanban-card[data-session="${id}"]` to find card elements. For sessionless cards, this returns `null` and the card is skipped.

**Logic:** Switch to `data-plan-id` as the primary selector (always present on every card), with `data-session` as legacy fallback.

**Implementation:**
```js
// BEFORE (line 4078):
const cardEl = document.querySelector(`.kanban-card[data-session="${id}"]`);

// AFTER:
const cardEl = document.querySelector(`.kanban-card[data-plan-id="${id}"]`) || document.querySelector(`.kanban-card[data-session="${id}"]`);
```

**Note:** The `currentCards.find` on line 4082 already uses `(c.sessionId || c.planId) === id` — update this to `(c.planId || c.sessionId) === id` for consistency with the primary-first model.

#### 2. `.card-btn.copy` event listener — planId-primary for optimistic move (lines 4967, 4991)

**Context:** The copy button handler extracts only `sessionId` and passes it to `moveCardsOptimistically`. For sessionless cards, `sessionId` is `''`, so `moveCardsOptimistically([''], ...)` never finds a match.

**Implementation:**

The `moveCardsOptimistically([sessionId], column, nextCol)` call on line 4991 then works because `sessionId` now contains the `planId` (always present) when no session exists.

#### 8. `promptOnDropResult` handler — planId-primary `currentCards.find` with `includes` (line 6317)

**Context:** This handler checks if any card's `sessionId` is in the returned `msg.sessionIds` array. For plan-only cards, `c.sessionId` is empty/undefined and won't match.

**Implementation:**
```js
// BEFORE:
const card = currentCards.find(c => msg.sessionIds.includes(c.sessionId));

// AFTER:
const card = currentCards.find(c => msg.sessionIds.includes(c.planId) || msg.sessionIds.includes(c.sessionId));
```

**Edge Case:** If `c.sessionId` is `undefined`, `includes(undefined)` returns `false`, so this is safe. `planId` is checked first since it's always present.

#### 9. Existing `sessionId || planId` patterns — flip to `planId || sessionId` for consistency (lines 4129, 4139, 4916, 4929, 4936, 4082)

**Context:** Several handlers already use the fallback pattern but in the wrong order (`sessionId || planId`). These should be flipped to `planId || sessionId` to match the primary-first identity model.

**Implementation:**
```js
// Line 4082 — moveCardsOptimistically currentCards.find:
const cardData = currentCards.find(c => (c.planId || c.sessionId) === id);

// Line 4129 — getSelectedInColumn:

// Line 4139 — getSelectedInRenderedContainer:

// Line 4916 — card click handler:

// Line 4936 — re-apply selection state:

```

**Note:** Line 4929 already sends both `sessionId` and `planId` to the backend — no logic change needed, just the `pid` variable computation is flipped.

#### 10. `promptOnDropResult` copy-button lookup — planId-primary (line 6236)

**Context:** The `promptOnDropResult` handler looks up the copy button by `data-session` to update its label. For sessionless cards, this query fails.

**Implementation:**
