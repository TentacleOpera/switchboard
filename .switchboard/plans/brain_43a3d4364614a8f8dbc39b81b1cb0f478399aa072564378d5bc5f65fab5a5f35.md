# Fix Verification Step Parsing for Spelling Typos and Direct Headers

In the Kanban view UAT ('test') tab, plan verification steps are not parsed under headers with spelling typos (e.g., "Verificaton Plan", "Verificaiton Steps") or when steps are written directly under a "## Verification Plan" header without manual subheadings. This plan addresses both issues.

## Goal

Fix the `_parseVerificationSteps` method in KanbanProvider to tolerate common spelling typos in "Verification" headers and to correctly extract steps placed directly under `## Verification Plan` without a manual-specific subheading trigger.

## Metadata

- **Tags:** [bugfix, frontend]
- **Complexity:** 3

## User Review Required

> [!NOTE]
> The proposed fix modifies the regex matching for UAT verification plan sections in the Kanban provider to be typo-tolerant and support direct lists. It preserves all existing parsing rules and tests.

## Edge-Case & Dependency Audit

- **Race Conditions:** None — `_parseVerificationSteps` is a pure synchronous function with no shared state.
- **Security:** No security implications — parsing plan markdown content.
- **Side Effects:** Setting `inManualStepsSection = true` by default means steps directly under `## Verification Plan` (before any `###` subheading) will now be captured. The `###` subheading check resets the flag to `false`, so automated test sections are still excluded. Plans with no subheadings at all will have their steps captured, which is the intended UX improvement.
- **Dependencies & Conflicts:** Pattern 3 only runs when Patterns 1 and 2 found zero steps (dedup guard at line 6557). No conflict with other patterns.

## Dependencies

- None

## Adversarial Synthesis

Key risks: (1) `inManualStepsSection = true` default could capture automated steps in degenerate plans that have `## Verification Plan` followed by numbered steps with no `###` subheadings at all. Mitigation: such plans are ambiguous and showing steps in the UAT tab is better UX than hiding them. (2) The `Verific[a-z]*` regex could match degenerate headers like `## Verific Plan`, but this is purely theoretical and won't occur in real plan files. Overall risk is very low.

## Proposed Changes

### [MODIFY] [KanbanProvider.ts](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/KanbanProvider.ts)

- **Context:** The `_parseVerificationSteps` method (line 6519) uses three regex patterns to extract manual verification steps from plan markdown. Pattern 3 (lines 6557-6593) handles `## Verification Plan` headers but has two bugs: (a) the regex at line 6558 requires exact spelling "Verification", missing common typos; (b) `inManualStepsSection` defaults to `false` at line 6561, so steps placed directly under the header (before any manual-specific subheading) are skipped.
- **Logic:** 
  1. Widen the Pattern 3 regex to match `Verific` followed by any lowercase letters, then either `Plan` or `Steps`. This tolerates typos like "Verificaton" and "Verificaiton" while also matching the valid variant "Verification Steps".
  2. Default `inManualStepsSection` to `true` so steps directly under the verification header are captured. The existing `###` subheading check (line 6572-6575) resets the flag to `false`, preserving the exclusion of automated test sections.
- **Implementation:**

  Line 6558 — widen the regex:
  ```diff
  -            const verificationPlanMatch = content.match(/##\s*Verification\s+Plan\s*\n([\s\S]*?)(?=\n##|$)/i);
  +            const verificationPlanMatch = content.match(/##\s*Verific[a-z]*\s+(?:Plan|Steps)\s*\n([\s\S]*?)(?=\n##|$)/i);
  ```

  Line 6561 — default `inManualStepsSection` to `true`:
  ```diff
  -                let inManualStepsSection = false;
  +                let inManualStepsSection = true;
  ```

- **Edge Cases:**
  - `## Verification Steps` (valid variant, previously unmatched) is now captured — this is correct behavior.
  - `## Verific Plan` (zero letters after "Verific") would match — purely theoretical, no real plan uses this.
  - Plans with `## Verification Plan` followed immediately by numbered steps (no `###` subheading) will now have those steps captured. The `###` reset mechanism ensures automated sections are still excluded.
