# Relocate Kanban Automation Button and Timers to Sub-Bar

Move the automation toggle button and active column countdown timers from the right side of the main controls strip into a new compact sub-bar directly above the columns (in the negative space). Flashing status notifications will appear directly to the right of the button/timers in this sub-bar.

## Goal

Relocate the automation toggle button (`#btn-autoban`) and countdown timers (`#autoban-timers-inline`) from the main controls strip into a dedicated sub-bar above the kanban board, and convert the currently absolute-positioned `#status-message` overlay into an inline flow element within that sub-bar.

## Metadata

- **Tags:** [frontend, UI, UX]
- **Complexity:** 4

## User Review Required

- Confirm that the status message appearing/disappearing as an inline element in the sub-bar is acceptable (minor content reflow within the sub-bar row; the board columns below do not shift).

## Complexity Audit

### Routine
- Adding new `.kanban-sub-bar` CSS class and HTML element
- Moving two existing elements (`#btn-autoban`, `#autoban-timers-inline`) from one container to another
- Adjusting `.kanban-board` `padding-top` value
- Adding `.sub-bar-status` class for inline status message styling
- Updating `showStatusMessage` JS handler to use `display: inline-block` instead of `display: ''`
- Updating `animationend` JS handler to use `display: none` consistently

### Complex / Risky
- Removing inline `position: absolute` styles from `#status-message` and converting it to a flow element ŌĆö if inline styles are not fully removed, the element will escape the sub-bar and overlay the board
- CSS specificity: the global `.strip-icon-btn` styles (32x32, bordered, opaque background) will cascade into the sub-bar button; sub-bar-specific overrides for `.is-active`, `.is-off`, and `:hover` states must be added or the button will look broken (glowing transparent borderless button)

## Edge-Case & Dependency Audit

- **Race Conditions:** None. The DOM moves are static HTML restructuring; JS references elements by `getElementById` which works regardless of container.
- **Security:** No impact. Pure UI restructuring.
- **Side Effects:** The `updateAutobanButtonState()` function (line 4345) sets `autobanBtn.style.display` based on `currentAutomationMode`. This continues to work since the element ID is unchanged. The `syncAutobanCountdownTimer()` function (line 4348) and `updateAutobanIndicators()` (line 5827) both reference `#autoban-timers-inline` by ID ŌĆö unaffected by the move.
- **Dependencies & Conflicts:** The `#status-message` inline styles (`position: absolute; top: 0; left: 0; right: 0; z-index: 100;`) MUST be removed. If any inline style remains, the status message will not flow inline in the sub-bar. The `showStatusMessage` handler must set `display: inline-block` (not `display: ''` which resolves to `block`) to ensure the element sits properly in the flex row.

## Dependencies

None.

## Adversarial Synthesis

Key risks: (1) The `#status-message` inline absolute-positioning styles must be fully removed or the element escapes the sub-bar, breaking the layout. (2) Sub-bar-specific CSS overrides for `.strip-icon-btn` interactive states are required or the button's `.is-active`/`.is-off`/`:hover` visuals will be incoherent on a borderless transparent button. (3) The JS `showStatusMessage` handler must use `display: inline-block` instead of `display: ''` to avoid block-level stacking in the flex row. Mitigations: All three are routine, well-scoped fixes that can be verified visually in a single manual test pass.
