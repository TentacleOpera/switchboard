# Dynamic Kanban Single Column Automation Tab Refactor (New)

This change refactors the local Switchboard automation tab in Single Column mode to make it fully dynamic. It addresses the lack of an overall header, fixes description text, orders the Column Rules section first, introduces a dropdown selector to choose which column to automate (respecting custom columns in the active Kanban layout), resolves transitions dynamically (no hardcoded created-to-completed jumps), and dynamically displays the corresponding terminal pool for the selected column.

## Goal

Refactor the Single Column automation tab to use a dynamic source column selector instead of hardcoded `'PLAN REVIEWED'`, enabling users to automate any Kanban column with the correct role pool and next-column transition resolved at runtime.

## User Review Required

> [!IMPORTANT]
> **Dynamic Role Resolution for Terminal Pools**: In Single Column mode, we will dynamically determine which role pool (e.g., Planner, Coder, Reviewer) to display based on the selected start column:
> - `CREATED` -> **Planner** pool
> - `PLAN REVIEWED` -> **Coder** / **Lead Coder** pool
> - `LEAD CODED` / `CODER CODED` -> **Reviewer** pool
> - Other/Custom Columns -> Dynamically mapped via `columnToPromptRole()` from `agentPromptBuilder.ts`.

## Complexity Audit

### Routine
- Adding `sourceColumn` field to `SingleColumnAutobanConfig` type and default
- Updating `normalizeSingleColumnConfig` to parse/validate `sourceColumn`
- Adding the overall "AUTOMATION" header div to the webview
- Updating `modeDescriptions['single-column']` text
- Reordering Column Rules section before Automation Rules section in the webview
- Adding the column dropdown element populated from `columnDefinitions`
- Parameterizing the terminal pool block by the resolved role instead of hardcoded `'coder'`
- Updating START/STOP button payloads to include `sourceColumn`

### Complex / Risky
- Making `_autobanColumnToRole` delegate to `columnToPromptRole` for dynamic role resolution — changes the core dispatch path for all automation modes, not just single-column
- Fixing `_persistAutobanState` line 5016 which hardcodes `rules['PLAN REVIEWED']` for interval extraction — must use the dynamic `sourceColumn` key or the single-column config's `intervalMinutes` directly
- Guarding against `columnToPromptRole` returning `null` for unmapped custom columns — must disable START and show a message to prevent dispatching to a null role

## Edge-Case & Dependency Audit

- **Race Conditions**: Dropdown change while autoban is running — `setAutomationModeFromKanban` already stops the engine before applying changes (line 5856-5858), so this is safe. No additional guard needed.
- **Security**: No new attack surface. The `sourceColumn` value flows from the webview through the existing `setAutomationMode` message path, which is already trusted input.
- **Side Effects**: Changing `_autobanColumnToRole` to delegate to `columnToPromptRole` affects ALL automation modes (multi-column also calls this method). The delegation must preserve existing behavior for built-in columns while adding support for custom columns.
- **Dependencies & Conflicts**: `columnToPromptRole` (agentPromptBuilder.ts line 977) already handles the full built-in + custom-agent mapping. No new dependency needed — just wire the existing function in.

## Dependencies

- None (all referenced modules are already imported in the target files)

## Adversarial Synthesis

Key risks: `_autobanColumnToRole` must delegate to `columnToPromptRole` for dynamic resolution (affects all modes, not just single-column), `_persistAutobanState` hardcodes `'PLAN REVIEWED'` on line 5016 and must be updated, and `columnToPromptRole` returns `null` for unmapped custom columns requiring a START-button guard. Mitigations: delegation preserves existing behavior for built-in columns, `_persistAutobanState` should read from `singleColumnConfig.intervalMinutes` directly, and the webview should disable START and show a warning when the resolved role is null.
