# Fix HTML Previews in Planning Panel via CSP and srcdoc Rendering

## Goal

Fix the blank white screens rendering issue in the Switchboard HTML preview panel (such as for `pii-fix-before-after.html` prototype mockup) by restoring `srcdoc` rendering with proper Content Security Policy (CSP) configurations.

### Core Problems & Root Cause Analysis

1. **VS Code Webview Iframe Restriction:**
   The codebase was previously updated to load HTML files using `iframe.src = webviewUri` in `planning.js` to avoid CSP inheritance issues. However, VS Code's webview sandbox completely blocks loading local files (even those converted using `asWebviewUri()`) directly inside nested iframes via `iframe.src`. This causes the iframe to remain blank or display a white screen.

2. **Strict Content Security Policy (CSP):**
   Using `iframe.srcdoc` inherits the CSP of the parent webview. The parent webview's CSP in `planning.html` currently blocks external scripts (`https:`) and dynamic javascript execution (`'unsafe-eval'`), which are both required by local HTML prototypes that rely on CDN scripts (like React, ReactDOM, Ant Design, and Babel Standalone from `https://unpkg.com`) and browser-side Babel compilation. In addition, external styles from `https:` are blocked.

3. **Missing `connect-src` for Fetch/XHR:**
   The current CSP relies on `default-src 'none'` as a catch-all. This means `connect-src` is also `'none'`, silently blocking any `fetch()` or XHR calls made from within prototype iframes (e.g., lazy-loaded CDN chunks or API calls from Babel-compiled code).

### Proposed Solution

1. **Restore `srcdoc` Priority in `planning.js`:**
   Modify the rendering logic in `handlePreviewReady` so that if `htmlContent` is available, the panel renders it using `iframe.srcdoc = htmlWithBase` (injecting a `<base>` tag referencing `webviewUri`). This allows local assets and relative paths to resolve naturally while avoiding the VS Code `iframe.src` scheme block.

2. **Relax CSP in `planning.html`:**
   Modify the CSP `<meta>` tag to allow `https:` and `'unsafe-eval'` for `script-src`, `https:` for `style-src`, and `https:` for `connect-src`. This permits React, Antd, and Babel CDN scripts to load and compile in-browser mockup prototypes, and allows XHR/fetch calls from those scripts.

3. **Iframe sandbox (already in place):**
   The `#html-preview-frame` iframe already has `sandbox="allow-scripts allow-same-origin"` (planning.html line 2198). This pre-existing attribute provides defence-in-depth: even with `allow-same-origin`, it blocks the prototype from navigating `window.top`, opening popups, or submitting forms cross-origin. No change required — this is documented here to confirm the security posture is intentional.

## Complexity Audit

### Routine
- Swapping two `else if` branches in `handlePreviewReady` (planning.js) — single-function change, no new logic introduced.
- Adding `https:` to `script-src`, `style-src`, `connect-src`, and `'unsafe-eval'` to the CSP meta tag (planning.html) — one-line change.
- Adding `imageImg.removeAttribute('src')` to the `htmlContent` branch — completing a pre-existing omission in the cleanup path.

### Complex / Risky
- **CSP scope creep:** The relaxed CSP applies to the entire webview, not just the iframe. If the webview's own JS or plan-rendered content were ever to execute untrusted scripts, `'unsafe-eval'` would amplify the attack surface. Mitigated by (a) `renderMarkdown()` sanitizing all plan content before `innerHTML` insertion, and (b) the iframe's pre-existing `sandbox` attribute which restricts what prototype code can do even if it escapes the iframe.

## Edge-Case & Dependency Audit

**Race Conditions:**
- On rapid auto-refresh (file watcher), `iframe.srcdoc` is reassigned while the iframe may be mid-load. This can cause a brief flash. The existing `requestId` guard (`if (requestId !== undefined && requestId !== state.previewRequestId) return;`) is not checked in the `html-folder` branch — the function returns early on line 1383 before the guard on line 1410. However, the backend already serializes requests via `_latestRequestIds`, so stale messages from previous requests will be suppressed at the source. Risk: **low**.

**Security:**
- `'unsafe-eval'` extends to the parent webview scope. Mitigated as noted above — no eval paths exist in the webview's own JavaScript (planning.js uses no `eval`, `Function()`, or dynamic script injection in its own code paths).
- `connect-src https:` allows the iframe's content to make outbound requests. This is intentional for CDN-dependent prototypes. Developers loading their own HTML files accept this risk.
- **Pre-existing `sandbox` attribute on iframe (line 2198):** `sandbox="allow-scripts allow-same-origin"` is already present. This is a required security control — it prevents prototype code from accessing `window.top`, navigating the parent frame, or opening popups. The `allow-same-origin` flag is necessary for the injected `<base>` tag to correctly resolve relative local asset URLs. Removing `allow-same-origin` would harden isolation further but would break relative path resolution for local assets.

**Side Effects:**
- The `srcdoc` approach means the browser fully re-parses and re-renders the HTML on each refresh, vs. `iframe.src` which could use the browser's navigation cache. This is preferable for prototyping (always fresh), with no negative side effects.
- The `<base>` tag injected by `injectBaseTag()` sets `href` to the file's own `webviewUri`. Relative paths in the prototype (e.g., `./assets/logo.png`) will resolve relative to the file's directory. These assets must be inside a directory covered by `localResourceRoots` (already configured by the backend in `_sendHtmlDocsReady`).

**Dependencies & Conflicts:**
- `injectBaseTag()` helper already exists in planning.js (lines 350-366). No new utility needed.
- Backend `_handleFetchPreview` for `html-folder` (PlanningPanelProvider.ts, lines 2099-2109) already sends both `htmlContent` and `webviewUri` together in the success path. The frontend change is purely a reordering of existing branches; no backend changes required.
- The `isImage` branch (lines 1349-1353) is unaffected and must remain the first condition check.

## Adversarial Synthesis

**Risk Summary:** The primary risk is the broadened CSP scope — `'unsafe-eval'` now applies to the entire Planning Panel webview, not just the prototype iframe. This is acceptable given the developer-only threat model (no untrusted user content is eval'd). The pre-existing `sandbox="allow-scripts allow-same-origin"` attribute on the iframe provides a defence-in-depth layer that the original plan failed to document — this was identified and confirmed during the improve-plan review. A secondary risk was the missing `connect-src https:` in the original plan (would have silently broken fetch/XHR from CDN scripts); this has been added as a required correction. Key mitigations: `renderMarkdown()` sanitizes all plan content before DOM insertion; `sandbox` restricts iframe escape vectors; the backend suppresses stale preview messages via `_latestRequestIds`.

```javascript
if (isImage && webviewUri) {
    // Image preview: hide iframe, show image container
    if (iframe) { iframe.style.display = 'none'; iframe.removeAttribute('src'); iframe.removeAttribute('srcdoc'); }
    if (imageContainer) { imageContainer.style.display = 'flex'; }
    if (imageImg) { imageImg.src = webviewUri + '?t=' + Date.now(); } // cache-buster for refresh
} else if (htmlContent) {
    // HTML preview: use srcdoc and inject base tag for relative asset resolution
    // (iframe.src with vscode-webview-resource: URIs is blocked by VS Code's sandbox)
    if (iframe) {
        iframe.style.display = '';
        iframe.removeAttribute('src');
        const htmlWithBase = injectBaseTag(htmlContent, webviewUri);
        iframe.srcdoc = htmlWithBase;
    }
    if (imageContainer) { imageContainer.style.display = 'none'; }
    if (imageImg) { imageImg.removeAttribute('src'); }
} else if (webviewUri) {
    // Fallback: iframe src if htmlContent not available (e.g., backend file read failed)
    if (iframe) {
        iframe.style.display = '';
        iframe.removeAttribute('srcdoc');
        iframe.src = webviewUri + '?t=' + Date.now(); // cache-buster for refresh
    }
    if (imageContainer) { imageContainer.style.display = 'none'; }
    if (imageImg) { imageImg.removeAttribute('src'); }
}
```

**Edge Cases:**
- The backend sends both `htmlContent` and `webviewUri` together on success (PlanningPanelProvider.ts lines 2101-2109). After this change, `htmlContent` will always win for valid HTML files.
- On backend file-read failure (lines 2112-2119), only `webviewUri` is sent. The `else if (webviewUri)` fallback ensures these cases still render (though likely blank via the `iframe.src` path — acceptable as a graceful degradation).
- `imageImg.removeAttribute('src')` added to `htmlContent` branch — completes the cleanup for the image container when switching away from image previews.
