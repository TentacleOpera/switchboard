# Mutual Exclusion of Subagents in Prompt Builder

Add two mutually exclusive subagent options to the "Add-ons" configuration section for all agent roles in the Switchboard Kanban interface:
1. **No Subagents**: Injects a directive forbidding subagent spawning.
2. **Custom Subagent**: Toggles an inline text input to specify a custom subagent name, injecting an instruction allowing its use.

---

## Goal

Replace the existing `useSubagents` boolean checkbox with a three-way `subagentPolicy` radio group that supports "Default", "No Subagents", and "Custom Subagent" options — eliminating mutual-exclusivity bugs and providing a clean UI for controlling subagent behavior in generated prompts.

## Metadata

- **Tags:** [frontend, backend, UI, UX]
- **Complexity:** 5

## User Review Required

> [!IMPORTANT]
> **Design Decision — Radio Group vs. Separate Checkboxes**:
> The original plan proposed two separate checkboxes (`noSubagents`, `customSubagent`) with JS-based mutual exclusivity. After adversarial review, this was changed to a **radio group** (`subagentPolicy`) with three options: "Default" (current behavior), "No Subagents" (strict prohibition), and "Custom Subagent" (with conditional text input). This eliminates the mutual-exclusivity bug class entirely and reuses the existing `type: 'radio'` rendering path in `renderRoleAddons`.
>
> **Interaction with existing `useSubagents`**: The old `useSubagents` checkbox is replaced by the `subagentPolicy` radio. When `subagentPolicy` is "Default", the parallel instruction follows the current default (sequential). When "Custom Subagent" is selected with a name, the parallel instruction authorizes that specific subagent. When "No Subagents" is selected, a strict prohibition overrides any parallel dispatch instruction.

## Complexity Audit

### Routine
- Adding `subagentPolicy` radio addon metadata to `ROLE_ADDONS` in `sharedDefaults.js` (replacing `useSubagents` entries)
- Adding `noSubagentsEnabled` and `customSubagentName` fields to `PromptBuilderOptions` interface
- Adding `NO_SUBAGENTS_DIRECTIVE` and `CUSTOM_SUBAGENT_DIRECTIVE` constants to `agentPromptBuilder.ts`
- Constructing `subagentBlock` in `buildKanbanBatchPrompt` with three-way logic
- Adding `noSubagentsByRole` and `customSubagentNameByRole` to `_getPromptsConfig` return object
- Backward-compatible migration: reading old `useSubagents` boolean and mapping to new `subagentPolicy` enum

### Complex / Risky
- Replacing the `useSubagents` checkbox with a radio group that has a **conditional text input** on one option — this extends the existing `type: 'radio'` rendering in `renderRoleAddons` (which currently only renders plain radio buttons, no conditional sub-fields)
- Updating all 13+ call sites in `KanbanProvider.ts` that pass `useSubagentsEnabled` to also pass `noSubagentsEnabled` and `customSubagentName` — missing any call site causes inconsistent prompt generation
- Ensuring the `agentsTabShowInlineForm` custom agent form also supports the new radio group (currently has no subagent policy control)

## Edge-Case & Dependency Audit

- **Race Conditions**: None — all state changes are synchronous UI updates followed by `saveRoleConfig(role)` and `refreshPreview()`.
- **Security**: The `customSubagentName` value is injected into prompt text. If a user enters malicious content (e.g., prompt injection strings), it would be embedded in the generated prompt. Mitigation: sanitize the name to alphanumeric + underscore only before injection.
- **Side Effects**: Changing `subagentPolicy` from "Custom Subagent" back to "Default" must clear `customSubagentName` in the saved config to avoid stale data.
- **Dependencies & Conflicts**: The existing `useSubagents` addon key is stored in persisted `roleConfigs` via `saveRoleConfig`. Old configs with `useSubagents: true` must be migrated to `subagentPolicy: 'default'` during `_getPromptsConfig` loading. The `gatherer` role currently has no `useSubagents` addon — it should not receive the `subagentPolicy` radio group (consistent with current exclusion).

## Adversarial Synthesis

Key risks: (1) The three-way conflict between the existing `useSubagents` and the two new options was the original plan's blind spot — resolved by replacing `useSubagents` with a `subagentPolicy` radio group. (2) 13+ call sites in KanbanProvider.ts must all be updated to pass the new options; missing any causes prompt inconsistency. (3) The conditional text input on the "Custom Subagent" radio option extends the existing radio rendering pattern and needs careful implementation. Mitigations: Radio group eliminates mutual-exclusivity bugs; enumerate all call sites explicitly in the plan; extend the existing `type: 'radio'` rendering with a `textInputOn` option.

### Backend Prompt Construction

#### [MODIFY] [agentPromptBuilder.ts](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/agentPromptBuilder.ts)
- **Context**: `PromptBuilderOptions` interface (lines 75-134) and `buildKanbanBatchPrompt` function (lines 265-408+).
- **Logic**:
  - Add to `PromptBuilderOptions` interface (after line 123, near `useSubagentsEnabled`):
    ```ts
    /** When true, injects strict no-subagent prohibition directive. Overrides useSubagentsEnabled. */
    noSubagentsEnabled?: boolean;
    /** When present and non-empty, injects directive authorizing use of this specific custom subagent. Overrides useSubagentsEnabled. */
    customSubagentName?: string;
    ```
  - Add directive constants (after line 229, near other directive exports):
    ```ts
    export const NO_SUBAGENTS_DIRECTIVE = "SUBAGENT POLICY: You are strictly forbidden from spawning or invoking any subagents. Handle all tasks yourself.";
    export const CUSTOM_SUBAGENT_DIRECTIVE_TEMPLATE = (name: string) =>
        `SUBAGENT POLICY: You are authorized to use the "${name}" subagent for this task. Do not spawn or invoke any other subagents.`;
    ```
  - Update `buildKanbanBatchPrompt` (around lines 286-293):
    - Extract new options:
      ```ts
      const noSubagentsEnabled = options?.noSubagentsEnabled ?? false;
      const customSubagentName = options?.customSubagentName?.trim() || undefined;
      ```
    - Replace the current `parallelInstruction` construction (lines 289-293) with a `subagentBlock` that handles three-way precedence:
      ```ts
      let subagentBlock = '';
      if (noSubagentsEnabled) {
          subagentBlock = NO_SUBAGENTS_DIRECTIVE;
      } else if (customSubagentName) {
          subagentBlock = CUSTOM_SUBAGENT_DIRECTIVE_TEMPLATE(customSubagentName);
          // Also include parallel instruction if multi-plan
          if (plans.length > 1) {
              subagentBlock += '\n\n' + `If your platform supports parallel sub-agents, dispatch one "${customSubagentName}" sub-agent per plan to execute them concurrently. If not, process them sequentially.`;
          }
      } else if (useSubagentsEnabled && plans.length > 1) {
          subagentBlock = `If your platform supports parallel sub-agents, dispatch one sub-agent per plan to execute them concurrently. If not, process them sequentially.`;
      }
      ```
    - Replace the `parallelInstruction` usage in `batchExecutionRules` (line 295) — remove the embedded `parallelInstruction` and instead append `subagentBlock` separately in the suffix block for each role's prompt construction.
  - For the **planner** role (lines 332-408): append `subagentBlock` after the existing suffix block construction (around line 392-398), similar to how `antigravityBlock` is appended.
  - For the **reviewer** role (lines 411-464): append `subagentBlock` in the `suffixBlock` array (line 452).
  - For the **lead/coder/intern/tester/other** roles: append `subagentBlock` in the suffix block wherever `batchExecutionRules` or `parallelInstruction` is currently used.
- **Implementation**: The key change is replacing the binary `useSubagentsEnabled` logic with three-way precedence: `noSubagentsEnabled` > `customSubagentName` > `useSubagentsEnabled`. The `subagentBlock` is constructed once and appended to the prompt suffix.
- **Edge Cases**: If `noSubagentsEnabled` is true AND `customSubagentName` is provided, `noSubagentsEnabled` takes precedence (prohibition wins). If `customSubagentName` is an empty string after trimming, treat as undefined (fall through to `useSubagentsEnabled`).
