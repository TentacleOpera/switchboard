# Custom Theme CSS Override Support for Webviews

## Metadata
**Complexity:** 3
**Tags:** ui, ux, feature, frontend

## Goal

Provide a mechanism for users and their autonomous coding agents to customize the look and feel (colors, fonts, layout, and icon styling) of all Switchboard extension webviews without relying on hardcoded themes or VS Code default themes.

### Core Problems & Background Context
Currently, the extension's webview panels (`KanbanProvider`, `PlanningPanelProvider`, `ReviewProvider`, `SetupPanelProvider`, `TaskViewerProvider`) use hardcoded `:root` CSS variables (such as a cyan/teal sci-fi "video game style" color scheme) inside their HTML templates. There is no way for a user (or a user's coding agent) to override these styles workspace-by-workspace. Some users find the default styling high-contrast or distracting and prefer a more native/minimalist aesthetic.

### Root Cause Analysis
Webviews are loaded from static HTML files (`kanban.html`, `planning.html`, etc.) inside the extension bundle. The styling is defined using inline `<style>` tags. Because the styling is bundled and rendered in isolated iframe/webview contexts, local workspace CSS files are ignored.

To solve this, the extension needs to dynamically read a workspace-specific theme file (e.g., `.switchboard/theme.css`) and inject it into the HTML before loading it into the webview.

---

## Proposed Changes

We will modify the five Webview providers to look for a `.switchboard/theme.css` stylesheet in the current workspace root directory. If the file exists, its content will be read synchronously (using `fs.readFileSync` to support both sync and async helper signatures) and injected as a `<style id="switchboard-custom-theme">` element right before `</head>` in the HTML content.

## Icon Recoloring & Customization Support

Users and agents can easily customize icons inside `.switchboard/theme.css` because they are rendered as standard `<img>` tags inside the webviews. 

1. **Recoloring existing icons with CSS filters**:
   CSS filters can be overridden to map colors. For example, to shift the cyan/teal glow of active elements to a warm orange:
   ```css
   .strip-icon-btn.is-active img,
   .btn-add-plan img,
   .column-icon-btn img {
       filter: sepia(1) saturate(5) hue-rotate(15deg) brightness(1) !important;
   }
   ```
2. **Replacing icons entirely**:
   Standard CSS allows replacing image content using `content: url(...)`:
   ```css
   .btn-add-plan img {
       content: url("data:image/svg+xml;utf8,<svg xmlns='http://www.w3.org/2000/svg' viewBox='0 0 24 24' fill='%23ff5500'><path d='M19 13h-6v6h-2v-6H5v-2h6V5h2v6h6v2z'/></svg>") !important;
       filter: none !important; /* Disable default filters */
   }
   ```

### Automated Tests
Currently, there are no layout integration tests for CSS themes. We will verify that the compilation/bundling succeeds.
- Run compilation:
  ```bash

  ```

### Manual Verification
1. Create a `.switchboard/theme.css` in the project root with the following overrides:
   ```css
   :root {
       --bg-color: #0d1117;
       --panel-bg: #161b22;
       --panel-bg2: #21262d;
       --accent-teal: #f0883e; /* orange */
       --glow-teal: 0 0 8px rgba(240, 136, 62, 0.4);
   }
   /* Shift flat icons to orange/yellow */
   .strip-icon-btn.is-active img,
   .btn-add-plan img,
   .column-icon-btn img {
       filter: sepia(1) saturate(6) hue-rotate(15deg) brightness(1.1) !important;
   }
   ```
2. Open the Kanban board, Artifacts Panel, and Task Viewer.
3. Confirm that the background, panel colors, accents, and icons are successfully recolored based on the `.switchboard/theme.css` definitions.
4. Remove the `.switchboard/theme.css` file and verify that the extension gracefully falls back to the default video game style theme.
