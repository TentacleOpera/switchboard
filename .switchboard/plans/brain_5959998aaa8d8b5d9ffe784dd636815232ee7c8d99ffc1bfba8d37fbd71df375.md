# Fix Antigravity Brain Plan Mirror Scoping and Debounce Issues

## Goal

Resolve the issue where the Switchboard system fails to detect and mirror newly created Antigravity brain plans from the shared `~/.gemini/antigravity-ide/brain` directory.

### Core Problems & Root Cause Analysis

1. **Scoping Cutoff and Time Windows (`TaskViewerProvider.ts`)**:
   - A plan is currently only auto-claimed if it is less than 15 seconds old (`isFreshUnregisteredCandidate` at line 11779) or falls within a dynamic scan `cutoff` window. If the watcher misses the creation event, or if the rescan runs after the file has aged past 15 seconds, the plan is permanently ignored and never mirrored.
   - The system also lacks a proper content-based workspace-scoping check for unregistered plans, resulting in either eager cross-workspace contamination or overly restrictive timing gates.

2. **Debounce Timer Clears Auto-Claim Flag**:
   - On macOS/Linux, creating and writing a plan file triggers a `rename` event (creation) followed by a `change` event (writing content) within milliseconds.
   - The file watcher debounce logic at line 8953 handles the `rename` event first (`allowAutoClaim = true`), setting a 300ms timeout.
   - Immediately after, the `change` event fires (`allowAutoClaim = false`), which clears the existing timeout and schedules a new one with `allowAutoClaim = false`.
   - As a result, the debounce callback always executes with `allowAutoClaim = false`, preventing the plan from ever being auto-claimed at creation.

3. **Stale/Rigid Test Assertions**:
   - Multiple assertions in `workspace-scope-regression.test.js`, `brain-delete-tombstone-regression.test.js`, and `brain-registry-rescue-regression.test.js` are failing because they assert exact regexes of legacy code structures that were altered during the SQLite/database-first refactoring.

## Edge-Case & Dependency Audit

### Race Conditions
- **Stale claims after workspace switch**: If `reinitializeBrainWatcher()` fires while a debounce timer is pending, the old callback may read a phantom entry from `_brainDebounceClaims`. Mitigation: clear `_brainDebounceClaims` alongside `_brainDebounceTimers` in `reinitializeBrainWatcher()` and `dispose()`.
- **Concurrent watcher + manual rescan**: Both the VS Code watcher and native fs.watch fallback share the same `_brainDebounceTimers` and `_brainDebounceClaims` maps. Since Node.js is single-threaded, there's no true data race, but both watchers could fire for the same file event. The debounce mechanism already handles this correctly — both will set entries in the claims set, and the last timer to fire will read the accumulated state.

### Security
- Content-based scoping reads files from the shared brain directory (`~/.gemini/antigravity-ide/brain/`). This directory is user-owned and local-only, so no security boundary is crossed. However, the plan content could theoretically contain crafted paths that match another workspace's root — this is a false-positive risk, not a security vulnerability.

### Side Effects
- The `_isPlanEligibleForWorkspace` enhancement adds file I/O (reading plan content and optionally transcript files) to what was previously a pure in-memory registry lookup. This adds negligible latency (files are local, typically <500KB) but could slow down high-frequency watcher events if many brain plans change simultaneously.

### Dependencies & Conflicts
- The transcript path `.system_generated/logs/transcript.jsonl` is an internal Antigravity implementation detail. If Antigravity changes this path structure, the fallback scoping check will silently degrade (return `eligible: false`), which is safe but reduces the effectiveness of the enhancement.
- The `_normalizeRegistryPlanId` and `_getRegistrySessionId` helpers (lines 9555, 9565) are stable internal APIs — no external dependency risk.

## Dependencies
- `sess_antigravity_brain_mirror` — Brain plan mirroring and watcher infrastructure

## Adversarial Synthesis
Key risks: (1) stale `_brainDebounceClaims` entries after workspace switch could cause phantom auto-claims, (2) test regexes for `_loadPlanRegistry` and `_getWorkspaceIdentityPath` reference code patterns that don't match the actual refactored code, (3) content-based scoping heuristic may false-positive on cross-workspace reference plans. Mitigations: (1) clear claims set in all cleanup paths, (2) fix test regexes to match actual variable names and use bounded matching, (3) content-based check is a fallback only — the primary registry check remains authoritative.

#### 2. Fix the VS Code Watcher Debounce Callback (`handleBrainEvent`, line 8953)
Update the VS Code file watcher debounce to add the stable path to `_brainDebounceClaims` when `allowAutoClaim` is true. Read and clear it within the debounced timeout:
```typescript
const handleBrainEvent = (uri: vscode.Uri, allowAutoClaim: boolean) => {
    const fullPath = uri.fsPath;
    if (!this._isBrainMirrorCandidate(fullPath)) return;

    const stablePath = this._getStablePath(fullPath);
    if (allowAutoClaim) {
        this._brainDebounceClaims.add(stablePath);
    }
    // Debounce: Windows fires multiple events per save (rename + change)
    const existing = this._brainDebounceTimers.get(stablePath);
    if (existing) clearTimeout(existing);
    this._brainDebounceTimers.set(stablePath, setTimeout(async () => {
        try {
            this._brainDebounceTimers.delete(stablePath);
            const finalAllowAutoClaim = this._brainDebounceClaims.has(stablePath);
            this._brainDebounceClaims.delete(stablePath);
            // Skip if we wrote this brain file ourselves (mirror→brain direction)
            if (this._recentBrainWrites.has(stablePath)) return;
            if (fs.existsSync(fullPath)) {
                await this._ensureTombstonesLoaded(workspaceRoot);
                await this._mirrorBrainPlan(fullPath, finalAllowAutoClaim, workspaceRoot);
            }
        } catch (e) {
            console.error('[TaskViewerProvider] Brain watcher debounce callback failed:', e);
        }
    }, 300));
};
```

#### 5. Update Scoping Check in `_isPlanEligibleForWorkspace` (line 10729)
Rename the `_workspaceRoot` parameter to `workspaceRoot` (remove unused-prefix underscore) and enhance the method to inspect the plan's markdown contents or the session's `transcript.jsonl` log for the workspace root path. This ensures that any unregistered brain plan is accurately scoped to its corresponding workspace:
```typescript
private _isPlanEligibleForWorkspace(stableBrainPath: string, workspaceRoot: string): { eligible: boolean; reason: string } {
    const planId = this._getPlanIdFromStableBrainPath(stableBrainPath);
    if (this._isPlanInRegistry(planId)) {
        return { eligible: true, reason: 'in_plan_registry' };
    }

    // Scoping check for unregistered plans: check if the plan contains paths belonging to this workspace
    try {
        const baseBrainPath = this._getBaseBrainPath(stableBrainPath);
        if (fs.existsSync(baseBrainPath)) {
            const content = fs.readFileSync(baseBrainPath, 'utf8');
            const normalizedRoot = this._getStablePath(workspaceRoot);

            if (content.toLowerCase().includes(normalizedRoot.toLowerCase())) {
                return { eligible: true, reason: 'workspace_path_match' };
            }

            // Fallback: check session transcript
            const sessionDir = path.dirname(baseBrainPath);
            const transcriptPath = path.join(sessionDir, '.system_generated', 'logs', 'transcript.jsonl');
            if (fs.existsSync(transcriptPath)) {
                const transcript = fs.readFileSync(transcriptPath, 'utf8');
                if (transcript.toLowerCase().includes(normalizedRoot.toLowerCase())) {
                    return { eligible: true, reason: 'workspace_transcript_match' };
                }
            }
        }
    } catch (e) {
        console.warn('[TaskViewerProvider] Failed to verify plan workspace eligibility:', e);
    }

    return { eligible: false, reason: 'not_in_plan_registry' };
}
```

**Note**: The `_getBaseBrainPath` call strips `.resolved` suffixes from the stable path. If the file on disk is a `.resolved` sidecar (e.g., `plan.md.resolved.1`), `fs.existsSync(baseBrainPath)` checks for the base `.md` file. If only the `.resolved` version exists, the content check is skipped and the method falls through to `not_in_plan_registry`. This is acceptable because `.resolved` sidecars are processed through a different code path that already handles scoping.
