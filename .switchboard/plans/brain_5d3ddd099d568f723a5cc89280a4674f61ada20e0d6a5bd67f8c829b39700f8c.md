# Audit App User IDs across app, RevenueCat and Airbridge

## Step 1: Understand the Goal
The objective is to establish a singular, unified identity layer across the application's infrastructure to ensure seamless data flow between RevenueCat, Airbridge, and Amplitude. Currently, the application is fragmented: RevenueCat authentication relies on the Firebase `uid`, whereas existing analytics platforms (Heap and Branch) identify users via the persistent MongoDB `_id`. This discrepancy will permanently fracture server-to-server subscription attribution if left unresolved. We must standardize the `appUserID` to `_id` and explicitly configure RevenueCat to capture both native device identifiers and cross-platform attributes.

## Step 2: Complexity, Edge-Case & Dependency Audit
*   **Complexity:** Moderate. The code modifications are highly localized to the authentication and purchase services, but the structural implications for data integrity are massive. 
*   **Edge Cases:**
    *   **Orphaned Subscriptions:** Shifting the `appUserID` from `uid` to `_id` means active users updating the app will present a new identifier to RevenueCat. This could temporarily hide their active entitlements until a "Restore Purchases" is triggered.
    *   **ATT Authorization (iOS):** Calling `Purchases.collectDeviceIdentifiers()` before the user has accepted the iOS App Tracking Transparency (ATT) prompt will result in a null IDFA.
*   **Dependencies & Conflicts:** This plan relies on the `_id` property being consistently returned in the authentication payload from the `authoriseUser` backend calls. It also serves as a hard dependency for the upcoming Amplitude and Airbridge SDK instrumentation tasks (due April 14), preparing the methods needed to pass those platform IDs into RevenueCat.

## 5. Exhaustive Implementation Spec

### Adversarial Synthesis

#### Grumpy Critique
You are blindly switching the RevenueCat `appUserID` from Firebase `uid` to MongoDB `_id` on an app that is live in production! Do you have any idea what that does? It orphans every existing subscriber's purchase history! When they log in with the new `_id`, RevenueCat will treat them as a brand new user with no active entitlements. The app will lock them out of premium features until they manually tap 'Restore Purchases'. Furthermore, adding `$airbridgeDeviceId` and `$amplitudeUserId` as optional parameters to `_setMetaData` is useless unless you actually update the callers to pass those IDs down, which you can't do yet because those SDKs aren't even installed!

#### Balanced Response
Grumpy is correct regarding the subscription fragmentation risk. Because we are fundamentally migrating the `appUserID` string, RevenueCat's aliasing behavior will treat the `_id` login as a new identity. The development team must ensure that the application gracefully handles "Restore Purchases" flows to recover access for existing users updating the app. Regarding `_setMetaData`, expanding its signature with optional parameters now is the correct architectural step. It future-proofs the service so that when the Amplitude and Airbridge SDKs are installed in the upcoming tasks, the engineers can simply pass the retrieved identifiers into this method without needing to refactor `PurchaseService.ts` again.

#### `src/library/services/Auth.tsx`

##### MODIFY `src/library/services/Auth.tsx`
*   **Context:** The `_initializeUser` method currently initializes the `PurchaseService` using `data.uid`, which creates a mismatch with the `_id` string utilized by the rest of the application's analytics.
*   **Logic:** Modify the `PurchaseService().initialise()` invocation to consume `data._id` instead. This ensures RevenueCat shares the exact same primary key as your backend database and downstream analytics, establishing a unified identity triangle.
*   **Implementation:**
```tsx
const _initializeUser = async (data: any, dispatch: any) => {
  const transformedMembers = transformMembersToAvatarState(data?.members || []);
  
  // CHANGED: Use data._id instead of data.uid to ensure universal identity matching
  await PurchaseService().initialise(data._id, dispatch); 
  
  dispatch(setLoginUserData({ ...data, members: transformedMembers || [] }));
  dispatch(setSelectedMember(transformedMembers?.length > 0 ? transformedMembers : {}));
  dispatch(setSessionStartTime(Date.now()));
  
  const { token, refreshToken, gcpToken } = data;
  CStorage.set('userToken', token); // This triggers useEffect in app to create AuthenticatedNavigation
  if (refreshToken) {
    CStorage.set(LocalStorage.RefreshToken, refreshToken);
  }
  CStorage.set('gcpToken', gcpToken.access_token);
  CStorage.set(LocalStorage.User, JSON.stringify(data));
};
```
*   **Edge Cases Handled:** By centralizing on the `_id` string at the exact moment of user hydration, we guarantee that all subsequent RevenueCat SDK calls are irrevocably bound to the database's primary key.

#### `src/library/services/PurchaseService.ts`

##### MODIFY `src/library/services/PurchaseService.ts`
*   **Context:** RevenueCat must be commanded to collect underlying hardware advertising IDs (like IDFA and GPS Ad ID) for Airbridge's probabilistic matching. Additionally, the service must be prepared to accept cross-platform identifiers to securely map S2S billing events to Amplitude and Airbridge user profiles.
*   **Logic:** 
    1.  Inject `await Purchases.collectDeviceIdentifiers();` directly after `Purchases.configure(...)` within the `_initialise` function.
    2.  Expand the `_setMetaData` parameters to accept optional strings for `airbridgeDeviceId`, `amplitudeUserId`, and `amplitudeDeviceId`. Append these to the dictionary passed to `Purchases.setAttributes()`.
*   **Implementation:**
```typescript
const _initialise = async (appUserID: string, dispatch: AppDispatch) => {
  if (__DEV__) await Purchases.setDebugLogsEnabled(true);

  let customerInfo: CustomerInfo;
  const apiKey =
    Platform.OS === 'android'
      ? await Config.getSecret('VIA_RC_ANDROID_KEY')
      : await Config.getSecret('VIA_RC_IOS_KEY');

  try {
    if ((await Purchases.isConfigured()) && !isNil(appUserID)) {
      ({ customerInfo } = await Purchases.logIn(appUserID));
    } else {
      await Purchases.configure({
        apiKey,
        appUserID: isNil(appUserID) ? null : appUserID,
      });
      await Purchases.enableAdServicesAttributionTokenCollection();
      
      // ADDED: Explicitly collect native device identifiers to support Airbridge S2S matching
      await Purchases.collectDeviceIdentifiers(); 
      
      customerInfo = await Purchases.getCustomerInfo();
    }

    Purchases.addCustomerInfoUpdateListener(updatedCustomer => {
      dispatch(setCustomerInfo(updatedCustomer));
    });

    dispatch(setCustomerInfo(customerInfo));
  } catch (e) {
    // crashlyticsError('RC_CONFIG_FAIL', e);
  }

  isPurchaseServiceInitialised = true;
  dispatch(setIsInitialised());
};

const _setMetaData = async (
  email: string, 
  displayName: string, 
  airbridgeDeviceId?: string, 
  amplitudeUserId?: string, 
  amplitudeDeviceId?: string
) => {
  // set name and email in RevenueCat for in-app purchases
  try {
    const attributes: Record<string, string> = {
      $email: email,
      $displayName: displayName,
    };

    // ADDED: Append optional cross-platform identifiers to the RevenueCat payload
    if (airbridgeDeviceId) {
      attributes['$airbridgeDeviceId'] = airbridgeDeviceId;
    }
    if (amplitudeUserId) {
      attributes['$amplitudeUserId'] = amplitudeUserId;
    }
    if (amplitudeDeviceId) {
      attributes['$amplitudeDeviceId'] = amplitudeDeviceId;
    }

    await Purchases.setAttributes(attributes);
    await Purchases.syncAttributesAndOfferingsIfNeeded();
  } catch (metaDataError) {
    console.error('RC_SET_METADATA_FAILED', metaDataError);
    crashlyticsError('RC_SET_METADATA_FAILED', metaDataError);
  }
};
```
*   **Edge Cases Handled:** By making the new parameters in `_setMetaData` optional, existing invocations across the app (such as in `OfferScreen.tsx` and `AuthenticatedNavigation.tsx`) will not break during compilation. The `collectDeviceIdentifiers()` call is safely insulated inside the configuration initialization block to ensure it only executes when the SDK is freshly booted.
