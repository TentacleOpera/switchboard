> [!CAUTION]
> **Yes, there are significant background operations that explain your IDE slowdown.** The combined load of ~20+ file watchers, 6+ setInterval timers, and multiple native `fs.watch` fallbacks running simultaneously across all open panels is substantial.

## 🔥 The Compounding Problem

When all three panels (Kanban, Planning, Design) are open simultaneously:

### Timer Stack (always running)
```
GlobalPlanWatcher:     every 10s  → fs reads + DB queries
PipelineOrchestrator:  every 1s   → state computation (when running)
Jules Status Poll:     every 30s  → API/status check
Autoban tick:          every Nm   → DB reads + dispatches (when enabled)
Autoban sweep:         every 60s  → DB reads (when enabled)
ContinuousSync idle:   every 60s  → mtime checks (when enabled)
Kanban countdown:      every 1s   → DOM updates (when autoban enabled)
Plan Scanner:          every Ns   → fs reads + DB sync (when enabled)
```

### Watcher Overlap (critical issue)
Multiple services watch the **exact same directories** redundantly:

| Directory | Watchers |
|-----------|----------|
| `.switchboard/plans/**/*.md` | **5 watchers**: TaskViewerProvider (VSCode + native), PlanningPanelProvider, GlobalPlanWatcherService (VSCode + native) |
| `.switchboard/state.json` | **4 watchers**: extension.ts (VSCode + native), TaskViewerProvider (VSCode + native) |
| Antigravity brain dirs | **3 watchers**: TaskViewerProvider (VSCode + native), PlanningPanelProvider |
| `.switchboard/` directory | **2 native watchers**: extension.ts, TaskViewerProvider |

### Total Active Watchers
With all panels open: **~20+ simultaneous file watchers**, roughly half of which are native `fs.watch` fallbacks that duplicate VS Code's built-in watchers.

## 🎯 Worst Offenders (Prioritized)

### 1. `GlobalPlanWatcherService` — 10-second filesystem scan 🔴
- Scans **all** mapped folders every 10 seconds
- Does directory listings + DB lookups on each tick
- **Even when no plans are changing**
- Configurable via `switchboard.planWatcher.scanIntervalMs` (default: 10000)

### 2. Redundant native `fs.watch` fallbacks 🔴
- Nearly every VS Code `FileSystemWatcher` has a parallel native `fs.watch` fallback
- This **doubles** the number of OS-level file descriptors and event processing
- The dedup guards (TTL maps, debounce) add their own memory/timer overhead

### 3. `PipelineOrchestrator` 1-second tick 🟠
- Ticks every second when the pipeline is running
- Each tick emits a state change event → potential webview update

### 4. Stacked plan watchers across services 🟠
- `.switchboard/plans/` is watched by 5 different watcher instances
- A single plan file save triggers **5 separate event handlers** → debounce chains → DB queries

---

## 💡 Immediate Mitigations

1. **Increase `switchboard.planWatcher.scanIntervalMs`** — Set to `60000` (60s) or higher in VS Code settings. This is the single biggest win.
   ```json
   "switchboard.planWatcher.scanIntervalMs": 60000
   ```

2. **Disable the plan scanner** if not needed:
   ```json
   "switchboard.planScanner.enabled": false
   ```

3. **Disable periodic plan watcher scan** if file watchers are sufficient:
   ```json
   "switchboard.planWatcher.periodicScanEnabled": false
   ```

4. **Close panels you're not actively using** — each panel adds ~5-8 additional file watchers.

5. **Disable Antigravity brain watchers** if not using research features:
   ```json
   "switchboard.research.antigravityBrainEnabled": false
   ```
