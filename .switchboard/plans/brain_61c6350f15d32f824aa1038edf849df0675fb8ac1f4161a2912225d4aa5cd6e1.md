# Refactor Custom Agent Settings Form in Kanban Tab

This implementation plan covers the removal of redundant settings from the custom agent configuration form (specifically drag & drop mode and column visibility), converting the "Design Doc Link" text field into a standard boolean checkbox that resolves to the global design doc, and simplifying the Prompt Add-ons section by removing the accordion toggle. It also updates the Kanban Column Setup menu to allow custom agents to be selected as assigned agents.

## Goal

Simplify the custom agent form by removing unused settings (drag & drop mode, kanban column visibility), converting the design doc link to a boolean checkbox that resolves the global design doc, and exposing custom agents in the kanban column assigned-agent dropdown.

## Metadata

- **Tags:** [frontend, backend, UI, UX]
- **Complexity:** 5

## User Review Required

- Confirm that converting Design Doc Link from a per-agent text input to a global-resolving checkbox is acceptable (agents will no longer have individual design doc URLs).
- Confirm that removing the "Show as Kanban Column" checkbox is acceptable (custom agents are now assignable to columns via the Setup tab instead).

## Complexity Audit

### Routine
- Remove HTML elements (drag & drop select, kanban checkbox, details/summary accordion)
- Update `agentsTabShowInlineForm` to stop referencing removed elements
- Update `agentsTabSaveCustomAgent` to hardcode removed values
- Add `designDoc?: boolean` to `CustomAgentAddons` interface
- Update `parseCustomAgentAddons` for backward compatibility with `designDocLink` string
- Replace `<details>` with `<div>` for Prompt Add-ons section
- Remove `removeAttribute('open')` call for deleted details element
- Update `renderKanbanAssignedAgentOptions` to append custom agent optgroup
- Remove `includeInKanban: false` from `handleRestoreKanbanDefaults`
- Update `autobanRoles` in setup panel to include all custom agents
- Remove dead test cases for custom agent column propagation

### Complex / Risky
- Backward compatibility: existing agents with `designDocLink` string or `includeInKanban: true` must round-trip correctly through the form
- Design doc resolution in `generateUnifiedPrompt` for custom agents requires reading global VS Code config and fetching Notion cache — currently only the planner/tester branches do this
- Legacy `includeInKanban: true` agents in persisted state will still create columns via `buildKanbanColumns` until the `customColumns` block is removed or a migration runs

## Edge-Case & Dependency Audit

- **Race Conditions**: None — all changes are synchronous form operations or single-threaded VS Code state reads.
- **Security**: No new attack surface. Design doc resolution reads from VS Code settings and Notion cache (existing patterns).
- **Side Effects**: Removing `includeInKanban` from the form means newly created agents always have `includeInKanban: false`. Existing agents with `includeInKanban: true` in state.json will retain that value until edited and re-saved. The `buildKanbanColumns` removal of `customColumns` ensures legacy agents don't produce phantom columns.
- **Dependencies & Conflicts**: The `kanbanOrder` field in `CustomAgentConfig` remains in the interface for backward compatibility but is no longer set meaningfully from the form (always 0). The sort at line 3171 falls through to `name.localeCompare(b.name)`, which is acceptable.

## Dependencies

- None

## Adversarial Synthesis

Key risks: legacy `designDocLink` string values lost on re-edit (mitigate by checking box on load and preserving link in persisted addons), legacy `includeInKanban: true` agents creating phantom columns (mitigate by removing `customColumns` block from `buildKanbanColumns`), and design doc resolution code duplication in `generateUnifiedPrompt` (mitigate by extracting a shared helper). Overall risk is moderate — changes are well-scoped with clear backward-compatibility paths.

- **Extract design doc resolution helper**: Add a private method to resolve the global design doc link and pre-fetch Notion cache:
  ```typescript
  private async _resolveGlobalDesignDoc(workspaceRoot: string): Promise<{ designDocLink?: string; designDocContent?: string }> {
      const config = vscode.workspace.getConfiguration('switchboard');
      const designDocEnabled = config.get<boolean>('planner.designDocEnabled', false);
      const designDocLink = designDocEnabled ? (config.get<string>('planner.designDocLink', '') || '').trim() : undefined;
      if (!designDocLink) return {};
      let designDocContent: string | undefined;
      if (designDocLink.includes('notion.so') || designDocLink.includes('notion.site')) {
          try {
              const notionService = this._getNotionService(workspaceRoot);
              designDocContent = (await notionService.loadCachedContent()) || undefined;
          } catch { /* non-fatal */ }
      }
      return { designDocLink, designDocContent };
  }
  ```
