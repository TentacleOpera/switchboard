# Reliable Switchboard Brain Plan Watcher and Claiming System

## Goal

Resolve real-time plan synchronization issues in the Switchboard extension by making file system watchers recursive and responsive to modifications, fixing startup-time directory existence checks, and coordinating cross-window auto-claiming so only the user's active/focused window claims new brain plans with the correct project scope.

### Problem Analysis & Root Cause

1. **Antigravity brain watcher is non-recursive and misses file changes.** `_setupAntigravityWatcher` (PlanningPanelProvider.ts:435) uses `RelativePattern(brainUri, '*')` — a shallow glob that only catches top-level entries. It also lacks an `onDidChange` listener, so updates to existing plan files are silently ignored. This caused 20+ plans to accumulate unmirrored until a manual full-sync (`switchboard.fullSync`) swept them up.

2. **Kanban plans watcher skips directories that don't exist at startup.** `_setupKanbanPlansWatcher` (PlanningPanelProvider.ts:474) guards with `if (!fs.existsSync(plansDir)) { continue; }` (line 488), so if `.switchboard/plans/` hasn't been created yet when the extension loads, that workspace root never gets a watcher — even after the directory is later created.

3. **Cross-window race condition on auto-claim.** Multiple IDE windows watching the same brain directory all race to auto-claim new plans. The existing atomic claim marker (`_tryClaimBrainPlan`, TaskViewerProvider.ts:10764) prevents double-claiming, but unfocused windows still attempt the claim, creating unnecessary I/O and log noise. There is no soft filter to let unfocused windows opt out early.

4. **Auto-claimed brain plans lack project scope.** When `_mirrorBrainPlan` (TaskViewerProvider.ts:11729) auto-claims a plan, it doesn't set the `project` field, so the plan appears unscoped in the Kanban board even when a project filter is active.

5. **Stale workspace root in brain watcher closures.** `_setupBrainWatcher` (TaskViewerProvider.ts:8919) captures `workspaceRoot` statically at line 8931 (`const workspaceRoot = this._resolveWorkspaceRoot()`). After a workspace switch, the captured value is stale, causing mirrors to be written to the wrong workspace's staging directory. Both the VS Code watcher callback (line 8973) and the fs.watch fallback (line 9013) are affected.

### What happened with the 20 new plans appearing?
Closing and restarting `planning.html` (or reloading the extension) ran a full sync command (`switchboard.fullSync`), which executes `_rescanAntigravityPlanSources` and does a complete disk sweep of the brain folder. This swept up and claimed all the previously missed plans, confirming that the real-time file system watchers were the failing point.

## Edge-Case & Dependency Audit

### Race Conditions
- **Debounce-window workspace switch:** If the user switches workspaces during the 300ms debounce window, `this._resolveWorkspaceRoot()` will return the new root, potentially mirroring the plan to the wrong workspace. **Mitigation:** Capture the workspace root at event-fire time and validate it still matches in the callback; if not, skip the mirror.
- **globalState propagation delay:** `globalState.update()` is async and eventually consistent across windows. Two windows may briefly both believe they're focused. **Mitigation:** The atomic claim marker in `_tryClaimBrainPlan` (writeFileSync with `flag: 'wx'`) is the authoritative guard; globalState is only a soft filter.
- **fs.watch double-fire on Windows:** Already handled by the existing 300ms debounce mechanism. No change needed.

### Security
- Claim marker files (`.switchboard_claim_<hash>.json`) are written to the shared brain directory. Malicious tampering could allow claim theft. **Mitigation:** Claim markers are advisory, not security-critical. The worst case is a plan appearing in the wrong workspace's Kanban, which is recoverable.

### Side Effects
- Removing the `fs.existsSync` guard in `_setupKanbanPlansWatcher` means watchers are now created for all workspace roots regardless of whether `.switchboard/plans/` exists. This is a minor increase in watcher count but has no functional side effect — VS Code's `createFileSystemWatcher` with `RelativePattern` gracefully handles non-existent intermediate directories.
- Adding `onDidChange` to the antigravity watcher will trigger `_sendLocalDocsReady` on any file modification under the brain directory, including non-plan files. Downstream processing already filters by supported extensions, so this is harmless noise.

### Dependencies & Conflicts
- The `project` column already exists in the Kanban database (added by V23 migration, KanbanDatabase.ts:395). No schema change required.
- `KanbanProvider.getProjectFilter()` (KanbanProvider.ts:3820) already returns the active project filter string or `null`. No change required.
- `KanbanPlanRecord.project` (KanbanDatabase.ts:30) is already typed as `project?: string`. No change required.

## Dependencies

- `sess_KanbanV23Migration` — V23 migration adds `project` column to plans table (already deployed)

## Adversarial Synthesis

Key risks: stale workspace root in fs.watch fallback (unaddressed in original plan), wrong-root mirroring during debounce-window workspace switches, and `||` vs `??` project fallback silently preserving old values on explicit clear. Mitigations: apply dynamic resolution to fs.watch fallback, add root-validation guard in debounce callbacks, switch to nullish coalescing for project field.

## Open Questions

None. The design leverages standard VS Code APIs and matches existing configurations.
