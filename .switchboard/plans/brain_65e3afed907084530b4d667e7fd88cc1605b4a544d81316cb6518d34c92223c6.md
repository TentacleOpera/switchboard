# Unifying Workspace Dropdown Behavior with Active Selection

Currently, in `kanban.html`, the workspace/project dropdown picker behaves inconsistently:
1. If no plan is selected, changing the dropdown immediately switches the workspace context/project filter.
2. If one or more plans are selected, changing the dropdown does not switch the workspace context/project filter immediately; it waits for the user to click the "Assign" button.

This plan unifies the behavior so that the workspace/project context changes instantly regardless of whether plans are selected. Selected plans will remain in memory (and the "Assign" button count will remain active). When the user clicks the "Assign" button, the selected plans will be assigned to the newly selected workspace/project, and they will then appear on the board.

## Goal

Remove the early-return guard in the workspace/project dropdown change listener so that switching workspaces or projects happens instantly even when plans are selected. Update the `selectedCards` data structure to track per-card workspace and project, and update the `renderBoard` pruning logic to retain cross-workspace/cross-project selections across board refreshes.

## Metadata

**Tags:** frontend, UI, UX
**Complexity:** 5

## User Review Required

> [!NOTE]
> Since switching workspaces instantly will update the board's display to show cards from the newly selected workspace, the selected cards from the previous workspace will disappear from the board view. However, they will remain in memory (as indicated by the count on the `ASSIGN (N)` button). When the user clicks `ASSIGN`, those plans will be assigned to the new workspace and will appear on the board.
>
> If the user changes their mind and wants to deselect them without assigning, they can switch back to the original workspace (where they will reappear on the board as selected) and deselect them, or we could potentially add a way to clear selection. (This plan preserves the existing selection clearing on assign/drop).

## Complexity Audit

### Routine
- Removing the early-return guard (`if (selectedCards.size > 0)`) in the dropdown `change` listener — a one-line deletion
- Adding `project?: string` to the `KanbanCard` interface in `KanbanProvider.ts` — purely additive
- Mapping `row.project` / `rec.project` in the two card-mapping blocks in `KanbanProvider.ts` — straightforward field pass-through since `project` already exists on `KanbanPlanRecord`
- Adding `data-project` HTML attribute to the card container `div` in `createCardHtml` — additive template change
- The ASSIGN button click handler requires a minor guard update (`typeof val === 'object' ? val.workspaceRoot : val`) — already largely present in the existing code

### Complex / Risky
- Changing `selectedCards` values from `string` (workspaceRoot) to `{ workspaceRoot, project }` objects — touches a dozen `.delete(key)` sites and the `sourceWorkspaces` extraction in the assign handler; requires careful verification that all consumers of `selectedCards.values()` handle the new format
- The `renderBoard` pruning logic is the gatekeeper for selection persistence; incorrect pruning logic could silently drop or retain wrong cards (e.g., cards that were legitimately deleted from the DB)

## Edge-Case & Dependency Audit

### Race Conditions
- Board refresh triggered immediately after a workspace switch may race with the in-flight `selectWorkspace` message; however, the backend already has debounce/throttle logic and `lastBoardSignature` comparison, so this is low risk.

### Security
- None. No privilege changes. No cross-origin data.

### Side Effects
- `selectedCards` can now grow beyond the currently visible workspace's cards. If a user selects from Workspace A, switches to Workspace B multiple times, and never clicks ASSIGN, memory growth is bounded by total plan count (negligible).
- The ASSIGN button remains enabled across workspace switches, which is the intended UX. No regression on existing ASSIGN flows.
- All other `selectedCards` consumers (context menu, drag-and-drop multi-select, merge buttons) use `selectedCards.keys()` or `selectedCards.has()`, which are unaffected by the value-type change.

### Dependencies & Conflicts
- `KanbanCard.project` depends on `KanbanPlanRecord.project` (already exists as `project?: string` in `KanbanDatabase.ts` line 30).
- The `data-project` attribute on the card DOM element is read as `el.dataset.project` in the card click handler — this is the origin of the stored project value.
- `updateReassignButtonVisibility()` only reads `selectedCards.size` — no change needed.
- All `selectedCards.delete(id)` calls delete by key (sessionId/planId), not by value — unaffected.

## Dependencies

None (no prior plans required).

## Adversarial Synthesis

Key risks: (1) The `selectedCards` value format change from string → object breaks the existing `sourceWorkspaces` extraction in the ASSIGN handler — mitigated by the `typeof val === 'object' ? val.workspaceRoot : val` guard, which must be applied consistently. (2) The `renderBoard` pruning loop must correctly distinguish between "card deleted from DB" (should prune) vs "card belongs to a different workspace" (should retain) — this requires comparing the stored `workspaceRoot` in the `selectedCards` value against `currentWorkspaceRoot`. (3) Partial migration risk if the card click listener sets the old string format in some code paths — verify the click listener at line 4725 is the only place `selectedCards.set(pid, ...)` is called. Mitigations: single atomic migration to `{ workspaceRoot, project }` format at the click listener; update the pruning loop and assign handler in the same PR.

**Step 3 — Update `renderBoard` pruning logic (~lines 4629–4636):**
```javascript
// BEFORE (lines 4629–4636)
// Prune stale selectedCards entries so ASSIGN count and merge buttons stay accurate
// when cards disappear from the board (deletion, workspace/project filter change)
const cardIds = new Set(cards.map(c => c.sessionId || c.planId).filter(Boolean));
for (const key of selectedCards.keys()) {
    if (!cardIds.has(key)) {
        selectedCards.delete(key);
    }
}

// AFTER — retain cross-workspace and cross-project selections; only prune
//         cards that were legitimately removed from *their own* workspace/project
const cardIds = new Set(cards.map(c => c.sessionId || c.planId).filter(Boolean));
for (const key of selectedCards.keys()) {
    const val = selectedCards.get(key);
    const cardWorkspaceRoot = typeof val === 'object' ? val.workspaceRoot : val;
    // If the card belongs to a different workspace, do NOT prune it — it's
    // invisible on the board right now but still tracked for cross-workspace ASSIGN.
    if (cardWorkspaceRoot && cardWorkspaceRoot !== currentWorkspaceRoot) {
        continue;
    }
    // Card belongs to the current workspace. Only prune if it's not in cardIds
    // (meaning it was deleted or completed from the DB).
    if (!cardIds.has(key)) {
        selectedCards.delete(key);
    }
}
```
