# Add Intern Coded Column to Autoban Config and Rules & Single Column Timer UI Fix

This plan fixes the issue where the Kanban automation engine does not respect or run for the `INTERN CODED` column (causing the automation timer to display `I: off`). It also resolves the UI issue in `single-column` mode where inactive column countdown timers are shown, limiting the inline indicators to only display the active source column's timer.

## Goal

Add `INTERN CODED` to the default autoban rules so the automation engine ticks for intern-coded plans, and filter the inline timer badges in the webview to show only the active column's timer in single-column mode (and hide them entirely in antigravity-batch mode).

> 3. The `AUTOBAN_SHARED_REVIEWER_COLUMNS` constant to include `INTERN CODED` for consistency (non-breaking — the shared-reviewer lane dedup is already bypassed by strict column isolation).

## Edge-Case & Dependency Audit

- **Race Conditions:** None. The autoban engine processes each column independently via `_enqueueAutobanTick` with queue serialization. Adding `INTERN CODED` does not introduce concurrent access.
- **Security:** No impact. No secrets, credentials, or user input validation changes.
- **Side Effects:** Adding `INTERN CODED` to `DEFAULT_AUTOBAN_RULES` means existing installations that previously had no `INTERN CODED` rule will now receive one via `normalizeAutobanConfigState` (which merges `DEFAULT_AUTOBAN_RULES` into `rawRules`). This is the desired behavior — it enables the previously-dormant column. Existing persisted configs that explicitly disable `INTERN CODED` will retain their override because `rawRules` spreads over defaults.
- **Dependencies & Conflicts:** The `columnToPromptRole` function (agentPromptBuilder.ts:1048) already maps `INTERN CODED` → `'reviewer'`, so the backend dispatch logic already knows how to route intern-coded plans. The `_autobanColumnToRole` method delegates to `columnToPromptRole`, so no changes needed there. The `_startAutobanEngine` method (TaskViewerProvider.ts:6860-6861) iterates `Object.entries(rules)` and already handles any enabled rule — adding `INTERN CODED` to defaults automatically enables it in the engine.

## Dependencies

- `sess_autobanState` — autoban config normalization and default rules
- `sess_kanbanHtml` — webview inline timer badge rendering

## Adversarial Synthesis

Key risks: (1) Legacy `CODED` rule must NOT propagate to `INTERN CODED` — the intern column was created after the CODED→LEAD/CODER split, so there is no legacy mapping. (2) The `singleColumnConfig` guard in the frontend must handle null/undefined gracefully to avoid a crash when the config hasn't been populated yet. (3) `AUTOBAN_SHARED_REVIEWER_COLUMNS` should include `INTERN CODED` for API consistency even though the shared-reviewer dedup path is bypassed. Mitigations: explicit `INTERN CODED` normalization independent of legacy `CODED`; null-safe `singleColumnConfig` access in the badge filter; additive-only change to the shared-reviewer constant.

  ```javascript
  // Filter badges by automation mode
  let filteredBadgeData = badgeData;
  if (currentAutomationMode === 'single-column') {
      const sourceCol = (autobanConfig.singleColumnConfig || {}).sourceColumn;
      const sourceIdx = autobanColumns.indexOf(sourceCol);
      filteredBadgeData = sourceIdx >= 0 ? [badgeData[sourceIdx]] : [];

  }
  // multi-column: show all badges (no filtering)
  ```

  Then replace all subsequent references to `badgeData` with `filteredBadgeData`:
  - Line 5904: `existingBadges.length !== filteredBadgeData.length`
  - Line 5905: `container.innerHTML = filteredBadgeData.map(b => ...`
  - Line 5913: `for (let i = 0; i < filteredBadgeData.length; i++)`
  - Line 5915: `const data = filteredBadgeData[i];`

  Context: The `autobanColumns` array is derived from `columnDefinitions.filter(col => col.autobanEnabled)` (line 3288), and `INTERN CODED` already has `autobanEnabled: true` in the backend column definitions (agentConfig.ts:96). The causal chain for the `I: off` bug is: `DEFAULT_AUTOBAN_RULES` lacks `INTERN CODED` → `normalizeAutobanConfigState` produces rules without `INTERN CODED` → backend broadcasts config without the rule → frontend `updateAutobanIndicators` finds no rule for the column → renders `I: off`.

  Edge Cases:
  - `autobanConfig.singleColumnConfig` may be null/undefined when the config hasn't been populated yet. The `(autobanConfig.singleColumnConfig || {}).sourceColumn` guard returns `undefined`, causing `sourceIdx` to be -1 and `filteredBadgeData` to be `[]` — safe fallback (no badges shown until config is ready).
  - In antigravity-batch mode, the container is emptied (innerHTML = '') but not hidden via display:none. This matches the existing "disabled" path (line 5880) and avoids layout shift issues.
