# Add Chronological Automation Safety Note to Kanban UI

## Goal

Add an informational banner in the Automation Tab of `kanban.html` below the mode selection dropdown, informing users that automation pulls plans chronologically (oldest-first), so they can safely run automation while drafting new plans.

## Metadata

- **Tags:** [frontend, UI, UX]
- **Complexity:** 2

## User Review Required

> [!IMPORTANT]
> The informational note will be styled to match the dark-mode aesthetic of Switchboard, using the existing `--accent-teal` accent (the codebase's informational color) with a left-border highlight.
> It will be conditionally displayed only for continuous automation modes (`single-column` and `multi-column`), and hidden when `antigravity-batch` (which is manual and one-shot) is selected.

### Proposed Wording
> 💡 **Safe Planning:** Automation pulls plans from **oldest to newest**. You can safely run automation while planning, as recently created plans are processed last.

## Complexity Audit

### Routine
- Single DOM element creation with inline styling (matches existing warning banner pattern at lines 6820, 7886)
- Conditional visibility based on `currentAutomationMode` variable (already available in scope)
- Insertion into existing container at a well-defined point

### Complex / Risky
- None

## Edge-Case & Dependency Audit

- **Race Conditions:** None. The safety note is a static read-only element. It has no interactive state and does not modify any shared state.
- **Security:** None. No user input is rendered; the text is hardcoded.
- **Side Effects:** None. The element is purely informational and does not trigger any messages or state changes.
- **Dependencies & Conflicts:** The note's visibility depends on `currentAutomationMode`, which is already set before `createAutobanPanel()` runs. Since `renderAutobanPanel()` (line 7394) destroys and rebuilds the entire panel on each render, the note is recreated with the correct visibility automatically — no separate visibility update is needed in the mode-change listener.

## Dependencies

- None

## Adversarial Synthesis

Key risks: (1) The original plan specified "soft blue accent border" but no `--accent-blue` CSS variable exists — using `--accent-teal` (the existing informational accent) maintains consistency. (2) The plan originally proposed a separate `updateSafetyNoteVisibility()` helper called from the mode-change listener, but since `renderAutobanPanel()` destroys and rebuilds the entire panel on mode change, visibility only needs to be set at construction time. Mitigations: Use `--accent-teal` for styling; set visibility inline during element creation based on `currentAutomationMode`.


  - **Do NOT add `guardInteraction()`** to the safety note — it is a read-only element with no interactive controls.
