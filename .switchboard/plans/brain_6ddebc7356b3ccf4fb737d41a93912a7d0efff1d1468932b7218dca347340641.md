# Secure Sandboxed HTML Preview for Planning Panel

## Goal

Resolve the issue where local HTML previews (like [pii-fix-before-after.html](file:///Users/patrickvuleta/Documents/GitHub/patrickwork/designs/pii-fix-before-after.html)) fail to render or execute JavaScript, resulting in a blank white screen.

The root cause is the parent webview's CSP using a dynamic `nonce`, which disables standard `'unsafe-inline'` script execution inside any `srcdoc` iframe that inherits the CSP. Since local HTML scripts lack this dynamic nonce, they are blocked.

To resolve this securely:
1. Remove `allow-same-origin` from the iframe sandbox to run the preview page in an isolated unique-origin context.
2. Inject a custom, local CSP into the HTML preview content that allows `'unsafe-inline'` and `'unsafe-eval'` for local scripts while preventing any access to the parent document or VS Code APIs.

## User Review Required

> [!NOTE]
> By removing `allow-same-origin` from the iframe sandbox, scripts in the HTML preview will run in a `null` origin. This prevents them from calling `window.parent` or using the VS Code API. This is the correct security design for previewing untrusted design files.

> [!NOTE]
> **Known limitation (webviewUri fallback path):** If the backend file read fails, `_handleFetchPreview` falls back to sending only a `webviewUri` (no `htmlContent`). In that fallback, `_injectLocalCsp` is never called, so the blank-screen issue persists for that edge case. This is acceptable since the fallback only fires on file I/O errors; normal operation always reads and sends `htmlContent`.

## Edge-Case & Dependency Audit

**Race Conditions**
- None. The CSP injection is synchronous and happens during file read before the message is posted. No async race conditions.

**Security**
- Without `allow-same-origin`, the sandboxed iframe runs under a `null` origin. Scripts inside cannot call `acquireVsCodeApi()`, cannot read `window.parent`, and cannot access the VS Code extension host. This is the correct security boundary.
- The injected CSP `<meta>` tag allows `'unsafe-inline'` and `'unsafe-eval'` inside the sandboxed null-origin frame only — these permissions do not propagate to the parent webview.
- The parent webview's `frame-src` already includes `about:srcdoc` (confirmed in `planning.html` line 6), so no changes are needed to permit `srcdoc` iframes.

**Side Effects**
- `injectBaseTag` in `planning.js` (line 350) also modifies the `<head>` on the client side after the server delivers `htmlContent`. Sequencing: server runs `_injectLocalCsp` → sends message → client runs `injectBaseTag`. Both inject into `<head>` start. The `<base>` tag (from client) lands before the CSP meta, but browsers apply `<meta>` CSP regardless of position within `<head>`, so this ordering is harmless.
- For HTML fragments without `<head>` or `<html>`, `_injectLocalCsp` prepends the CSP meta, then `injectBaseTag` prepends the `<base>` tag. Final order in document: `<base>` then CSP meta. This is still valid per spec.

**Dependencies & Conflicts**
- `planning.js` does **not** require modification — the fix is entirely server-side in `PlanningPanelProvider.ts` plus the HTML attribute change in `planning.html`.
- The `injectBaseTag` function in `planning.js` (line 350-366) is unaffected and continues to work correctly alongside the new CSP injection.

## Adversarial Synthesis

Key risks: (1) The webviewUri-only fallback path bypasses CSP injection, leaving a residual blank-screen edge case on file I/O failure. (2) The ordering of server-side CSP meta injection vs. client-side `<base>` tag injection is technically benign but should be documented. Mitigations: (1) is documented as a known limitation in User Review Required — fixing it would require a broader refactor of the fallback strategy and is out of scope. (2) The CSP meta applies regardless of `<head>` position, confirmed by the HTML living standard.

### [MODIFY] [PlanningPanelProvider.ts](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/PlanningPanelProvider.ts)

#### Step 1 — Add `_injectLocalCsp` private helper

Add the helper method anywhere inside the `PlanningPanelProvider` class (e.g., after `_getHtml` at line ~676). The method injects a permissive CSP `<meta>` tag into the HTML preview content. Because the iframe sandbox removes `allow-same-origin`, these permissions are isolated to the null-origin frame.

```typescript
private _injectLocalCsp(html: string): string {
    const cspTag = `<meta http-equiv="Content-Security-Policy" content="default-src 'none'; script-src 'unsafe-inline' 'unsafe-eval' https: vscode-webview: vscode-webview-resource: vscode-resource:; style-src 'unsafe-inline' https: vscode-webview: vscode-webview-resource: vscode-resource:; img-src data: blob: https: vscode-webview: vscode-webview-resource: vscode-resource:; font-src data: https: vscode-webview: vscode-webview-resource: vscode-resource:; connect-src https: data:;">`;
    const headMatch = html.match(/<head\b[^>]*>/i);
    if (headMatch && headMatch.index !== undefined) {
        const index = headMatch.index + headMatch[0].length;
        return html.slice(0, index) + '\n  ' + cspTag + html.slice(index);
    }
    const htmlMatch = html.match(/<html\b[^>]*>/i);
    if (htmlMatch && htmlMatch.index !== undefined) {
        const index = htmlMatch.index + htmlMatch[0].length;
        return html.slice(0, index) + '\n<head>' + cspTag + '</head>' + html.slice(index);
    }
    return cspTag + '\n' + html;
}
```

> [!NOTE]
> **Reviewer amendment:** The original plan's CSP snippet was missing `font-src` and `connect-src` directives. Both were added to the implementation. See Code Review Results below.
