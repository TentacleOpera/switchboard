# Kanban Function Bar Buttons Reorganization and Layout Collapse

Move the "Collapse Coders" and "Copy Switchboard Chat" buttons from the top controls strip to the sub-bar below it, positioned after the automation start button and its controls (inline timers, pause, reset, status message). This relieves visual cramping in the top bar.

## Goal

Relocate two icon buttons (`btn-collapse-coders`, `btn-copy-chat-workflow`) from the `controls-strip` div to the `kanban-sub-bar` div to reduce visual clutter in the top bar and group them logically with automation controls.

## Metadata

- **Tags:** [frontend, UI, UX]
- **Complexity:** 2

## User Review Required

> [!NOTE]
> The buttons will automatically inherit the styles of `.kanban-sub-bar .strip-icon-btn`, which removes their borders and makes them transparent unless hovered. This aligns them visually with the style of the other buttons in the sub-bar (like `btn-autoban`). Specifically, `btn-collapse-coders` in its `is-active` state will lose the border highlight it currently gets from `.strip-icon-btn.is-active` in the controls-strip, and will instead use the sub-bar's borderless `.kanban-sub-bar .strip-icon-btn.is-active` style (background glow only, no border change). This is consistent with how `btn-autoban` renders its active state.

## Open Questions

None. All layout requirements and active/inactive behavior states have been fully clarified by the user.

## Complexity Audit

### Routine
- Cut two `<button>` elements from one div and paste them into another in the same HTML file
- CSS inheritance is automatic — `.kanban-sub-bar .strip-icon-btn` already handles all visual states (default, hover, `.is-active`, `.is-off`, `.is-paused`)
- JS event listeners use `getElementById` and are position-independent — no JS changes required

### Complex / Risky
- None

## Edge-Case & Dependency Audit

- **Race Conditions:** None. This is a static DOM repositioning with no async logic.
- **Security:** No impact. No data flows or permissions are affected.
- **Side Effects:** After the move, `btn-delete-project` (a `strip-btn` with 🗑 emoji) will be the last element in the left section of `controls-strip`, adjacent to `btn-cli-triggers`. This is a minor visual adjacency change with no functional impact.
- **Dependencies & Conflicts:** The `#status-message` element has `style="display:none;"` by default. When automation is inactive, it is hidden and the flex layout collapses the gap, so `btn-copy-chat-workflow` and `btn-collapse-coders` will sit directly after `btn-autoban` with only the flex `gap: 10px` spacing. When automation is active, the timers, pause, reset, and status message appear between `btn-autoban` and the moved buttons, pushing them rightward. Both states are the intended behavior per the plan.

## Dependencies

None.

## Adversarial Synthesis

Key risks: The `btn-collapse-coders.is-active` visual state changes from bordered+glow (controls-strip style) to borderless+glow (sub-bar style), which is a minor visual regression for that specific button. Mitigations: This is explicitly acknowledged and accepted — the sub-bar active style is consistent with sibling buttons like `btn-autoban`. No functional behavior is affected; all JS event listeners are ID-based and position-independent.
