# Red Team Findings — Worktree Terminal Creation

## Failure Modes & Risk Analysis

### 1. Registry Lookup State Synchronization
- **Risk**: Reading `this._readTerminalRegistryState` inside the loop for batch state updates could get stale data if writes are pending.
- **Line Reference**: [TaskViewerProvider.ts#L7620](file:///Users/patrickvuleta/Documents/GitHub/worktrees/switchboard/remote-sync-2/src/services/TaskViewerProvider.ts#L7620)
- **Review**: The state is written asynchronously during the `this.updateState` invocation in `_createAutobanTerminal`. Since `updateState` executes sequentially through its own internal lock queue, the values written are flushed to disk/memory. However, fetching `this._readTerminalRegistryState` immediately afterwards could cause issues if the queue has not finished processing.
- **Mitigation**: Rather than querying `_readTerminalRegistryState` over and over, we can use the return values of `_createAutobanTerminal` directly. `_createAutobanTerminal` knows the role and name. We can return both `{ role, suffixedUniqueName }` from `_createAutobanTerminal` directly.

### 2. PID Resolution Delay
- **Risk**: A background PID resolution promise may write to state after the batch updates have completed.
- **Line Reference**: [TaskViewerProvider.ts#L7087](file:///Users/patrickvuleta/Documents/GitHub/worktrees/switchboard/remote-sync-2/src/services/TaskViewerProvider.ts#L7087)
- **Review**: Since PID resolution writes to `state.terminals[suffixedNameForPid]` using `updateState` (which is queued), it will safely preserve the terminal entry created during `_createAutobanTerminal` execution. There is no conflict.

### 3. Disposable Leak on Terminal Closure
- **Risk**: The shell start listener is bound to `vscode.window.onDidStartTerminalShellExecution`. If a terminal is closed before this fires, the listener timeout will handle disposal after 5 seconds, but it would be cleaner to check if the terminal has been closed.
- **Line Reference**: [TaskViewerProvider.ts#L7154](file:///Users/patrickvuleta/Documents/GitHub/worktrees/switchboard/remote-sync-2/src/services/TaskViewerProvider.ts#L7154)
- **Review**: If `terminal.exitStatus` is defined, the terminal is dead. The timeout checks `terminal.exitStatus === undefined` before sending commands. This prevents errors.
