# Plan: Fix Kanban Prompt Dispatch & Contextual Accuracy

## Goal

Fix the Kanban board workflow so that moving plans from any column (including renamed or custom columns) generates the correct agent prompt with accurate contextual labels, eliminating misleading hardcoded text.

## Metadata

- **Tags:** backend, bugfix, workflow, reliability
- **Complexity:** 6

## User Review Required

No. This is a pure bugfix with no product behavior changes.

## Complexity Audit

### Routine
- Add `sourceColumnLabel?: string` to `PromptBuilderOptions` in `agentPromptBuilder.ts`.
- Update the `coder` and `lead` role intro strings to use the dynamic label.
- Pass `sourceColumnLabel` through `_generateBatchPlannerPrompt`, `_generateBatchExecutionPrompt`, and `_generateBatchTesterPrompt` in `KanbanProvider.ts`.

### Complex / Risky
- Refactor `_generatePromptForColumn` role lookup from a hardcoded `columnToPromptRole` ID switch to a `KanbanColumnDefinition.kind`-based dispatch, then route ALL resolved roles through `_generatePromptForDestinationRole` instead of the partial if/else chain. This changes the core path for ALL Kanban prompt generation. An incorrect `kind→role` mapping would misroute prompts for every column.
- Extend `_generatePromptForDestinationRole` to handle non-execution roles (`researcher`, `splitter`, `analyst`, `ticket_updater`, custom agents) that currently fall through to execution. Custom agent roles (`custom_agent_*`) must be handled without calling `buildKanbanBatchPrompt` (which throws for unknown roles).

## Edge-Case & Dependency Audit

- **Race Conditions:** None. Prompt generation is synchronous after async DB reads.
- **Security:** None. No auth or data exposure changes.
- **Side Effects:** ALL Kanban prompt generation paths will use the new dispatch logic. A bug in the `kind→role` mapping would affect every drag-and-drop and button prompt.
- **Dependencies & Conflicts:** `columnToPromptRole` is still used by autoban and other subsystems. Ensure it remains compatible. Check if any tests assert exact prompt strings containing "from PLAN REVIEWED".

## Dependencies

None.

## Adversarial Synthesis

Key risks: (1) The `kind→role` mapping must cover ALL `kind` values including `'custom-user'` and `'custom-agent'`; missing cases re-introduce the exact bug for those column types. (2) Routing source-column dispatch through `_generatePromptForDestinationRole` must handle custom agent roles without crashing `buildKanbanBatchPrompt`. (3) The `PLAN REVIEWED` special-case override (skip `role: 'planner'`, use execution) is a fragile hardcoded exception. Mitigations: exhaustive kind→role switch with explicit fallback comments, custom agent role guard in `_generatePromptForDestinationRole`, document the PLAN REVIEWED override with a TODO for a cleaner abstraction.

## Context & Root Cause

This plan addresses a critical logic error in the Kanban board workflow where moving plans into the "Planned" column triggers the wrong agent role and generates misleading instructions.

The bug manifests when a user moves a plan from an initial column (e.g., "New" or a renamed version like "TEST") into the "Planned" column (`PLAN REVIEWED`). Instead of receiving the expected **Planner** prompt ("Please improve these plans..."), the user receives a **Lead Coder** prompt ("Please execute these plans that are coming from PLAN REVIEWED").

### Why this happens:
1.  **Renamed Column IDs**: The system currently maps columns to roles via a static list of hardcoded IDs (e.g., `CREATED`, `PLAN REVIEWED`). If the source column is a custom column or a renamed built-in, `columnToPromptRole` returns `null`.
2.  **Incorrect Fallback**: When the role is unresolved, `KanbanProvider.ts` defaults to the "Execution" path, triggering the `lead` role.
3.  **Hardcoded Prompt Templates**: The `coder` role template in `agentPromptBuilder.ts` contains hardcoded strings like `"from PLAN REVIEWED"`. This text is misleading when the plans are actually coming from a different column (like "TEST") or are *entering* `PLAN REVIEWED` rather than leaving it.
4.  **Incomplete Source-Column Dispatch**: The if/else chain in `_generatePromptForColumn` (lines 2300–2329) only routes `'planner'`, `'reviewer'`, and `'tester'` roles. All other roles (including `'researcher'`, `'splitter'`, `'analyst'`, `'ticket_updater'`) fall through to `_generateBatchExecutionPrompt`, generating an execution prompt instead of the role-specific prompt. This is a pre-existing bug exposed by the new dispatch logic.

> **Clarification:** Built-in column IDs (`CREATED`, `PLAN REVIEWED`, etc.) remain stable even when labels are renamed. The bug primarily affects custom user columns (which receive new IDs not in `columnToPromptRole`'s switch) and any built-in column not explicitly listed there (e.g., `CONTEXT GATHERER`). The hardcoded "from PLAN REVIEWED" text is misleading regardless of the actual source column.

### 2. `src/services/KanbanProvider.ts`

#### [MODIFY] `_generatePromptForColumn`
- **File:** `src/services/KanbanProvider.ts`
- **Line:** 2273–2330
- Refactor the role lookup logic to use the active board's column definitions, then route ALL roles through `_generatePromptForDestinationRole` (replacing the partial if/else chain).
- **Logic Change**:
  - Build column definitions from the active workspace:
    ```typescript
    const allColumns = this._buildKanbanColumns(
        await this._getCustomAgents(workspaceRoot),
        await this._getCustomKanbanColumns(workspaceRoot)
    );
    ```
  - Find the source column definition:
    ```typescript
    const columnDef = allColumns.find(c => c.id === column);
    ```
  - Extract the display label:
    ```typescript
    const sourceColumnLabel = columnDef?.label || column;
    ```
  - Determine role using `columnDef.role` (with `PLAN REVIEWED` override) or `kind` fallback:
    ```typescript
    let role: string | null = null;
    // PLAN REVIEWED's role ('planner') is for destination dispatch;
    // source dispatch should use execution. See TODO below.
    if (columnDef?.role && column !== 'PLAN REVIEWED') {
        role = columnDef.role;
    }
    if (!role && columnDef) {
        switch (columnDef.kind) {
            case 'created': role = 'planner'; break;
            case 'coded': role = 'reviewer'; break;
            case 'reviewed': role = 'tester'; break;
            case 'review': role = null; break; // execution fallback (RESEARCHER/SPLITTER handled by columnDef.role above)
            case 'custom-user': role = null; break; // custom-user columns have role set via columnDef.role; if missing, fall through
            case 'custom-agent': role = null; break; // custom-agent columns have role set via columnDef.role; if missing, fall through
            case 'gather': role = null; break; // CONTEXT GATHERER has dragDropMode:disabled — not reachable
            case 'completed': role = null; break; // not a source column
        }
    }
    if (!role) {
        role = columnToPromptRole(column); // final fallback for unknowns
    }
    ```
  - **Key change**: Replace the if/else chain (planner/reviewer/tester/execution) with a single call to `_generatePromptForDestinationRole(cards, role, workspaceRoot, sourceColumnLabel)`. This ensures ALL resolved roles (including `researcher`, `splitter`, `analyst`, `ticket_updater`) are routed correctly, not just the 3 previously handled.
  - The destination-column custom-agent check (lines 2280–2292) remains unchanged — it runs first and short-circuits.

> **TODO:** The `column !== 'PLAN REVIEWED'` special case is a hardcoded exception. A cleaner abstraction would be a `sourceRole` field on `KanbanColumnDefinition`, but that is out of scope for this bugfix.

- Extend the final fallback to handle non-execution roles correctly:
  ```typescript
  // Built-in non-execution roles that buildKanbanBatchPrompt supports
  if (role === 'researcher' || role === 'splitter' || role === 'analyst' || role === 'ticket_updater') {
      const repoScopeMap = new Map<string, string>();
      const db = this._getKanbanDb(workspaceRoot);
      if (await db.ensureReady()) {
          for (const card of cards) {
              const plan = await db.getPlanBySessionId(card.sessionId);
              if (plan?.repoScope) repoScopeMap.set(card.sessionId, plan.repoScope);
          }
      }
      const defaultPromptOverrides = await this._getDefaultPromptOverrides(workspaceRoot);
      const promptsConfig = await this._getPromptsConfig(workspaceRoot);
      return buildKanbanBatchPrompt(role, this._cardsToPromptPlans(cards, workspaceRoot, repoScopeMap), {
          defaultPromptOverrides,
          workspaceRoot,
          sourceColumnLabel,
          gitProhibitionEnabled: promptsConfig.gitProhibitionByRole?.[role] ?? true,
          switchboardSafeguardsEnabled: promptsConfig.switchboardSafeguardsByRole?.[role] ?? true
      });
  }
  // Custom agent roles (custom_agent_*) — NOT routed through buildKanbanBatchPrompt
  // which throws for unknown roles. Return a generic plan-file-link prompt.
  if (role?.startsWith('custom_agent_')) {
      const repoScopeMap = new Map<string, string>();
      const db = this._getKanbanDb(workspaceRoot);
      if (await db.ensureReady()) {
          for (const card of cards) {
              const plan = await db.getPlanBySessionId(card.sessionId);
              if (plan?.repoScope) repoScopeMap.set(card.sessionId, plan.repoScope);
          }
      }
      const { planList } = buildPromptDispatchContext(this._cardsToPromptPlans(cards, workspaceRoot, repoScopeMap));
      return `Please process the following plans.\n\nPLANS TO PROCESS:\n${planList}`;
  }
  ```
