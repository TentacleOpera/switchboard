# Configurable Bottom Status Bar Items and Controls

## Goal

Hide the 'agent open: allowed' status bar toggle by default and add a new "Status Bar" tab in the Setup panel to control visibility of that toggle plus five new quick-access status bar buttons (Open Terminals, Clear Terminals, Reset Terminals, Open Kanban, Open Planning Panel).

## Metadata

- **Tags:** [frontend, UI, UX]
- **Complexity:** 5

## User Review Required

> [!IMPORTANT]
> - **Default Off Behavior**: All status bar items (including the existing 'agent open: allowed' toggle) will default to **off** and will not be displayed on the status bar unless manually checked in the new **Status Bar** tab in settings.
> - **Icon Choices**: Using standard VS Code Octicons (`$(terminal)`, `$(trash)`, `$(sync)`, `$(project)`, and `$(note)`) for the new buttons to ensure native look and feel.
> - **Artifacts Button Command**: The "Open Planning Panel" status bar button invokes `switchboard.openPlanningPanel`, the same command used by the existing Quick Planning button in the sidebar (`implementation.html` line 2145). This is a status bar shortcut for the same functionality — no new command needed.

### Complex / Risky
- **Behavioral change to existing status bar item**: The `fileOpeningPreventionStatusBarItem` (line 1672) is currently shown unconditionally via `.show()` on every activation. Changing to conditional visibility controlled by `switchboard.statusBar.showAgentOpenToggle` is a breaking change for existing users who expect the toggle to always be visible.
- **`clearAllTerminals` must match existing sidebar logic**: The sidebar's Clear Terminals button (`implementation.html` line 2083) already guards with `term.role && term.alive` before sending `/clear`. The new status bar command iterates `registeredTerminals` (a `Map<string, vscode.Terminal>`) which doesn't have an `alive` flag — the equivalent guard is `terminal.exitStatus === undefined`. Both paths must achieve the same result: only clear live terminals.
- **Interaction between `showAgentOpenToggle` and `preventAgentFileOpening`**: Two orthogonal settings now control one UI element — visibility vs. state. The `updateStatusBarVisibility()` helper must define clear precedence: `showAgentOpenToggle=false` hides the item regardless of `preventAgentFileOpening` value.

## Edge-Case & Dependency Audit

- **Race Conditions**: `clearAllTerminals` iterates `registeredTerminals` and sends `/clear` to each. The existing sidebar Clear button (`implementation.html` line 2083) guards with `term.role && term.alive`. The status bar command must use the equivalent guard: `terminal.exitStatus === undefined` before each `sendRobustText` call. This ensures both the sidebar and status bar clear buttons behave identically — only clearing live terminals.
- **Security**: No new secrets or auth surfaces. Status bar buttons invoke existing commands that already have their own access controls.
- **Side Effects**: Changing `switchboard.statusBar.showAgentOpenToggle` from `true` to `false` hides the toggle immediately via `onDidChangeConfiguration`. If the user was relying on the toggle to quickly re-enable file opening, they must now go to Setup → Status Bar to re-enable it.
- **Dependencies & Conflicts**: The `fileOpeningPreventionStatusBarItem` variable is module-scoped in `extension.ts` (line 34). The new status bar items must also be module-scoped for access in `onDidChangeConfiguration`. No conflicts with other planned features.

## Dependencies

- None

## Adversarial Synthesis

Key risks: existing users lose the always-visible agent-open toggle (mitigated by one-click re-enable in Status Bar tab), `clearAllTerminals` must match the existing sidebar clear logic by guarding with `exitStatus === undefined` (same behavior, different API surface), and `showAgentOpenToggle`/`preventAgentFileOpening` interaction needs explicit precedence (mitigated by defining visibility as independent of state). All configuration properties must use `scope: "window"` since status bar items are per-window, not per-resource.
