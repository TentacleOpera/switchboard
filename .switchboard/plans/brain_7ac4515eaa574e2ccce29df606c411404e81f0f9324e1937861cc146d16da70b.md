# Switchboard Documentation — Full Readability Audit

## Executive Summary

The docs have **good bones** — the site structure, sidebar nav, Astro layout, and CSS are all solid. The problem is the *content*. It reads like an auto-generated UI inventory: it tells you what buttons exist but never teaches you how to actually use the product. There are zero screenshots in 31 pages for a visual VS Code extension, massive content duplication, stub pages that dead-end readers, broken links on the most critical pages, and no concrete examples of the most important concept in the product (plan files).

## The 7 Root Problems

### 1. Zero screenshots — anywhere

31 pages documenting a VS Code extension with a Kanban board, terminal grid, themes, and inspect mode. Not one screenshot or diagram. For a product whose main value is its *visual control surface*, this is the single biggest readability failure.

> [!CAUTION]
> The theme page describes two themes ("Afterburner" and "Claudify") without showing either one. The Kanban board page describes 11 columns, card controls, and a toolbar — without ever showing the board.

### 4. Massive content duplication

The same 200-word block appears in 2-3 places:

| Topic | Duplicated across |
|-------|-------------------|
| Features (create, modes, worktrees) | features-worktrees.md ↔ project-panel.md |
| NotebookLM workflow | planning-artifacts.md ↔ quota-optimization.md |
| Project Manager | project-manager.md ↔ implementation-sidebar.md ↔ claude-desktop.md |
| Slash commands | settings-commands.md ↔ claude-desktop.md ↔ quick-start.md |
| Feature-scoped overrides | features-worktrees.md ↔ prompts.md |

**Impact:** Duplicated content drifts out of sync and forces users to wonder which version is canonical.

### 5. Tautological descriptions — "X does X"

Pages describe UI labels by restating the label:
- "Git Branch Strategy: How the agent handles git branches"
- "Design briefs for guiding design work"
- "Research tab: Browse local folder research outputs"
- "Settings export: writes settings to a file"

This tells the user **nothing new**. It's the documentation equivalent of a variable named `data`.

### 6. Describes WHAT exists but never teaches HOW

Most pages are structured as: *"Here are the buttons. Here is what each button label says."* Very few pages walk through a complete task:

- **No page shows what a plan file looks like** — the atomic unit of the entire product
- **No CLI command examples** for any provider (Claude, Gemini, Cursor, etc.)
- **No before/after of a prompt** showing how add-ons modify it
- **No constitution example** — what does a good rule look like?
- **No worked example** of the full plan→code→review→complete cycle

### 7. Unexplained jargon

Terms used without definition on first occurrence:
- **Host-agnostic** — used 4+ times, never defined
- **Control Plane** — appears in setup-panel, multi-repo, architecture — never explained
- **Band A / Band B** — complexity classification, defined only in agent-roles.md's last 3 lines
- **WIP 1** — appears in project-manager.md, never defined
- **Tick** — used in automation.md for the orchestration engine, never defined
- **Scaffolds** — setup-panel.md uses this dev jargon for end users

## Prioritized Action Plan

### 🚨 P0 — Fix immediately (broken user experience)

- [ ] **Fix all broken links** — quick-start.md (3 links), design-panel.md (next href), implementation-sidebar.md (prev href + next label mismatch)
- [ ] **Merge or delete stub pages** — UAT and Comms should either be expanded to 200+ words or merged into kanban-board/index.md as subsections
- [ ] **Expand agents-tab page** — Add CLI command examples for Claude Code, Gemini, Antigravity, Cursor. Explain what visibility toggles, limit-dispatch, and terminal count actually do

### 🟡 P1 — High-impact readability improvements

- [ ] **Add screenshots** — At minimum: the Kanban board (with labeled columns), the Setup Panel, the Implementation Sidebar, both themes (Afterburner vs Claudify), and Inspect Mode. This alone transforms the docs.
- [ ] **Create a "Plan File Anatomy" page** — Show what a real plan file looks like with annotations. This is the single most important missing concept page.
- [ ] **Eliminate duplication** — Pick ONE canonical source for Features, Project Manager, NotebookLM, slash commands. Other pages get 1-sentence summaries with links.
- [ ] **Add a Glossary page** — Define Host, Control Plane, Band A/B, WIP, Plan, Feature, Worktree, Constitution, PRD, Targeted Pass, Orchestration, Tick

### 🟢 P2 — Content quality improvements

- [ ] **Convert button inventories → task guides** — Restructure kanban-board, settings-commands, and prompts around user tasks ("Set up your first automation", "Configure pair programming", "Import from ClickUp")
- [ ] **Group the settings table** — Split the 65-row settings-commands.md table into logical sections with headers
- [ ] **Add code/config examples everywhere** — CLI startup commands per provider, example constitution rules, example prompt with add-ons applied, example JSON response from Local API
- [ ] **Replace tautological descriptions** — Every UI description should answer "why would I use this?" not "what does the label say?"
- [ ] **Remove pre-release docs** — The Automated Triage Pipeline section in troubleshooting.md documents a hidden feature. Ship it or remove the docs.
- [ ] **Add an architecture diagram** — The architecture page lists class names but never shows how data flows between panels, DB, file watchers, and terminals
