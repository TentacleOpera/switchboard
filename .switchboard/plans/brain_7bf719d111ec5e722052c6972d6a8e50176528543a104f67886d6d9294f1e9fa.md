# Resolve Stale MCP Server State

The Kanban misclassification bug persists because running MCP server processes are locking the workspace-local runtime files, preventing updates from the latest source code.

## Investigation Results (The "Why")

Detailed investigation revealed a multi-layered failure in the MCP server's update and synchronization lifecycle:

### 1. The Stale Logic (Concrete Divergence)
- **Source Code**: [src/mcp-server/register-tools.js](file:///c:/Users/patvu/Documents/GitHub/switchboard/src/mcp-server/register-tools.js) correctly imports [deriveKanbanColumn](file:///c:/Users/patvu/Documents/GitHub/switchboard/src/services/kanbanColumnDerivation.js#10-53) from `../services/kanbanColumnDerivation`. This modern version includes the `challenge` workflow signal.
- **Runtime State**: The file currently in [.switchboard/MCP/register-tools.js](file:///c:/Users/patvu/Documents/GitHub/switchboard/.switchboard/MCP/register-tools.js) is a **stale version** from over 24 hours ago. It contains an **inline** [deriveColumn](file:///c:/Users/patvu/Documents/GitHub/switchboard/.switchboard/MCP/register-tools.js#1920-1930) function that is missing the `challenge` signal, causing code-reviewed plans to be misclassified as `CREATED`.

### 2. The Process Lock (Silent Failure)
- There are multiple orphan `node` processes (e.g., PIDs **572**, **43996**) running the MCP server directly from the `.switchboard/MCP/` directory.
- These processes, likely spawned by external CLI agents (Copilot/Qwen), create a **file lock** on `register-tools.js`.
- When the VS Code extension activates, it attempts to "repair" the runtime via `ensureWorkspaceMcpServerFiles`, but because the file is locked, the overwrite fails or is skipped, and the extension proceeds with the stale version.

### 3. Deployment Logic Flaws
- `ensureWorkspaceMcpServerFiles` performs a "surgical" copy of the `mcp-server` folder but **does not copy the `src/services` folder**. 
- This means that even if a developer modifies `src/mcp-server` and the copy succeeds, the server will fail at runtime because the relative import `../services/kanbanColumnDerivation` points to a non-existent path in the `.switchboard/MCP/` sandbox. 
- This confirms that the MCP server **must** be bundled (`npm run compile`) for external service imports to work in the current architecture.

### 4. Cleanup Gaps
- The `cleanWorkspace` routine (run on extension activation) wipes `inbox`, `outbox`, and `cooldowns`, but **ignores the `MCP` directory**. This allows stale artifacts to survive "Clean Workspace" commands.

### Related Symptom: Sidebar Status Lights Going Yellow

The sidebar MCP status indicator periodically drops from green to yellow. This is a downstream symptom of the same stale-runtime bug. The mechanism:

1. **`checkMcpConnection()`** (extension.ts:2695) checks whether `mcp-server.js` exists on disk and whether the `mcpServers` VS Code config has an active `switchboard` entry. The sidebar shows green when both pass, yellow when either fails.
2. **If the stale MCP runtime crashes** (e.g. because it hits a `MODULE_NOT_FOUND` for `../services/kanbanColumnDerivation`), the child process exits. The extension sets `mcpServerProcess = null`. The next health check finds no process → yellow.
3. **Config read contention**: `vscode.workspace.getConfiguration().get('mcpServers')` can throw if the config file is being written to simultaneously (e.g. by an agent updating settings). This sets `configReadFailed = true` → yellow.
4. **External CLI agents** (Copilot, Qwen, etc.) launch their own MCP servers from `.switchboard/MCP/`. If the stale runtime crashes for them, those agents lose MCP connectivity — but the sidebar light only tracks the extension-spawned process.

**Fixing Steps 1-3 of this plan will resolve the yellow-light flickering**, because stable runtime files prevent crash-driven process exits, and `TRANSIENT_DIRS` cleanup prevents stale artifacts from accumulating.

## Proposed Changes

### Step 1 (Critical): Fix the Deployment Copy to Include Dependencies

#### [MODIFY] [extension.ts](file:///c:/Users/patvu/Documents/GitHub/switchboard/src/extension.ts)
- In `ensureWorkspaceMcpServerFiles()` (line 91-107), after copying `mcp-server/` to `.switchboard/MCP/`, **also copy `src/services/kanbanColumnDerivation.js`** to `.switchboard/services/kanbanColumnDerivation.js` so the `require('../services/kanbanColumnDerivation')` resolves correctly in the sandbox.
- **Alternative (preferred):** Instead of copying extra files, modify the deployment to write a **bundled** version. Since `npm run compile` already produces a webpack bundle of the MCP server with all dependencies inlined, copy from `dist/mcp-server/` instead of `src/mcp-server/`. This eliminates the cross-directory import problem entirely.

### Step 2: Add `MCP` to Clean Workspace Transient Dirs

#### [MODIFY] [cleanWorkspace.ts](file:///c:/Users/patvu/Documents/GitHub/switchboard/src/lifecycle/cleanWorkspace.ts)
- Change `TRANSIENT_DIRS` from `['inbox', 'outbox', 'cooldowns']` to `['inbox', 'outbox', 'cooldowns', 'MCP']`.
- This forces a full re-copy of the MCP server runtime on every workspace clean, eliminating stale file persistence.

### Step 3: Add Lock-Release Warning in Deployment

#### [MODIFY] [extension.ts](file:///c:/Users/patvu/Documents/GitHub/switchboard/src/extension.ts)
- In `ensureWorkspaceMcpServerFiles()`, wrap the `copyDirectoryRecursive` call in a try-catch. If `EBUSY` or `EPERM`, emit a high-priority warning to the Switchboard output channel: `"[MCP] WARNING: Could not update MCP runtime files (file locked by external process). Kill orphan node processes and restart."`.

### Step 4: Kill Orphans on Activation (Optional Enhancement)

#### [MODIFY] [extension.ts](file:///c:/Users/patvu/Documents/GitHub/switchboard/src/extension.ts)
- Before calling `ensureWorkspaceMcpServerFiles()`, scan for node processes whose CWD matches `.switchboard/MCP` and kill them. This is aggressive and should be gated behind a setting or user confirmation.

## Adversarial Review

### 🔴 Grumpy Critique

1. **CRIT: The "preferred" alternative breaks workspace mode.** The setting `switchboard.runtime.workspaceMode` exists specifically so developers can run the MCP server from mutable source files instead of the immutable bundle. Switching the deployment to copy from `dist/` kills this workflow. You MUST check `isWorkspaceRuntimeModeEnabled()` before choosing the source directory.

2. **MAJOR: Killing orphan processes is platform-fragile.** On Windows you'd use `wmic` or `tasklist`, on macOS/Linux `lsof` or `ps`. The code will be ugly and unreliable. This should be a manual step, not automatic.

3. **MINOR: Adding MCP to TRANSIENT_DIRS means every "Clean Workspace" nukes the runtime.** This forces a re-copy on next activation. For users with slow disks or large MCP runtimes, this adds startup latency.

### 🟢 Balanced Synthesis

1. **Agreed on CRIT #1.** The deployment function should check `isWorkspaceRuntimeModeEnabled()`. If true, copy from `src/mcp-server/` AND `src/services/` (both directories). If false (default), copy from `dist/mcp-server/` (bundled). Updated the plan.

2. **Agreed on MAJOR #2.** Remove Step 4 (orphan killing) from the implementation plan. It should remain a manual cleanup step documented in the plan's verification section.

3. **Disagree on MINOR #3.** The MCP runtime directory is ~130KB. Re-copy takes <100ms. The alternative — allowing stale files to persist across "Clean Workspace" commands — is the root cause of this entire bug.

### Challenge Review Action Plan

1. **[REQUIRED]** Gate deployment source on `isWorkspaceRuntimeModeEnabled()`: workspace mode → copy `src/mcp-server/` + `src/services/kanbanColumnDerivation.js`; default → copy `dist/mcp-server/`.
2. **[REQUIRED]** Add `'MCP'` to `TRANSIENT_DIRS` in `cleanWorkspace.ts`.
3. **[REQUIRED]** Add EBUSY/EPERM handling in `ensureWorkspaceMcpServerFiles()`.
4. **[REMOVED]** Strike orphan-killing automation (Step 4). Document manual cleanup instead.
