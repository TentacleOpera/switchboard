# Filter Kanban Plans Columns Dropdown by Active Columns

In `planning.html` (the Kanban Plans tab), the columns filter dropdown currently displays all possible columns (built-in and custom). This includes columns corresponding to inactive/disabled agents, which adds unnecessary noise to the dropdown filter.

This change filters the column definitions sent to the `planning.html` webview so that only active columns (matching the user's configuration in the setup/agents tab) or columns that currently contain plans (occupied columns) are listed in the dropdown.

## Goal

Filter the Planning Panel's Kanban Plans column dropdown to exclude columns for inactive agents (unless those columns contain plans), mirroring the existing `_filterDynamicColumns` behavior in `KanbanProvider`.

## Metadata

- **Tags:** [frontend, UI]
- **Complexity:** 3

## User Review Required

> [!NOTE]
> The dynamic column filtering mirrors the Kanban board's behavior. If an agent is inactive but there are plans in that column, the column will remain visible in the dropdown so the user can filter those plans.

## Complexity Audit

### Routine
- Reading `visibleAgents` from the same `state.json` already parsed in `_getKanbanColumnDefinitions`
- Applying a 3-condition filter matching the existing `_filterDynamicColumns` pattern in `KanbanProvider`
- Passing the workspace-specific plans array to the method at the existing call site

### Complex / Risky
- None

## Edge-Case & Dependency Audit

- **Race Conditions:** None — `_getKanbanColumnDefinitions` is called within the `fetchKanbanPlans` handler which already has a request-ID guard (line 1320). The state.json read is synchronous-within-async and single-threaded in the VS Code extension host.
- **Security:** No concerns — `state.json` is a local workspace file already read by this method.
- **Side Effects:** None — the filter only removes columns from the dropdown; it does not modify any persisted state or affect the main Kanban board.
- **Dependencies & Conflicts:** Relies on `KanbanColumnDefinition.hideWhenNoAgent` (already defined in `agentConfig.ts` line 68) and `KanbanPlanSummary.column` (line 34). No new dependencies introduced.

## Dependencies

- None

## Adversarial Synthesis

Key risks: multi-workspace column merging may show a column from one workspace that is hidden in another; `visibleAgents` may be absent from `state.json` in fresh workspaces. Mitigations: per-workspace filtering naturally handles the merge (a column visible in any workspace survives dedup); the `!== false` check treats missing roles as visible, so absent `visibleAgents` defaults to showing all columns.

#### [MODIFY] [PlanningPanelProvider.ts](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/PlanningPanelProvider.ts)

**Context:** `_getKanbanColumnDefinitions` (lines 3179–3196) currently reads `state.json` for `customAgents` and `customKanbanColumns`, calls `buildKanbanColumns()`, and returns all columns unfiltered. The `fetchKanbanPlans` handler (lines 1317–1387) calls this method per workspace root and merges the results into `mergedColumns`, which is sent to the webview as `msg.columns` — the source for the column filter dropdown.

**Logic:** Extend `_getKanbanColumnDefinitions` to also read `visibleAgents` from the already-parsed `state.json` and apply the same 3-condition filter used by `KanbanProvider._filterDynamicColumns` (lines 2158–2170).

**Implementation:**

1. **Modify `_getKanbanColumnDefinitions` signature** (line 3179):
   - Change from `(workspaceRoot: string)` to `(workspaceRoot: string, plans?: KanbanPlanSummary[])`
   - The `plans` parameter carries the workspace-specific plans so we can check for occupied columns.

2. **Read `visibleAgents` from state.json** (inside the existing try block, after line 3191):
   ```typescript
   const visibleAgents: Record<string, boolean> = state.visibleAgents || {};
   ```
   - Clarification: No need to import `parseCustomAgents` or build a defaults map. The filter uses `!== false`, so roles absent from `visibleAgents` are treated as visible (matching the KanbanProvider pattern). Custom agents not in the map pass the filter naturally.

3. **Apply the filter after `buildKanbanColumns()`** (after line 3195, before return):
   ```typescript
   const allColumns = buildKanbanColumns(customAgents, customKanbanColumns);
   if (!plans || plans.length === 0) {
       return allColumns.filter(col => {
           if (!col.hideWhenNoAgent) return true;
           if (col.role && visibleAgents[col.role] !== false) return true;
           return false;
       });
   }
   const occupiedColumns = new Set(plans.map(p => p.column));
   return allColumns.filter(col => {
       if (!col.hideWhenNoAgent) return true;
       if (col.role && visibleAgents[col.role] !== false) return true;
       if (occupiedColumns.has(col.id)) return true;
       return false;
   });
   ```
   - This mirrors `KanbanProvider._filterDynamicColumns` (lines 2164–2169) exactly: keep columns that are not `hideWhenNoAgent`, have an active role, or are occupied by plans.

4. **Update the call site** in `fetchKanbanPlans` handler (line 1361):
   - Change from:
     ```typescript
     const colDefs = await this._getKanbanColumnDefinitions(root);
     ```
   - To:
     ```typescript
     const colDefs = await this._getKanbanColumnDefinitions(root, plans);
     ```
   - `plans` is already available in scope (line 1335: `const plans = await this._getKanbanPlans(root)`).

**Edge Cases:**
- **No `visibleAgents` in state.json:** `state.visibleAgents` is `undefined`, so `visibleAgents` defaults to `{}`. The `!== false` check treats all roles as visible — correct behavior for a fresh workspace.
- **No `hideWhenNoAgent` on column:** Columns without `hideWhenNoAgent` (e.g. "New", "Planned", "Completed", custom-user columns) always pass the first filter condition and are always shown.
- **Multi-workspace merge:** Each workspace root is filtered independently. If workspace A hides "Intern" but workspace B shows it, the column survives the dedup merge (line 1363: `seenColumnIds` check). This is correct — the dropdown should show a column if it's relevant in any workspace.
- **Plans with unknown column IDs:** If a plan references a column ID not in the built definitions, `occupiedColumns.has(col.id)` simply won't match for that column. No error.

### Remaining Risks

- **Default drift:** If `KanbanProvider._getVisibleAgents` adds or changes built-in defaults, the copy in `PlanningPanelProvider` must be updated in sync. A future refactoring could extract the defaults map to a shared constant in `agentConfig.ts`.
