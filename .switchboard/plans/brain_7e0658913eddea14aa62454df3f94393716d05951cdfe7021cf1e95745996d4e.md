# Fix Local Folder Configuration Saving and Synchronization Mismatch

## Goal

Resolve issues preventing the Switchboard Planning panel from successfully saving and loading local folder configurations, by adding a legacy-setting fallback for reads, redirecting writes to the correct configuration target, and normalizing path comparisons.

## Metadata

- **Tags:** [bugfix, frontend, reliability]
- **Complexity:** 4

## User Review Required

> [!NOTE]
> This change is backward-compatible and self-contained. It ensures existing single-path configurations (`localFolderPath`) are loaded as fallbacks, and redirects setting updates to the folder settings level (`.vscode/settings.json`) if available.

## Open Questions

None.

## Complexity Audit

### Routine
- Adding `_getConfigurationTarget()` helper method to `LocalFolderService` — follows existing `vscode.workspace.getWorkspaceFolder()` pattern
- Replacing `vscode.ConfigurationTarget.Workspace` with `this._getConfigurationTarget()` in four methods — mechanical substitution
- Adding legacy `localFolderPath` fallback in `getFolderPaths()` — single `if` guard after existing array read
- Normalizing `root === activeRoot` comparisons with `path.resolve()` — two identical substitutions

### Complex / Risky
- Legacy setting divergence: if `localFolderPath` (singular) is always merged into the returned array, removing a folder via UI (which updates the array) won't remove it because the fallback re-adds it. **Mitigation**: fallback only applies when `localFolderPaths` is empty/undefined, so once the user adds/removes any folder, the array is non-empty and the fallback is skipped.
- `path.resolve()` does NOT normalize case on case-insensitive filesystems (macOS, Windows). If `root` and `activeRoot` differ only in casing, the comparison will still fail. This is a marginal edge case — the most common real-world mismatches (trailing slashes, relative paths) are handled by `path.resolve()`.

## Edge-Case & Dependency Audit

- **Race Conditions**: None. Configuration reads and writes are synchronous VS Code API calls; the debounce in `_sendHtmlDocsReady()` already guards against rapid re-reads.
- **Security**: No new attack surface. The legacy fallback reads a user-owned setting; the configuration target change doesn't alter which values are written, only where they're stored.
- **Side Effects**: Writing to `ConfigurationTarget.WorkspaceFolder` instead of `Workspace` changes which `.vscode/settings.json` file is updated (folder-level vs. workspace-level). In single-folder workspaces, these are the same file. In multi-root workspaces, the folder-level file is more specific and correct for `scope: "resource"` settings.
- **Dependencies & Conflicts**: The `switchboard.research.localFolderPath` (singular) key is NOT registered in `package.json`'s `contributes.configuration`. VS Code still returns values for unregistered keys stored in `.vscode/settings.json`, but this is permissive behavior. No other code in `src/` (excluding `.bak` files and test mocks) references the singular key. The `planning.js` webview has a `localFolderPathUpdated` message handler — this is unrelated (it handles the webview-side state, not the VS Code setting).

## Dependencies

None.

## Adversarial Synthesis

Key risks: legacy-setting divergence (mitigated by empty-only fallback), case-insensitive path comparison (marginal, documented limitation), and unregistered-setting fragility (low probability of VS Code behavior change). Mitigations are built into the fallback scope and the plan avoids over-engineering by not adding case normalization I/O overhead.

#### [MODIFY] [LocalFolderService.ts](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/LocalFolderService.ts)

1. **Add singular configuration fallback** (lines 67-74):
   Modify `getFolderPaths()` to check the legacy string setting `research.localFolderPath` if `research.localFolderPaths` is empty or undefined. The fallback only applies when the array is empty, preventing divergence once the user interacts with the UI.

   ```typescript
   getFolderPaths(): string[] {
       const config = vscode.workspace.getConfiguration('switchboard', vscode.Uri.file(this._workspaceRoot));
       const paths = config.get<string[]>('research.localFolderPaths', []);

       // Legacy fallback: read singular setting if array is empty
       if (paths.length === 0) {
           const legacyPath = config.get<string>('research.localFolderPath', '');
           if (legacyPath) {
               paths.push(legacyPath);
           }
       }

       const seen = new Set<string>();
       return paths
           .map(p => this.resolveFolderPath(p))
           .filter(p => p && !seen.has(p) && seen.add(p) as unknown as boolean);
   }
   ```

   Note: No equivalent legacy fallback is needed for `getHtmlFolderPaths()` — no singular `htmlFolderPath` setting exists in the codebase or `.vscode/settings.json`.

2. **Add configuration target helper** (insert after constructor, ~line 29):
   Add a private method that determines the correct configuration target based on whether the workspace root corresponds to a workspace folder.

   ```typescript
   private _getConfigurationTarget(): vscode.ConfigurationTarget {
       const folder = vscode.workspace.getWorkspaceFolder(vscode.Uri.file(this._workspaceRoot));
       return folder ? vscode.ConfigurationTarget.WorkspaceFolder : vscode.ConfigurationTarget.Workspace;
   }
   ```

3. **Target folder-scoped updates** (lines 81-100, 225-244):
   Replace `vscode.ConfigurationTarget.Workspace` with `this._getConfigurationTarget()` in four methods:
   - `addFolderPath()` — line 89
   - `removeFolderPath()` — line 99
   - `addHtmlFolderPath()` — line 233
   - `removeHtmlFolderPath()` — line 243

   Each replacement is a single-argument change:
   ```typescript
   // Before:
   await config.update('research.localFolderPaths', newPaths, vscode.ConfigurationTarget.Workspace);
   // After:
   await config.update('research.localFolderPaths', newPaths, this._getConfigurationTarget());
   ```

#### [MODIFY] [PlanningPanelProvider.ts](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/PlanningPanelProvider.ts)

1. **Fix path comparison mismatch** (lines 1703, 1796):
   In `_sendLocalDocsReady()` and `_sendHtmlDocsReady()`, compare `root` and `activeRoot` by resolving them to their canonical paths using `path.resolve()` to avoid mismatches caused by trailing slashes or relative path differences.

   **`_sendLocalDocsReady()` — line 1703:**
   Change:
   ```typescript
   if (root === activeRoot || !activeRoot) {
   ```
   To:
   ```typescript
   if (!activeRoot || path.resolve(root) === path.resolve(activeRoot)) {
   ```

   **`_sendHtmlDocsReady()` — line 1796:**
   Change:
   ```typescript
   if (root === activeRoot || !activeRoot) {
   ```
   To:
   ```typescript
   if (!activeRoot || path.resolve(root) === path.resolve(activeRoot)) {
   ```

   Note: `path.resolve()` normalizes trailing slashes and relative paths but does NOT normalize case on case-insensitive filesystems. The `_resolveWorkspacePath()` method at line 214 also uses `root === activeRoot` but serves a different purpose (file resolution skip logic) and is out of scope for this fix.
