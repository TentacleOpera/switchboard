# HTML Previews Tab UI and UX Enhancements

## Goal

The HTML Previews tab currently lacks key usability features:
1. When first opened, the HTML preview area is a blank white space (due to the empty iframe being visible with its default background), which looks like a blank page and provides no information to the user.
2. When loading larger files, there is no loading indicator inside the preview pane. The only visual change is the small status text in the top bar, which is easy to miss and makes the interface feel unresponsive.
3. The folder management button is placed far to the right in the top controls strip, whereas in other tabs like Local Docs, it is placed in the sidebar header as a "Manage Folders" button.

**Root cause analysis:**
- The iframe at `planning.html:2295` has `background: white` inline style, causing a jarring white flash on the dark-themed UI when no content is loaded.
- `loadDocumentPreview` for `html-folder` (planning.js:636-662) only updates the `status-html` text element — it never shows any in-pane visual feedback.
- The `btn-manage-html-folders` button lives in the controls-strip (planning.html:2285) with `style="margin-left: auto;"`, inconsistent with the Local Docs sidebar pattern where `sidebar-folders-btn` is placed inside the `sidebar-toggle-row` (planning.js:1008-1022).

This plan details the changes to:
- Introduce a beautiful dark-themed placeholder view as the initial state for the HTML Previews tab.
- Add a loading spinner inside the preview pane to provide clear feedback when loading files.
- Move the folders button into the HTML Previews sidebar toggle row to match the Local Docs sidebar design.

## Complexity Audit

### Routine
- Adding CSS `@keyframes spin` animation for the loading spinner
- Adding `#html-initial-state` and `#html-loading-state` HTML containers to the preview pane
- Changing iframe `background: white` to `background: var(--panel-bg)` to prevent white flash
- Hiding/showing initial/loading/preview states via `style.display` toggles in JS
- Removing the `btn-manage-html-folders` button from the controls-strip in HTML
- Adding a `sidebar-folders-btn` to the `sidebar-toggle-row` in `renderHtmlDocs` (mirroring Local Docs pattern at planning.js:1008-1022)

### Complex / Risky
- Dangling event listener: The init-time `btn-manage-html-folders` click listener (planning.js:3447-3456) references a DOM element that will be removed from the controls-strip. Must remove or guard this listener to prevent silent failure. The dynamically-created button in `renderHtmlDocs` must attach its own click listener at creation time.
- Loading spinner must respect the `previewRequestId` guard: only hide the spinner when the returned `requestId` matches `state.previewRequestId`, to avoid hiding the spinner prematurely on rapid file clicks (race condition on fast successive selections).

## Edge-Case & Dependency Audit

- **Race Conditions:** Rapid file clicks increment `previewRequestId` (planning.js:649). The existing guard at `handlePreviewReady` (line 1443) checks `requestId !== state.previewRequestId`. The loading spinner show/hide must participate in this same guard — only hide spinner when requestId matches. For `handlePreviewError` (line 1575), the same guard exists.
- **Security:** Error messages rendered in the iframe via `srcdoc` template literals must escape HTML (existing code at line 1587 uses `error.replace(/</g, '&lt;')`). New error display in the initial/loading states must follow the same pattern.
- **Side Effects:** Removing `btn-manage-html-folders` from the controls-strip means the init-time `getElementById` at line 3447 will return `null`. The existing `if (btnManageHtmlFolders)` guard handles this gracefully, but the listener attachment code should be removed entirely to avoid dead code.
- **Dependencies & Conflicts:** The `renderHtmlDocs` function (line 885) replaces `treePaneHtml.innerHTML` entirely (line 890), destroying all child elements and their listeners. The Manage Folders button must be recreated with a fresh listener each time `renderHtmlDocs` runs — matching the Local Docs pattern at lines 1008-1022.

## Dependencies
- None (self-contained UI changes)

## Adversarial Synthesis

Key risks: Dangling init-time event listener for the removed Folders button causing silent failure; white iframe background flashing through dark-themed overlay containers. Mitigations: Remove the init-time listener code for `btn-manage-html-folders` and attach listeners at creation time in `renderHtmlDocs`; change iframe default background to `var(--panel-bg)` and use z-indexed overlay containers for initial/loading states.
