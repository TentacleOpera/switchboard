# Sync Planner Metadata Updates to Kanban Database

Ensure that when a planner agent writes metadata updates (complexity, tags, dependencies, and topic) to a mirror plan file (e.g., `.switchboard/plans/brain_*.md` or `ingested_*.md`), the updates are immediately parsed and synced to the Kanban database, and the Kanban board UI is refreshed.

Also, refine metadata regex patterns to handle case/formatting variations (e.g. `**Complexity**: 5` vs `**Complexity:** 5`).

## Goal

Sync planner-agent metadata changes from mirror plan files into the Kanban SQLite database and refresh the board UI, so that complexity, tags, dependencies, and topic updates appear immediately without manual reload.

## Metadata

- **Tags:** backend, database, workflow, reliability
- **Complexity:** 5

## User Review Required

- Confirm that using granular `update*ByPlanFile` DB methods (instead of `upsertPlans`) is acceptable — this avoids overwriting existing DB fields like `routedTo`, `dispatchedAgent`, `clickupTaskId`, etc.
- Confirm that silent no-op on non-existent DB records (brand-new plans not yet registered) is acceptable behavior.

## Complexity Audit

### Routine
- Regex pattern updates in `planMetadataUtils.ts` (4 patterns, mechanical substitution)
- Case-insensitive path comparison in `KanbanProvider.refreshIfShowing` (2-line change)
- Calling existing granular DB update methods (`updateComplexityByPlanFile`, `updateTagsByPlanFile`, `updateDependenciesByPlanFile`, `updateTopicByPlanFile`)

### Complex / Risky
- Inserting metadata sync into the staging watcher callback without disrupting the existing mirror→brain / mirror→source sync flow — wrong insertion point would silently break the feature or the existing sync
- Ensuring both `isBrainMirror` and `isIngestedMirror` branches receive the metadata sync, with correct `relativeMirror` path computation for each branch

## Edge-Case & Dependency Audit

- **Race Conditions:** The staging watcher debounces at 500ms. Metadata sync runs after the debounce fires and the mirror→source sync completes. Since the DB update methods are synchronous (better-sqlite3), there is no TOCTOU risk between read and write. However, if two mirror edits arrive within the 500ms window, the second debounce callback will re-parse and re-update, which is idempotent and safe.
- **Security:** No user-controlled input reaches SQL directly — all values pass through `parsePlanMetadata` (regex-extracted strings) and into parameterized DB methods. No injection risk.
- **Side Effects:** If `parsePlanMetadata` throws on malformed content, the entire staging watcher callback could fail. Must wrap metadata sync in try/catch to protect the existing mirror→source sync flow.
- **Dependencies & Conflicts:** The `_getWorkspaceIdForRoot(workspaceRoot)` method must be called to obtain the `workspaceId` required by all `*ByPlanFile` DB methods. This method is already available in `TaskViewerProvider` and uses a cached value when possible.

## Dependencies

- None

## Adversarial Synthesis

Key risks: inserting metadata sync at the wrong point in the staging watcher callback (before the return in the brain branch, or after the source write in the ingested branch), and metadata sync failures disrupting the existing mirror→source sync. Mitigations: use granular `update*ByPlanFile` methods instead of `upsertPlans` to avoid field overwrite, wrap all metadata sync in try/catch with console.warn, and specify exact line-number insertion points.
