# Move Configuration Components to Central Setup Panel

## Goal
Streamline the Switchboard sidebar by relocating configuration-heavy accordion components (Setup, Database Operations, Custom Agents, and Default Prompt Overrides) into a central editor webview panel. The sidebar will retain only the "Live Feed" and "Terminal Operations" accordions, providing a cleaner, more focused operational view.

## User Review Required
> [!NOTE]
> Moving these components requires shifting their associated JavaScript event listeners and message-passing architecture from `TaskViewerProvider` to the new or updated central setup panel. Users will now click an "Open Setup" button in the sidebar to configure Custom Agents, Prompts, and Database paths.

## Complexity Audit

### Routine
- Removing HTML sections for `startup-section` (Setup), `db-sync-fields` (Database Operations), `custom-agent-list` (Custom Agents), and `default-prompt-override-summary` from `src/webview/implementation.html`.
- Moving the corresponding UI configuration styles and JavaScript event listeners (e.g., `btn-add-custom-agent`, `btn-customize-default-prompts`, `db-reset-btn`) to the central setup panel's HTML/JS.
- Adding a button in the `implementation.html` Terminal Operations section to trigger the central setup view.

### Complex / Risky
- **Message Handling Migration:** `TaskViewerProvider.ts` currently handles messages like `saveStartupCommands`, `saveDefaultPromptOverrides`, and `createAgentGrid` from the sidebar. Splitting the UI means the central panel must either share this message handler or have its own `SetupPanelProvider` that updates `state.json` and notifies the sidebar to refresh.
- **State Synchronization:** If the user updates "Custom Agents" in the central panel, the sidebar's Agent List must re-render immediately to reflect the new agents. 

## Edge-Case & Dependency Audit
- **Race Conditions:** If the user interacts with the sidebar (e.g., triggering an agent) while simultaneously saving configuration changes in the central setup panel, the sidebar might read a partially updated `state.json`.
- **Security:** No new security risks; existing validation for `state.json` and command inputs applies.
- **Side Effects:** The `implementation.html` script relies heavily on reading DOM elements to save state (e.g., `document.querySelectorAll('.agent-visible-toggle')`). Relocating these elements means the sidebar can no longer save them directly; state must be managed by the central panel.
- **Dependencies & Conflicts:** 
  - Cannot query active Kanban plans directly as `get_kanban_state` is unavailable in this cognitive step, but no known active plans inherently conflict with extracting `implementation.html` markup.
  - Dependencies include `agentConfig.ts` (for parsing overrides/custom agents) and the `TaskViewerProvider` lifecycle.

## Proposed Changes

### 1. Extract Configuration HTML from Sidebar
**File:** `src/webview/implementation.html`
- Delete the `CUSTOM AGENTS` div, `#custom-agent-list`, and `#btn-add-custom-agent`.
- Delete the `DEFAULT PROMPT OVERRIDES` div, `#default-prompt-override-summary`, and `#btn-customize-default-prompts`.
- Delete the entire `<div class="startup-section">` (SETUP).
- Delete the entire `Database Operations` `<div class="system-section">`.
- Keep `live-feed-toggle` and `terminal-operations-toggle`.
- Add a new button in the Terminal Operations accordion: `<button id="btn-open-central-setup" class="secondary-btn w-full">OPEN CENTRAL SETUP</button>`.

### 2. Extract Configuration JavaScript
**File:** `src/webview/implementation.html`
- Remove event listeners for `btn-add-custom-agent`, `btn-customize-default-prompts`, `btn-save-startup`, `db-reset-btn`, and `db-update-location-btn`.
- Remove the `custom-prompts-modal` and `custom-agent-modal` HTML blocks and their associated logic.

### 3. Create Central Setup Webview Panel
**File:** `src/webview/setup.html` (New) and `src/services/SetupPanelProvider.ts` (New/Update)
- Migrate the extracted HTML and JS logic into a dedicated editor webview (similar to the Kanban board implementation).
- Implement message passing (`saveStartupCommands`, `saveDefaultPromptOverrides`, `getDbPath`) in the new provider.

### 4. Wire Synchronization & Commands
**File:** `src/extension.ts` & `src/services/TaskViewerProvider.ts`
- Register `switchboard.openSetupPanel` command to launch the new webview.
- When the Setup Panel saves configurations to `state.json`, invoke `vscode.commands.executeCommand('switchboard.refreshUI')` to ensure the sidebar `TaskViewerProvider` re-reads the state and updates the Agent List.

## Adversarial Synthesis

### Grumpy Critique
You can't just rip DOM elements out of `implementation.html`! Half of the initialization logic relies on checking if those elements exist. For instance, `saveStartupCommands` iterates over `.agent-visible-toggle` and `#plan-ingestion-folder-input`. If you move those inputs to a central panel, the sidebar's `saveStartupCommands` handler will overwrite the configuration with empty values because it can't find the elements in its DOM! 

### Balanced Response
Grumpy is correct about the silent data loss risk. We cannot simply delete the UI without refactoring the state saving mechanism. 
**Action Plan:**
1. The sidebar's `TaskViewerProvider` must be stripped of its `saveStartupCommands` and `saveDefaultPromptOverrides` message handlers. 
2. The new `SetupPanelProvider` will exclusively own saving these configurations to `state.json`.
3. The sidebar will transition to a pure read-only consumer of these settings, updating its UI dynamically whenever `switchboard.refreshUI` triggers a state reload.
