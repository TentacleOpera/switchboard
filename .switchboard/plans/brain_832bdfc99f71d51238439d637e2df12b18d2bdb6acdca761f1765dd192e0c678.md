# Guard Kanban Board Against Background Workspace Hijacking

This plan addresses a persistent bug where editing a plan file in a background workspace (e.g., by a background reviewer or coder agent) causes the active Kanban board to switch its workspace or go completely black.

## Goal

Prevent background workspace file-watcher events from hijacking the active Kanban board by adding workspace-match guards to the two unguarded UI refresh entry points (`_refreshRunSheets` and `refreshWithData`).

## Metadata

- **Tags:** [bugfix, UI, reliability]
- **Complexity:** 3

## User Review Required

> [!IMPORTANT]
> The fix adds path-comparison guards to prevent UI updates for non-active workspaces. These guards will run automatically and silently reject any background UI updates that target non-selected workspaces, keeping the active board stable. Background database and metadata updates will still be processed.

## Complexity Audit

### Routine
- Adding a path-comparison guard to `_refreshRunSheets` (single early-return block, reuses existing `getCurrentWorkspaceRoot`/`resolveEffectiveWorkspaceRoot` APIs)
- Adding a path-comparison guard to `refreshWithData` (single early-return block, uses existing `_currentWorkspaceRoot` field)
- Both guards follow the same pattern already established in `refreshUI` (line 2089) and `refreshIfShowing` (line 1015)
- Existing unit test in `TaskViewerProvider.refreshUI.test.ts` already validates the guard pattern; extending it is straightforward

### Complex / Risky
- None

## Edge-Case & Dependency Audit

- **Race Conditions:** The guard reads `_currentWorkspaceRoot` which is set by `setCurrentWorkspaceRoot` (line 653). Since JS is single-threaded in the extension host, the read is atomic relative to the guard check. No race window exists.
- **Security:** No security implications — this is a UI stability guard, not an auth/privilege boundary.
- **Side Effects:** The guard's early return in `_refreshRunSheets` skips both the kanban board update AND the sidebar dropdown update for the non-active workspace. This is correct behavior — the sidebar should also not switch context for background workspaces. When the user manually switches to that workspace, `setCurrentWorkspaceRoot` fires → `refreshUI` runs → full refresh occurs.
- **Dependencies & Conflicts:** `refreshUI` (line 2079) already has an identical guard. The new guards in `_refreshRunSheets` and `refreshWithData` are defense-in-depth — they catch calls that bypass `refreshUI` (14 direct call sites to `_refreshRunSheets`, 1 direct call site to `refreshWithData`). No conflict with the existing guard.

## Dependencies

- None

## Adversarial Synthesis

Key risks: (1) Guard in `_refreshRunSheets` blocks legitimate sidebar updates when the user switches workspace rapidly (guard reads stale `_currentWorkspaceRoot`). Mitigation: `setCurrentWorkspaceRoot` fires `onWorkspaceChange` event which triggers `refreshUI` with the new root, bypassing the guard. (2) Double-guard creates redundant logging noise. Mitigation: `_refreshRunSheets` guard fires first; `refreshWithData` guard is a safety-net that rarely triggers.

## Proposed Changes

We will modify two key files to check if the workspace being refreshed matches the user's active workspace selection.

### Switchboard Extension Services

#### [MODIFY] [TaskViewerProvider.ts](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/TaskViewerProvider.ts)

We will update the `_refreshRunSheets` method (line 13457) to verify that the workspace root of the runsheets matches the currently selected workspace root on the Kanban board. If they differ, the UI refresh will be skipped.

**Context:** `_refreshRunSheets` is called from 14+ sites. Many of these receive an explicit `workspaceRoot` from file-watcher callbacks that may target a background workspace. The method currently resolves the workspace root (line 13458-13462) but never checks whether it matches the board's active selection before calling `refreshWithData` (line 13498).

**Logic:** After resolving `resolvedWorkspaceRoot` (line 13462), compare it against the Kanban board's current workspace root. If they differ, return early — the board and sidebar should not update for a non-active workspace.

**Implementation:** Insert the guard block between lines 13462 and 13464 (after `resolvedWorkspaceRoot` is computed, before the DB read at line 13465):

```typescript
    private async _refreshRunSheets(workspaceRoot?: string) {
        const selectedWorkspaceRoot = workspaceRoot
            ? this._resolveWorkspaceRoot(workspaceRoot)
            : this._resolveWorkspaceRoot();
        if (!selectedWorkspaceRoot) return;
        const resolvedWorkspaceRoot = this._kanbanProvider?.resolveEffectiveWorkspaceRoot(selectedWorkspaceRoot) || selectedWorkspaceRoot;

        // Guard: only refresh if resolvedWorkspaceRoot matches the currently selected workspace root in the Kanban board
        const currentRoot = this._kanbanProvider?.getCurrentWorkspaceRoot();
        if (currentRoot) {
            const resolvedCurrentRoot = this._kanbanProvider?.resolveEffectiveWorkspaceRoot(currentRoot) || currentRoot;
            if (path.resolve(resolvedCurrentRoot) !== path.resolve(resolvedWorkspaceRoot)) {
                console.log(
                    `[TaskViewerProvider] _refreshRunSheets: resolvedWorkspaceRoot ${resolvedWorkspaceRoot} differs from current ${resolvedCurrentRoot} — skipping runsheet refresh`
                );
                return;
            }
        }
```

**Edge Cases:**
- When `currentRoot` is `null` (initialization), the guard is skipped — allowing the first refresh to proceed. This matches the behavior in `refreshUI` (line 2089).
- When `workspaceRoot` is not provided (e.g., line 13446), `_resolveWorkspaceRoot()` returns the active workspace root, so the guard passes.

#### [MODIFY] [KanbanProvider.ts](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/KanbanProvider.ts)

As a secondary layer of defense, we will update the `refreshWithData` method (line 1037) to ensure it only updates cards and workspace selections if the incoming data matches the currently active workspace root.

**Context:** `refreshWithData` is the single point where card data is sent to the webview (line 1161: `updateBoard` message). It currently has no workspace-match check, so any caller that passes data for a non-active workspace will cause the board to switch or go black.

**Logic:** After resolving `workspaceRoot` (line 1044), compare it against `_currentWorkspaceRoot`. If they differ, return early.

**Implementation:** Insert the guard block after line 1044 (after `resolvedWorkspaceRoot` is computed, before the DB read at line 1045):

```typescript
    public async refreshWithData(activeRows: import('./KanbanDatabase').KanbanPlanRecord[], completedRows: import('./KanbanDatabase').KanbanPlanRecord[], workspaceRoot: string, projects?: string[]) {
        if (!this._panel) {
            console.warn('[KanbanProvider] refreshWithData: no panel — skipping');
            return;
        }

        try {
            const resolvedWorkspaceRoot = path.resolve(workspaceRoot);
            if (this._currentWorkspaceRoot && path.resolve(this._currentWorkspaceRoot) !== resolvedWorkspaceRoot) {
                console.log(`[KanbanProvider] refreshWithData: resolvedWorkspaceRoot ${resolvedWorkspaceRoot} differs from current ${this._currentWorkspaceRoot} — not refreshing board`);
                return;
            }
```

**Edge Cases:**
- When `_currentWorkspaceRoot` is `null` (initialization), the guard is skipped — allowing the first data load to proceed.
- The guard uses `path.resolve()` on both sides for normalization, matching the pattern in `refreshIfShowing` (line 1015).
