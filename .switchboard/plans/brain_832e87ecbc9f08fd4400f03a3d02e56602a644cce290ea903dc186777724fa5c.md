# Fix New Plan Project Assignment for Workspace Mappings

Whenever the background `GlobalPlanWatcherService` detects a new plan file, it fails to assign it to the currently selected project if the user has a mapped child workspace active (e.g. `Autism360App`). This occurs because the project filter is saved in the watcher using the active child workspace root path, while the watcher processes new plans using the effective parent workspace root path. This plan resolves child workspace roots to their effective parent roots before storing the current project filter in the watcher.

## Goal

Fix the root-key mismatch in `GlobalPlanWatcherService._currentProjects` so that new plans detected via file watchers inherit the project filter set from a mapped child workspace.

- **Complexity:** 3

### Complex / Risky
- None

## Edge-Case & Dependency Audit

- **Race Conditions:** `setCurrentProject` is called from `KanbanProvider.setProjectFilter`, which fires from the webview message handler — always after `initialize()` and `buildMappingIndexFromDbs` have completed. If called before the mapping index is built, `resolveEffectiveWorkspaceRootFromMappings` returns the child root unchanged (safe fallback, same as current behavior). No race in practice.
- **Security:** No security implications — purely internal state management.
- **Side Effects:** Clearing `_currentProjects` on mapping refresh means the project filter is lost when mappings change. This is acceptable because mapping changes are rare and the user can re-select the filter.
- **Dependencies & Conflicts:** Depends on `resolveEffectiveWorkspaceRootFromMappings` from `WorkspaceIdentityService`, which is already used elsewhere in the codebase (e.g., `ensureWorkspaceIdentity`). No conflicts with existing callers.

## Dependencies

- None

## Adversarial Synthesis

Key risks: (1) If `setCurrentProject` is called before the mapping index is built, resolution silently falls back to the child root — same as the current bug. Mitigated by the fact that the webview only sends `setProjectFilter` after initialization completes. (2) Stale `_currentProjects` entries after mapping changes. Mitigated by clearing the map on `_refreshWatchers`. Overall risk is low; the fix is a well-scoped one-line resolution with cheap defensive measures.

## Proposed Changes

### [MODIFY] GlobalPlanWatcherService.ts

**File:** `src/services/GlobalPlanWatcherService.ts`

#### Context

`setCurrentProject(workspaceRoot, project)` (line 40) stores the project filter keyed by `workspaceRoot`. When the user has a mapped child workspace active (e.g., `Autism360App`), `KanbanProvider.setProjectFilter()` passes the child root as `workspaceRoot` (line 3771 of `KanbanProvider.ts`). But `_handlePlanFile` (line 423) looks up the project using the effective parent root (from `_getAllMappedFolders`), causing a key mismatch and the project defaults to empty string.

#### Logic

1. Add top-level import: `import { resolveEffectiveWorkspaceRootFromMappings } from './WorkspaceIdentityService';` (near line 7, alongside existing imports).
2. In `setCurrentProject` (line 40), resolve the workspace root before caching:
   ```typescript
   public setCurrentProject(workspaceRoot: string, project: string | null): void {
       const effectiveProject = project === KanbanDatabase.UNASSIGNED_PROJECT_FILTER ? '' : project;
       const effectiveRoot = resolveEffectiveWorkspaceRootFromMappings(workspaceRoot);
       if (effectiveRoot !== workspaceRoot) {
           this._outputChannel?.appendLine(`[GlobalPlanWatcher] setCurrentProject: resolved ${workspaceRoot} → ${effectiveRoot} for project "${effectiveProject}"`);
       }
       if (effectiveProject) {
           this._currentProjects.set(effectiveRoot, effectiveProject);
       } else {
           this._currentProjects.delete(effectiveRoot);
       }
   }
   ```
3. In `_refreshWatchers` (line 179), clear `_currentProjects` to prevent stale keys after mapping changes:
   ```typescript
   private async _refreshWatchers(): Promise<void> {
       // Clear stale project filters when mappings change
       this._currentProjects.clear();
       // ... existing code ...
   }
   ```

#### Implementation

- Line 7 area: Add `import { resolveEffectiveWorkspaceRootFromMappings } from './WorkspaceIdentityService';`
- Lines 40–49: Replace `setCurrentProject` body with the resolved-root version above.
- Line 179: Add `this._currentProjects.clear();` as the first line of `_refreshWatchers`.

#### Edge Cases

- If `resolveEffectiveWorkspaceRootFromMappings` returns the same root (no mapping), behavior is identical to current code.
- If mapping index is not yet built, the function returns the child root unchanged — safe fallback.
