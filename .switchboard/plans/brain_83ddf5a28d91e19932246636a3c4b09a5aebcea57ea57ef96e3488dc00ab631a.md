# Implement Explicit Tags and Metadata System

## Goal
Implement a dedicated tagging system that allows AI agents to search the Kanban database by predefined tags (e.g., authentication, backend) and complexity. This decouples tags from the plan title (`topic`) and replaces the brittle markdown-parsing heuristic for complexity with explicit, planner-assigned metadata.

## User Review Required
> [!NOTE]
> This plan requires a database schema migration (v3) to add a `tags` column to the `plans` table. Existing plans will have empty tags until updated. 
> You will need to confirm the list of predefined tags you want the planner to use (e.g., `frontend, backend, authentication, database, UI, devops`).

## Complexity Audit
### Routine
- Updating `src/services/agentPromptBuilder.ts` to instruct the Planner to output an explicit `## Metadata` block containing Tags and Complexity.
- Extending `src/mcp-server/register-tools.js` so `get_kanban_state` accepts a `tag` parameter.
### Complex / Risky
- Database Migration: Updating `src/services/KanbanDatabase.ts` and `src/services/KanbanMigration.ts` to execute an `ALTER TABLE` statement, adding the `tags` column, and ensuring the `syncPlansMetadata` loop correctly passes this new field.

## Edge-Case & Dependency Audit
- **Race Conditions:** None. SQLite handles schema migrations safely via standard synchronous execution on initialization.
- **Security:** When querying tags via the MCP tool, we must use safe parameterized SQL queries (e.g., `LIKE ?` with `%tag%`) to prevent injection.
- **Side Effects:** The `KanbanProvider` will need to read the new metadata block instead of relying exclusively on the old `## Complexity Audit` section, though it should retain a fallback for older plans.
- **Dependencies & Conflicts:** This builds on the recent database-driven architecture.

## Adversarial Synthesis
### Grumpy Critique
So we're doing another database migration just to add a tags column? Fine, but if you store tags as a JSON array in SQLite, querying them without the JSON1 extension is a nightmare. And if you let the Planner hallucinate tags, the database is going to be polluted with 50 variations of `backend` (`back-end`, `server`, `api`). Also, if you break `syncPlansMetadata` while adding this, you'll desync the entire board like what happened when complexity was dropped in migration v2!

### Balanced Response
Grumpy is right about the storage and hallucination risks. To avoid SQLite JSON querying issues, we will store tags as a simple comma-separated string (e.g., `,backend,authentication,`) which makes standard `LIKE '%,backend,%'` filtering perfectly safe and fast. To prevent tag hallucination, the planner prompt will be strictly constrained to a predefined list of tags. We will also carefully thread the new `tags` field through `KanbanDatabase.updateMetadataBatch` to ensure it syncs safely alongside the existing `complexity` and `topic` fields.
