# Polish planning.html Document and Plan Previews Spacing and Colors

## Goal

Align `planning.html`'s CSS variables, preview backgrounds, typography, and inline code styling to match the pitch-black dark aesthetic and tighter spacing of `kanban.html`.

## Metadata

- **Tags:** [frontend, UI, UX]
- **Complexity:** 3

## User Review Required

> [!NOTE]
> This change will align `planning.html`'s background variables (`--panel-bg` and `--panel-bg2`) with `kanban.html`'s color scheme, making it pitch-black/dark grey. It also overrides the editor-background fallback in preview containers to ensure a consistent look. This deliberately opts out of VS Code's `--vscode-editor-background` theme variable in preview panes in favor of a fixed dark palette. The webview panel is always rendered in dark mode, so this has no impact on light-theme users.

## Complexity Audit

### Routine
- Updating three CSS custom property values (`--panel-bg`, `--panel-bg2`, `--kanban-bg`) in `:root`
- Changing preview pane background from VS Code variable to custom variable
- Adding explicit background to `.kanban-preview-pane`
- Adjusting font-size, line-height, and margin values on existing selectors
- Adding inline `code` styling with hardcoded amber colors
- Setting heading color to `#ffffff`
- Updating `comment-popup` stale fallback color

### Complex / Risky
- None

## Edge-Case & Dependency Audit

- **Race Conditions:** None — CSS-only changes, no runtime logic.
- **Security:** No security implications.
- **Side Effects:** Changing `--panel-bg` from `#1a1a1a` to `#000000` propagates to 32 references across the file (body, tab bars, tree panes, controls strips, card gradients, comment popups, etc.). All current usages already target a dark background, so the shift from dark-grey to pitch-black is visually consistent. The `color-mix()` gradient in `.kanban-plan-item` (line 1364) will still blend correctly with the new base values.
- **Dependencies & Conflicts:** The existing `pre code` reset (lines 825–834) strips background, padding, and border from code inside `<pre>` blocks. The new inline `code` styles must NOT override this reset — they are separate selectors (`#markdown-preview code` etc. at line 836), and the `pre code` rule has higher specificity due to the compound selector, so no conflict exists.

## Dependencies

None.

## Adversarial Synthesis

Key risks: (1) Global `--panel-bg`/`--panel-bg2` variable changes propagate to 32 references — but all are dark-background contexts where pitch-black is an improvement, not a regression. (2) Removing `--vscode-editor-background` fallback opts out of VS Code theme adaptivity in preview panes — acceptable because webview panels are always dark-themed. Mitigations: the `pre code` reset is preserved by existing specificity; `--kanban-bg` and `comment-popup` fallback are updated for consistency.
