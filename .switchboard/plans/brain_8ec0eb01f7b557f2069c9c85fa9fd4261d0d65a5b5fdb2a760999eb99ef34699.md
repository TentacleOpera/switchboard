> [!IMPORTANT]
> **The two core problems, restated:**
> 1. Wall-of-text pages that dump every button/setting without structure
> 2. No plain-English explanation of *when you'd use this* and *what the workflow is*
>
> **The fix pattern for every page:**
> Every page opens with a **"When to use this"** paragraph (1-3 sentences, plain English, describes the use case) followed by a **"The workflow"** section (numbered steps showing the simple path). Reference material goes *below the fold* in collapsible `<details>` blocks.

---

## Global Rules (apply to every page)

### Rule 1: Every page opens with use case + workflow

Before any reference tables or button inventories, every page must answer:
- **Why does this exist?** (1-2 sentences — what problem does it solve?)
- **What's the simple workflow?** (3-5 numbered steps — the happy path)

### Rule 2: Reference material goes in collapsible sections

Dense tables (settings, all columns, all buttons) get wrapped in `<details><summary>` blocks so they don't dominate the page. The summary label should be descriptive: "All columns reference" not "Details".

### Rule 3: Separate essential from optional

When listing agents, columns, settings, or features — clearly label what's **essential for basic use** vs. **optional/advanced**. Don't present 11 columns as if the user needs all 11.

### Rule 4: One canonical source per concept

Each concept (Features, Project Manager, NotebookLM, slash commands) has ONE authoritative page. Every other mention is a 1-sentence summary with a link. No 200-word block duplication.

### Rule 5: Fix all broken links

Every `href` in frontmatter and body content must be verified against actual file paths before any content work begins.
