# Support Intern Role in Autoban Terminal Pools

Allow the `intern` role to be registered, created, and reconciled as part of the autoban terminal pools, aligning the backend with dynamic routing capabilities in the Kanban automation tab.

## Goal

Add `intern` as a first-class built-in autoban pool role so that intern terminals can be created, managed, and accounted for identically to coder, lead, and reviewer terminals — closing the gap between the existing dynamic routing logic (which already routes low-complexity plans to `intern`) and the terminal pool infrastructure (which currently rejects `intern` as an unsupported role).

## Metadata

- **Tags:** [backend, workflow]
- **Complexity:** 3

## User Review Required

> [!IMPORTANT]
> The `intern` role is already utilized in dynamic routing for low-complexity plans (complexity scores 1-4). This change simply adds the backend capability to support dedicated `intern` terminal pools, ensuring they are validated, created, and managed identically to coder, lead, and reviewer terminals.

## Complexity Audit

### Routine
- Adding `'intern'` to the `_autobanPoolRoles` built-in array (single literal insertion)
- Adding a `case 'intern': return 'Intern'` branch to `_getAutobanRoleLabel` (single switch case)
- Adding `rolesToCheck.add('intern')` in the dynamic-routing branch of `_allEnabledAutobanRolesExhausted` (single line)

### Complex / Risky
- None

## Edge-Case & Dependency Audit

- **Race Conditions:** None. All three changes are synchronous, deterministic code paths with no async state mutation.
- **Security:** No new attack surface. The `intern` role has the same terminal creation path as existing roles, gated by `_autobanPoolRoles` validation.
- **Side Effects:** Adding `'intern'` to `_autobanPoolRoles` causes the Kanban automation tab's terminal pool UI to display an "Intern" pool section automatically (driven by `_autobanPoolRoles` iteration). This is the intended effect.
- **Dependencies & Conflicts:** The `complexityScale.ts` module already defines `scoreToRoutingRole` mapping scores 1-4 → `'intern'` and `getFallbackRole('intern')` → `'coder'`. The `KanbanProvider.resolveRoutedRole` already handles intern routing with pair-programming bypass. The `columnToPromptRole` already maps `'INTERN CODED'` → `'reviewer'`. No conflicts — all downstream consumers already support `intern`.

## Dependencies

- None

## Adversarial Synthesis

Key risks: (1) terminal naming inconsistency if `_getAutobanRoleLabel` case is omitted, producing lowercase "intern" names alongside capitalized peers; (2) premature autoban stoppage if `_allEnabledAutobanRolesExhausted` omits intern from the PLAN REVIEWED check. Mitigations: both are simple, deterministic code additions with no conditional logic or state mutation.

## Proposed Changes

### Switchboard Extension Backend

#### [MODIFY] [TaskViewerProvider.ts](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/TaskViewerProvider.ts)

- **`_autobanPoolRoles`** (line 5039): Include `'intern'` in the built-in pool roles array so the backend recognizes it as a valid automation agent.
  - **Context:** This method returns the list of valid autoban pool roles. `_createAutobanTerminal` (line 5591) validates the requested role against this list — without `'intern'`, creating an intern terminal throws `"Unsupported autoban pool role"`.
  - **Logic:** Change `const builtIn = ['planner', 'coder', 'lead', 'reviewer']` → `const builtIn = ['planner', 'coder', 'lead', 'reviewer', 'intern']`.
  - **Implementation:** Single literal insertion at line 5040.
  - **Edge Cases:** The `_normalizeAutobanPoolRole` method (line 5054) normalizes `'intern'` → `'intern'` via `_normalizeAgentKey`, which is correct. No normalization mismatch.

- **`_getAutobanRoleLabel`** (line 5078): Add a case for `'intern'` returning `'Intern'` to display the correct label when creating/managing terminals.
  - **Context:** This method provides the human-readable label used by `getNextAutobanTerminalName` (line 5615) to generate terminal names like "Intern 1", "Intern 2". Without the explicit case, the `default` branch returns `'intern'` (lowercase), producing names like "intern 1" alongside "Coder 1" and "Lead Coder 2" — inconsistent capitalization.
  - **Logic:** Add `case 'intern': return 'Intern'` before the `default` case at line 5083.
  - **Implementation:** Single switch case insertion.
  - **Edge Cases:** None. The label is used only for display and terminal naming.

- **`_allEnabledAutobanRolesExhausted`** (line 5358): Include `'intern'` in the set of roles to check under dynamic routing for the `'PLAN REVIEWED'` column. This ensures autoban doesn't stop prematurely when only an intern terminal is ready and waiting.
  - **Context:** When the routing mode is dynamic (not `all_coder` or `all_lead`), lines 5374-5375 add `'coder'` and `'lead'` to `rolesToCheck`. The method then iterates over `rolesToCheck` calling `_selectAutobanTerminal` for each — if *any* role has an available terminal, it returns `false` (not exhausted). Without `'intern'`, if coder and lead terminals are exhausted but an intern terminal is available, the method incorrectly returns `true`, causing autoban to stop.
  - **Logic:** Add `rolesToCheck.add('intern')` after line 5375 (alongside the existing `'coder'` and `'lead'` additions).
  - **Implementation:** Single line insertion.
  - **Edge Cases:** The dispatch fallback chain at lines 7208-7218 already handles intern → coder → lead escalation. The exhaustion check is separate — it determines *autoban stoppage*, not dispatch ordering. Adding intern here is not redundant; it prevents premature autoban shutdown when only intern capacity remains.

7. Verify that the terminal shows status **READY** and is designated as **BACKUP**.
8. Verify the terminal name uses capitalized "Intern" (e.g., "Intern 1"), not lowercase "intern 1".
