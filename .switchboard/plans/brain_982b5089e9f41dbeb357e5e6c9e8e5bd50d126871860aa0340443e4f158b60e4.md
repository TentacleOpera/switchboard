# Local Docs Sidebar Clipboard Import Integration

## Goal
The goal of this change is to streamline the document import workflow by removing the separate 'Clipboard Import' tab and instead placing a contextual 'Import from Clipboard' button next to each configured local folder in the Local Docs sidebar tree pane.

### Background & Core Problems
Currently, importing a document from the clipboard requires navigating away from the Local Docs tab to a dedicated 'Clipboard Import' tab. In that tab, the user has to manually select the destination folder from a dropdown and trigger the import. This context switching is inefficient and clutters the interface with a tab that only serves a single form submission.

### Root Cause Analysis
The clipboard import logic was designed as a standalone utility tab (`#clipboard-content`) rather than being integrated contextually where folders are browsed. Integrating the clipboard trigger directly onto the root folder subheaders eliminates the need for the tab and dropdown selection entirely, resolving both the UX friction and tab clutter.

## Edge-Case & Dependency Audit

- **Race Conditions:** If the user clicks Import on two different folder headers in quick succession, the `_importInProgress` guard in PlanningPanelProvider (line 3264) prevents concurrent imports. The frontend should also disable all `.folder-import-btn` buttons during an in-flight import (the plan's proposed code already does this).
- **Security:** No security implications — clipboard read is already gated by the extension host.
- **Side Effects:** Removing the Clipboard Import tab removes the "COPY AGENT PROMPT" button (`#btn-copy-agent-prompt`). This button copies a prompt template that asks an agent to output plans in the `--- PLAN ---` separator format. This functionality is also available via the Command Palette (`Switchboard: Import Plan from Clipboard`), so the removal is safe.
- **Dependencies & Conflicts:** The `importResearchDoc` message handler in PlanningPanelProvider already accepts `folderPath` (line 1263), so no backend changes are needed. The `importResearchDocResult` message handler in planning.js (around line 2640) currently resets `importResearchDocClipboardBtn` — this must be updated to also reset `.folder-import-btn` buttons.

## Adversarial Synthesis
Key risks: `prompt()` is blocked by webview CSP and must be replaced with a CSP-compatible title entry mechanism (recommend: skip title, rely on backend H1 extraction); the `#btn-import-research-doc-clipboard` button exists in both the Clipboard Import tab and the Research tab — only the Clipboard tab instance should be removed; the `importResearchDocResult` handler must be updated to reset `.folder-import-btn` buttons. Mitigations: omit the title prompt on inline import (backend already handles empty titles via H1/timestamp fallback at PlanningPanelProvider line 3283-3290); preserve the Research tab's Step 3 section intact; update the result handler to target the new button class.

#### [MODIFY] [planning.js](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/webview/planning.js)
- Remove event listeners for the Clipboard Import tab buttons (`#btn-copy-agent-prompt`, `#btn-import-plans`). These are at lines 157-185. The `#btn-import-research-doc-clipboard` event listener (lines 187-221) must be kept — it serves the Research tab's Step 3 button.
- In `renderLocalDocs` (around line 1454-1460), modify the creation of `.source-folder-header` to append a label span and a `.folder-import-btn`:
  ```javascript
  const sourceHeader = document.createElement('div');
  sourceHeader.className = 'folder-subheader source-folder-header';
  
  const labelSpan = document.createElement('span');
  const folderName = sourceFolder.split(/[\\/]/).filter(Boolean).pop() || sourceFolder;
  labelSpan.textContent = folderName;
  sourceHeader.appendChild(labelSpan);
  
  const importBtn = document.createElement('button');
  importBtn.className = 'folder-import-btn';
  importBtn.textContent = 'Import';
  importBtn.title = `Import document from clipboard into ${folderName}`;
  importBtn.addEventListener('click', (e) => {
      e.stopPropagation();
      
      // Disable all import buttons during operation
      document.querySelectorAll('.folder-import-btn').forEach(btn => {
          btn.disabled = true;
          btn.textContent = '...';
      });
      
      const statusEl = document.getElementById('status');
      if (statusEl) {
          statusEl.style.color = '';
          statusEl.textContent = 'Importing from clipboard...';
      }
      
      vscode.postMessage({
          type: 'importResearchDoc',
          folderPath: sourceFolder
          // No docTitle — backend extracts from H1 or generates timestamp
      });
  });
  sourceHeader.appendChild(importBtn);
  docList.appendChild(sourceHeader);
  ```
  **Note on title entry:** The original plan proposed using `prompt()` for a custom title, but `prompt()` is blocked by the webview's Content-Security-Policy. Instead, the inline Import button omits the title parameter. The backend (`_handleImportResearchDoc`, PlanningPanelProvider line 3283-3290) already handles empty titles by extracting the H1 header from the clipboard content or generating a timestamp-based title. Users who want to specify a custom title can use the Research tab's Step 3 form which has a dedicated title input field.

- In the `importResearchDocResult` message handler (around line 2640), add reset logic for `.folder-import-btn` buttons:
  ```javascript
  // Reset inline folder import buttons
  document.querySelectorAll('.folder-import-btn').forEach(btn => {
      btn.disabled = false;
      btn.textContent = 'Import';
  });
  ```
  This should be added in both the success and error branches of the handler, alongside the existing `importResearchDocClipboardBtn` reset logic (which serves the Research tab's Step 3 button and must be preserved).
