## Stage 1 — Grumpy Principal Engineer

*Another day, another set of event listeners bolted onto a 9,700-line god class. Let's see what broke.*

- **NIT — Double-fetch on panel creation.** `onDidChangeViewState` at [PlanningPanelProvider.ts:434-442](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/PlanningPanelProvider.ts#L434-L442) fires immediately on the FIRST visibility event (panel create = visible), posting `refreshKanbanPlans` → boot `fetchKanbanPlans` at [project.js:391](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/webview/project.js#L391) already ran. Guaranteed double-fetch on every fresh open. The request-ID guard dedupes on the host and the second fetch supersedes the first (correct data), but it's a wasted full DB scan. **No webview debounce was added** — the plan's Complexity Audit explicitly recommended one ("the webview should debounce or skip if a fetch is already in flight") and it was not implemented.

- **NIT — Direct `postMessage` in `_hydratePanel`.** [PlanningPanelProvider.ts:758](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/PlanningPanelProvider.ts#L758) uses `this._projectPanel?.webview.postMessage(...)` (direct) instead of `this.postMessageToProjectWebview(...)` (queued). In `_hydratePanel`, the webview HTML was just set at lines 717-719. If `onDidChangeViewState` fires before the webview JS has parsed, the message is lost. However, this is mitigated: the boot `fetchKanbanPlans` at [project.js:391](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/webview/project.js#L391) runs synchronously on parse, making the `refreshKanbanPlans` redundant anyway. The message loss is invisible because the boot fetch already covers the same path. Harmless but sloppy.

- **NIT — `_fullKanbanPlansSent` not reset on dispose.** The flag at [PlanningPanelProvider.ts:150](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/PlanningPanelProvider.ts#L150) is per-instance (not per-panel). The dispose handler at [PlanningPanelProvider.ts:417-429](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/PlanningPanelProvider.ts#L417-L429) does NOT reset it. This is **correct** — the flag resets at [line 3501](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/PlanningPanelProvider.ts#L3501) on each new `fetchKanbanPlans` request start. It tracks "has ANY full response been sent for the current request cycle", not "has this panel ever received data". Not resetting on dispose is the right choice.

- **CLEAN — `openProject()` existing-panel refresh.** [PlanningPanelProvider.ts:351-356](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/PlanningPanelProvider.ts#L351-L356). The `if (this._projectPanelReady)` guard correctly prevents posting to an unready webview. Uses direct `postMessage` which is fine — the panel is known-ready.

- **CLEAN — Guard-race fix.** [PlanningPanelProvider.ts:3496-3577](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/PlanningPanelProvider.ts#L3496-L3577). The `_fullKanbanPlansSent = false` reset at [line 3501](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/PlanningPanelProvider.ts#L3501), fallback send at [lines 3550-3563](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/PlanningPanelProvider.ts#L3550-L3563), and normal send + flag set at [lines 3566-3577](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/PlanningPanelProvider.ts#L3566-L3577) are correctly wired. The losing fetch sends its (possibly stale) full response, and the winning fetch sends its fresher response last — postMessage ordering guarantees the fresher one wins in the webview.

- **CLEAN — `refreshKanbanPlans` handler in `project.js`.** [project.js:451-453](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/webview/project.js#L451-L453). Simple echo: `vscode.postMessage({ type: 'fetchKanbanPlans', requestId: Date.now() })`. Correct — `Date.now()` ensures a monotonically increasing requestId.

- **CLEAN — KanbanProvider `onDidChangeViewState`.** Both [KanbanProvider.ts:1297-1305](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/KanbanProvider.ts#L1297-L1305) (open) and [KanbanProvider.ts:1363-1371](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/KanbanProvider.ts#L1363-L1371) (deserialize) call `this._refreshBoard(this._currentWorkspaceRoot)`, guarded by `e.webviewPanel.visible && this._currentWorkspaceRoot`. The `_refreshBoard` method has its own snapshot-hash dedup at [KanbanProvider.ts:1720-1726](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/KanbanProvider.ts#L1720-L1726), so no-op re-renders are suppressed. No double-registration risk — each is in a distinct code path (open vs deserialize).
