## Root Cause: Cutoff Filter in `_rescanAntigravityPlanSources`

**File**: [`TaskViewerProvider.ts`](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/TaskViewerProvider.ts#L11094-L11127)

```typescript
private async _rescanAntigravityPlanSources(workspaceRoot: string): Promise<void> {
    const now = Date.now();
    const cutoff = this._lastAntigravityRescanAt > 0
        ? this._lastAntigravityRescanAt - 2000   // ← PROBLEM: only 2s lookback on subsequent calls
        : now - TaskViewerProvider.ANTIGRAVITY_RESCAN_WINDOW_MS; // 30-min window for first scan
    this._lastAntigravityRescanAt = now;

    // ...for each candidate file:
    const isRecent = stats.birthtimeMs >= cutoff || stats.mtimeMs >= cutoff;

    if (!existingEntry && !hasDbRow && !isRecent) {
        continue;  // ← Skips plan if not in registry, not in DB, and not recent
    }
```

**How the failure unfolds:**

1. The plan file was written at **20:59:04**.
2. `_syncFilesAndRefreshRunSheets` (which calls `_rescanAntigravityPlanSources`) runs for the first time after startup.
   - On the **first call** (e.g. 21:00:00), `_lastAntigravityRescanAt == 0`, so `cutoff = 21:00:00 - 30min = 20:30:00`. The file at 20:59:04 is **recent** → plan should be picked up. ✅ *This should work...*
3. **BUT**: On every subsequent `_syncFilesAndRefreshRunSheets` call (e.g. 21:01:00), `cutoff = 21:00:00 - 2s = 20:59:58`. The file mtime is 20:59:04 < 20:59:58 → `isRecent = false`. Since the plan is not in the registry or DB (it was never successfully claimed), it is **permanently skipped**.

So the question becomes: **why wasn't it picked up on the first 30-minute scan?**

The answer lies in the `_mirrorBrainPlan` call that would have fired — let me trace the exact first-scan scenario. There are two possibilities:

### Possibility A: Extension was Already Running with Prior Rescan State
If VS Code was running continuously since before 20:59:04, `_lastAntigravityRescanAt` was already > 0 by the time the plan was written. This means the first `_syncFilesAndRefreshRunSheets` call *after* the plan was written used `cutoff = lastScan - 2s`. If `lastScan` was at e.g. 20:58:00 and the plan was written at 20:59:04, then `cutoff = 20:57:58`, the plan *is* recent — **this path would work**.

### Possibility B: Watcher Fired but `_mirrorBrainPlan` Rejected the Plan
The `onDidCreate` watcher may have fired for `implementation_plan.md`, calling `_mirrorBrainPlan` directly (bypassing the rescan). In this case `_mirrorBrainPlan` may have returned early at one of its guards. Looking at the code:

```typescript
// Line 11770-11774
const isFreshUnregisteredCandidate =
    !existingEntry &&
    !runSheetKnown &&
    !fs.existsSync(mirrorPath) &&
    (Date.now() - fileCreationTimeMs) <= TaskViewerProvider.NEW_BRAIN_PLAN_AUTOCLAIM_WINDOW_MS;
//  NEW_BRAIN_PLAN_AUTOCLAIM_WINDOW_MS = 15_000 (15 seconds)
```

The `fileCreationTimeMs` is the **directory birth time** (20:57:24), not the file write time. At 20:59:04 (when the plan was written), `Date.now() - fileCreationTimeMs = 100 seconds > 15s`. So `isFreshUnregisteredCandidate = false`.

This means the `wouldAutoClaim` path depends on `allowAutoClaim` being true. If the watcher calls `_mirrorBrainPlan` with `allowAutoClaim = false` (the default), then `wouldAutoClaim = false`, and the plan is skipped with "Mirror skipped (eligibility)" log message.

## The Actual Bug

Looking at the watcher registration code, `_mirrorBrainPlan` is called **without** `allowAutoClaim = true` in the watcher path — only in the rescan path. So the watcher fires, `allowAutoClaim = false`, `isFreshUnregisteredCandidate = false` (because >15s since dir creation), `wouldAutoClaim = false`, plan is skipped.

Then when the rescan runs (from `_syncFilesAndRefreshRunSheets`), it passes `allowAutoClaim = true`, but **the cutoff filter** prevents the file from being seen as "recent" if the rescan runs more than 2 seconds after the last rescan.

**The combined bug**: For plans written into a session directory that is older than 15 seconds (i.e. agent started session dir, then did research/planning before writing the plan), the watcher won't auto-claim them AND the rescan will only find them if the file mtime falls within the `[lastRescanAt - 2s, now]` window. This window is too narrow for plans written in a different session where rescans happen frequently.

---

## Fix Required

The fix in `_rescanAntigravityPlanSources` needs to handle the case where a plan file has **never been seen before** regardless of its age. The current `!existingEntry && !hasDbRow && !isRecent` guard is too aggressive — it skips unregistered, unclaimed plans that aren't "recent" relative to the last rescan.

### Option 1 (Minimal): Widen the rescan window
Change the cutoff for subsequent scans to use the full 30-minute window instead of `lastScan - 2s`:

```typescript
// Current (wrong):
const cutoff = this._lastAntigravityRescanAt > 0
    ? this._lastAntigravityRescanAt - 2000
    : now - TaskViewerProvider.ANTIGRAVITY_RESCAN_WINDOW_MS;

// Fix: always use the 30-minute window
const cutoff = now - TaskViewerProvider.ANTIGRAVITY_RESCAN_WINDOW_MS;
```

The `_lastAntigravityRescanAt` debounce guard (if it exists) should still prevent excessive full scans.

### Option 2 (Targeted): Skip the `isRecent` guard for unclaimed plans
The guard exists to avoid re-processing already-known plans on every rescan. It's safe to process unclaimed plans (no registry entry, no DB row) regardless of recency:

```typescript
// Remove the isRecent gate for unclaimed plans:
if (!existingEntry && !hasDbRow) {
    // Always process unclaimed plans — they need to be claimed regardless of age
} else if (!isRecent) {
    continue;  // Skip plans already known to the system that haven't changed recently
}
```

### Option 3: Fix `NEW_BRAIN_PLAN_AUTOCLAIM_WINDOW_MS` to use file mtime instead of dir birthtime
The `isFreshUnregisteredCandidate` check uses the **directory creation time** (`fileCreationTimeMs = stat.birthtimeMs`) as a proxy for "how new is this plan". But the plan file itself may have been written much later. Using `mtimeMs` of the file would be more accurate.
