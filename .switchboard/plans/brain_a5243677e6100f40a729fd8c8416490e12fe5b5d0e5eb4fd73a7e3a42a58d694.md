# Restore Targeted Gitignore Strategy as Default for Workspace Exclusion System

The `WorkspaceExcludeService` was implemented with blanket exclusion rules
(`.switchboard/*` and `.agent/*`) as defaults, replacing the targeted `.gitignore`
approach that was used by the original `_runGitignoreMigrationV1`. The targeted
approach selectively ignored only machine-local files (the kanban database and
integration configs) while preserving `.switchboard/plans/`, `.switchboard/reviews/`,
`.agent/` and shared protocol documents so they remain tracked by git and visible
to agents. This change restores that behavior as the default strategy, removes the
unneeded blanket-gitignore preset (reachable via `custom` instead), and redesigns
the Setup UI's ignore rules display to be read-only for preset strategies and
editable only when the user selects `custom`.

## Goal

1. **Replace** the existing strategy enum (`localExclude`, `gitignore`, `none`) with
   a new set: `targetedGitignore`, `localExclude`, `custom`, `none`.
2. Make `targetedGitignore` the new default, writing a hardcoded targeted rule set
   — matching the actual `.gitignore` used by this repo — to `.gitignore`.
3. Remove `gitignore` as a standalone preset. Blanket-ignoring `.switchboard/*` and
   `.agent/*` is harmful to agent context gathering; anyone who truly needs it can
   use `custom` and type the rules themselves.
4. The ignore rules textarea in the Setup panel becomes **read-only** for all preset
   strategies (`targetedGitignore`, `localExclude`). It displays what will be
   written. Only `custom` allows editing.
5. `localExclude` remains a first-class preset (writes to `.git/info/exclude`) so
   users on shared repos who want zero `.gitignore` pollution have a named option.
6. No migration is required: the blanket configuration has not been shipped.

### Strategy Matrix

| Strategy | Target | Rules | Who it's for |
|---|---|---|---|
| `targetedGitignore` (default) | `.gitignore` | Hardcoded targeted set | Most users — team-friendly |
| `localExclude` | `.git/info/exclude` | Stored `ignoreRules` setting | Solo devs on shared repos |
| `custom` | `.gitignore` or `.git/info/exclude` (user picks via sub-option) | Editable textarea | Power users |
| `none` | — | — | Fully manual |

## Complexity Audit

### Routine
- `package.json`: Drop `"gitignore"` from the enum, add `"targetedGitignore"` and
  `"custom"`. Change `default` from `"localExclude"` to `"targetedGitignore"`.
- `WorkspaceExcludeService.ts`: Add a `targetedGitignore` branch (hardcoded targeted
  rules → `.gitignore`). Rename/repurpose the old `gitignore` branch to `custom`
  (reads from `ignoreRules` setting, target file determined by sub-option or
  defaults to `.gitignore`). Reuse existing append-only logic throughout.
- `TaskViewerProvider._normalizeGitIgnoreConfig()`: Update valid strategy union
  to `targetedGitignore | localExclude | custom | none`; update fallback default
  to `'targetedGitignore'`.
- `setup.html`: Replace `gitignore` option with `targetedGitignore` and `custom`.
  Make the textarea `readonly` by default. JS listener removes `readonly` only when
  strategy is `'custom'`.

### Complex / Risky
- **Autosave signature divergence**: The setup panel uses a signature-based autosave
  that compares `collectSetupSavePayload()` serializations. The read-only textarea
  must still serialize its display value correctly per strategy, and the signature
  must not drift between hydration and the first user action. Requires careful
  review of `sanitizeGitIgnoreRules` + `collectSetupSavePayload` — for preset
  strategies the serialized `gitIgnoreRules` should be the display array, not the
  stored `ignoreRules` setting (which would be stale for `targetedGitignore`).
- **Default normalization**: `_normalizeGitIgnoreConfig` currently falls back any
  unrecognized strategy to `'localExclude'`. After this change it must fall back to
  `'targetedGitignore'`. Hardcoded `'localExclude'` fallback strings in the webview
  JS (`lastGitIgnoreConfig` initializer at line 799–802 of `setup.html`) must be
  updated too.
- **`custom` target file**: For simplicity, `custom` always writes to `.gitignore`
  (same target as the old `gitignore` preset). A sub-option for `localExclude`
  custom rules is out of scope; users who want that can use `localExclude` preset.

The following is the exact set of targeted rules `targetedGitignore` will write to
`.gitignore`. It matches the pattern already established in this repo's `.gitignore`:

```

!.switchboard/reviews/
!.switchboard/plans/
!.switchboard/sessions/
!.switchboard/CLIENT_CONFIG.md

.switchboard/kanban.db
```

> [!IMPORTANT]
> `.agent/` is intentionally **NOT** excluded by this targeted strategy. Workflows,
> personas, and skills in `.agent/` are shared team artifacts that agents need to
> find via normal search.

> [!NOTE]
> The `localExclude` and `gitignore` preset strategies continue to use the
> `switchboard.workspace.ignoreRules` setting as before (for append-only behavior).
> For those strategies the textarea will display the current setting value read-only.
> Only `custom` allows editing.

## Edge-Case & Dependency Audit

### Dependencies & Conflicts
- `add_git_ignore_ui_to_setup_menu.md` (already Coded/Reviewed) — **this plan
  modifies the same files** (`setup.html`, `TaskViewerProvider.ts`,
  `WorkspaceExcludeService.ts`, `package.json`). Both plans must be implemented
  sequentially; this plan builds on top of the existing UI. No revert of that plan
  is needed.
- No other active plans in New/Planned appear to conflict with this plan.

### Edge Cases
- **Append-only idempotency with negation rules**: The targeted rule set includes
  negation patterns (e.g., `!.switchboard/plans/`). The existing line-exact Set
  comparison in `WorkspaceExcludeService` correctly handles these because negation
  lines are matched verbatim. No special handling required.
- **Strategy displayed to user must match rules shown**: The textarea must recompute
  its display content from the strategy any time the dropdown changes, not just on
  hydration. The `renderGitIgnoreConfig()` function must cover this.
- **User switching from `custom` back to a preset**: The textarea should revert to
  the preset read-only display for that strategy. The stored `ignoreRules` setting
  retains the user's custom array but is not written to disk unless `custom` is
  selected again.
- **`targetedGitignore` always targets `.gitignore`**: It does not respect
  `localExclude`. The target file is hardcoded.

### `src/services/WorkspaceExcludeService.ts`

#### [MODIFY] `WorkspaceExcludeService.ts` — Add `targetedGitignore` branch

**Context:** The `apply()` method currently handles `localExclude`, `gitignore`,
and `none`. A new branch must handle `targetedGitignore`.

**Logic:**
1. Define a `TARGETED_RULES` static constant array containing the exact targeted
   rule set documented above, including blank-line separators and comment lines.
2. In `apply()`, update the default strategy fallback from `'localExclude'` to
   `'targetedGitignore'`.
3. Add an `if (strategy === 'targetedGitignore')` branch that:
   - Sets `targetFile` to `<workspaceRoot>/.gitignore`.
   - Uses `TARGETED_RULES` instead of the `rules` config array.

**Implementation:**

```typescript
import * as vscode from 'vscode';
import * as fs from 'fs';
import * as path from 'path';

export class WorkspaceExcludeService {
    private static readonly HEADER_COMMENT = '# Switchboard managed exclusions';

    /**
     * Targeted rules written by the `targetedGitignore` strategy.
     * Mirrors the repo's own .gitignore pattern: ignore machine-local state
     * while preserving plans, reviews, sessions, and shared protocol docs.
     */
    private static readonly TARGETED_RULES: string[] = [
        '# Switchboard runtime state (per-session, not shareable)',
        '.switchboard/*',
        '!.switchboard/reviews/',
        '!.switchboard/plans/',
        '!.switchboard/sessions/',
        '!.switchboard/CLIENT_CONFIG.md',
        '!.switchboard/README.md',
        '!.switchboard/SWITCHBOARD_PROTOCOL.md',
        '!.switchboard/workspace-id',
        '# ClickUp integration config (workspace-specific IDs)',
        '.switchboard/clickup-config.json',
        '# Linear integration config (workspace-specific IDs and sync map)',
        '.switchboard/linear-config.json',
        '.switchboard/linear-sync.json',
        '# Notion integration config and page content cache',
        '.switchboard/notion-config.json',
        '.switchboard/notion-cache.md',
        '# Switchboard kanban database \u2014 machine-local, do not commit.',
        '.switchboard/kanban.db',
        '.switchboard/*.db-shm',
        '.switchboard/*.db-wal',
    ];

    constructor(private readonly workspaceRoot: string) {}

    async apply(): Promise<void> {
        const config = vscode.workspace.getConfiguration('switchboard.workspace');
        const strategy: string = config.get('ignoreStrategy', 'targetedGitignore');
        const rules: string[] = config.get('ignoreRules', []);

        if (strategy === 'none') {
            console.log('[WorkspaceExcludeService] Strategy is "none" — skipping exclusion management.');
            return;
        }

        const gitRoot = path.join(this.workspaceRoot, '.git');
        if (!fs.existsSync(gitRoot) || !fs.statSync(gitRoot).isDirectory()) {
            console.log('[WorkspaceExcludeService] No .git directory found — skipping.');
            return;
        }

        let targetFile: string;
        let effectiveRules: string[];

        if (strategy === 'targetedGitignore') {
            targetFile = path.join(this.workspaceRoot, '.gitignore');
            effectiveRules = WorkspaceExcludeService.TARGETED_RULES;
        } else if (strategy === 'localExclude') {
            const infoDir = path.join(gitRoot, 'info');
            fs.mkdirSync(infoDir, { recursive: true });
            targetFile = path.join(infoDir, 'exclude');
            effectiveRules = rules;
        } else if (strategy === 'custom') {
            // Custom: user-managed rules written to .gitignore
            targetFile = path.join(this.workspaceRoot, '.gitignore');
            effectiveRules = rules;
        } else {
            console.warn(`[WorkspaceExcludeService] Unknown strategy "${strategy}" — skipping.`);
            return;
        }

        let existingContent = '';
        try {
            existingContent = await fs.promises.readFile(targetFile, 'utf-8');
        } catch (e: any) {
            if (e.code !== 'ENOENT') throw e;
        }

        const existingLines = new Set(existingContent.split('\n').map(l => l.trim()));
        // For targeted strategy, only check non-comment, non-empty lines for dedup
        const missingRules = effectiveRules.filter(rule => {
            const trimmed = rule.trim();
            if (trimmed === '' || trimmed.startsWith('#')) return false; // always include structural lines separately
            return !existingLines.has(trimmed);
        });

        if (missingRules.length === 0) {
            console.log('[WorkspaceExcludeService] All rules already present — no changes needed.');
            return;
        }

        const block = [
            '',
            WorkspaceExcludeService.HEADER_COMMENT,
            ...effectiveRules,
            ''
        ].join('\n');

        await fs.promises.appendFile(targetFile, block, 'utf-8');
        console.log(`[WorkspaceExcludeService] Appended ${missingRules.length} rule(s) to ${path.basename(targetFile)}`);
    }

    /**
     * Returns the targeted rule set for display purposes (used by the Setup UI).
     */
    static getTargetedRules(): string[] {
        return [...WorkspaceExcludeService.TARGETED_RULES];
    }
}
```

> [!NOTE]
> The `missingRules` check skips comment and empty lines from deduplication since
> those are structural separators. The actual idempotency guard is on the data lines.
> This means comment lines may be re-appended if a user previously ran a partial
> migration. This is an acceptable cosmetic nit (pre-existing behavior from the old
> plan's review) and is deferred.

## Adversarial Synthesis

### Grumpy Critique

> Oh look, a plan that adds a fourth enum value and then smuggles a 21-line hardcoded
> rule array into the *webview* as a JS constant. Now you have the truth in three
> places: `WorkspaceExcludeService.TARGETED_RULES`, the plan document, and
> `TARGETED_RULES_DISPLAY` in `setup.html`. The day someone adds a new integration
> and updates the service, the webview display will silently show stale rules. Users
> will see the wrong preview. Support tickets follow.
>
> The `missingRules` dedup logic is also wrong for the targeted strategy. You skip
> comment lines from the dedup check, then append the *entire* `effectiveRules` block
> (including comments) when even one data line is missing. That means a second run
> can append duplicate comment headers if the file already contains some but not all
> data lines.
>
> And the `ignoreRules` default change from `['.switchboard/*', '.agent/*']` to `[]`
> is a silent breaking change for any workspace that used `localExclude` or
> `gitignore` strategies and relied on those defaults because they never explicitly
> saved a value. Their textarea will now show nothing. They click APPLY and write
> zero rules.

### Balanced Response

**Point 1 — Duplicated rule array.** Valid concern. Mitigation: the Setup panel
should request the targeted rules from the backend rather than duplicating them.
Add a `getTargetedRules()` static method to `WorkspaceExcludeService` (already
included in the implementation above). The setup panel's `gitIgnoreConfig` message
should include a `targetedRulesDisplay: string[]` field when strategy is
`targetedGitignore`, populated by `WorkspaceExcludeService.getTargetedRules()`.
This eliminates the duplication at the cost of a simple backend call.
**Action: Extend `handleGetGitIgnoreConfig()` to include `targetedRulesDisplay`.**

**Point 2 — Dedup logic for comment lines.** Partially valid. The current block
includes all lines including comments in the append block. A better approach: check
if the *header comment* (`HEADER_COMMENT`) is already present as an anchor. If any
targeted data line is missing AND the header is absent, append the full block. If
the header is present, only append missing data lines individually. This avoids
duplicate comment headers. **Action: Tighten the idempotency check.**

**Point 3 — `ignoreRules` default change breaking `localExclude` users.**
Valid. The fix is simple: leave the `ignoreRules` package.json default as
`[".switchboard/*", ".agent/*"]` for backward compatibility. `WorkspaceExcludeService`
already reads the stored setting — existing workspaces that have explicitly or
implicitly set `localExclude` will continue to see their stored rules. New workspaces
will get `targetedGitignore` as the default strategy and the targeted hardcoded rules,
completely bypassing `ignoreRules`. The `handleGetGitIgnoreConfig()` fallback default
for `ignoreRules` stays as `['.switchboard/*', '.agent/*']` rather than `[]`.
**Action: Revert `ignoreRules` default to `['.switchboard/*', '.agent/*']` in both
`package.json` and `handleGetGitIgnoreConfig()`.**

**Refinement — Extended `handleGetGitIgnoreConfig()` return shape:**

```typescript
public handleGetGitIgnoreConfig(): {
    strategy: 'targetedGitignore' | 'localExclude' | 'custom' | 'none';
    rules: string[];
    targetedRulesDisplay: string[];
} {
    const config = vscode.workspace.getConfiguration('switchboard.workspace');
    const { strategy, rules } = this._normalizeGitIgnoreConfig(
        config.get<string>('ignoreStrategy', 'targetedGitignore'),
        config.get<string[]>('ignoreRules', ['.switchboard/*', '.agent/*'])
    );
    return {
        strategy,
        rules,
        targetedRulesDisplay: WorkspaceExcludeService.getTargetedRules()
    };
}
```

The webview then uses `message.targetedRulesDisplay` rather than a local constant
for the targeted preview, eliminating the duplication while keeping the display
instant on dropdown change (the last received value is cached in JS state).
