# Plan: Fix Child Directory .switchboard Pollution in Multi-Root Workspaces

This plan addresses the issue where `.switchboard/` directories and `api-server-port.txt` files are incorrectly created inside mapped child workspace directories in multi-root workspaces.

## Goal
Ensure that mapped child workspace directories are never polluted with `.switchboard/` folders, `kanban.db` files, `workspace-id` files, or `api-server-port.txt` files. All Switchboard state, database operations, and API ports must reside exclusively in the parent (Control Plane) workspace folder.

## User Review Required

> [!IMPORTANT]
> **Automatic Cleanup In Child Folders**
> As requested, on extension startup, we will perform an automated validation and cleanup sweep. If a mapped child workspace directory contains a `.switchboard` directory, we will:
> 1. Safely remove auto-generated files (like `api-server-port.txt` and `workspace-id`).
> 2. If the folder is empty or only contains these auto-generated files, completely delete the `.switchboard` folder to clean up pre-existing pollution at work.
> 3. If there is a pre-existing `kanban.db` with active plans in the child folder, we will log a warning and prompt the user before deletion, or safely reconcile/backup the database before cleaning it up.

> [!IMPORTANT]
> **Export Gap: `resolveEffectiveWorkspaceRootFromMappings`**
> `WorkspaceIdentityService.resolveEffectiveWorkspaceRootFromMappings` is currently a module-level *private* function. For `TaskViewerProvider` to call it in `_startLocalApiServer`, a new `export` must be added to `WorkspaceIdentityService.ts`. This is a required prerequisite for the startup race fix.

## Complexity Audit

### Routine
- Exporting `resolveEffectiveWorkspaceRootFromMappings` from `WorkspaceIdentityService.ts` (single-line addition).
- Fixing `_filterMappedRoots` path resolution to align with the expanded-home logic used in `WorkspaceIdentityService`.
- Aligning `KanbanDatabase._redirectToParentIfMapped` path resolution (already uses `_expandHome`; verify it matches the service).

### Complex / Risky
- **Automated cleanup in `_validateNoSwitchboardPollution`**: Deletion of filesystem artifacts on startup is destructive if wrong. Must guard: (a) only delete known auto-generated filenames; (b) check `kanban.db` for active plans before deletion; (c) use `vscode.window.showWarningMessage` when plans are found; (d) persist user consent so the dialog does not recur on every restart.
- **Startup race in `_startLocalApiServer`**: `LocalApiServer` is instantiated with `workspaceRoot = this._getWorkspaceRoot()` which may return `roots[0]` — a child root — before the mapping is resolved. Fixing requires calling the (newly exported) `resolveEffectiveWorkspaceRootFromMappings` on that root *before* passing it to `new LocalApiServer(...)`.

---

## Edge-Case & Dependency Audit

- **Race Conditions**: `_validateNoSwitchboardPollution` is called `void` (fire-and-forget) in the constructor. A concurrent `_startLocalApiServer` could write a port file to a child root in the window before cleanup runs. The cleanup should delete the port file *after* it is written by the server start, or the server start should be guarded to never write to child roots (which `_filterMappedRoots` already does for the *port file loop*, but not for the `LocalApiServer` constructor root itself).
- **Security**: Cleanup only touches known filenames (`api-server-port.txt`, `workspace-id`, `kanban.db`). Never recursively delete arbitrary contents.
- **Side Effects**: If `kanban.db` exists in a child root with real plans, a destructive wipe would cause data loss. The reconciliation/backup path in `User Review Required` is the mitigation.
- **Dependencies & Conflicts**: The export of `resolveEffectiveWorkspaceRootFromMappings` must not create a circular import. `TaskViewerProvider` already imports from `WorkspaceIdentityService` (line 65), so this is safe.

##### Fix C: Automated Cleanup in `_validateNoSwitchboardPollution` (lines 621–663)

**Context**: Currently only logs a warning. No cleanup is performed. This is the largest gap between plan and implementation.

**Logic** (guarded deletion):
1. For each `resolvedRoot` that is a mapped child root:
   - Check if `path.join(root, '.switchboard')` exists.
   - If yes, enumerate its contents.
   - **Safe-to-delete filenames**: `api-server-port.txt`, `workspace-id`.
   - Delete safe files unconditionally.
   - Check if `kanban.db` exists. If yes, check `KanbanDatabase.dbFileHasPlans(dbPath)`.
     - If no plans → delete `kanban.db`.
     - If has plans → call `vscode.window.showWarningMessage(...)` asking user to confirm. Store consent in `globalState` to avoid re-prompting on restart.
   - If `.switchboard/` is now empty, `fs.promises.rmdir` it.

**Implementation skeleton**:
```typescript
private async _validateNoSwitchboardPollution(): Promise<void> {
    // ... existing mapping detection logic ...
    for (const root of allRoots) {
        const resolvedRoot = path.resolve(root);
        if (!mappedChildRoots.has(resolvedRoot)) continue;

        const switchboardDir = path.join(resolvedRoot, '.switchboard');
        if (!fs.existsSync(switchboardDir)) continue;

        // Delete safe auto-generated files
        const safeFiles = ['api-server-port.txt', 'workspace-id'];
        for (const file of safeFiles) {
            try { await fs.promises.unlink(path.join(switchboardDir, file)); } catch { /* ignore ENOENT */ }
        }

        // Handle kanban.db
        const dbPath = path.join(switchboardDir, 'kanban.db');
        if (fs.existsSync(dbPath)) {
            const hasPlans = await KanbanDatabase.dbFileHasPlans(dbPath);
            if (!hasPlans) {
                await fs.promises.unlink(dbPath);
            } else {
                const consentKey = `switchboard.pollutionCleanupConsent.${resolvedRoot}`;
                const alreadyConsented = this._context.globalState.get<boolean>(consentKey, false);
                if (!alreadyConsented) {
                    const answer = await vscode.window.showWarningMessage(
                        `Switchboard: Child workspace "${root}" has a stray .switchboard/kanban.db with active plans. Remove it?`,
                        'Remove', 'Keep'
                    );
                    if (answer === 'Remove') {
                        await fs.promises.unlink(dbPath);
                        await this._context.globalState.update(consentKey, true);
                    }
                }
            }
        }

        // Remove empty .switchboard dir
        try {
            const remaining = await fs.promises.readdir(switchboardDir);
            if (remaining.length === 0) {
                await fs.promises.rmdir(switchboardDir);
            }
        } catch { /* ignore */ }
    }
}
```

**Edge Cases**:
- `ENOENT` on unlink is expected when files don't exist — swallow silently.
- `rmdir` only removes the directory if empty — safe.
- User consent is stored per-root in `globalState` to avoid dialog on every restart.
