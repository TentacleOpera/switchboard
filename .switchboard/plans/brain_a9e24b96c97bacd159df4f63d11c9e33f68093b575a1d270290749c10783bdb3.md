# Fix "Suggest Epics" Agent Workflow

The current agent-driven "suggest epics" flow (the prompt used in this session) has five
distinct deficiencies surfaced by the test run, plus a sixth data-availability defect
surfaced during plan review. This plan addresses all of them as changes to the prompt/
instructions that drive the agent, plus one downstream behaviour change in how the agent
reads plan content, plus one supporting change to the `kanban-board.md` generator so the
agent can read plan identity and epic assignment without falling back to the raw DB query.

## Goal

Fix the "Suggest Epics" agent workflow so it (a) reads the lightweight `kanban-board.md`
snapshot instead of the 1.3 MB `get-state.js` JSON blob, (b) scopes to active columns with
BACKLOG opt-in, (c) enforces a minimum group size of 2, (d) reads plan bodies before
grouping, and (e) detects cross-provider epics and flags plan deficiencies — while first
correcting a fatal data-availability gap: `kanban-board.md` today carries neither the
`planId` (needed by `create-epic.js`) nor any epic-assignment signal, so the agent must
have those fields surfaced in the snapshot before the data-source swap is viable.

## Problem Analysis

### Problem 1 — Wrong data source (direct DB query instead of kanban-board.md)
The agent ran `get-state.js`, which hits the SQLite DB directly and emits a massive JSON
blob (1.3 MB). The board already publishes a clean, human-readable snapshot at
`.switchboard/kanban-board.md` that lists every plan with its file path and title,
grouped by column. The agent should read that file instead. Benefits:
- No raw DB I/O — the file is already maintained by the extension
- Output is plain markdown: trivially parseable, small, no JSON noise

**However**, the original plan assumed `kanban-board.md` carried everything `get-state.js`
does "minus `epicId`." Review of `KanbanDatabase.exportStateToFile` (lines 5407–5411)
disproved this: the snapshot line format is `- [${plan.planFile}](${filePath}) — ${plan.topic}`
and carries **neither `planId` nor `isEpic`/`epicId`**. Both are required by the agent:
- `planId` is the primary key `create-epic.js` passes to the extension's
  `createEpicFromPlanIds`, which does `db.getPlanByPlanId(pid)`. Plan filenames are
  content slugs (e.g. `investigate_completion_and_archival_workflows.md`), NOT planIds
  (e.g. `0b481492-dc18-41d6-8216-cced4256418f` or `sess_1781000229386`). Deriving planId
  from the filename is impossible for regular plans.
- `epicId`/`isEpic` is the only way to detect already-assigned subtasks, because plan
  files have **no YAML frontmatter** (confirmed by inspecting actual plan files in
  `.switchboard/plans/`); epic linkage lives in the DB only.

**Resolution (revised):** enrich `kanban-board.md` with `planId` and epic assignment via an
additive HTML-comment suffix on each plan line (see Change 0 / Proposed Changes →
KanbanDatabase.ts). This preserves the visible board format (existing tests use substring
assertions that remain satisfied) while giving the agent the two missing fields. The agent
then reads `planId` and skip-tags from the comment, never from the filename or frontmatter.

The path-based signal "files under `.switchboard/epics/` are epics" remains valid as a
secondary check (epic files are moved to that directory by `promoteToEpic` /
`createEpicFromPlanIds`), but it cannot detect subtasks (which live in `plans/` with an
`epicId` in the DB), so it is not sufficient on its own.

### Problem 2 — BACKLOG should be opt-in
BACKLOG was included in the initial sweep even though the user deliberately moved those
cards out of the active workflow. The agent's default scope should be **CREATED, CONTEXT
GATHERER, and PLAN REVIEWED** only. After proposing epics for the active columns, the agent
should ask: *"Do you want me to also analyse BACKLOG?"* and wait for a yes before doing so.

### Problem 3 — Single-card epics are meaningless
An epic with one member is not a grouping — it is just a renamed plan. The agent must
enforce a **minimum of 2 plans per proposed epic**. A genuinely standalone plan should be
listed as standalone, not wrapped in a solo epic.

### Problem 4 — Surface-level grouping misses cross-provider relationships
The agent grouped by surface keyword (Linear, Notion) rather than by underlying
*capability theme*. The "Linear Channel Issues: Analyst Chat & Extension Command Interface"
plan and the "/switchboard-remote Orientation Skill" plan both exist to enable the same
user journey: *driving Switchboard from outside the local machine via a remote channel*.
The agent needs to **read the body of each candidate plan** (not just its title) before
proposing groupings. Title-only matching leads to over-conservative, under-connected epics.

### Problem 5 — Cross-provider epics not detected (plan deficiency flag)
"Remote Agent Orientation" should span both Linear and Notion remote surfaces — they share
the same fundamental architecture (poll + comment routing). The agent produced separate
epics per provider instead of one unified epic. Beyond mis-grouping, the deeper issue is
that a plan like "Linear Channel Issues" may be **superseded or partially covered** by a
newer, broader plan. The agent should be able to flag this as a *plan deficiency*
(overlap, redundancy, or gap) inside its proposal, so the user can decide whether to merge,
deprecate, or keep them separate before any epic is created.

### Problem 6 — Snapshot lacks planId and epic-assignment signal (NEW, surfaced in review)
See Problem 1's revised resolution. Without enriching `kanban-board.md`, the data-source
swap is non-viable: the agent would either pass slugs to `create-epic.js` (silent failure
on every epic) or re-group already-assigned subtasks (duplicate epics). This is the
foundational change that makes Changes 1–5 safe.

## Edge-Case & Dependency Audit

- **Race Conditions:** `kanban-board.md` is regenerated on every `_persist` via a
  fire-and-forget `exportStateToFile` (line 5461), using a `.tmp` + atomic rename
  (lines 5423–5425). The agent reads it once at SCAN time; if the board changes mid-run
  the agent works from a stale snapshot. This is acceptable — the CONFIRM gate lets the
  user reject stale groupings, and `create-epic.js` re-validates each planId against the
  live DB via the extension. No new race is introduced.
- **Security:** No new surface. The HTML comment only echoes data already in the DB
  (`planId`, `topic`). `topic` is user-authored and could contain `-->` to break the
  comment, but this only corrupts the agent's parse of that one line (it does not inject
  into the extension — `create-epic.js` routes through the running extension's API server,
  which re-resolves planIds server-side). Low risk; noted as an edge case in
  KanbanDatabase.ts changes.
- **Side Effects:** Enriching the snapshot changes a file ~4,000 installs already have
  generated. Because the change is purely additive (a trailing comment) and the writer is
  internal (regenerated on every board mutation), no migration is needed — existing files
  are overwritten on the next board change. No `*.migrated.bak` required since no user data
  is transformed, only the export format.
- **Dependencies & Conflicts:** None. `exportStateToFile` already calls `getBoard`, which
  returns `isEpic` and `epicId` on every `KanbanPlanRecord` (interface lines 57–58), so no
  new DB column or query is needed.

## Dependencies

- None. This plan is self-contained.

## Adversarial Synthesis

Key risks: (1) the original plan's data-source swap rested on two false assumptions —
that plan filenames encode planId (they don't; they're slugs) and that plan files carry
`epicId` frontmatter (they don't; linkage is DB-only) — either of which alone would make
the swap produce silent failures or duplicate epics; (2) the rewritten prompt's grouping
heuristics are qualitative and cannot be unit-tested. Mitigations: enrich
`kanban-board.md` with `planId` + epic tags via an additive HTML-comment suffix (one small
change to `exportStateToFile`), instruct the agent to parse those tags instead of
filenames/frontmatter, keep path-based `epics/` detection as a secondary signal, and gate
all creation behind the existing CONFIRM step plus server-side planId re-validation in
`create-epic.js`.

### Change 1 — Data source: read kanban-board.md, not get-state.js

**Old step 1 prompt:**

```

**New step 1 prompt:**
```

The agent extracts `planId` from the trailing HTML comment on each line

`epic` are epics — skip them. Plans whose comment carries `subtask-of:...` are already

valid as a secondary epic signal, but cannot detect subtasks and must not be the only
signal.)

*(Striken: false for regular plans — filenames are content slugs, not planIds.)*
~~To detect plans already assigned to an epic, the agent may grep the plan file for
`epicId:` in its YAML frontmatter~~ *(Striken: plan files have no frontmatter; epic

### Change 2 — Default scope: active columns only, BACKLOG opt-in

**Column scope (hardcoded in the prompt):**
- **Include by default:** CREATED, CONTEXT GATHERER, PLAN REVIEWED
- **Exclude by default:** BACKLOG, LEAD CODED, CODER CODED, CODE REVIEWED, and all
  post-coding columns (these are either in-flight or deliberately parked)

After delivering the proposal for the active columns the agent appends:

> "BACKLOG has N ungrouped plans. Want me to analyse those too? (yes / no)"

And stops. If the user says yes, it repeats the read-and-propose flow for BACKLOG only,
appending the BACKLOG proposals to the existing message thread. `N` is the `backlogCount`
value interpolated into the prompt by `_buildSuggestEpicsPrompt`.

### Change 3 — Minimum group size: 2 plans

The agent's grouping logic enforces:
- A proposed epic must have **≥ 2 member plans**
- Any plan that would be the sole member of a group goes into the "standalone" section
- The standalone section is informational — it does NOT result in epic creation

### Change 4 — Read plan bodies before grouping

After collecting the list of candidate plans from kanban-board.md the agent:
1. Reads each candidate plan file in full (using `view_file` or `cat`)
2. Extracts: goal summary, dependency section, tags, and the first paragraph of
   problem analysis
3. Uses this content — not just titles — to determine thematic groupings

This is the most expensive step but the most important for grouping quality. The agent
should read plans in parallel where possible. For BACKLOG (opt-in), it reads plan bodies
only after the user confirms.

**Practical limit:** if there are more than ~25 candidate plans in the active columns,
the agent should read only the plans whose titles suggest thematic overlap (i.e., do a
first-pass title cluster, then deep-read within each cluster to validate/refine).

### Change 5 — Cross-provider epic detection + plan deficiency flags

After reading plan bodies the agent looks for:

**a) Cross-provider groupings:** Plans that address the same user capability on different
providers (Linear vs Notion vs ClickUp) should be grouped into one epic, not one epic per
provider. The canonical test: *"If we removed the provider name from the plan title, would
these plans describe the same thing?"* If yes → same epic.

**b) Plan deficiency flags:** When two plans in the same proposed epic appear to overlap
significantly in scope, the agent flags this inline with one of:
- `⚠️ POSSIBLE OVERLAP` — both plans address the same code paths; one may supersede the other
- `⚠️ REDUNDANCY` — one plan appears to be a subset of another
- `⚠️ GAP` — the epic theme implies a third plan that doesn't exist yet

These flags are informational only. They appear in the proposal message alongside the
grouped plan list. The user resolves them (merge, deprecate, keep) before or during
the CONFIRM step.

## Updated Flow (full revised prompt)

```
You are grouping loose Switchboard plans into epics. Follow this flow exactly — do not
create any epic before the user approves.

1. SCAN
   Read the board snapshot:
     cat <workspaceRoot>/.switchboard/kanban-board.md

   Scope: CREATED, CONTEXT GATHERER, and PLAN REVIEWED columns only.
   Ignore BACKLOG and all post-coding columns.
   Each plan line ends with an HTML comment, e.g.:
     - [.switchboard/plans/foo.md](...) — Foo <!-- planId:abc-123 -->
     - [.switchboard/epics/epic-def.md](...) — Bar Epic <!-- planId:def-456 epic -->
     - [.switchboard/plans/baz.md](...) — Baz <!-- planId:ghi-789 subtask-of:"Bar Epic" -->
   Skip lines tagged `epic` (they are epics) or `subtask-of:...` (already assigned).
   Use the `planId:` value from the comment — NOT the filename — when calling
   create-epic.js. (A path under .switchboard/epics/ also indicates an epic, but
   subtask detection requires the `subtask-of` tag — do not rely on filenames alone.)

2. READ PLAN BODIES
   For each candidate plan in scope, read the full plan file.
   Extract: goal, problem summary, dependencies, tags.
   Use this — not just titles — to determine groupings.

3. PROPOSE (single message, all groups at once)
   Group by underlying capability theme, not by surface keyword.
   Cross-provider plans that address the same capability go into one epic.
   Minimum 2 plans per epic. Single-plan "groups" go in the Standalone section.
   Flag ⚠️ POSSIBLE OVERLAP / REDUNDANCY / GAP where detected.
   For each proposed epic: epic name, member plans with planId and one-line summary.
   List genuinely standalone plans separately. Then stop and wait.

   After the proposal, ask: "BACKLOG has N ungrouped plans. Analyse those too? (yes/no)"

4. CONFIRM
   Wait for user approval or edits. Do not touch the database until confirmed.

5. EXECUTE
   For each approved group:
     node .agents/skills/kanban_operations/create-epic.js \
       "<epic name>" '["planId1","planId2",...]' "<workspaceRoot>"
   Use assign-to-epic.js for plans added to an existing epic.
```

`<workspaceRoot>` and the BACKLOG count `N` are interpolated by `_buildSuggestEpicsPrompt`
at copy time (see KanbanProvider.ts changes).
