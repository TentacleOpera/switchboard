# Persona Protocol Hardening

## Goal
Update the `switchboard_operator.md` persona to ensure all agents correctly identify and use the session-specific brain directory for implementation plans.

## Metadata
**Tags:** infrastructure, bugfix
**Complexity:** 4

## User Review Required
> [!NOTE]
> **Clarification:** This plan hardens persona guidance only. It does not expand scope into runtime code, migration logic, or broad documentation cleanup. Because the target file is itself under `.agent/personas/`, implementation should be executed by a coding persona or other authorized editor if the active agent is currently bound by the Switchboard Operator self-edit prohibition.

## Complexity Audit
### Routine
- Editing one markdown rule in `.agent/personas/switchboard_operator.md` to replace ambiguous wording with an explicit session-scoped path.
- Preserving the existing fallback behavior that creates timestamped local plans in `.switchboard/plans/` when the Antigravity brain directory is unavailable.
- Keeping the descriptive-title requirement in Rule 5 intact while tightening only the path language.

### Complex / Risky
- Persona/runtime drift: the wording change must align with the already-implied session-directory convention instead of inventing a new storage layout.
- The target is a persona file, and Rule 8 in `.agent/personas/switchboard_operator.md` explicitly says persona files cannot self-edit under that persona.
- Adjacent docs/tests still contain some root-level brain-path examples, so this plan must document that residual inconsistency without silently widening scope.

## Edge-Case & Dependency Audit
- **Race Conditions:** None in runtime execution because this plan only changes persona text. The real risk is planning ambiguity: different agents may interpret "brain-linked artifact" differently and choose incompatible target paths. Making the path explicit removes that coordination hazard.
- **Security:** Low risk. The change narrows instructions to a specific filesystem convention and introduces no new credentials, network calls, or secret handling. **Clarification:** keep `<conversation-id>` as a placeholder in documentation; do not hardcode a real session identifier into the persona file.
- **Side Effects:** Agents following the updated persona will stop treating `~/.gemini/antigravity/` or `~/.gemini/antigravity/brain/implementation_plan.md` as an acceptable default target. That is the intended outcome, but it may expose stale assumptions in other documentation or tests that still mention a root-level brain file. This plan does not broaden scope to clean up those references.
- **Dependencies & Conflicts:** `get_kanban_state` succeeded. Active **New** items: none. Active **Planned** items: this plan, `Restore Targeted Gitignore Strategy as Default for Workspace Exclusion System`, and `Visual Kanban Structure Reordering Plan`. Based on those active-plan titles plus the `.switchboard/plans/` filenames, there is no direct overlap with `.agent/personas/switchboard_operator.md`. **Clarification:** historically related items such as `Prevent Cross-Workspace Brain Plan Leakage` and `Fix: ID Scoping & Rescue Brain Plans` exist in non-active columns; they are relevant background but are excluded from active dependency/conflict analysis by rule.

## Adversarial Synthesis
### Grumpy Critique
*Oh good, we're "hardening" a protocol by changing one sentence in a persona file and pretending ambiguity was the only problem. That's adorable.*

1. *"Brain-linked artifact" was vague, yes, but if the rest of the ecosystem still contains root-level examples, then congratulations: you've upgraded one instruction while leaving a trail of contradictory breadcrumbs everywhere else.*
2. *You're editing a persona file whose own Rule 8 says persona files must not self-edit. If the implementer sleepwalks through that contradiction, the process will violate its own policy before the ink is dry.*
3. *A sloppy rewrite could accidentally change more than path specificity — maybe the fallback behavior, maybe the descriptive-title requirement, maybe the environment check. Then you've turned a documentation clarification into a silent behavioral spec change.*
4. *And let's not kid ourselves: if you don't explicitly say this is documentation-only scope, somebody will "helpfully" start patching runtime code, tests, or migration logic because they smelled a path inconsistency. That's scope creep wearing a fake mustache.*

### Balanced Response
Those concerns are valid, so the plan is intentionally narrow and explicit:

1. The implementation is limited to Rule 5 in `.agent/personas/switchboard_operator.md`; it does not authorize cleanup of every related reference in the repo.
2. The self-edit conflict is documented up front so the work can be routed to a coding persona or another authorized editor rather than performed under the restricted operator persona.
3. The diff preserves the fallback clause and the descriptive-title requirement verbatim, changing only the ambiguous path wording and the environment check target.
4. Residual mismatches elsewhere are called out as side effects and follow-up candidates, not bundled into this plan. That keeps the plan honest, surgical, and implementable.
