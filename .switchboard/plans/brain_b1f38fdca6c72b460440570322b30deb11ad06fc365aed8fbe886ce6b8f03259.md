# replicate 'Review' Feature in Switchboard

This plan outlines the implementation of a contextual 'Review' feature for plans in Switchboard, similar to the one found in the Antigravity IDE.

## Proposed Changes

### [Component Name] Switchboard Extension Core

#### [NEW] [ReviewProvider.ts](file:///c:/Users/patvu/Documents/GitHub/switchboard/src/services/ReviewProvider.ts)

- Create a new `ReviewProvider` class to manage the Review Webview.
- Implement logic to render Markdown with selectable blocks.
- Add event handlers for "Submit Comment" which will send a message to the active terminal/agent.

#### [MODIFY] [extension.ts](file:///c:/Users/patvu/Documents/GitHub/switchboard/src/extension.ts)

- Register the new `ReviewProvider`.
- Add a NEW command `switchboard.reviewPlan` (do **NOT** replace `switchboard.openPlan` â€” existing "View" button behaviour is preserved).
- Add a new `Review` icon button to the Kanban card (in `kanban.html`) that calls `switchboard.reviewPlan`.

### [Component Name] Webview UI

#### [NEW] [review.html](file:///c:/Users/patvu/Documents/GitHub/switchboard/src/webview/review.html)

- Create a dedicated HTML/CSS/JS for the Review Webview.
- Use VS Code's built-in `markdown.api.render` API for rendering â€” **not** `marked` â€” to remain CSP-compliant.
- Implement text selection via `mouseup` + `window.getSelection()` in the webview JS, posting `{ selectedText, selectionRect }` back to the extension host.
- Render the floating popup inside the webview itself (not the extension host).
- Implement WebviewPanel lifecycle boilerplate: `onDidDispose`, `retainContextWhenHidden: true`, and a `reveal()` method (matching the `KanbanProvider.ts` pattern).

## Final Review Results

### Implemented Well
- A `switchboard.reviewPlan` command was correctly registered and wired to a new `ReviewProvider.ts`.
- The logic safely manages context (`ReviewPlanContext`) and gracefully falls back to `fs` if `getReviewTicketData` doesn't provide a session.

### Issues Found
- **[CRITICAL]** The original implementation of `switchboard.reviewPlan` *overwrote* the existing Kanban View button, explicitly violating the directive: `Add a NEW command switchboard.reviewPlan (do NOT replace switchboard.openPlan â€” existing "View" button behaviour is preserved).`
- **[CRITICAL]** The subsequent "Ticket View" refactor of the webview wiped out the Markdown renderer and the text selection/commenting popup required by the plan: `Implement logic to render Markdown with selectable blocks` and `Implement text selection via mouseup + window.getSelection() in the webview JS`. This completely broke the core capability requested.

### Fixes Applied
- **Kanban Board:** Restored the `switchboard.openPlan` logic (mapped to `viewPlan`) to the original View icon, and added a distinct `Review Plan Ticket` button next to it.
- **Review Webview:** Re-introduced the VS Code `markdown.api.render` pipeline inside `ReviewProvider.ts` to attach `renderedHtml` alongside the raw ticket text.
- **Review Webview:** Added a "Preview" toggle to `review.html` that swaps between the Ticket View editor and the rendered Markdown. Re-wired the `mouseup` + `getSelection()` comment popup logic so users can select text inside the preview and dispatch contextual feedback via `switchboard.sendReviewComment` exactly as originally designed.

### Validation Results
- Executed `npm run compile`. All assets successfully bundled. No TypeScript type mismatches encountered between the extended `ReviewTicketData` schema and the message passing layer.

### Remaining Risks
- The hybrid "Ticket Editor / Markdown Preview" design adds state complexity to the webview JS. Switching rapidly between preview and edit modes could theoretically lead to unhandled selection state edge cases if text is mutated simultaneously.
