# Imported Plan

syncPlansMetadata has a guard at line 127 that skips re-parsing for plans already in the DB, regardless of whether the plan file has changed:

typescript
// KanbanMigration.ts line 127
if (row.complexity === 'Low' || row.complexity === 'High') {
    resolvedComplexity = row.complexity;  // ← uses cached value, never re-reads file
} else if (resolveComplexity) {
    // only calls getComplexityFromPlan if snapshot row carries 'Unknown'
}
But the deeper issue is what value row.complexity carries when this code runs. The row is built by _buildKanbanRecordFromSheet (line 870–876), which calls getComplexityFromPlan at construction time — so by the time syncPlansMetadata sees a row, it already has the freshly-parsed value from the file.

The real failure is in _buildKanbanRecordFromSheet: it passes rawPlanFile (the path stored in the sheet JSON) to getComplexityFromPlan. For this brain-sourced plan (antigravity_ prefix), rawPlanFile likely points to the brain path (~/.gemini/antigravity/brain/...), not the workspace copy at .switchboard/plans/. When I updated the plan I wrote to .switchboard/plans/brain_bf878...md — but the self-heal reads the brain copy, which still has the old content without ## Metadata.

In short: The file watcher / sync-board refresh re-parses planFile as stored in the session sheet — which for brain-sourced plans is the brain/appdata path, not the workspace path you edited. The system doesn't detect that the workspace mirror changed, so complexity stays stale.

This is a real bug. The fix would be: when re-parsing complexity, try both brainSourcePath and mirrorPath/localPlanPath, and take the result from whichever file has the more-recently-modified timestamp or the most information (i.e., has a ## Metadata section).
