# Fix Antigravity Brain Plan Mirroring — Three Compounding Bugs

## Metadata
**Complexity:** 3
**Tags:** bugfix, reliability, brain-watcher, testing

## Goal

Fix three compounding bugs in the Antigravity brain plan mirroring pipeline that together cause plans to be silently dropped and never appear on the Kanban board. These bugs were discovered during UAT of the prior fix (`brain_5a7b1232`), which restored the `_rescanAntigravityPlanSources` call site but did not address defects within the rescan logic itself.

### Core Problems & Root Cause Analysis

**UAT Failure Context**: Session `e96f4c0d` (`implementation_plan.md` for "Fix HTML Previews") was created at 20:57:24 with the plan written at 20:59:04 (97 seconds later). Session `a283c2f5` (`uat_failure_investigation.md`) was created at 21:03 with the plan written at 21:06. Neither plan appeared on the Kanban board despite the `_rescanAntigravityPlanSources` call being correctly restored.

---

**Bug 1 — Rescan Cutoff Filter Drops Unclaimed Plans (`_rescanAntigravityPlanSources`, line 11121)**

The rescan uses a sliding cutoff window to skip files that haven't changed since the last scan:
```typescript
const cutoff = this._lastAntigravityRescanAt > 0
    ? this._lastAntigravityRescanAt - 2000   // only 2s lookback on subsequent calls
    : now - ANTIGRAVITY_RESCAN_WINDOW_MS;     // 30-min window on first call

const isRecent = stats.birthtimeMs >= cutoff || stats.mtimeMs >= cutoff;

if (!existingEntry && !hasDbRow && !isRecent) {
    continue;  // ← silently skips unclaimed plans outside the window
}
```

After the first rescan runs, all subsequent rescans use `cutoff = lastRescanAt - 2000ms`. Any plan file with an mtime older than 2 seconds before the last rescan is not "recent". If that plan is also unclaimed (not in registry, not in DB), it is **permanently skipped on every future rescan**. This affects any plan written into a session directory more than 2 seconds before a subsequent `_syncFilesAndRefreshRunSheets` call, which is nearly every real-world case.

**Bug 2 — `_isLikelyPlanFile` Rejects Non-Standard Filenames (line 11297-11305)**

The plan file validator has a fast-pass for `implementation_plan.md` only. All other filenames must pass a content heuristic requiring 2+ recognized Switchboard section headers OR 1 section + metadata tag:
```typescript
if (baseFilename === 'implementation_plan.md') { return true; }  // fast pass

// Otherwise, requires ≥2 of these specific sections:
// Goal, Goals, Metadata, User Review Required, Proposed Changes,
// Verification Plan, Edge-Case & Dependency Audit, etc.
```

Any Antigravity-generated plan that uses different section names (e.g. `## Summary`, `## Root Cause`, `## Fix Required`) silently fails this check and is rejected at line 11802 in `_mirrorBrainPlan`. This affects every plan file not named `implementation_plan.md` that uses non-standard sections.

**Bug 3 — Regression Test Does Not Cover Cutoff Filter Behavior**

`brain-rescan-regression.test.js` only verifies that `_rescanAntigravityPlanSources` is called within `_syncFilesAndRefreshRunSheets`. It does not test the `isRecent` cutoff logic, so Bug 1 was undetectable via automated testing.

## Proposed Changes

### [TaskViewerProvider.ts](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/TaskViewerProvider.ts)

#### Fix Bug 1 — Flip the cutoff guard (line 11121)

The guard should only skip plans that are **already known** to the system and haven't changed recently. Unclaimed plans (no registry entry, no DB row) must always be processed — they cannot be "already handled" by definition.

**Current (wrong):**
```typescript
if (!existingEntry && !hasDbRow && !isRecent) {
    continue;
}
```

**Fix:**
```typescript
// Only skip plans already registered/in-DB that haven't changed since the last scan.
// Unclaimed plans (no registry entry, no DB row) must always be processed — they
// cannot have been previously handled and may have been missed by the watcher.
if ((existingEntry || hasDbRow) && !isRecent) {
    continue;
}
```

- **Risk**: Slightly more iterations per rescan for large brain dirs. Mitigated by the existing `_lastAntigravityRescanAt` debounce — full rescans are already bounded to the first call per 30-minute window for normal frequency.
- **Side effects**: None. `_mirrorBrainPlan` has its own dedupe guard (`_recentMirrorProcessed`, 5s TTL) and mtime check that prevents redundant processing of already-mirrored files.

#### Fix Bug 2 — Relax `_isLikelyPlanFile` for brain-sourced files (line 11297)

Brain-sourced files (files within `~/.gemini/antigravity*/brain/`) are always agent-created artifacts written intentionally. Unlike files in user-managed plan folders, they don't need the strict heuristic to filter out accidental .md files. Treat them like `isAdditionalFolder = true`: any .md with an H1 heading is accepted.

**Current:**
```typescript
const baseFilename = path.basename(this._getBaseBrainPath(filePath)).toLowerCase();
if (baseFilename === 'implementation_plan.md') {
    return true;
}
// ... strict heuristic for all other filenames
```

**Fix** — add a brain-source bypass before the filename check:
```typescript
// Brain-sourced files are always agent-created artifacts; accept any .md with an H1.
if (this._isAntigravitySourcePath(filePath)) {
    return hasH1;  // hasH1 is already computed above at line 11288
}

const baseFilename = path.basename(this._getBaseBrainPath(filePath)).toLowerCase();
if (baseFilename === 'implementation_plan.md') {
    return true;
}
// ... existing strict heuristic for non-brain files
```

- **Risk**: A brain session directory containing a non-plan .md with an H1 would be ingested. Acceptable — if an agent writes an H1-headed .md to its brain dir, it is by definition an intentional document. The `EXCLUDED_BRAIN_FILENAMES` list (`task.md`, `walkthrough.md`, etc.) already guards against the known noise files.
- **Side effects**: None. All other guards in `_mirrorBrainPlan` still apply.

### [brain-rescan-regression.test.js](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/test/brain-rescan-regression.test.js)

#### Fix Bug 3 — Add structural test for the cutoff guard logic (new assertion)

Add a fourth structural assertion verifying that the cutoff guard skips known-and-unchanged plans (not unclaimed ones):

```javascript
// Assert: the cutoff skip guard is conditioned on existingEntry || hasDbRow, not their absence.
// This prevents regression of the "unclaimed plans permanently skipped" bug.
const cutoffGuardPattern =
    /if\s*\(\s*\(\s*existingEntry\s*\|\|\s*hasDbRow\s*\)\s*&&\s*!isRecent\s*\)/;
assert.ok(
    cutoffGuardPattern.test(source),
    'Expected rescan cutoff guard to skip only known plans (existingEntry || hasDbRow), not unclaimed ones.'
);
```
