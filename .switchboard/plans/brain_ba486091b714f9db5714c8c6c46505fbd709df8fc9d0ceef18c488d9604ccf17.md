# Fix Planning Panel Doc Previews and Folder Saving

## Goal
Fix folder persistence, HTML document preview loading, and cross-tab preview error pollution within the Switchboard Planning Panel across single and multi-root workspaces.

## Problem Description & Causes
This plan addresses three bugs in the Switchboard Planning panel related to document previews, folder configuration persistence, and tab isolation.

### 1. Folder modal says "No folders configured" on reopen
- **Problem**: When a folder is configured via the folders modal on the Local Docs or HTML Previews tab, files load in the sidebar. However, reopening the modal shows "No folders configured".
- **Cause**: In `PlanningPanelProvider.ts`, the backend helper methods `_sendLocalDocsReady` and `_sendHtmlDocsReady` scan all workspace roots. They use a deduplication `Set` (`scannedPaths`). If a folder path is configured in multiple roots (e.g. globally or mirrored), it is added to `scannedPaths` during the loop. When the loop processes the active root, the folder is skipped due to `if (folderPath && scannedPaths.has(folderPath)) continue;`. Consequently, `configuredFolderPaths` is sent to the webview as empty (`[]`), causing the webview to overwrite `state.localFolderPaths` and clear the modal list.

### 2. HTML previews display as blank white space
- **Problem**: Opening HTML documents in the HTML Previews tab shows a blank white screen instead of rendering the file.
- **Cause**: The webview's Content Security Policy (CSP) defines `frame-src {{WEBVIEW_CSP_SOURCE}};`. In modern VS Code, `asWebviewUri` converts local file paths to `https:` scheme URIs (e.g. `https://file%2B.vscode-resource.vscode-cdn.net/...`). Because the scheme `https:` is missing from `frame-src` in the CSP, the browser blocks loading the local webview URI inside the iframe.

### 3. Local doc preview shows "sourceFolder is not a configured HTML folder path" after visiting HTML previews
- **Problem**: If you view an HTML preview, and then click back to the Local Docs tab, the local preview pane displays the error: `Error: sourceFolder is not a configured HTML folder path`.
- **Cause (Error spillover)**: In `planning.js`, `handlePreviewError` classifies sources as either online or local (offline). Since `html-folder` is not an online source (`ONLINE_SOURCES` = `['clickup', 'linear', 'notion']`), `handlePreviewError` defaults to rendering the error in the local preview area (`markdownPreview`).
- **Cause (Validation mismatch — multi-root)**: In `PlanningPanelProvider.ts`, `_handleFetchPreview` resolves folders via `this._getLocalFolderService(workspaceRoot)`. The `workspaceRoot` parameter comes from `_handleMessage` (line 742), which always uses the *active* workspace root. The webview's `fetchPreview` message does NOT include a `workspaceRoot` — it only sends `sourceFolder` (planning.js line 540-546). In a multi-root workspace, if the user clicks a file from Root A while the active editor is in Root B, the service is created for Root B, which doesn't have the folder configured → validation fails. The same root mismatch also affects auto-refresh: `_setupActiveDocWatcher` (line 553-554) re-resolves the root via `_getWorkspaceRoot()` at trigger time, which may differ from the root where the folder was originally configured.

### Complex / Risky
- Implementing `_getLocalFolderServiceForFolder` in [PlanningPanelProvider.ts](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/PlanningPanelProvider.ts): This new helper scans all workspace roots to find which root has a given `sourceFolder` configured, prioritizing the active workspace root. It is required because the webview does not send `workspaceRoot` in `fetchPreview` messages — only `sourceFolder` — so the backend must discover the correct root from the folder path alone. Without this, multi-root workspaces will always validate against the wrong root when the user clicks a file from a non-active root.
- Fixing the auto-refresh root resolution: the `_setupActiveDocWatcher` callback (line 550–562) re-resolves `workspaceRoot` via `_getWorkspaceRoot()` at trigger time, which may differ from the root where the folder was originally configured. Fix: replay the cached `_activePreviewWorkspaceRoot` instead of re-resolving.

## Edge-Case & Dependency Audit
- **Race Conditions**: Auto-refresh triggered by `activeDocWatcher` (line 541) while the user is rapidly switching tabs or workspaces. The watcher already has a generation guard (`gen !== this._watcherGeneration`) and a path guard (`filePath !== this._activePreviewPath`). Adding `_activePreviewWorkspaceRoot` ensures the correct root is used even if the active root changes between preview load and auto-refresh trigger.
- **Security**: Adding `https:` to the CSP `frame-src` allows iframes to load any `https:` URL. Clickjacking and XSS risks are mitigated by the iframe's sandbox attributes. **Verification required**: confirm that the `<iframe id="html-preview-frame">` in `planning.html` has `sandbox="allow-scripts allow-same-origin"` (or equivalent restrictive sandbox). If no sandbox is present, one must be added alongside the CSP change.
- **Side Effects**: Computing `configuredFolderPaths` directly from the active root's service means the webview receives only the active root's folder paths. This is the intended behavior — the modal should show folders configured for the active workspace. File enumeration across roots remains unchanged.
- **Multi-Root**: `_getLocalFolderServiceForFolder` must prioritize the active root when multiple roots configure the same folder path (e.g. a shared global path). This matches the existing priority logic in `_sendLocalDocsReady`/`_sendHtmlDocsReady` where `path.resolve(root) === path.resolve(activeRoot)` is checked first.
- **Dependencies & Conflicts**: None.

## Dependencies
None.

## Adversarial Synthesis
- **Risk Summary**: Key risks: CSP widening may allow unintended iframe loads if sandbox attributes are absent; multi-root folder-to-root resolution may misroute when the same folder path is configured in multiple roots; auto-refresh root re-resolution can misroute validation. Mitigations: verify/add iframe sandbox attributes; prioritize active root in `_getLocalFolderServiceForFolder` to resolve ambiguity; cache workspace root at preview-load time and replay during auto-refresh.

     ```typescript
     /**
      * Find the LocalFolderService for the workspace root that has the given

      * multiple roots configure the same folder path.

     private _getLocalFolderServiceForFolder(
         sourceFolder: string | undefined,
         workspaceRoot: string,
         sourceId: 'local-folder' | 'html-folder' = 'local-folder'
     ): LocalFolderService | null {
         if (!sourceFolder) { return null; }
         const allRoots = this._getWorkspaceRoots();
         const activeRoot = this._getWorkspaceRoot();

         // Try active root first (matches existing priority logic)
         if (activeRoot) {
             const service = this._getLocalFolderService(activeRoot);
             const paths = sourceId === 'html-folder'
                 ? service.getHtmlFolderPaths()
                 : service.getFolderPaths();
             const resolved = service.resolveFolderPath(sourceFolder);
             if (paths.includes(resolved)) {
                 return service;
             }
         }

         // Fall back to scanning all roots
         for (const root of allRoots) {
             if (activeRoot && path.resolve(root) === path.resolve(activeRoot)) continue; // already tried
             const service = this._getLocalFolderService(root);
             const paths = sourceId === 'html-folder'
                 ? service.getHtmlFolderPaths()
                 : service.getFolderPaths();
             const resolved = service.resolveFolderPath(sourceFolder);
             if (paths.includes(resolved)) {
                 return service;
             }
         }

         // Fallback: use the provided workspaceRoot's service (preserves current behavior)
         return this._getLocalFolderService(workspaceRoot);
     }
     ```
