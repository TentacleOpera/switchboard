# Filter Kanban Plans by Workspace without Project Assignment

This plan details changes to the Switchboard Kanban view to ensure that selecting a workspace without a project assignment filters out any plans that belong to sub-projects, only showing plans directly assigned to that workspace (unassigned to any project).

## Goal

When a user selects a workspace in the Kanban dropdown without choosing a specific project, only plans with no project assignment should be shown — not a rollup of all plans across all projects.

## Metadata

- **Tags:** [frontend, backend, bugfix]
- **Complexity:** 4

## User Review Required

- Should completed plans also be filtered by project? Currently `_refreshBoardImpl` calls `getCompletedPlans()` without any project filter, so completed plans from ALL projects always appear regardless of the active filter. This is a pre-existing inconsistency, not introduced by this change, but worth deciding whether to fix now.

## Complexity Audit

### Routine
- Replacing `null` with `KanbanDatabase.UNASSIGNED_PROJECT_FILTER` in backend initialization and message handlers
- Updating frontend dropdown change handler to pass `__unassigned__` instead of `null`
- Updating test assertion to check for `__unassigned__` instead of `null`
- Replacing `|| null` with `?? null` in backend-to-frontend message payloads (defensive)

### Complex / Risky
- Ensuring all 5 frontend call sites that convert `__unassigned__` → `null` are updated consistently; missing any one causes the same bug on a different UI path
- The `activeProjectFilter` frontend variable must track `'__unassigned__'` instead of `null` to prevent stale comparisons that trigger unnecessary `setProjectFilter` messages

## Edge-Case & Dependency Audit

- **Race Conditions:** None identified. The project filter is set synchronously before the async board refresh.
- **Security:** No security implications. The sentinel value is a UI filter, never stored as a plan's project name (already guarded in `GlobalPlanWatcherService.setCurrentProject` line 44).
- **Side Effects:** Changing `_projectFilter` default from `null` to `'__unassigned__'` means `_refreshBoardImpl` (line 1826) will now always call `getBoardFilteredByProject` instead of `getBoard`, since `'__unassigned__' !== null`. This is the desired behavior — it ensures the unassigned filter is applied.
- **Dependencies & Conflicts:** The `KanbanDatabase.UNASSIGNED_PROJECT_FILTER` constant (`'__unassigned__'`) is already defined and used by `KanbanDatabase.getBoardFilteredByProject`, `getPlansByColumn`, `getPlansWithDependencies`, and `getCompletedPlansFilteredByProject`. No new constant needed.

## Dependencies

- None

## Adversarial Synthesis

Key risks: (1) Frontend has 5+ call sites converting `__unassigned__` → `null` that must all be updated; missing any reintroduces the bug on that UI path. (2) The `activeProjectFilter` JS variable must consistently store `'__unassigned__'` instead of `null`, or dropdown comparison logic will misfire. Mitigations: enumerate all call sites explicitly; use `?? null` instead of `|| null` defensively; update `activeProjectFilter` assignments in reassign handler.
