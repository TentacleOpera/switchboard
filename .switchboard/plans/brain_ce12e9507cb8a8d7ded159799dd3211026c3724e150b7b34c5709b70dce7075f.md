# Add Design System Browser Tab to Planning Webview

## Goal

Add a new "Design System" tab in the `planning.html` panel right before "HTML Previews". This tab allows users to configure and browse a local folder containing design system guidelines (e.g. `design.md` or other text/markdown files) and image assets (e.g. `.png`, `.jpg`, `.svg`). The right-hand side should render a preview of markdown documents or image files based on selection, support editing of design system markdown files, and have the ability to set a file as the active design/planning context.

### Background Context & Problem Analysis
Currently, Switchboard has tabs for Local Docs (markdown docs), Online Docs (Clickup/Linear/Notion), HTML Previews (prototype iframe + image assets), Kanban, and Research. However, there is no dedicated tab for viewing and maintaining design systems or UI design documents alongside visual design assets (images/SVGs) without executing them in an iframe. To support product design-driven engineering, developers need a unified "Design System" view where they can browse assets (images) and read/edit specifications (`design.md`) in a side-by-side view.

The Design System tab differs from Local Docs in that it must handle **mixed content** (both text docs and image assets) in the same tree, render image previews inline (not in an iframe), and allow editing of markdown design specs. It differs from HTML Previews in that it does not use an iframe for HTML rendering — it focuses on markdown + image viewing/editing.

- Adding `designFolderPaths`, `designPreviewCollapsed` to JS state object
- Wiring up Design Modal close/backdrop/refresh/add buttons (follows HTML folder modal pattern)

### Complex / Risky
- Mixed-content file watcher: the design tab needs a glob pattern covering both text extensions and image extensions (`**/*.{md,txt,markdown,rst,adoc,png,jpg,jpeg,gif,svg}`), which is broader than either existing watcher. Must ensure the watcher doesn't double-fire with existing Local Docs or HTML Previews watchers when the same folder is configured in multiple tabs.
- Edit mode for design markdown: the `editMode`/`dirtyFlags`/`externalChangePending` state in `planning.js` currently only tracks `local` and `kanban` tabs. Adding a `design` key requires updating all tab-switch dirty-check branches and `exitEditMode`/`enterEditMode` to handle the new key, including the confirm dialog when switching away with unsaved changes.
- `_handleFetchPreview` must support `sourceId === 'design-folder'` with image detection (reuse `IMAGE_EXTENSIONS` set already in `PlanningPanelProvider`) and markdown content delivery, plus the `_setupActiveDocWatcher` for the fetched file.
- `saveFileContentResult` handler must support `tab === 'design'` for saving edited markdown back to disk.

## Edge-Case & Dependency Audit

- **Race Conditions:** File watcher debounce for design folder changes must use a separate timer (`_designDocsDebounce`) distinct from `_htmlDocsDebounce` to avoid cross-tab refresh interference. The `_sendDesignDocsReady` method must follow the same debounce pattern as `_sendHtmlDocsReady`.
- **Security:** Path traversal prevention in `fetchPreview` for `design-folder` must validate `sourceFolder` against configured design folder paths (same pattern as `html-folder` validation at line 2119-2132 of `PlanningPanelProvider.ts`).
- **Side Effects:** Adding `design-folder` as a new `sourceId` in `_handleFetchPreview` must not break existing `local-folder` or `html-folder` branches. The `_mapLocalFilesToTreeNodes` function can be reused as-is since it already maps `{id, name, kind, parentId, hasChildren, metadata}`.
- **Dependencies & Conflicts:** The `TreeNode` interface in `ResearchImportService.ts` does not need modification for this plan (no `title` field needed here — this tab uses filenames). The `fetchPreview` handler already has a `sourceId` switch; adding `design-folder` is additive. The `appendToPlannerPrompt` (Set as Active Context) and `linkToDocument` handlers must also support `design-folder` as a sourceId.
- **CSP / localResourceRoots:** Design folders may be outside the workspace (e.g. `~/design-system`). The webview's `localResourceRoots` (PlanningPanelProvider.ts line 242-248) must include configured design folder paths, otherwise `webviewUri` for images will be blocked by CSP and images won't render. This must be updated in the `open()` method and dynamically when folders are added/removed.

## Dependencies

- `sess_designSystemTab` — Design System tab implementation (this plan)

## Adversarial Synthesis

Key risks: (1) Mixed-content watcher may double-fire when a folder is configured in both Local Docs and Design System tabs, causing redundant refreshes. Mitigation: use a separate debounce timer and deduplicate by absolute path in `_sendDesignDocsReady`. (2) Edit mode state expansion introduces a new dirty-check branch that could be missed during tab switching. Mitigation: mirror the exact pattern used for `local` and `kanban` dirty checks in the tab click handler (lines 95-117 of `planning.js`). (3) Image preview in the design tab must use `webviewUri` rendering (same as HTML Previews image handling) rather than attempting UTF-8 read on binary files.
