# Add 'Assign to Workspace' Button to Kanban Board

Provide a simple way to correct plans that were assigned to the wrong workspace. This change adds an "Assign to Workspace" button next to the workspace selector dropdown in the Kanban board. When clicked, it will update the `workspace_id` field in the database for all selected plans to match the selected dropdown workspace, deselect the plans, and automatically refresh the board.

## User Review Required

> [!NOTE]
> This is a database-only mutation. No physical plan files (`.switchboard/plans/*.md`) will be moved or renamed. Only their database mappings will be updated.

> [!IMPORTANT]
> Since the plans now belong to the target workspace's ID, they will instantly disappear from the current view after the update, and appear on the board of the target workspace when that workspace is opened/selected.

```typescript
public async reassignWorkspaceByPlanFile(
    planFile: string, 
    oldWorkspaceId: string, 
    newWorkspaceId: string
): Promise<boolean> {
    const normalized = this._ensureRelativePlanFile(planFile);
    return this._persistedUpdate(
        'UPDATE plans SET workspace_id = ?, updated_at = ? WHERE plan_file = ? AND workspace_id = ?',
        [newWorkspaceId, new Date().toISOString(), normalized, oldWorkspaceId]
    );
}
```

### Backend Extension Services

#### [MODIFY] [KanbanProvider.ts](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/KanbanProvider.ts)

Add message receiver for `'reassignPlansWorkspace'` inside `_handleMessage(msg)` to resolve the target workspace ID and execute the update sequentially.

```typescript
case 'reassignPlansWorkspace': {
    const sessionIds: string[] = msg.sessionIds;
    const targetWorkspaceRoot: string = msg.targetWorkspaceRoot;
    
    if (!targetWorkspaceRoot || !Array.isArray(sessionIds) || sessionIds.length === 0) {
        break;
    }

    const db = this._getKanbanDb(targetWorkspaceRoot);
    if (await db.ensureReady()) {
        const targetWorkspaceId = await this._readWorkspaceId(targetWorkspaceRoot) 
            || await db.getWorkspaceId() 
            || await db.getDominantWorkspaceId();

        if (targetWorkspaceId) {
            let successCount = 0;
            for (const sessionId of sessionIds) {
                const plan = await db.getPlanBySessionId(sessionId);
                if (plan) {
                    const ok = await db.reassignWorkspaceByPlanFile(
                        plan.planFile, 
                        plan.workspaceId, 
                        targetWorkspaceId
                    );
                    if (ok) successCount++;
                }
            }

            await this._refreshBoard(this._currentWorkspaceRoot);
            vscode.window.showInformationMessage(
                `Successfully reassigned ${successCount} plans to the target workspace.`
            );
        }
    }
    break;
}
```

### Remaining Risks

1. **Separate-DB topology**: If source and target workspaces have different parent DBs (not shared via `workspaceDatabaseMappings`), `getPlanBySessionId` won't find the plan in the target DB, and all reassignments will fail. The 0-count warning now surfaces this, but a future enhancement should detect this case proactively (e.g., compare `resolveEffectiveWorkspaceRoot` for source vs target).
2. **UNIQUE constraint collision**: If a plan with the same `plan_file` already exists in the target workspace, the UPDATE silently fails. The partial-failure warning now surfaces this, but a future enhancement could pre-check for collisions and show a specific error.
3. **`confirm()` in webview**: The native `confirm()` dialog works in VS Code webviews but may not match the extension's visual style. A future enhancement could use a VS Code-native confirmation via message passing.
