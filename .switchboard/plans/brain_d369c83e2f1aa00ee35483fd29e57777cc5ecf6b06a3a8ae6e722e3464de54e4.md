# Unified Premium Sidebar Styling for Planning Webview

## Metadata
**Complexity:** 5
**Tags:** frontend, ui, ux, refactor

---

## Goal

Provide a visually consistent, premium experience across all sidebars in the Switchboard Planning webview. We will unify the tree views (Local Docs, Online Docs, HTML Previews) to use the dark gradient, glowing card hover/selected states, and left border highlights of the Kanban Plans tab as their styling base.

### Core Problems & Background Context
Currently, the Kanban Plans sidebar looks modern and polished, using a customized card style with:
- Dark gradient backgrounds
- Dynamic border glow on hover/selection
- Bright left border highlights

In contrast, the sidebars for Local Docs, Online Docs, and HTML Previews look plain, using flat grey backgrounds, basic borders, and simple highlight states. This inconsistency detracts from the app's overall aesthetics.

### Root Cause Analysis
The tree structures in the non-Kanban sidebars are rendered using the `.tree-node` class. However, the DOM structure nests the `.tree-children` container *inside* the parent `.tree-node` element. 
If we were to apply card backgrounds, borders, and margins directly to `.tree-node`, child items would be rendered *inside* the parent card's borders when expanded, creating an ugly nested-card appearance.
Additionally:
- Manual tree items for imported docs in [planning.js](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/webview/planning.js) use hardcoded inline styles (`style.cssText` overrides) that ignore CSS classes.
- Section headers like `.source-folder-header` and `.source-header-row` lack unified styling and visual cues.

## Complexity Audit

### Routine
- Copying `.kanban-plan-item` gradient/border/glow CSS properties to `.tree-node` — direct reuse of existing pattern
- Removing inline `style.cssText` overrides from imported doc items and source-header-row elements
- Adding `.source-folder-header` and `.source-header-row` premium styling with left accent border
- Updating `.tree-children` indentation from `margin-left: 16px` (already the current value, no change needed)
- Changing `.antigravity-artifact-node` from `padding-left: 24px` to `margin-left: 24px`

### Complex / Risky
- **DOM restructuring of `renderNode`**: Introducing `.tree-node-container` wrapper changes the parent-child relationship of `.tree-node` and `.tree-children` from parent-child to siblings. This affects 7+ call sites that query `.tree-children` inside a `.tree-node` element (expand/collapse handlers at lines 562-576, `handleChildrenReady` at line 1337, `renderNode` return value consumers at lines 967, 973, 1103, 1109, 1120, 1348, 1748). Each must be updated to traverse to the `.tree-node-container` first.
- **Collapsed sidebar CSS**: The existing `.content-row.collapsed .tree-node` rules (lines 208-217) must be extended to also handle `.tree-node-container` for margin/padding overrides, or the collapsed icon-only sidebar will break.
- **Depth-based `marginLeft` migration**: Currently set on the `.tree-node` wrapper (line 540: `wrapper.style.marginLeft`). After restructuring, this must move to `.tree-node-container` so the entire card (including its border) is indented, not just the inner row content.

## Edge-Case & Dependency Audit

- **Race Conditions**: None identified. The DOM restructuring is synchronous and the webview is single-threaded.
- **Security**: No security implications. Changes are purely visual/CSS.
- **Side Effects**: 
  - `findTreeNode()` (line 334) searches `.tree-node` elements by `data-source-id`/`data-node-id`. After restructuring, these attributes remain on the inner `.tree-node` row, so the function continues to work unchanged.
  - `state.selectedEl` is set to `.tree-node` elements in multiple places. After restructuring, the `.selected` class must remain on the `.tree-node` row (not the container) so that selection highlighting targets the visible card.
  - The `handleChildrenReady` function (line 1337) uses `parentEl.querySelector('.tree-children')` where `parentEl` is found via `findTreeNode`. After restructuring, `.tree-children` is a sibling of `.tree-node` inside `.tree-node-container`, so the query must change to `parentEl.parentElement.querySelector('.tree-children')` or `parentEl.closest('.tree-node-container').querySelector('.tree-children')`.
- **Dependencies & Conflicts**: The cyberpunk theme overrides (`.cyber-theme-enabled .kanban-plan-item`) at lines 1572-1578 should be extended to also cover `.tree-node` for consistent neon glow in cyberpunk mode.

## Adversarial Synthesis
Key risks: (1) DOM restructuring breaks 7+ selector call sites that assume `.tree-children` is a child of `.tree-node`; (2) Collapsed sidebar CSS must be updated for the new wrapper element; (3) Inline style removal scope is broader than initially specified — icon/label/controls sub-elements also have `style.cssText` overrides. Mitigations: Grep exhaustively for `.tree-children` and `.tree-node` querySelector calls before restructuring; enumerate all inline style overrides; add collapsed-state rules for `.tree-node-container`.

## Proposed Changes

### [MODIFY] planning.js — DOM Restructuring & Inline Style Removal

**Context**: The `renderNode` function (lines 524-614) creates a single `.tree-node` div containing icon, label, deleteBtn, and `.tree-children`. This nesting prevents card styling because children render inside the parent card's borders. Additionally, imported doc items (lines 1913-1965) and source-header-row elements (lines 1256-1284) use inline `style.cssText` that overrides CSS classes.

**Logic**: Introduce a `.tree-node-container` wrapper that holds `.tree-node` (the clickable row) and `.tree-children` (the child list) as siblings. Migrate all inline styles to CSS classes.

**Implementation**:

1. **Refactor `renderNode` (lines 524-614)**:
   - Create an outer `container` div with `className = 'tree-node-container'`.
   - The existing `wrapper` div retains `className = 'tree-node'` and contains only icon, label, and deleteBtn.
   - Create `childContainer` with `className = 'tree-children'` as before, but append it to `container` (not `wrapper`).
   - Move the depth-based margin from `wrapper.style.marginLeft` (line 540) to `container.style.marginLeft`.
   - Update the expand/collapse click handler (lines 562-576): Instead of `wrapper.addEventListener('click', ...)`, the handler stays on `wrapper` (the row), but the toggle target changes from `childContainer` (which is now a sibling, not a child) to the `childContainer` reference captured in closure scope — this already works since `childContainer` is in the same function scope.
   - Return `{ wrapper: container, childContainer }` instead of `{ wrapper, childContainer }` so that callers appending the result to the DOM get the container (which includes both the row and the children).

2. **Update `handleChildrenReady` (line 1337)**:
   - Change `const childContainer = parentEl.querySelector('.tree-children')` to `const childContainer = parentEl.closest('.tree-node-container')?.querySelector('.tree-children')`.
   - This is the most critical fix — without it, expanding folders will fail silently.

3. **Refactor imported doc item creation (lines 1913-1965)**:
   - Remove `wrapper.style.cssText` (line 1919) entirely — the `.tree-node` CSS class will handle layout.
   - Remove `icon.style.cssText` (line 1923) — the `.tree-node .icon` CSS rule already sets `font-size: 12px`; remove the inline `14px` override.
   - Remove `label.style.cssText` (line 1945) — the `.tree-node .label` CSS rule handles color and font-size.
   - The wrapper already has `className = 'tree-node'`, so it will pick up the new premium card styling.

4. **Refactor source-header-row creation (lines 1256-1284)**:
   - Remove `headerRow.style.cssText` (line 1258) — move all layout properties to the `.source-header-row` CSS class.
   - Remove `header.style.cssText` (line 1263) — the `.source-header` CSS class already defines most of these properties; add the missing ones (color, font-weight) to the class.
   - Remove `controlsContainer.style.cssText` (line 1269) — add a `.source-controls` CSS rule with `display: flex; align-items: center; gap: 8px`.
   - Remove `refreshBtn.style.cssText` (line 1276) — add a `.source-refresh-btn` CSS rule with the current inline values.

5. **Antigravity section (lines 1138-1218)**:
   - Session rows (line 1164: `className = 'tree-node folder-subheader'`) should NOT become cards. Add a CSS rule `.tree-node.folder-subheader` that overrides the card styling to remain a plain section header.
   - Artifact rows (line 1186: `className = 'tree-node antigravity-artifact-node'`) should become cards. These have no `.tree-children`, so no restructuring needed.

**Edge Cases**:
- The `renderNode` return value is destructured as `{ wrapper }` at lines 967, 973, 1103, 1109, 1120, 1348, 1748. After the change, `wrapper` refers to the `.tree-node-container`, so `docList.appendChild(wrapper)` will correctly append the container (which includes the row and any children). No caller changes needed.
- The `loadDocumentPreview` function (line 621) uses `findTreeNode(sourceId, docId)` which returns the `.tree-node` row. The `wrapper.dataset.sourceFolder` access on line 622 will still work since `data-source-folder` is on the row element.
