# Add Project Layer to Switchboard Kanban UI

This plan describes the implementation of a Project dropdown and "ASSIGN TO PROJECT" button in the Switchboard Kanban webview, allowing users to associate plans with specific projects, filter the board by project, and auto-assign new plans to the active project.

## User Review Required

> [!IMPORTANT]
> The database schema already supports project-level scoping via the `repo_scope` field on the `plans` table. Therefore, no SQL migration is needed. The project layer will map directly to this existing `repo_scope` field.

> [!NOTE]
> The project dropdown will populate with all unique non-empty `repo_scope` values from the workspace database. Selecting a project from the dropdown will apply a workspace filter and query the board using the existing `getBoardFiltered` method.

## Open Questions

None. The user has agreed to the design and layout of the new project sub-header.

### Backend Provider Layer

#### [MODIFY] [KanbanProvider.ts](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/KanbanProvider.ts)
- Initialize `this._repoScopeFilter` in the constructor from the VS Code workspaceState storage (`kanban.repoScopeFilter`).
- Clear the active project filter when switching workspaces in `setCurrentWorkspaceRoot`.
- In `refreshWithData`, fetch the unique project scopes list using `db.getUniqueRepoScopes` and pass it inside the `updateWorkspaceSelection` postMessage back to the webview alongside `activeFilter` (`this._repoScopeFilter`).
- Handle incoming webview messages:
  - `setProjectFilter`: update `_repoScopeFilter` state, persist it, and trigger board refresh.
  - `reassignPlansProject`: invoke `reassignPlansProject` on `TaskViewerProvider`.

#### [MODIFY] [TaskViewerProvider.ts](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/TaskViewerProvider.ts)
- Implement `_applyRepoScopeToPlanContent(content: string, repoScope: string): string` to parse a markdown plan's metadata block and add, replace, or remove the `- **Repo:** <repoScope>` line under `## Metadata`.
- Implement `reassignPlansProject(sessionIds: string[], targetProject: string): Promise<void>` to read the plan file on disk, update the metadata inline, update the database entry using `updateMetadataBatchByPlanFile`, and refresh the UI.
- Update `_createInitiatedPlan` to auto-assign the active `_repoScopeFilter` when creating a new plan or importing from clipboard.

### Frontend Webview Layer

#### [MODIFY] [kanban.html](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/webview/kanban.html)
- Add the `settings-strip` markup containing:
  - `<select id="project-select" class="workspace-select">`
  - `<button class="strip-btn is-teal" id="btn-reassign-project" disabled>ASSIGN TO PROJECT</button>`
- Update `updateWorkspaceSelection` message handler in the webview to populate `#project-select` options dynamically with projects sent from the backend, and set the active selection.
- Update `updateReassignButtonVisibility` to enable/disable `#btn-reassign-project` and update its label dynamically (e.g. `ASSIGN TO PROJECT (N)`) when card selections change.
- Handle `#project-select` `change` event to post `setProjectFilter` back to the host.

4. Select `my-test-project` from the dropdown and verify that the Kanban board filters to show only plans assigned to it.
5. Create a new plan while `my-test-project` is selected. Verify that the new plan is auto-assigned to `my-test-project` (and contains `**Repo:** my-test-project` in its metadata section on disk).
6. Select the plan, click **ASSIGN TO PROJECT** again, and select/assign to empty/unassign. Verify that it is unassigned and correctly removed from the project filter view.
