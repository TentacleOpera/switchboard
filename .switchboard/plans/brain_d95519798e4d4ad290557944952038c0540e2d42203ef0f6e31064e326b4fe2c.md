# Pause and Resume Kanban Automation Timers

This plan introduces a "Pause" / "Resume" state for the Kanban automation engine. Instead of completely turning the engine off (which clears active dispatch counters and resets the next run duration), this feature allows freezing the countdown timers at their current remaining duration and resuming them from that exact point when requested.

## Goal

Add a Pause/Resume toggle to the Kanban sub-bar that freezes all automation countdown timers in place and resumes them from the exact frozen point, without resetting dispatch counters or session state.

## Metadata

- **Tags:** [frontend, backend, UI, UX, workflow, reliability]
- **Complexity:** 5

## User Review Required

> [!NOTE]
> - **Visual Placement:** The new Pause/Resume button will be placed in the Kanban sub-bar, directly to the left of the "Reset Timers" button.
> - **Icon Rendering:** We will use a clean, self-contained SVG for the Pause/Play icon rather than introducing new binary PNG dependencies.
> - **Toggling Behavior:** When paused, the countdown timers will show the frozen remaining times. The pause button will switch its icon to a "Play" symbol and its tooltip/tooltip description to "Resume automation timers".

## Complexity Audit

### Routine
- Adding `paused` and `pausedRemainingMs` fields to `AutobanConfigState` type and normalizer
- Registering the new command in `extension.ts`
- Adding the `toggleAutobanPause` message case in `KanbanProvider._handleMessage`
- Adding the Pause/Resume button HTML and CSS in `kanban.html`
- Updating `updateAutobanButtonState()` to toggle icon/tooltip
- Adding the click event listener for the new button

### Complex / Risky
- Pause/resume timer management in `TaskViewerProvider` — mixing `setTimeout` (one-shot resume) with `setInterval` (regular ticks) in `_autobanTimers`, plus clearing the empty-column sweep timer on pause and recreating it on resume
- `_tryRestoreAutoban` must check `paused` state to avoid auto-starting the engine after extension reload while paused
- Config changes (e.g., interval minutes) while paused produce stale `pausedRemainingMs` values — must recalculate on config update

---

## Edge-Case & Dependency Audit

- **Race Conditions:** An in-flight async tick (on `_autobanTickQueue`) may update `_autobanLastTickAt` after pause reads it. Impact: remaining time may be off by a few seconds at most. Acceptable for MM:SS display. No mutex needed.
- **Security:** No new attack surface. The pause command follows the same VS Code command registration pattern as existing autoban commands.
- **Side Effects:** Pausing clears `_autobanTimers` and the empty-column sweep timer. Resuming re-creates them. Stopping the engine entirely (`enabled = false`) while paused must clear all paused state references.
- **Dependencies & Conflicts:** The `_tryRestoreAutoban` method (called on extension reload) must respect `paused` state. The `resetAutobanTimersFromKanban` method must clear paused state if called while paused. Config updates via `updateAutobanConfigFromKanban` must recalculate `pausedRemainingMs` when paused.

---

## Dependencies

- None — this is a self-contained feature addition.

## Adversarial Synthesis

Key risks: `_tryRestoreAutoban` auto-starting the engine after reload while paused (functional bug), empty-column sweep timer auto-stopping the engine while paused (data loss), and stale `pausedRemainingMs` on config changes while paused (incorrect resume timing). Mitigations: guard `_tryRestoreAutoban` with `paused` check, clear sweep timer on pause and recreate on resume, recalculate `pausedRemainingMs` on config changes while paused.
