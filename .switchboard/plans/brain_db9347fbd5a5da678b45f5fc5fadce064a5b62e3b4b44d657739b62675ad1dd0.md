# Fix HTML Preview Blank Screen in Planning Panel

Fix the HTML preview panel showing a blank screen by correcting CSS layout properties and allowing VS Code webviews to load files from configured HTML/workspace directories.

## Goal

Fix the HTML Previews tab in the Planning Panel so that selecting an HTML document renders it correctly inside the iframe instead of showing a blank/collapsed screen. Two root causes: (1) missing CSS flexbox rules for `#html-preview-content` cause the content area to collapse to zero height, and (2) `localResourceRoots` omits workspace folders, causing VS Code to block `asWebviewUri`-resolved file loads.

### Grumpy Review Findings (Stage 1)
- **CSS Selectors [NIT]**: The CSS flex properties were added correctly to `#html-preview-content`, which prevents the iframe collapse. The iframe and container inherit the height correctly now.
- **URI Encoding [MAJOR]**: The plan requested `...this._getWorkspaceRoots().map(root => vscode.Uri.file(root))`. *Are you trying to break Remote Development workspaces?* `_getWorkspaceRoots()` returns `fsPath`, which drops the `vscode-remote://` scheme. Passing a raw path through `vscode.Uri.file()` hardcodes a `file://` scheme. If a user is on WSL or SSH, the webview will block remote resources because it expects the `vscode-remote://` scheme! You should map `vscode.workspace.workspaceFolders` directly to extract the pristine `folder.uri`!

## Complexity Audit

### Routine
- Adding `#html-preview-content` to existing CSS selector groups (lines 115-128 of `planning.html`) — follows the exact same pattern already used by `#local-content`, `#online-content`, `#kanban-content`
- Adding workspace folder URIs to the `localResourceRoots` array in `open()` (line 239 of `PlanningPanelProvider.ts`) — straightforward `vscode.Uri.file()` mapping

### Complex / Risky
- `_sendHtmlDocsReady()` (line 1856-1874) rebuilds `localResourceRoots` from scratch each time it runs. If workspace folders are added in `open()` but NOT included in this dynamic rebuild, they will be overwritten and lost on the next HTML docs refresh. Both sites must be updated in concert.

## Edge-Case & Dependency Audit

- **Race Conditions**: `_sendHtmlDocsReady` is debounced by 300ms. The `refreshSource: html-folder` message is sent on webview init (planning.js line 3201). If a user clicks an HTML file before the debounced callback fires and updates `localResourceRoots`, the `asWebviewUri` call in `_handleFetchPreview` could produce a URI that the webview refuses to load. Mitigated by the short debounce window and the fact that the tree won't be populated until `htmlDocsReady` arrives anyway.
- **Security**: Adding workspace folders to `localResourceRoots` broadens the webview's file access scope. This is acceptable because (a) the webview already has script execution enabled, (b) `asWebviewUri` is the VS Code-approved mechanism for loading local files, and (c) the `_handleFetchPreview` method already validates paths against configured folder paths before resolving.
- **Side Effects**: The `_sendLocalDocsReady` method does NOT update `localResourceRoots`, but this is not a bug — local-folder previews are rendered as markdown text content (sent via `msg.content`), not via `asWebviewUri`/iframe. No changes needed there.
- **Dependencies & Conflicts**: No other plans or features depend on the HTML preview tab's CSS or `localResourceRoots` configuration.

## Dependencies

## Adversarial Synthesis

Key risks: (1) `_sendHtmlDocsReady` overwrites `localResourceRoots` from scratch, so workspace folders added only in `open()` would be lost on the next HTML folder refresh — both sites must be updated. (2) CSS specificity is safe (`#html-preview-content.active` at 1,1,0 beats `.research-tab-content.active` at 0,2,0) but should be verified during implementation. Mitigations: include workspace folders in both the initial `open()` and the dynamic rebuild in `_sendHtmlDocsReady`; use the same ID-selector pattern already proven by `#local-content` etc.

### Planning Panel Webview

#### [MODIFY] [planning.html](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/webview/planning.html)

**Context**: The `#html-preview-content` div (line 1871) has `class="research-tab-content"`. When this tab becomes active, the generic `.research-tab-content.active` rule (line 111-113) sets `display: block`, which does not support flex children. The inner `.content-row` (line 1876) has `flex: 1` but won't expand without a flex parent, causing the preview area to collapse to zero height.

Other tab content areas (`#local-content`, `#online-content`, `#kanban-content`) have explicit ID-based CSS overrides (lines 115-128) that set `display: flex`, `flex-direction: column`, `height: calc(100vh - 40px)`, and `background: var(--panel-bg)`. `#html-preview-content` is missing from these selectors.

**Logic**: Add `#html-preview-content` to both selector groups. CSS specificity ensures `#html-preview-content.active` (1,1,0) overrides `.research-tab-content.active` (0,2,0).

**Implementation**:

1. At line 115-122, add `#html-preview-content` to the selector list:
   ```css
   #local-content,
   #online-content,
   #kanban-content,
   #html-preview-content {
       display: none;
       flex-direction: column;
       height: calc(100vh - 40px);
       background: var(--panel-bg);
   }
   ```

2. At line 124-128, add `#html-preview-content.active` to the active selector list:
   ```css
   #local-content.active,
   #online-content.active,
   #kanban-content.active,
   #html-preview-content.active {
       display: flex;
   }
   ```

**Edge Cases**: The `#clipboard-content` and `#notebook-content` tabs (lines 306-319) have their own separate styling with `padding` and `gap` — they should NOT be merged into this selector group. Only `#html-preview-content` needs to be added to the existing group.

### Extension Services

#### [MODIFY] [PlanningPanelProvider.ts](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/PlanningPanelProvider.ts)

**Context**: The webview's `localResourceRoots` controls which local files the webview is allowed to load via `asWebviewUri`. Currently, only the extension's own `dist`, `webview`, and `node_modules` directories are included at creation time (line 239). The `_sendHtmlDocsReady` method dynamically adds configured HTML folder paths (line 1856-1874), but workspace folders are never included. If an HTML file references resources from the workspace (CSS, images), or if the HTML folder itself is under the workspace, VS Code will block the load.

**Logic**: Add all workspace folder URIs to `localResourceRoots` in both the initial `open()` call and the dynamic rebuild in `_sendHtmlDocsReady()`.

**Implementation**:

1. In `open()` method (line 239-244), add workspace folder URIs to `localResourceRoots`:
   ```typescript
   localResourceRoots: [
       vscode.Uri.joinPath(this._extensionUri, 'dist'),
       vscode.Uri.joinPath(this._extensionUri, 'webview'),
       vscode.Uri.joinPath(this._extensionUri, 'node_modules'),
       ...this._getWorkspaceRoots().map(root => vscode.Uri.file(root))
   ]
   ```

2. In `_sendHtmlDocsReady()` (line 1856-1874), add workspace folder URIs to the `baseRoots` array so they survive the dynamic rebuild:
   ```typescript
   const baseRoots = [
       vscode.Uri.joinPath(extensionUri, 'dist'),
       vscode.Uri.joinPath(extensionUri, 'webview'),
       vscode.Uri.joinPath(extensionUri, 'node_modules'),
       ...this._getWorkspaceRoots().map(root => vscode.Uri.file(root))
   ];
   ```

**Edge Cases**:
- `_getWorkspaceRoots()` returns an empty array when no workspace is open, so the spread operator safely produces no additional entries.
- Workspace folder URIs are already `vscode.Uri.file()` instances from `folder.uri.fsPath`, matching the pattern used for `extraRoots`.
- The `_sendLocalDocsReady` method does NOT need `localResourceRoots` updates because local-folder previews are rendered as text content (not via `asWebviewUri`).
