# Subtask Navigation and Parent Restoration in Project Panel

Provide subtask back-navigation in the Project Integration tab (ClickUp / Linear). Currently, clicking 'Back to List' when inspecting a subtask detail returns the user all the way to the root task list. We want to introduce a sibling button 'Back to Parent Task' that appears dynamically only when viewing a subtask. Clicking it will seamlessly load the parent task's details.

## Goal

Add a "← BACK TO PARENT" button to the task detail view in the Project Integration sidebar panel that appears only when the currently inspected task is a subtask (i.e., has a valid `parentId`). Clicking it loads the parent task's detail view via the existing `loadLinearTaskDetails` / `loadClickUpTaskDetails` functions.

## User Review Required

> [!NOTE]

Both 'Back to List' and 'Back to Parent Task' will exist side-by-side to allow the user a choice between step-by-step upward navigation and immediate exit.

> [!NOTE]

The UUID-based path (line 800–819 in `LinearSyncService.ts`) is missing `parent { id }` in its GraphQL query — this is the only gap to fix. The identifier-based path (line 822+) uses `_buildIssueListQuery()` which **already includes** `parent { id }` (line 420) — no change needed there.

> [!IMPORTANT]
> **ClickUp parentId must be verified** before implementing the ClickUp render path. The plan assumes `selectedClickUpIssue.task.parentId` is populated — but this depends on the ClickUp backend normalizer flattening `parent.id` to `parentId`. Coder must confirm this field exists on the ClickUp task object before coding the UI logic.

### Complex / Risky
- **ClickUp `parentId` provenance**: If the ClickUp task normalizer (backend) doesn't expose `parentId` as a flat field on the task object, the button will silently never appear for ClickUp subtasks. Must verify before writing render logic.
- **Parent not in loaded list**: When clicking "Back to Parent", the parent task may not be in the currently loaded `linearProjectIssues` list (filtered out, different project). `loadLinearTaskDetails` will still succeed via a fresh API call, but the loading-card DOM query at line 3717 will return `null` and the spinner won't appear on a card. This is a cosmetic gap — the detail will load correctly.

## Edge-Case & Dependency Audit

- **Race Conditions**: The existing 30-second safety timeout in `loadLinearTaskDetails` / `loadClickUpTaskDetails` handles stalled requests. No new async risk introduced.
- **Security**: No new user input is accepted. `parentId` is read from the task object returned by the backend — not from user-supplied text. XSS is not a concern.
- **Side Effects**: The new button's `hidden` class must be reset correctly on every render call in both `renderSidebarLinearTaskDetail` and `renderSidebarClickUpTaskDetail`. Failing to hide the button when a root-level task is loaded would leak stale `data-parent-id` from a previous view.
- **Dependencies & Conflicts**: The `backButton` variable in `createProjectPanel()` references `#sidebar-task-back-button`. The new button `#sidebar-task-back-to-parent-button` is a distinct element — no naming collision. `renderSidebarLinearProjectPanel()` at line 3685 mutates `backButton.textContent` — this only affects the "BACK TO LIST" button, not the new parent button.
