# Kanban Automation Bug Fixes: Timer Minimum & Column Isolation

Two critical bugs in the Kanban automation feature:
1. The automation timer interval cannot be set below 5 minutes in single-column mode, although it should support down to 1 minute now that terminal pools exist.
2. In both single-column and multi-column automation modes, cards are simultaneously moved from multiple columns at once instead of being isolated to the ticking source column. The root cause is `_getAutobanReviewerLaneColumns`, which expands shared reviewer columns (LEAD CODED, CODER CODED) into a combined lane — causing cards from both columns to be processed when only one column's timer fires.

## Goal

Fix both automation modes to enforce a 1-minute minimum timer interval (single-column only — multi-column already supports 1 min) and strictly isolate card processing to the ticking source column only, in both single-column and multi-column modes.

## Metadata

- **Tags:** [bugfix, frontend, backend, workflow]
- **Complexity:** 5

## User Review Required

> [!IMPORTANT]
> - **Single-Column Isolation**: Running single-column automation will strictly isolate transitions to the configured source column only. All other columns will be ignored/treated as disabled during the run.
> - **Multi-Column Isolation**: In multi-column mode, each column's tick will only process cards from that specific column. The shared reviewer lane expansion (LEAD CODED + CODER CODED combined) will be removed — each column ticks independently and processes only its own cards.
> - **Timer Minimum**: Both single-column and multi-column modes will support a minimum interval of 1 minute. (Multi-column already does; single-column needs the fix.)

### Complex / Risky
- Filtering `_getEnabledAutobanSourceColumns` for single-column mode: affects the empty-column sweep (`_stopAutobanIfNoValidTicketsRemain` → `_autobanHasEligibleCardsInEnabledColumns`), session exhaustion checks, and the tick engine — all must behave consistently
- Removing `_getAutobanReviewerLaneColumns` shared-reviewer lane expansion for BOTH single-column and multi-column modes: the shared reviewer lane (LEAD CODED + CODER CODED combined) is the root cause of cross-column processing in both modes. Removing it means `shouldSkipSharedReviewerAutobanDispatch` dedup logic becomes a no-op (harmless — the `_autobanTickQueue` serialization and `_activeDispatchSessions` map already prevent concurrent/duplicate dispatch)
- `_startAutobanEngine` single-column timer isolation: the root cause is that `normalizeAutobanConfigState` merges `singleColumnSyntheticRules` with `DEFAULT_AUTOBAN_RULES`, re-enabling all columns in `this._autobanState.rules`. ## Edge-Case & Dependency Audit

- **Race Conditions**: `_stopAutobanEngine` (line 6882) resets `_autobanTickQueue = Promise.resolve()` (line 6894), which could discard an in-flight tick if a mode switch triggers stop/start in rapid succession. However, mode switches are user-initiated and non-concurrent, so this is not a real risk.
- **Security**: No security implications — these are internal automation timing and filtering changes.
- **Side Effects**: The empty-column sweep timer (`_autobanEmptyColumnSweepTimer`, line 6868) calls `_stopAutobanIfNoValidTicketsRemain` → `_autobanHasEligibleCardsInEnabledColumns` → `_getEnabledAutobanSourceColumns`. If `_getEnabledAutobanSourceColumns` is not filtered for single-column mode, the sweep checks ALL columns and may never auto-stop (because cards in non-source columns keep it alive). This is why the `_getEnabledAutobanSourceColumns` fix is critical.
- **Dependencies & Conflicts**: Multi-column mode already uses `min = '1'` and `Math.max(1, ...)` (kanban.html line 6987-6998) and `normalizeFiniteCount(..., 1)` (autobanState.ts line 207). The single-column path is the only one with the 5-minute floor. No conflict between modes. The shared reviewer lane removal affects both modes equally — `_autobanTickQueue` serialization and `_activeDispatchSessions` already prevent duplicate dispatch, so the lane expansion is redundant.

## Adversarial Synthesis

Key risks: (1) `_getEnabledAutobanSourceColumns` must be filtered for single-column mode or the empty-column sweep and exhaustion checks will operate on wrong columns; (2) removing `_getAutobanReviewerLaneColumns` shared-reviewer lane expansion for both modes makes `shouldSkipSharedReviewerAutobanDispatch` a no-op, but the `_autobanTickQueue` serialization and `_activeDispatchSessions` map already prevent duplicate dispatch. Mitigations: the single-column guard is on `automationMode`; the lane removal is unconditional (same bug exists in both modes) and the existing serialization mechanisms make the shared lane redundant.

### Stage 2: Balanced Synthesis

The plan's original assessment that `shouldSkipSharedReviewerAutobanDispatch` "effectively becomes a no-op (harmless)" was **incorrect**. The guard is not harmless — it creates a cross-column false-skip scenario because:

1. `_autobanLaneLastDispatchAt` uses a shared `'coded-reviewer'` key for both LEAD CODED and CODER CODED dispatches.
2. `shouldSkipSharedReviewerAutobanDispatch` compares `lastDispatchAt >= latestTickAt` for the `reviewerLaneColumns` list.
3. With isolation, `reviewerLaneColumns = [sourceColumn]` only, so `latestTickAt` reflects only the ticking column's previous tick time.
4. But `lastDispatchAt` may have been set by the OTHER shared reviewer column's dispatch — causing a false "already dispatched" conclusion.

**Fix applied**: Remove the entire shared-reviewer dedup mechanism (guard block, dispatch timestamp write, field declaration, unused imports). The `_autobanTickQueue` serialization and `_activeDispatchSessions` map already prevent concurrent/duplicate dispatch, making the shared-reviewer dedup redundant even before isolation — and actively harmful after it.
