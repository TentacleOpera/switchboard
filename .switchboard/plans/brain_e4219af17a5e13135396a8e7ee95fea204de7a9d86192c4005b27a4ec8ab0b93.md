# Duplicated & Redundant Watchers — Analysis

> [!IMPORTANT]
> I traced every event handler chain end-to-end. Below are the **confirmed duplications** — watchers/timers that do the same work as another watcher/timer and can be consolidated or removed.

---

## 1. 🔴 `state.json` — 4 watchers, 2 are redundant

**What's watching:**

| # | Location | Type | Handler |
|---|----------|------|---------|
| A | [extension.ts:1546](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/extension.ts#L1546) | VS Code FSW | `syncTerminalRegistryWithState()` → `taskViewerProvider.refresh()` |
| B | [extension.ts:1565](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/extension.ts#L1565) | Native `fs.watch` | Same handler as A (calls `debouncedStateSync`) |
| C | [TaskViewerProvider.ts:8872](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/TaskViewerProvider.ts#L8872) | VS Code FSW | `_refreshConfiguredPlanWatcher()` → `this.refresh()` |
| D | [TaskViewerProvider.ts:8897](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/TaskViewerProvider.ts#L8897) | Native `fs.watch` | Same handler as C (calls `refreshState`) |

**The duplication:**
- **A calls `taskViewerProvider.refresh()`** — the exact same method C calls via `this.refresh()`. Both trigger a full board re-render.
- A also calls `syncTerminalRegistryWithState()` which is the only unique work — but this could be merged into C's handler.
- B is a native fallback for A. D is a native fallback for C. The native fallbacks themselves duplicate their VS Code counterparts.

**Verdict:** 
- ✂️ **Remove A + B** (the `extension.ts` pair). Move the `syncTerminalRegistryWithState()` call into TaskViewerProvider's `refreshState` handler so it's done in one place.
- This eliminates **2 watchers** and prevents **double `refresh()` calls** on every state.json change.

## 2. 🔴 `.switchboard/plans/**/*.md` — 5+ watchers, major overlap

**What's watching the same plan files:**

| # | Service | Type | On create/change handler |
|---|---------|------|-------------------------|
| A | [GlobalPlanWatcher:298](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/GlobalPlanWatcherService.ts#L298) | VS Code FSW | `_handlePlanFile()` → parses metadata, **upserts into DB**, fires `onPlanDiscovered` |
| B | [GlobalPlanWatcher:342](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/GlobalPlanWatcherService.ts#L342) | Native `fs.watch` | Same as A |
| C | [TaskViewerProvider:9040](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/TaskViewerProvider.ts#L9040) | VS Code FSW | `_handlePlanCreation()` → parses plan, **upserts into DB**, creates runsheet |
| D | [TaskViewerProvider:9111](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/TaskViewerProvider.ts#L9111) | Native `fs.watch` | Same as C |
| E | [PlanningPanelProvider:527](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/PlanningPanelProvider.ts#L527) | VS Code FSW | Triggers `fetchKanbanPlans` re-fetch for planning panel UI |

**The duplication:**

When a single `.md` file is created/changed in `.switchboard/plans/`:

1. **A fires** → reads file, parses metadata, does DB `upsertPlans()`, fires `onPlanDiscovered`
2. **B fires** → same as A (native fallback for same service)
3. **C fires** → reads file, parses plan, does **separate** DB upsert via `_registerPlan()`, creates runsheet
4. **D fires** → same as C (native fallback)
5. **E fires** → triggers a full kanban plan list re-fetch

That's potentially **2 DB upsert operations** (A+C) for the same file, plus 2 native fallback events that mostly re-trigger the same work.

A and C have internal dedup guards (`_pendingCreations`, `_planCreationInFlight`) but those only catch **self-initiated** writes. An externally dropped plan file **does** hit both handlers.

**Additionally:** GlobalPlanWatcher runs a **10-second periodic filesystem scan** ([line 124](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/GlobalPlanWatcherService.ts#L124)) that does the exact same `_scanForNewFiles()` → `_handlePlanFile()` work that its own file watchers already handle. The scan exists as a safety net for watcher reliability, but it means:
- Every 10 seconds: directory listing + `fs.stat()` per file + DB query for all existing plans
- This runs **even when the file watchers are working perfectly**

**Verdict:**
- ✂️ **Remove the GlobalPlanWatcher native `fs.watch`** (B) — it duplicates A and the 10-second scan already acts as a safety net
- ✂️ **Remove the TaskViewerProvider native `fs.watch` for plans** (D) — it duplicates C, and GlobalPlanWatcher already covers missed events via its periodic scan
- ✂️ **Increase the periodic scan interval** from 10s to 60-120s — it's a safety net for missed events, not the primary discovery path. The file watchers (A + C) are the primary path.
- Consider longer-term: **consolidate A and C** so only GlobalPlanWatcher does DB ingestion, and TaskViewerProvider subscribes to `onPlanDiscovered` for runsheet creation instead of running its own parallel watcher

## 4. 🟠 GlobalPlanWatcher periodic scan vs. its own file watchers

This deserves special emphasis because it's the **single biggest CPU waster**:

```
File watchers (A)  →  detect plan file events  →  _handlePlanFile()  →  DB upsert
                                                                            ↓
Periodic scan      →  readdir + stat every 10s →  _handlePlanFile()  →  DB upsert (same)
```

The periodic scan at [line 124](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/GlobalPlanWatcherService.ts#L124) does:
1. `fs.promises.readdir()` recursively on every mapped workspace's `.switchboard/plans/`
2. `fs.promises.stat()` on every `.md` file found
3. `db.getAllPlans()` to get existing plans
4. Compares and calls `_handlePlanFile()` for new ones

**This runs every 10 seconds even when the watchers already caught everything.** The only scenario where this catches something the watchers missed is when VS Code's `FileSystemWatcher` silently drops an event (rare).

**Verdict:**
- ✂️ **Set default to 120s or disable by default** — this is a safety net, not a primary detection mechanism. The watchers handle 99.9% of cases.
