# Duplicated & Redundant Watchers — Analysis

## Goal

Reduce the constant background cost of Switchboard's watcher/scan machinery **without changing plan-pickup latency or removing any event-coverage layer**, and clean up redundant UI around manual scanning. Four concrete deliverables:

1. **Cheapen the periodic scan instead of slowing it** — keep the 10s cadence (instant-feeling pickup preserved), but eliminate the per-cycle `db.getAllPlans()` query and per-file `fs.stat()` by tracking seen paths in memory. ~95% of the scan's CPU cost disappears with zero latency change.
2. **Eliminate the double `refresh()` per state.json change** — consolidate the two extension.ts state.json watchers into TaskViewerProvider's existing (kept) watcher pair, so each state change triggers exactly one sync + one board re-render.
3. **Repurpose the kanban toolbar "Import plans" button** (`btn-import-plans`, kanban.html:2215) into a **"Scan folders now"** button. Its current function — scan configured folders and pick unclaimed plans — is an exact duplicate of the Import button in planning.html's Kanban Plans tab (`btn-import-kanban-plans`, planning.html:3131); both invoke `switchboard.importUnclaimedPlans`. The kanban one becomes an immediate on-demand scan (fires `GlobalPlanWatcher.triggerScan` across all watch folders, then refreshes the board).
4. **Unify scan timing under the existing Plan Scanner tab in setup.html** — a full Plan Scanner UI already exists (setup.html:478 tab; markup ~1003–1065 with enable toggle, "Scan Speed" dropdown 3s–60s, IDE presets, custom sources), but it only governs the *external-sources* scanner (`switchboard.planScanner.*`). The internal `.switchboard/plans` fallback scan (`switchboard.planWatcher.*`, GlobalPlanWatcher) has no UI at all. Rather than adding a second, confusingly-similar control elsewhere, extend the existing "Scan Speed" dropdown to govern **both** timers (and add an "Off" option). Both services already react live to config changes (`GlobalPlanWatcherService.ts:88-94`; TVP `startPlanScanner` listener at `TaskViewerProvider.ts:11320-11327`), so this is a settings-write change only.

**Root cause (original trace preserved in full under Original Analysis):** watcher redundancy accreted in layers — each service added a VS Code FSW plus a native `fs.watch` fallback for the same paths, and `GlobalPlanWatcherService` added a 10s periodic scan on top. The original plan proposed removing fallback watchers and slowing the scan to 120s; review showed both moves trade reliability/latency for CPU in an extension whose core promise is *instant* plan pickup from gitignored and out-of-workspace directories where VS Code's FSW is documented (in this codebase's own comments) to miss events. **All native-watcher removals and the 120s default are therefore explicitly deferred** (see Deferred Items); this revision keeps every detection layer and removes cost instead.

## Edge-Case & Dependency Audit

- **Race Conditions:**
  - state.json write-vs-read race: extension.ts's removed watcher carried a 200ms debounce specifically to avoid "Unexpected end of JSON input" (`extension.ts:1534-1536`). TVP's `refreshState` currently has **no** debounce — the debounce must be ported, not dropped.
  - `syncTerminalRegistryWithState` self-coalesces via `syncInFlight`/`syncPending` (`extension.ts:1493-1511`); safe to call repeatedly from the hook.
  - Scan vs. watcher race on the same new file: unchanged — both already route through `_debounceHandleFile` (per-file 300ms coalescing), and the seen-set only changes *whether* the scan looks at a path, not how events are handled.
  - "Scan folders now" pressed while a periodic scan is running: `triggerScan` is independent of the `_scanInProgress` guard (it re-handles all files unconditionally). Double work is possible but idempotent (upserts). Acceptable for a manual button.
- **Security:** None — no new inputs beyond a settings dropdown writing two pre-declared, range-validated settings.
- **Side Effects:**
  - `switchboard.importUnclaimedPlans` command stays registered (extension.ts:878) — planning.html's Import button still routes to it via `PlanningPanelProvider.ts:1571`. Only the kanban webview entry point changes.
  - Seen-set memoization means the periodic scan no longer re-checks *known* files against the DB each cycle. The scan never updated existing plans anyway (it skips paths already in the DB), so behavior for known files is unchanged; only wasted work is removed.
  - Removing extension.ts's `fs.watch` also removes its `.switchboard` mkdir side effect (`extension.ts:1554-1563`); TVP's `_setupStateWatcher` performs an equivalent mkdir (`TaskViewerProvider.ts:8893-8895`), so bootstrap is preserved.
- **Dependencies & Conflicts:**
  - `GlobalPlanWatcherService` config listener (`:88-94`) already restarts the scan when `scanIntervalMs`/`periodicScanEnabled` change — the Setup dropdown needs no service-side changes.
  - `package.json` declares `scanIntervalMs` min 2000 / max 300000 — all preset values fit.
  - Webpack rebuild (`npm run compile`) required before manual verification — deferred per session directive.
  - Dev-only project: no migrations or back-compat shims; clean breaks preferred.

## Dependencies

- None

## Adversarial Synthesis

Key risks: (1) the state.json consolidation could reintroduce a documented JSON-parse race or strand events fired before the sync hook is wired — mitigated by porting the 200ms debounce, null-checking the hook, and keeping both of TVP's state watchers (FSW + native); (2) the seen-paths scan cache could miss a deleted-then-recreated plan file — mitigated by rebuilding the set from each cycle's listing so recreated paths register as new. All native-watcher removals from the original analysis are deferred, so plan-pickup latency and event coverage are unchanged by this revision.

**The duplication:**

When a single `.md` file is created/changed in `.switchboard/plans/`:

1. **A fires** → reads file, parses metadata, does DB `upsertPlans()`, fires `onPlanDiscovered`
2. **B fires** → same as A (native fallback for same service)
3. **C fires** → reads file, parses plan, does **separate** DB upsert via `_registerPlan()`, creates runsheet
4. **D fires** → same as C (native fallback)
5. **E fires** → triggers a full kanban plan list re-fetch

That's potentially **2 DB upsert operations** (A+C) for the same file, plus 2 native fallback events that mostly re-trigger the same work.

A and C have internal dedup guards (`_pendingCreations`, `_planCreationInFlight`) but those only catch **self-initiated** writes. An externally dropped plan file **does** hit both handlers.

The scan exists as a safety net for watcher reliability, but it means:
- Every 10 seconds: directory listing + `fs.stat()` per file + DB query for all existing plans
- This runs **even when the file watchers are working perfectly**

**Verdict:**
- ✂️ **Remove the GlobalPlanWatcher native `fs.watch`** (B) — it duplicates A and the 10-second scan already acts as a safety net
- ✂️ **Remove the TaskViewerProvider native `fs.watch` for plans** (D) — it duplicates C, and GlobalPlanWatcher already covers missed events via its periodic scan
- ✂️ **Increase the periodic scan interval** from 10s to 60-120s — it's a safety net for missed events, not the primary discovery path. The file watchers (A + C) are the primary path.
- Consider longer-term: **consolidate A and C** so only GlobalPlanWatcher does DB ingestion, and TaskViewerProvider subscribes to `onPlanDiscovered` for runsheet creation instead of running its own parallel watcher

> **Revision 2:** All three removals/changes **deferred** (see Deferred Items): B is the sole watcher for non-workspace mapped folders; D removal orphans runsheet creation on missed FSW events (TVP doesn't subscribe to `onPlanDiscovered`); the scan is create-only (skips DB-known paths, `:177`) so it cannot back up change-events, and 60-120s worst-case pickup is a UX regression. ### 4. 🟠 GlobalPlanWatcher periodic scan vs. its own file watchers

This deserves special emphasis because it's the **single biggest CPU waster**:

```
File watchers (A)  →  detect plan file events  →  _handlePlanFile()  →  DB upsert
                                                                            ↓
Periodic scan      →  readdir + stat every 10s →  _handlePlanFile()  →  DB upsert (same)
```

The periodic scan at [line 124](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/GlobalPlanWatcherService.ts#L124) does:

2. `fs.promises.stat()` on every `.md` file found

4. Compares and calls `_handlePlanFile()` for new ones

**This runs every 10 seconds even when the watchers already caught everything.** The only scenario where this catches something the watchers missed is when VS Code's `FileSystemWatcher` silently drops an event (rare).

**Verdict:**
- ✂️ **Set default to 120s or disable by default** — this is a safety net, not a primary detection mechanism. The watchers handle 99.9% of cases.

> **Revision 2:** **Interval change rejected — cadence stays 10s.** The waste is steps 2–3 (stat-per-file + `getAllPlans()` every cycle), not the readdir; a recursive readdir on a small plans dir is near-free. ## Proposed Changes

### 1. `src/services/GlobalPlanWatcherService.ts` — cheap periodic scan (cadence unchanged)

- **Context:** `_scanForNewFiles` (lines 139–197) runs every cycle: `getAllPlans()` DB query, recursive readdir, `fs.stat()` on every `.md` file, path-set comparison. The DB query and stats execute even when nothing changed.
- **Logic:** Memoize the previous cycle's file listing per workspace. A cycle's real work happens only for paths not present last cycle. Deleted-then-recreated files re-register as new because the set is rebuilt from each cycle's listing.
- **Implementation:**
  1. Add field: `private _scanSeenPaths = new Map<string, Set<string>>(); // workspaceRoot → absolute .md paths from last scan cycle`
  2. Rework `_scanForNewFiles(workspaceRoot)`:
     - Recursive readdir collecting all `.md` absolute paths into `currentPaths: Set<string>` (keep the existing `scanDir` walker, but collect paths instead of processing inline).
     - `const prevPaths = this._scanSeenPaths.get(workspaceRoot);`
     - `this._scanSeenPaths.set(workspaceRoot, currentPaths);`
     - **First cycle** (`prevPaths === undefined`): preserve current behavior exactly — run the existing DB-check + stat + mtime-guard logic over all files (this is what imports files that predate extension startup but are missing from the DB). Seed complete.
     - **Subsequent cycles:** `newPaths = [...currentPaths].filter(p => !prevPaths.has(p))`. If empty → return (no DB query, no stats — this is the steady-state cycle: one readdir, nothing else). Otherwise run the existing logic (`getAllPlans()` existing-path sets, `fs.stat` mtime guards at lines 179–183, `_debounceHandleFile`) over `newPaths` only.
  3. Clear the workspace's entry in `_scanSeenPaths` when its watcher is disposed in `_refreshWatchers` (folder removed from config), so re-adding a folder re-seeds.
  4. `triggerScan` (manual, line 553) is unaffected — it deliberately re-handles every file.
- **Edge Cases:**
  - File created and deleted between cycles: never seen — same as today (watchers handle it or it's gone).
  - Path case/normalization: store paths normalized the same way the walker produces them (`path.join` output) — comparison is internal-set-only, no cross-source comparison.
  - Memory: a few hundred strings per workspace — negligible.

### 2. `src/extension.ts` + `src/services/TaskViewerProvider.ts` — single state.json pipeline

#### `src/extension.ts`

- **Context:** State-watcher block at lines 1533–1573 (FSW + `debouncedStateSync` + native `fs.watch` on `.switchboard/`). `syncTerminalRegistryWithState` closure at 1493–1511 with `syncInFlight`/`syncPending`/`resolveSyncCompletionWaiters` — the closure stays here; only the *invocation* moves.
- **Implementation:**
  1. Delete lines 1533–1573 (`stateSyncTimer`/`debouncedStateSync`, the `stateWatcher` FSW + subscriptions, the `fsStateWatcher` native fallback). TVP's `_setupStateWatcher` already mkdirs the state dir (`TaskViewerProvider.ts:8893-8895`), so the bootstrap mkdir at 1554–1563 is preserved elsewhere. Verify `isAllowedSwitchboardLocation` has other callers before touching it; if this was its only use, remove it too (clean break).
  2. In the same `if (workspaceRoot)` block (after the initial `await syncTerminalRegistryWithState(runtimeStateRoot);` at line 1518), wire the hook:
     ```ts
     taskViewerProvider.setStateSyncHook(async () => {
         const currentRoot = kanbanProvider!.getCurrentWorkspaceRoot();
         const stateRoot = currentRoot ? (resolveEffectiveStateRoot(currentRoot) || currentRoot) : runtimeStateRoot;
         await syncTerminalRegistryWithState(stateRoot);
     });
     ```
  3. Keep `onDidOpenTerminal` (1521), `onDidCloseTerminal` (1528), and the window-focus sync (1581) untouched — recovery paths.

#### `src/services/TaskViewerProvider.ts`

- **Context:** `_setupStateWatcher` at 8865–8906; `refreshState` at 8876–8880; self-write guard `_selfStateWriteUntil` (field at 269, set at 1626). **Both watchers (FSW + native) are kept.**
- **Implementation:**
  1. Add field + setter (match existing style, cf. `setKanbanProvider` at 1655):
     ```ts
     private _stateSyncHook?: () => Promise<void>;
     public setStateSyncHook(hook: () => Promise<void>): void {
         this._stateSyncHook = hook;
     }
     ```
  2. Rework `refreshState` to add the ported debounce and hook call:
     ```ts
     // Debounce: state.json is written frequently; without it, syncs can race the
     // write itself and cause "Unexpected end of JSON input" parse errors.
     let stateRefreshTimer: NodeJS.Timeout | undefined;
     const refreshState = () => {
         if (Date.now() < this._selfStateWriteUntil) return; // suppress self-triggered events
         if (stateRefreshTimer) clearTimeout(stateRefreshTimer);
         stateRefreshTimer = setTimeout(() => {
             void this._refreshConfiguredPlanWatcher();
             const sync = this._stateSyncHook ? this._stateSyncHook() : Promise.resolve();
             void sync.catch(() => { /* sync errors are non-fatal */ }).finally(() => this.refresh());
         }, 200);
     };
     ```
- **Edge Cases:**
  - Events before the hook is wired (TVP constructor runs `_setupStateWatcher` at line 420): null-check falls through to plain refresh; the explicit activation-time `syncTerminalRegistryWithState` call covers the startup window.
  - `_setupStateWatcher` re-entry (also called at 3376): fresh timer closure per call; old watchers disposed at method top; a stale 200ms timer firing once is harmless.
  - Self-write suppression now also skips terminal sync for ≤500ms windows — accepted; `onDidOpenTerminal` and window-focus syncs cover the gap.
