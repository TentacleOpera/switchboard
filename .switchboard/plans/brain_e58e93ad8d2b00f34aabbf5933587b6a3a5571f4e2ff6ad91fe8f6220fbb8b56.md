# Setup Webview Theme Tab and Docs Button Relocation

## Goal
The `setup.html` layout is being optimized to group user settings and visual preferences more logically. The "Visual Theme" section is being moved from the Artifacts Panel tab into its own dedicated "Theme" tab (replacing the sparse "Docs" tab), and the "Open Docs" button is being relocated to the Setup tab's "Switchboard guide" section for better grouping. Additionally, the animation toggle UI is being changed from negative wording ("Disable Cyber Theme Animation") to positive wording ("Enable Artifacts panel animation"). Since the animation defaults to ON for new users, the checkbox will default to checked.

**Root Cause Analysis:** The current "Docs" tab contains only a single "OPEN DOCS" button — an entire tab for one action is poor use of tab bar real estate. The "Visual Theme" settings are buried at the bottom of the Artifacts Panel tab, which is primarily about source selection and document caching — theme settings are conceptually unrelated. The negative wording ("Disable Cyber Theme Animation") is confusing because the checkbox being checked means animation is OFF, which is counterintuitive. These issues stem from incremental feature additions without a holistic tab organization pass.

## User Review Required
- **Theme radio buttons are cosmetic-only for now.** This plan adds theme radio buttons with "Afterburner" as the only real option, persisted via `vscode.setState` (webview-local only, not a VS Code setting). If you want theme selection to persist across panel reopenings or be configurable via VS Code settings, that requires additional backend work not included here. Confirm this scope is acceptable.
- **Conditional animation visibility.** When a non-Afterburner theme is selected, the animation settings section will be hidden. The backend `disableCyberAnimation` setting is NOT changed when switching themes — it remains as-is. This means if a user enables animation, switches to a non-Afterburner theme (where animation settings are hidden), then switches back to Afterburner, their animation preference is preserved. Confirm this behavior is desired.

### Complex / Risky
- Checkbox state inversion logic (positive UI ↔ negative backend config) requires careful mapping with explicit naming and comments
- Persisted tab state migration: users with `activeTabId: 'docs'` in `vscode.getState()` will see no active tab after rename — must remap 'docs' → 'theme' in `initTabs()`
- Conditional show/hide of animation settings based on theme selection requires JS logic that correctly reads radio state on load and on change

## Edge-Case & Dependency Audit

- **Race Conditions:** If the Theme tab is activated before the backend responds with `cyberAnimationDisabledSetting`, the checkbox may briefly show an incorrect state. The existing `runSetupHydration()` wrapper handles this — no new race condition, but the inversion logic must be applied inside the hydration callback.
- **Security:** No new security concerns. Theme selection is local to the webview.
- **Side Effects:** Moving `#btn-open-docs` to a different DOM container does not break its event listener (line 2807 uses `getElementById`), but the element must exist in the DOM at the time the listener is bound. Since setup.html binds listeners at the bottom of the script, the element will be present regardless of which container it's in.
- **Dependencies & Conflicts:** The `tabLoadCallbacks` currently has a pre-existing bug: the Artifacts Panel tab uses `data-tab="artifacts-panel"` but its callback is registered under key `'planning-panel'` (line 1436), so the callback never fires. This plan does NOT fix that bug — it's noted as a known issue for a separate plan. The new `'theme'` callback must use the correct key matching the `data-tab` attribute.

## Dependencies
- None

## Adversarial Synthesis
Key risks: (1) Persisted tab state with `activeTabId: 'docs'` will cause a blank panel after rename — requires migration in `initTabs()`. (2) Checkbox inversion logic between positive UI wording and negative backend config is error-prone — requires explicit variable naming and inline comments. (3) Theme radio button selection has no persistence beyond `vscode.setState`, meaning it resets on panel reopen. Mitigations: Add tab ID migration, use `isDisabled` variable naming with comments, and clearly document that theme persistence is webview-local only for this iteration.

**JavaScript — Theme radio button logic**
- Add change listener on theme radio buttons to toggle animation settings visibility:
  ```js
  document.querySelectorAll('input[name="theme-selection"]').forEach(radio => {

          const animationSettings = document.getElementById('theme-animation-settings');
          if (animationSettings) {
              // Only show animation settings for Afterburner theme

  });
  ```
