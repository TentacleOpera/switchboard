# Imported Plan

🐛 Bug Report: Complexity Display Desync
I have analyzed the codebase and the plan file to track how the metadata is parsed and displayed.

1. The ID Mismatch in Database Lookup (KanbanProvider.ts)
Inside KanbanProvider.getComplexityFromPlan(), the code attempts to retrieve the complexity from the Kanban Database as a "Secondary priority". It hashes the file path using SHA-256 (const planId = crypto.createHash(...)) and passes this hash to db.getPlanBySessionId(planId).

The Bug: The session_id column in the database stores the file's basename (e.g., brain_077...), not a hash. Because the lookup searches for a hash instead of the filename, the database lookup silently returns null, skipping the true database record and forcing a fallback to text-parsing for every refresh.

2. Complexity Repair / DB Self-Healing Bypass (_refreshBoardImpl)
When the UI triggers _refreshBoardImpl and encounters an Unknown complexity from the database, it attempts to "self-heal" the DB record by manually re-parsing the markdown text using the **Complexity:** Low regex.

The Bug: While **Complexity:** Low does successfully parse out of the markdown content using the regex /\*\*Complexity:\*\*\s*(Low|High)/i, the repair step (db.updateMetadataBatch()) is failing to reflect accurately on the Kanban view because the ID lookup is out of sync. This means the internal memory fallback isn't patching the final cards sent to the frontend.

3. State Desync Between Providers
The Ticket Dropdown is governed by a different provider (TaskViewerProvider), which accurately resolves the Low complexity assignment because its lookup mechanisms properly parse the local state/markdown without relying on the broken DB lookup.
The Kanban Webview relies heavily on the KanbanDatabase state to hydrate its cards. Because the DB lookup queries using the wrong ID variant (Hash vs Basename), KanbanProvider continues sending the default Unknown value in the updateBoard frontend payload.
Summary Fix Requirements: The primary fix involves correcting the parameter passed to db.getPlanBySessionId inside getComplexityFromPlan(). The Kanban backend must query using the actual session_id (the basename) instead of using a hashed directory path.
