# Add Reset Timer Button to Kanban Automation Subbar

## Goal

Add a "Reset Timer" button next to the automation countdown timers in the Kanban sub-bar. When clicked, it resets all active automation countdown intervals to their full duration and immediately dispatches the next plan(s) via the existing tick queue.

## Metadata

- **Tags:** [frontend, UI, workflow]
- **Complexity:** 3

## User Review Required

No breaking changes or high-risk design decisions are proposed. The feature is additive and follows existing message/command patterns.

## Open Questions

No open questions remain. The requirement is clear and fits naturally with the existing decoupled timer architecture.

## Complexity Audit

### Routine
- Adding an HTML button element to the existing sub-bar (follows `btn-autoban` pattern at line 2144)
- Adding a `case` handler in `KanbanProvider._handleMessage` (follows `toggleAutoban` pattern at line 4376)
- Registering a VS Code command in `extension.ts` (follows `setAutobanEnabledFromKanban` pattern at line 1080)
- Adding a public method on `TaskViewerProvider` (follows `resetAutobanPoolsFromKanban` pattern at line 5993)
- Controlling button visibility inside `updateAutobanButtonState()` (existing function at line 4383)

### Complex / Risky
- **Reset must avoid race condition with in-flight ticks.** Calling `_startAutobanEngine()` directly would call `_stopAutobanEngine()` which clears `_autobanTickQueue` and `_activeDispatchSessions`, potentially allowing a double dispatch. Instead, the reset method must only clear the `setInterval` timers, reset `_autobanLastTickAt`, and enqueue a new tick on the *existing* queue — preserving serialization and dispatch guards.

---

## Edge-Case & Dependency Audit

- **Race Conditions:** In-flight `_autobanTickColumn` calls must not be disrupted. The reset must enqueue on the existing `_autobanTickQueue` rather than replacing it. `_activeDispatchSessions` must NOT be cleared during reset (unlike full engine stop).
- **Security:** No security implications — this is a UI-triggered backend action following the existing webview→command→provider pattern.
- **Side Effects:** Reset enqueues an immediate tick per enabled column, which will dispatch plans to terminals. This is the intended behavior. Multiple rapid clicks could enqueue multiple ticks; a 2-second visual cooldown on the button (disable after click, re-enable on next `updateAutobanConfig` message) prevents accidental spam.
- **Dependencies & Conflicts:** The `antigravity-batch` mode does not use interval timers, so the reset button must be hidden in that mode (consistent with `btn-autoban` visibility logic at line 4394). If automation is enabled but all rules are disabled, the reset button appearing is harmless (no rules fire, enqueue is a no-op).

## Dependencies

None — this feature is self-contained within the existing automation subsystem.

## Adversarial Synthesis

Key risks: (1) double-dispatch race if reset calls `_startAutobanEngine()` naively, (2) missing `extension.ts` command registration. Mitigations: implement a targeted `resetAutobanTimersFromKanban()` that preserves the tick queue and dispatch guards; follow the established command registration pattern. - **Line ~5993** (after `resetAutobanPoolsFromKanban`) — Implement `resetAutobanTimersFromKanban()`:
  ```typescript
  /** Called by Kanban reset-timer button to restart countdown intervals and fire an immediate tick. */
  public async resetAutobanTimersFromKanban(): Promise<void> {
      if (!this._autobanState.enabled) { return; }

      // Clear only the setInterval timers — do NOT clear the tick queue or dispatch guards.
      for (const [, timer] of this._autobanTimers) {
          clearInterval(timer);
      }
      this._autobanTimers.clear();

      // Restart each enabled column's interval with a fresh timestamp and immediate tick.
      const { rules, batchSize } = this._autobanState;
      for (const [column, rule] of Object.entries(rules)) {
          if (!rule.enabled) { continue; }
          if (this._autobanState.automationMode === 'single-column' &&
              column !== this._singleColumnAutobanState.sourceColumn) {
              continue;
          }
          const intervalMs = Math.max(rule.intervalMinutes, 1) * 60 * 1000;
          this._autobanLastTickAt.set(column, Date.now());

          // Enqueue immediate tick on the EXISTING queue (preserves serialization)
          this._enqueueAutobanTick(column, batchSize);

          const timer = setInterval(() => {
              this._enqueueAutobanTick(column, batchSize);
          }, intervalMs);
          this._autobanTimers.set(column, timer);
      }

      this._postAutobanStateNow();
  }
  ```
  **Critical:** This method does NOT call `_stopAutobanEngine()` (which would clear `_autobanTickQueue` and `_activeDispatchSessions`, risking double dispatch). It only clears the `setInterval` handles and re-creates them, preserving the serialization queue and dispatch session guards.
