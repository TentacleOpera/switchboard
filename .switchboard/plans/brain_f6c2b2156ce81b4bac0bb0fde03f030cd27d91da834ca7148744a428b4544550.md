# Planning Panel Local Docs and Multi-Root Workspace Synchronization

## Goal

Resolve the issues in the "Artifacts" (planning) panel where:
1. The **Local Docs** tab starts in a blank "Loading" state and fails to show any configured folders or local documents on initial load.
2. Configuration of local, HTML, and design folders is bound to specific workspace folders instead of being global, causing them to disappear or change based on which editor is focused.
3. Multi-root workspace folders are scanned but there is no mechanism to filter the local, HTML, or design files by repository.

**Root Cause Analysis:**
- **Issue 1 (blank Loading state):** The webview sends `fetchRoots` on startup (planning.js line 4109), which triggers `_handleFetchRoots()` → `_sendLocalDocsReady()`. However, `_sendLocalDocsReady()` has a content dedup check (PlanningPanelProvider.ts lines 2096-2104) that skips re-posting when the signature is unchanged. On initial load, `_lastLocalDocsSignature` is `''`, so the first send should succeed — but the webview's message listener may not be attached yet when the backend responds, causing the response to be lost. Additionally, `fetchRoots` does not explicitly send `listLocalFolders` or `refreshSource: 'local-folder'`, so the Local Docs tab relies entirely on the `localDocsReady` message from `fetchRoots`, which can be lost in the timing gap.
- **Issue 2 (workspace-folder-scoped settings):** The three folder path settings (`research.localFolderPaths`, `research.htmlFolderPaths`, `research.designFolderPaths`) are declared with `"scope": "resource"` in package.json (lines 334, 341, 348), and `LocalFolderService._getConfigurationTarget()` (line 58-61) returns `WorkspaceFolder` when the root is a known workspace folder. This means each workspace folder stores its own folder paths, and switching the active editor changes which paths are visible.
- **Issue 3 (no workspace filter):** The backend already tags each file node with `metadata.root` (PlanningPanelProvider.ts line 2011) via `_mapLocalFilesToTreeNodes()`, but the webview has no UI to filter by workspace root.

## User Review Required

> [!IMPORTANT]
> **Configuration Target Change**
> Changing folder settings (`research.localFolderPaths`, `research.htmlFolderPaths`, `research.designFolderPaths`) from workspace-folder scoped target to user-level global configuration target will mean that the same set of folders (e.g. `designs`, `docs`) will resolve relative to all workspace roots in any workspace, or be absolute paths. This is the desired behavior for a consistent global setting, but will write to the user's global `settings.json`.

> [!NOTE]
> **Relative Path Resolution in Multi-Root**
> With global settings, a relative path like `"docs"` is stored once in the global config. Each `LocalFolderService` instance resolves it against its own `_workspaceRoot` (LocalFolderService.ts line 77), so `"docs"` becomes `/rootA/docs` for root A and `/rootB/docs` for root B. This is the desired behavior — each root's "docs" folder is distinct. The `resolveFolderPath()` method already handles this correctly and does not need to change.

### Complex / Risky
- Threading `workspaceItems` through the full data flow: backend payload → message handler (`handleLocalDocsReady`, `handleHtmlDocsReady`, `designDocsReady`) → render functions (`renderLocalDocs`, `renderHtmlDocs`, `renderDesignDocs`) → dropdown population and node filtering — this touches 3 parallel render pipelines and must be consistent across all three
- Existing workspace-scoped settings in `.vscode/settings.json` become lower-priority after the scope change; users may not realize their per-workspace overrides are being ignored in favor of the global value. No data loss occurs, but the behavioral change may surprise users who intentionally set different folders per workspace.

## Edge-Case & Dependency Audit

- **Race Conditions:** The initial `fetchRoots` message (planning.js line 4109) fires immediately when the IIFE executes. If the webview's `onDidReceiveMessage` listener is not yet attached on the extension host side, the message is lost and the panel stays blank. Adding a redundant `refreshSource: 'local-folder'` after `fetchRoots` provides a second chance for the backend to respond once the listener is ready.
- **Security:** No new security concerns. The workspace filter dropdown only filters already-scanned files; it doesn't expose new file system paths.
- **Side Effects:** Changing `package.json` scope from `"resource"` to `"application"` affects all users of the extension, not just those with multi-root workspaces. Single-root users will see their folder paths move from `.vscode/settings.json` to global `settings.json` on next edit. This is the intended behavior for consistency.
- **Dependencies & Conflicts:** The `_buildKanbanWorkspaceItems()` method (PlanningPanelProvider.ts line 789) is already used by the Kanban tab and returns `Array<{ label: string; workspaceRoot: string }>`. Reusing this same method for the workspace filter dropdowns ensures label consistency across tabs. No conflict with existing Kanban usage.

## Adversarial Synthesis

Key risks: (1) Missing `package.json` scope change would silently break the entire feature — VS Code ignores `ConfigurationTarget.Global` writes when the setting is declared as `resource`-scoped. (2) The `workspaceItems` data flow must be threaded through three parallel render pipelines consistently; a missed step in any pipeline would leave that tab's filter broken. (3) Existing per-workspace settings become lower-priority after the scope change, which may surprise users who intentionally configured different folders per workspace. Mitigations: The package.json change is a simple 3-line edit that must be verified first; the render pipeline changes follow an identical pattern across all three tabs, reducing the chance of inconsistency; the scope change is documented in User Review Required above.

      const select = document.getElementById(selectElementId);
      if (!select) return;
      const current = selectedValue || '';
      select.innerHTML = '<option value="">All Workspaces</option>';
      for (const item of workspaceItems) {
          const option = document.createElement('option');
          option.value = item.workspaceRoot;
          option.textContent = item.label;
          if (item.workspaceRoot === current) {
              option.selected = true;
          }
          select.appendChild(option);
      }
  }
