# Implementation Plan - Fix Plan Import Clipboard Error

Ensure that plan renames (such as those triggered by title changes after clipboard imports) do not suffer from race conditions with the `GlobalPlanWatcherService` file deletion detection, nor result in stale `session_id` references or duplicate DB rows.

## User Review Required

> [!IMPORTANT]
> The solution uses a transient suppression mechanism (`_recentRenames` registry) within the `GlobalPlanWatcherService` to ignore deletion events for recently-renamed plans. The suppression entry automatically expires after 2 seconds.

## Proposed Changes

### Database Layer

#### [MODIFY] [KanbanDatabase.ts](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/KanbanDatabase.ts)
- Add a new public method `updateSessionId(oldSessionId: string, newSessionId: string): Promise<boolean>` that updates the `session_id` of a plan row to the new relative plan path.

### Watcher Layer

#### [MODIFY] [KanbanProvider.ts](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/KanbanProvider.ts)
- Add a public getter `globalPlanWatcher` to access the `GlobalPlanWatcherService` from other components like `TaskViewerProvider`.

#### [MODIFY] [GlobalPlanWatcherService.ts](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/GlobalPlanWatcherService.ts)
- Add a transient `_recentRenames` `Set<string>` to track recently renamed files' old relative paths.
- Add a public method `registerRename(oldRelativePath: string): void` that registers a rename and schedules its eviction from `_recentRenames` after 2000 milliseconds.
- Update `_handlePlanDelete` to check `_recentRenames` and skip database row deletion if the file was just renamed.

### Extension Host Layer

#### [MODIFY] [TaskViewerProvider.ts](file:///Users/patrickvuleta/Documents/GitHub/switchboard/src/services/TaskViewerProvider.ts)
- Update `_renameSessionPlanFile` to:
  - Register the old relative plan path with `GlobalPlanWatcherService.registerRename` BEFORE renaming the file on disk.
  - Call `db.updateSessionId(sessionId, nextRelative)` right after `db.updatePlanFile` to ensure the session ID is updated in SQLite.
- Update the `savePlanText` request handler in `updateReviewTicket` to:
  - Track the current active session ID (using `activeSessionId`).
  - Use `activeSessionId` instead of the original request's stale `sessionId` for all downstream operations including run sheet updates, metadata updates, and `getReviewTicketData(activeSessionId)`.
