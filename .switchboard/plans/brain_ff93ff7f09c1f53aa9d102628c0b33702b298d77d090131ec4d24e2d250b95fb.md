# Live Feed Accordion Exclusivity

## Goal
Include the **Live Feed** accordion (`#panel-activity`) in the sidebar's mutually exclusive navigation system. Currently, the Terminal Operations, Setup (startup), and Database Operations accordions call `collapseAllAccordions()` before opening, but the Live Feed does not — meaning it can remain open while another accordion is expanded, violating the exclusivity invariant.

## Metadata
**Tags:** frontend, UI
**Complexity:** Low

## User Review Required
> [!NOTE]

## Complexity Audit

### Routine
- Add `data-accordion="true"` to `#panel-activity` so `collapseAllAccordions()` can find and collapse it
- Update the `liveFeedToggle` click handler to call `collapseAllAccordions('panel-activity')` before toggling open, matching the exact pattern used by `termOpsToggle`, `startupToggle`, and `db-sync-toggle-btn`

### Complex / Risky
- None

## Edge-Case & Dependency Audit

- **Race Conditions:** None. All DOM mutations are synchronous within a single click handler. No async operations or message round-trips are involved.
- **Security:** None. No user data, no IPC, no external calls.
- **Side Effects:**
  - When another accordion calls `collapseAllAccordions()`, it will now also collapse Live Feed if open. This is the desired behaviour — it completes the exclusivity contract.
  - `collapseAllAccordions()` inspects `aria-hidden` via `hasAttribute('aria-hidden')`. `#panel-activity` already has `aria-hidden="true"` in the HTML, so the utility will correctly update it to `"true"` on collapse.
  - The toggle handler already sets `aria-hidden` manually (`liveFeedFields.setAttribute('aria-hidden', isOpen ? 'false' : 'true')`). After adding `data-accordion="true"`, collapses triggered via `collapseAllAccordions()` will also update `aria-hidden` automatically (the utility already handles this at line 2254–2256). No double-update issue — the utility only acts if the panel `hasAttribute('aria-hidden')`, and sets it to `'true'`.
  - `#panel-activity` uses both classes `tab-panel` and `panel-fields`. The `collapseAllAccordions()` utility only queries `[data-accordion="true"]` — the class names are irrelevant to the selector. No unintended panels will be targeted.
- **Dependencies & Conflicts:**
  - **"Sidebar Exclusive Accordion Navigation"** (`CODE REVIEWED`) — the plan that introduced `collapseAllAccordions()` is already shipped. This plan is an additive follow-up with zero conflict.
  - No other active Kanban plans touch `#panel-activity` or the Live Feed toggle.

## Adversarial Synthesis

### Grumpy Critique
*"Two changes. TWO. And somehow you still nearly botched it. Let me enumerate your near-misses:*

*1. **You didn't check whether `#panel-activity` has the `open` class at startup.** The terminal-operations-fields starts `open` by default (`class="panel-fields open"`), but `#panel-activity` starts without it (`aria-hidden="true"`, no `open` class). Fine — but what if a future developer adds `open` to `#panel-activity` at startup to pre-open the feed? `collapseAllAccordions()` would immediately wipe it on first interaction with any other accordion. Not a bug today, but the plan is silent on this constraint.*

*2. **The handler calls `collapseAllAccordions('panel-activity')` only when `willOpen === true`.** The original handlers for `termOpsToggle` and `startupToggle` guard it with `if (willOpen)`. This is correct — you don't want to call collapse when the user is closing the panel. The original plan says 'call `collapseAllAccordions` before opening' without specifying the conditional guard. An inattentive coder could call it unconditionally and trigger a collapse of all other accordions every time the user closes the Live Feed — an annoying and incorrect UX.*

*3. **`aria-hidden` is managed in two places.** The live handler already does `liveFeedFields.setAttribute('aria-hidden', isOpen ? 'false' : 'true')`. When `collapseAllAccordions()` collapses `#panel-activity` due to another accordion opening, it sets `aria-hidden='true'` via its own code path. This is actually fine — they agree — but neither the original plan nor the spec explicitly calls this out. If someone reads the code naively they might think there's a conflict.*"

### Balanced Response
The Grumpy critique is valid on point (2) — the implementation spec below is explicit: the call to `collapseAllAccordions('panel-activity')` is guarded by `if (willOpen)`, exactly matching the established pattern. Points (1) and (3) are acknowledged as documentation/awareness items: the plan notes that `#panel-activity` must NOT have the `open` class at startup (matching current state), and explicitly documents that `aria-hidden` is handled in both the direct toggle and the utility with consistent results. No code changes are needed for these, but the coder must not remove the existing `liveFeedFields.setAttribute('aria-hidden',…)` line — it provides the direct-close path.

**Change B — JavaScript toggle handler (~line 2286)**

  3. Then toggle the `open` class on `liveFeedFields` and sync `liveFeedChevron`.
  4. Update `aria-hidden` as before (this direct path handles the case where the user closes the panel — the utility does not run on close).

```diff
+                    // Collapse all other accordions before opening Live Feed

+                }
+                const isOpen = liveFeedFields.classList.toggle('open');
```

Replacement:
```
        if (liveFeedToggle && liveFeedFields && liveFeedChevron) {
            liveFeedToggle.addEventListener('click', () => {
                const willOpen = !liveFeedFields.classList.contains('open');
```
