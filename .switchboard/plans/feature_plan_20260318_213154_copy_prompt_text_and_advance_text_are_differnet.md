# copy prompt text and advance text are differnet

## Goal
When using the kanban buttons, the prompts generated by the "Copy Prompt" and "Advance" buttons are different, even for same types of plans. They should be identical to ensure consistency whether a user copies the prompt manually or lets the system auto-dispatch it.

## Proposed Changes
1. **Centralize Prompt Generation**: Extract `_buildKanbanBatchPrompt` from `src/services/TaskViewerProvider.ts` (around line 2310) into its own shared utility file (e.g., `src/services/agentPromptBuilder.ts`) OR expose it as a public static/instance method.
2. **Adopt Unified Prompt in KanbanProvider**: Modify `src/services/KanbanProvider.ts` to use this centralized `_buildKanbanBatchPrompt` instead of its own `_generateBatchPlannerPrompt` and `_generateBatchLowComplexityPrompt` methods.
3. **Handle Kanban Columns -> Roles**: Since `_buildKanbanBatchPrompt` requires a `role` (planner, coder, lead, reviewer), `KanbanProvider` should map the target column to the corresponding role using its existing `_columnToRole` or similar logic before generating the prompt.
4. **Preserve Dynamic Complexity (Plan 1 Integration)**: Ensure the unified prompt builder incorporates the dynamic wording for high vs. low complexity plans correctly.

### Step 4: Resolve role mapping conflicts
**Current Issue**: Two different column-to-role mappings exist:
- `KanbanProvider._columnToRole` (line 1333): PLAN REVIEWED → planner
- `TaskViewerProvider._autobanColumnToRole` (line 2203): PLAN REVIEWED → lead

**Resolution**: 
- `_columnToRole` is for **manual column transitions** (user clicks "move to next stage")
- `_autobanColumnToRole` is for **autoban routing** (automatic dispatch based on complexity)
- The unified prompt builder should use the **autoban role mapping** for consistency with auto-dispatch behavior
- Update `KanbanProvider._generatePromptForColumn` to use autoban role mapping when generating copy prompts

## Adversarial Review

**Grumpy Critique**: 
"This plan wants to 'centralize' prompt generation but doesn't address the fundamental architectural mismatch: `KanbanProvider` operates on `KanbanCard` objects with relative paths and column IDs, while `TaskViewerProvider` operates on session sheets with absolute paths and role strings. You can't just 'extract' `_buildKanbanBatchPrompt` and call it from both places—the data structures are incompatible. Also, the plan says 'map column to role using `_columnToRole`' but that method returns different roles for the same column depending on context (line 1333 maps PLAN REVIEWED → planner, but line 2203 maps PLAN REVIEWED → lead). Which one is correct? The plan doesn't say. And what about the `focusDirective` and `batchExecutionRules`—do those apply to the Copy Prompt button too, or only to auto-dispatch? This is a mess."

**Balanced Synthesis**: 
Valid concerns about data structure mismatches and role mapping inconsistencies. The solution requires a **data adapter layer** that normalizes both `KanbanCard` and session sheet data into a common format (`{ topic: string; absolutePath: string; complexity?: string }`) before calling the unified prompt builder. The role mapping conflict is real and must be resolved:
- `_columnToRole` (KanbanProvider line 1333) is for **manual column transitions**
- `_autobanColumnToRole` (TaskViewerProvider line 2203) is for **autoban routing**
- The unified prompt builder should use the **autoban role mapping** for consistency

The `focusDirective` and `batchExecutionRules` should apply to **all batch prompts** (both Copy and Advance) for consistency. Implementation steps now include explicit data adapters for both contexts, role mapping reconciliation, and extraction of all helper method dependencies. This is correctly classified as Band B complexity due to multi-file coordination and data structure normalization requirements.
