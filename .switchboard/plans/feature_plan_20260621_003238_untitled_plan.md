# Fix: Create Plan Opens Raw Markdown File Instead of Project Panel with Edit Mode

## Goal

When the user clicks a "Create" plan button (in kanban.html or project.html), three things are wrong:
1. It opens a raw markdown file in the editor instead of the project panel's plan view with edit mode ON.
2. It auto-inserts placeholder headings (`## Goal`, `## Proposed Changes`, `## Verification Plan`, `## Open Questions` with `- TODO` bullets) that the user must manually delete.
3. The filename is always `feature_plan_<timestamp>_untitled_plan.md` because the title is hardcoded as `'Untitled Plan'` â€” even when the user writes a real plan, the filename never updates.

This plan fixes all three: (1) stop opening the raw `.md` file and instead open the project panel with edit mode auto-activated, (2) create the plan with just a single H1 header, (3) prompt for a title at creation time and rename the file on save when the H1 changes.

### Problem Analysis

**Symptom:** Clicking "Create" in the Kanban tab (project.html) or the Kanban board (kanban.html) opens a raw markdown file in the VS Code editor with pre-filled placeholder headings and a useless `untitled_plan` filename. The user expected the project panel's plan preview to open with edit mode enabled, a blank plan body, and a meaningful filename.

**Root cause chain:**

1. The "Create" button sends a `createPlan` message to the extension.
2. Both `PlanningPanelProvider.ts` (line 2439) and `KanbanProvider.ts` (line 5748) handle this by calling `vscode.commands.executeCommand('switchboard.initiatePlan')`.
3. `switchboard.initiatePlan` (extension.ts:675-677) calls `taskViewerProvider.createDraftPlanTicket()`.
4. `createDraftPlanTicket()` (TaskViewerProvider.ts:15429-15454) does three things:
   a. Calls `_buildDraftPlanContent('Untitled Plan')` (line 15432) which generates a template with four placeholder headings each containing `- TODO`.
   b. Calls `activatePlanInProjectPanel()` (line 15446) which sends `activateKanbanTabAndSelectPlan` to the project webview â€” this correctly switches to the Kanban tab and selects the new plan in the sidebar.
   c. Calls `_openPlanInReviewPanel()` (line 15448) which executes `vscode.commands.executeCommand('vscode.open', vscode.Uri.file(planFileAbsolute))` â€” this opens the raw `.md` file in the editor as plain text, stealing focus from the project panel.

5. The user sees the raw markdown file in the editor (step 4c) rather than the project panel's plan preview (step 4b), because the editor tab opens on top of the project panel.
6. The plan file contains the four placeholder headings (step 4a) which the user must manually delete before writing their plan.
7. The title is hardcoded as `'Untitled Plan'` (line 15430). The filename is generated by `_createInitiatedPlan()` (line 15987-15988): `feature_plan_${timestamp}_${slug}.md` where `slug = _toPlanSlug(title)`. Since the title is always `'Untitled Plan'`, the slug is always `untitled_plan`. The filename is never updated even if the user changes the H1 in the editor.

**Why the filename never updates:** The save handler (`saveFileContent` in PlanningPanelProvider.ts:3204-3289) writes content to the existing file path but never checks whether the H1 has changed or renames the file. The kanban DB stores `plan_file` as a column, and there's already a `db.updatePlanFile(sessionId, newRelativePath)` method (KanbanDatabase.ts:1662) plus an established rename pattern in extension.ts:3068-3092 â€” the save handler just doesn't use them.

**Why the raw file opens:** `_openPlanInReviewPanel()` (TaskViewerProvider.ts:15425-15427) unconditionally calls `vscode.open` on the plan file. This was likely intended as a fallback for when the project panel isn't available, but it fires even when `activatePlanInProjectPanel()` succeeds.

**Why placeholder headings are inserted:** `_buildDraftPlanContent()` (TaskViewerProvider.ts:15400-15424) hardcodes the four-section template. The user wants a blank canvas.

### Constraints

- `activatePlanInProjectPanel()` is only called from `createDraftPlanTicket()` â€” no other callers.
- `_openPlanInReviewPanel()` is only called from `createDraftPlanTicket()` â€” no other callers.
- `_buildDraftPlanContent()` is only called from `createDraftPlanTicket()` â€” no other callers.
- The project panel's edit mode is managed by `enterEditMode('kanban')` in project.js.
- The `kanbanPlanPreviewReady` handler in project.js (line 236-256) populates the preview content and enables the Edit button â€” this is the trigger point for auto-entering edit mode.
- The kanban DB already has `updatePlanFile(sessionId, newRelativePath)` (KanbanDatabase.ts:1662) â€” proven by extension.ts:3088.
- The save handler (`saveFileContent` in PlanningPanelProvider.ts:3204) already has the content and file path â€” adding a rename is ~15 lines.
- No confirmation dialogs (per CLAUDE.md).
- Must rebuild after editing `src/webview/*` (`npm run compile`).

### Complex / Risky

- **Slug algorithm duplication** â€” the rename-on-save inlines slug logic that already exists in `_toPlanSlug()` (TaskViewerProvider.ts:15387). The inlined copy lacks the `|| 'new_plan'` fallback. See Adversarial Synthesis and Proposed Changes Â§6 for the mitigation (extract to shared utility or add fallback).

## Edge-Case & Dependency Audit

**Race Conditions:**
- **TOCTOU on rename:** The original plan used `fs.existsSync(newPath)` then `fs.promises.rename()`. Between the check and the rename, another process could create a file at the target path. **Mitigation:** Use try/catch on the rename directly (matching the established pattern in extension.ts:3068-3092) instead of existsSync-then-rename. See Proposed Changes Â§6 (refined).
- **`_pendingAutoEdit` stale state:** If the user creates a plan but clicks a different plan before `kanbanPlanPreviewReady` fires, `_pendingAutoEdit` would still be true and auto-enter edit mode on the wrong plan. **Mitigation:** Clear `_pendingAutoEdit = false` in the plan-item click handler (project.js:736-741) before `loadKanbanPlanPreview`.
- **Concurrent saves:** If two webview panels save the same plan simultaneously, both could attempt a rename. The try/catch approach handles this â€” the second rename fails (file no longer exists at old path) and is caught non-fatally.

**Security:**
- The `showInputBox` title is sanitized by `_toPlanSlug()` (strips non-alphanumerics) before being used in a filename. No path traversal risk â€” the slug cannot contain `/`, `..`, or other path separators.
- The rename target path is constrained to `path.dirname(resolved)` (the plans directory), which is already within an allowed workspace root (validated by the `isAllowed` check at line 3223). No escape outside the workspace.

**Side Effects:**
- **Kanban DB `plan_file` column mutation:** The rename updates the DB via `updatePlanFile()`. If the DB lookup fails (plan not yet scanned), the file is renamed on disk but the DB retains the old path. The plan scanner will reconcile on the next sweep â€” it scans the plans directory by filename and inserts/updates entries. Non-fatal but the sidebar may briefly show a stale filename until the next `fetchKanbanPlans` refresh.
- **`fetchKanbanPlans` refresh after save:** The refined `saveFileContentResult` handler sends `fetchKanbanPlans` to refresh the sidebar list. This is an additional message round-trip on every kanban/epics save, even when no rename occurred. Acceptable â€” the plan list is small and the refresh is cheap.
- **Editor tabs:** If the raw `.md` file was previously open in an editor tab (from the old `_openPlanInReviewPanel` behavior), removing that call means no editor tab opens. This is the intended behavior â€” the project panel is the sole editing surface.

**Dependencies & Conflicts:**
- **`_toPlanSlug()` duplication:** The rename-on-save logic needs the same slug algorithm as `_toPlanSlug()` (TaskViewerProvider.ts:15387-15393). Since `_toPlanSlug` is a private method on TaskViewerProvider and the rename runs in PlanningPanelProvider, the slug logic must either be extracted to a shared utility or duplicated with the `|| 'new_plan'` fallback. See Proposed Changes Â§6 for the chosen approach.
- **`KanbanDatabase.forWorkspace()` + `ensureReady()` + `getPlanByPlanFile()` + `updatePlanFile()`:** All verified to exist with the expected signatures. `forWorkspace` is used 21 times in PlanningPanelProvider.ts â€” established pattern.
- **`activatePlanInProjectPanel()` callers:** Only called from `createDraftPlanTicket()` (verified). Adding the `autoEdit` parameter with a falsy default is safe for any future callers.
- **`_openPlanInReviewPanel()` callers:** Only called from `createDraftPlanTicket()` (verified). Safe to remove.
- **`_buildDraftPlanContent()` callers:** Only called from `createDraftPlanTicket()` (verified). Safe to simplify.
- **Plan scanner reconciliation:** When a plan is renamed on save before the scanner has inserted it into the DB, `getPlanByPlanFile` returns null and the DB update is skipped. The file is renamed on disk. The scanner sweeps the plans directory on its next run and will pick up the new filename. Whether it cleans up a stale DB entry for the old filename depends on scanner orphan-cleanup logic â€” this is a known gap, documented but non-fatal (the content is saved, the file is renamed, the DB reconciles eventually).

## Dependencies

- None â€” this plan is self-contained. All referenced APIs (`showInputBox`, `fs.promises.rename`, `KanbanDatabase.forWorkspace`, `updatePlanFile`, `getPlanByPlanFile`, `activatePlanInProjectPanel`, `enterEditMode`) are already present in the codebase and verified.

## Adversarial Synthesis

Key risks: (1) slug algorithm duplicated across TaskViewerProvider and PlanningPanelProvider without the `|| 'new_plan'` fallback â€” will drift and produce empty slugs on all-special-char H1s; (2) TOCTOU race on rename via existsSync-then-rename â€” less robust than the established try/catch pattern in extension.ts:3068; (3) `renamedFilePath` relative path computed with potentially-undefined `_getWorkspaceRoot()` instead of the already-resolved `wsRoot` â€” produces absolute paths in multi-root workspaces; (4) removing `_openPlanInReviewPanel()` with no fallback if the project panel fails to open â€” user sees nothing. Mitigations: extract/add slug fallback, use try/catch rename, use `wsRoot` for relative path, keep a degraded fallback. All four are reflected in the refined Proposed Changes below.

## Proposed Changes

### 1. `src/services/TaskViewerProvider.ts` â€” Prompt for title at creation time

**Context:** `createDraftPlanTicket()` (line 15430) hardcodes `const title = 'Untitled Plan'`. The filename and H1 are derived from this title, so they're always `untitled_plan`.

**Implementation:** Show a VS Code input box before creating the plan. If the user enters a title, use it. If they cancel or enter nothing, fall back to `'Untitled Plan'`.

Replace lines 15429-15432:

```ts

    const title = await vscode.window.showInputBox({
        prompt: 'Plan title (used for filename and H1 header)',
        placeHolder: 'Untitled Plan',
        value: 'Untitled Plan',
        validateInput: (v) => {
            // Reject titles that would produce an empty slug
            if (!v || !v.trim()) return null; // allow empty (falls back to Untitled Plan)
            return null;
        }
    }) || 'Untitled Plan';
    const createdAt = new Date().toISOString();
    const idea = this._buildDraftPlanContent(title);
```

**Edge Cases:**
- **User presses Escape / cancels:** `showInputBox` returns `undefined` â†’ falls back to `'Untitled Plan'`. No plan is created if we return early. Actually, we should create the plan with the default title â€” the user might just want to skip the prompt. So `|| 'Untitled Plan'` is correct.
- **User clears the input and hits Enter:** `showInputBox` returns `''` â†’ `|| 'Untitled Plan'` kicks in. Same as cancel.
- **User enters a title with special characters:** `_toPlanSlug()` (line 15387-15393) strips non-alphanumerics and replaces with underscores. E.g., "Fix: Login Bug!" â†’ `fix_login_bug`. Safe.

### 6. `src/services/PlanningPanelProvider.ts` â€” Rename plan file on save when H1 changes

**Context:** The `saveFileContent` handler (line 3204-3289) writes content to the existing file path but never renames the file even if the H1 has changed. This means a plan created as "Untitled Plan" stays `untitled_plan.md` forever, even after the user writes a real plan with a meaningful H1.

The codebase already has the rename pattern (extension.ts:3068-3092): `fs.promises.rename(old, new)` + `db.updatePlanFile(sessionId, newRelative)`. The save handler already has the content and file path â€” adding a rename is straightforward.

**Implementation:** After the `fs.promises.writeFile(resolved, content, 'utf8')` call (line 3284), before sending `saveFileContentResult`, add rename logic.

**Refinements from adversarial review (applied to the code below):**
1. **Slug fallback:** Added `|| 'new_plan'` to match `_toPlanSlug()` (TaskViewerProvider.ts:15392). Without this, an H1 of all special characters (e.g. `# !!!`) produces an empty slug and the rename is silently skipped. **DRY note:** The ideal fix is to extract `_toPlanSlug` into a shared static utility (e.g. `PlanSlug.toSlug(title)`) and call it from both TaskViewerProvider and PlanningPanelProvider. If time permits during implementation, do the extraction. If not, the inlined copy MUST include the `|| 'new_plan'` fallback to stay consistent. Mark the duplication with a `// TODO: extract to shared PlanSlug utility` comment.
2. **TOCTOU race fix:** Replaced `fs.existsSync(newPath)` + `fs.promises.rename()` with a direct `fs.promises.rename()` inside try/catch. This matches the established pattern in extension.ts:3068-3092. If the target file already exists (collision), the rename throws `EEXIST`/`EPERM` and is caught â€” the content is already saved to the original path, so this is non-fatal.
3. **Relative path fix:** Use the already-resolved `wsRoot` (computed inside the rename block) for `path.relative()` when building `renamedFilePath`, NOT `this._getWorkspaceRoot()`. In multi-root workspaces, `_getWorkspaceRoot()` can return undefined, producing `path.relative('', absolutePath)` = absolute path, which mismatches the kanban DB's relative-path convention.

```ts
this._lastPanelWriteTimestamp = Date.now();
await fs.promises.writeFile(resolved, content, 'utf8');

// Rename plan file if the H1 has changed and produces a different slug
let renamedTo: string | undefined;
let renameWsRoot: string | undefined;  // track which workspace root was used for the rename
if (tab === 'kanban' || tab === 'epics') {
    try {
        const h1Match = content.match(/^#\s+(.+)$/m);
        const h1Title = h1Match ? h1Match[1].trim() : '';
        if (h1Title) {
            // Generate the slug the file *should* have
            // TODO: extract to shared PlanSlug utility â€” duplicated from _toPlanSlug() in TaskViewerProvider.ts:15387
            const newSlug = h1Title
                .toLowerCase()
                .replace(/[^a-z0-9]+/g, '_')
                .replace(/^_+|_+$/g, '') || 'new_plan';
            const currentBasename = path.basename(resolved);
            const currentSlug = currentBasename.replace(/^feature_plan_\d{8}_\d{6}_/, '').replace(/\.md$/, '');
            if (newSlug !== currentSlug) {
                const timestamp = currentBasename.match(/^feature_plan_(\d{8}_\d{6})_/)?.[1] || '';
                const newBasename = `feature_plan_${timestamp}_${newSlug}.md`;
                const newPath = path.join(path.dirname(resolved), newBasename);
                // Try rename directly â€” if target exists (collision), rename throws and is caught.
                // This matches the established pattern in extension.ts:3068 (no existsSync pre-check).
                await fs.promises.rename(resolved, newPath);
                renamedTo = newPath;
                // Update kanban DB if available
                const wsRoot = this._getWorkspaceRoot() || (allRoots.length > 0 ? allRoots[0] : undefined);
                renameWsRoot = wsRoot;
                if (wsRoot) {
                    const db = KanbanDatabase.forWorkspace(wsRoot);
                    if (await db.ensureReady()) {
                        const oldRelative = path.relative(wsRoot, resolved).replace(/\\/g, '/');
                        const newRelative = path.relative(wsRoot, newPath).replace(/\\/g, '/');
                        const plan = await db.getPlanByPlanFile(oldRelative, await db.getWorkspaceId() || '');
                        if (plan) {
                            await db.updatePlanFile(plan.sessionId, newRelative);
                        }
                    }
                }
            }
        }
    } catch (renameErr) {
        // Rename failure is non-fatal â€” the content was already saved to the original path.
        // Common causes: target file exists (collision), cross-device rename, file locked.
        renamedTo = undefined;  // ensure we don't report a rename that didn't happen
        console.error('[PlanningPanelProvider] Plan rename on save failed:', renameErr);
    }
}

saveDestPanel?.webview.postMessage({
    type: 'saveFileContentResult',
    success: true,
    tab,
    // Use renameWsRoot (the root used for the DB lookup), NOT this._getWorkspaceRoot().
    // In multi-root workspaces _getWorkspaceRoot() can be undefined â†’ absolute path â†’ DB mismatch.
    renamedFilePath: renamedTo && renameWsRoot
        ? path.relative(renameWsRoot, renamedTo).replace(/\\/g, '/')
        : undefined
});
```

**Key details:**
- The timestamp prefix (`feature_plan_20260621_003238_`) is preserved from the original filename â€” only the slug portion changes.
- The rename only fires if the new slug differs from the current slug. If the H1 hasn't changed, no rename.
- If the target filename already exists (collision), `fs.promises.rename` throws and is caught â€” the content is still saved to the original file. No existsSync pre-check (matches extension.ts:3068 pattern).
- The kanban DB `plan_file` column is updated via the existing `updatePlanFile()` method.
- The `renamedFilePath` is returned to the webview so it can update its internal state.

**Edge Cases:**
- **No H1 in content:** `h1Match` is null â†’ no rename. The file keeps its current name.
- **H1 is "Untitled Plan":** `newSlug` = `untitled_plan`. If the file was already `untitled_plan`, no change. If the file was something else (e.g., user renamed it externally), it would rename back â€” but this is an edge case that shouldn't happen in practice.
- **H1 is all special characters (e.g. `# !!!`):** `newSlug` = `new_plan` (via the `|| 'new_plan'` fallback, matching `_toPlanSlug()`). The file renames to `feature_plan_<timestamp>_new_plan.md`. Without the fallback, the slug would be empty and the rename would be silently skipped â€” this was the original plan's bug, now fixed.
- **Collision with existing file:** `fs.promises.rename` throws `EEXIST`/`EPERM` â†’ caught by the `catch` block. `renamedTo` is set to `undefined`. Content is saved to the original path. No data loss. (Refined from the original existsSync-then-rename approach to match the established pattern in extension.ts:3068.)
- **Kanban DB not ready:** `db.ensureReady()` returns false â†’ skip DB update. The file is still renamed on disk; the DB will be reconciled by the plan scanner on the next sweep (which matches plans by file content/headers).
- **Plan not yet in kanban DB (created but scanner hasn't run):** `getPlanByPlanFile` returns null â†’ DB update skipped. File is renamed on disk. The plan scanner will pick up the new filename on its next sweep. Known gap â€” non-fatal, content is saved.
- **Non-kanban/epics tabs:** The rename only fires for `tab === 'kanban' || tab === 'epics'`. Constitution saves are not affected.
- **Cross-device rename:** `fs.promises.rename` may fail across devices/partitions. The `catch` block handles this â€” the content is already saved, so this is non-fatal.
- **Multi-root workspace with undefined `_getWorkspaceRoot()`:** The rename uses `allRoots[0]` as fallback for `wsRoot`. The `renamedFilePath` is computed with `renameWsRoot` (the resolved root), not `this._getWorkspaceRoot()`, avoiding absolute-path mismatches with the kanban DB.

## Risks & Edge Cases

- **Title prompt friction:** The input box adds one step to the creation flow. The default value is pre-filled ("Untitled Plan"), so the user can just press Enter to skip. This is the right tradeoff â€” meaningful filenames are worth one keystroke.
- **Project panel not open:** If the project panel hasn't been opened yet, `activatePlanInProjectPanel()` calls `openProject()` first. The `autoEdit` flag is deferred until the preview loads, so there's no race.
- **`_pendingAutoEdit` stale state:** If the user creates a plan but then clicks a different plan before the preview loads, `_pendingAutoEdit` would still be true. **Mitigation:** Clear `_pendingAutoEdit = false` in the plan item click handler.
- **Rename on save â€” DB lookup failure:** If the kanban DB can't find the plan by old `plan_file`, the DB update is skipped but the file is still renamed on disk. The plan scanner will reconcile the DB on the next sweep. Non-fatal.
- **Rename on save â€” race with external editor:** If the file is open in an external editor and the rename fails, the `catch` block logs the error and the content is already saved to the original path. Non-fatal.
- **No kanban provider or workspace root:** If `this._kanbanProvider` is null or `workspaceRoot` is empty, the plan is created on disk and the refined fallback opens the raw `.md` file via `_openPlanInReviewPanel()` so the user sees something. The user can find it in the sidebar on the next refresh.
