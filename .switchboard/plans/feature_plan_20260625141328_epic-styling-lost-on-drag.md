# Fix: Epic Card Styling & Promote-to-Epic Button Reset After Drag to Another Column

## Goal

When a user drags an epic card from one Kanban column to another, two visible regressions occur: (1) the epic card's `epic-card` CSS class (visual styling — border, badge, etc.) disappears, and (2) the "PROMOTE TO EPIC" sidebar button becomes active again, indicating the card is no longer recognized as an epic. The fix ensures epic styling and identity are preserved across all drag-and-drop code paths by including subtask session IDs in `moveCards` deltas and keeping `lastBoardSignature` in sync after optimistic DOM moves.

### Problem

When a user drags an epic card from one Kanban column to another, two visible regressions occur:
1. The epic card's `epic-card` CSS class (visual styling — border, badge, etc.) disappears
2. The "PROMOTE TO EPIC" sidebar button becomes active again, indicating the card is no longer recognized as an epic

### Root Cause Analysis

After extensive code tracing, the root cause is a **state inconsistency between the webview's `currentCards` and the backend DB after an epic cascade move**, combined with a **missing `lastBoardSignature` update in `handleDrop`**.

Here's the detailed flow:

**1. Epic cascade moves epic + subtasks in DB, but `moveCards` delta only includes the epic's ID**

When `moveCardToColumn` detects an epic (`KanbanProvider.ts:4592`), it calls `db.updateColumnWithEpicCascade(sessionId, subtaskSessionIds, targetColumn)` which moves BOTH the epic and all its subtasks to the target column in a single DB transaction (`KanbanDatabase.ts:3757-3783`).

However, the `moveCards` delta message sent to the webview (`KanbanProvider.ts:5322, 5336`) only contains the epic's `sessionId` — not the subtask `sessionId`s:

```js
// KanbanProvider.ts:5322
this._panel?.webview.postMessage({ type: 'moveCards', sessionIds, targetColumn });
```

The webview's `moveCards` handler (`kanban.html:6045-6062`) only updates cards whose IDs are in `idsToMove`:

```js
currentCards = currentCards.map(card => {
    const cardKey = card.planId || card.sessionId;
    if (idsToMove.has(cardKey) || (card.sessionId && idsToMove.has(card.sessionId))) {
        changed = true;
        return { ...card, column: targetCol };
    }
    return card;
});
```

This updates the **epic** card's column but leaves all **subtask** cards with their **old column** in `currentCards`. The DB now has subtasks in the new column, but `currentCards` has them in the old column. Subtask cards DO appear on the Kanban board (the `getBoard` query at `KanbanDatabase.ts:2172` returns ALL active plans with no `isEpic` or `epic_id` filter), so they are present in `currentCards`.

**2. `handleDrop` mutates `currentCards` in place without updating `lastBoardSignature`**

The `handleDrop` function (`kanban.html:5838-5872`) performs an optimistic DOM move and mutates `currentCards` directly at line 5862:

```js
card.column = effectiveTargetColumn;
```

But it does **NOT** update `lastBoardSignature` (unlike `moveCardsOptimistically` which does at line 4321). This means `lastBoardSignature` still reflects the **pre-drag** state.

**3. Signature mismatch triggers spurious full re-render**

After the `moveCards` delta sets `lastBoardSignature = buildBoardSignature(currentCards)` (with epic in new column, subtasks in OLD column), any subsequent `updateBoard` from a file watcher or scheduled refresh will have fresh DB data where **both epic and subtasks are in the new column**. The `buildBoardSignature` comparison (`kanban.html:6114`) detects a difference (subtask columns changed) and calls `renderBoard(nextCards)` — a full DOM replacement.

**4. The full re-render replaces all card DOM elements**

`renderBoard` sets `container.innerHTML = sortedItems.map(card => createCardHtml(card)).join('')` (`kanban.html:5105`). Each card HTML is generated by `createCardHtml`, which checks `card.isEpic` at line 5328:

```js
const epicClass = card.isEpic ? ' epic-card' : '';
```

While the fresh `nextCards` from `updateBoard` **should** have `isEpic: !!row.isEpic` correctly set (verified at `KanbanProvider.ts:1231`), the spurious re-render destroys the optimistic UI state and can race with other pending messages. If a `moveCards` delta arrives between the `updateBoard` signature check and the `renderBoard` call, or if the `updateBoard` arrives from a watcher that reads a stale DB snapshot, the epic card can be re-rendered without `isEpic` or with stale selection state.

**5. Selection state desync**

After `handleDrop` clears selection (`kanban.html:5877-5880`), the `moveCards`-triggered `renderBoard` re-attaches click listeners. If the user clicks the card to re-select it, the click handler reads `isEpic` from `currentCards` (`kanban.html:5127`):

```js
isEpic: cardData?.isEpic || false,
```

If `currentCards` was replaced by a stale `updateBoard` (from the `else` branch at line 6119: `currentCards = nextCards`) where the card data is fresh but the signature matched (no re-render), the `isEpic` should still be correct. But if the signature comparison is wrong due to the `handleDrop` mutation not updating `lastBoardSignature`, a re-render with stale data could produce a card without `isEpic`.

**6. Scope gap — `promptOnDrop` path has the same bug (discovered during plan improvement)**

The `promptOnDrop` handler (`KanbanProvider.ts:5631-5720`) sends `moveCards` deltas at **three** separate locations (lines 5671, 5698, 5707) with only the directly-moved session IDs — not subtask IDs from epic cascades. This means dragging an epic in prompt mode suffers the identical styling-loss bug. The original plan only addressed `moveCardForward` and `moveCardBackwards`; the `promptOnDrop` path was missed.

Additionally, a grep for `type: 'moveCards'` reveals **21 send sites** across `KanbanProvider.ts`. Many are button-driven paths (`promptSelected`, `promptAll`, `_distributePlannerDispatch`, etc.) that also call `moveCardToColumn` and would have the same bug for epics. This plan focuses on the drag-and-drop paths (the reported bug), but a helper method is recommended to make the fix DRY and extensible to all paths in future work.

## Complexity Audit

### Routine
- Adding `lastBoardSignature = buildBoardSignature(currentCards)` after the optimistic DOM move in `handleDrop` — mirrors existing pattern at `kanban.html:4321`
- The `moveCards` webview handler already preserves `isEpic` via spread (`{ ...card, column: targetCol }`) — no change needed on the webview side for the subtask-ID fix
- The DB layer (`updateColumnWithEpicCascade`) already correctly cascades the move — the bug is purely in the webview/backend messaging contract
- No schema changes, no new APIs, no migration needed

### Complex / Risky
- **Multiple `moveCards` send sites**: The `promptOnDrop` handler has 3 separate `moveCards` send sites (lines 5671, 5698, 5707) that all need the subtask-ID fix. Missing any one leaves a partial bug.
- **Redundant DB queries**: The plan's inline approach re-queries `getPlanBySessionId` + `getSubtasksByEpicId` after `moveCardToColumn` already did the same queries internally (lines 4590-4596). This doubles DB queries per epic move. The helper method approach mitigates this but doesn't eliminate it.
- **21 total `moveCards` send sites**: Only 5 are covered by this plan (the drag-and-drop paths). The remaining 16 are button-driven paths with the same latent bug. This plan does not fix them but the helper method makes future fixes trivial.

## Edge-Case & Dependency Audit

### Race Conditions
- **`moveCards` delta vs `updateBoard` race**: If a file-watcher `updateBoard` arrives between the `moveCards` delta and the `lastBoardSignature` update in `handleDrop`, the signature mismatch could still trigger a spurious re-render. Fix #2 (updating `lastBoardSignature` in `handleDrop` immediately after the DOM mutation) minimizes this window but does not eliminate it for the 350ms `setTimeout` delay before the backend message is sent (`kanban.html:5881`).
- **CLI trigger path**: `triggerAction`/`triggerBatchAction` do not send `moveCards`; they call `_scheduleBoardRefresh` (~100ms delay). Fix #2 prevents the spurious re-render here, but the 350ms gap between `handleDrop`'s optimistic move and the `updateBoard` from the refresh is the vulnerable window.

### Security
- No security implications — all operations are internal DB writes and webview state updates.

### Side Effects
- **Subtask cards on the board**: Subtask cards appear on the Kanban board (confirmed: `getBoard` returns all active plans). Including their IDs in `moveCards` will cause them to visually move to the target column in the webview, which is correct since the DB cascade already moved them.
- **Subtask cards NOT on the board**: If a subtask is filtered out (e.g., ghost plan filtering at `KanbanProvider.ts:1185-1205`), its ID in the `moveCards` delta is harmless — the webview handler simply won't find it in `currentCards` and `changed` won't be set for it.

### Dependencies & Conflicts
- No external dependencies. All methods used (`getPlanBySessionId`, `getSubtasksByEpicId`, `updateColumnWithEpicCascade`) are existing internal APIs.
- No conflicts with other in-flight changes expected — the fix is additive (collecting and passing more IDs).

## Adversarial Synthesis

Key risks: (1) The `promptOnDrop` path was missed by the original plan — 3 additional `moveCards` send sites need the subtask-ID fix or prompt-mode epic drags will still lose styling. (2) The inline re-query approach doubles DB calls per epic move; a shared helper method eliminates duplication and makes future expansion to the remaining 16 `moveCards` send sites trivial. (3) The 350ms `setTimeout` gap in `handleDrop` before the backend message is sent creates a race window where a file-watcher `updateBoard` can arrive with stale `lastBoardSignature`. Mitigations: helper method for DRY coverage of all drag paths; `lastBoardSignature` update immediately after DOM mutation (before the `setTimeout`); document the remaining button-driven paths as known future work.

## Proposed Changes

### 1a. `src/services/KanbanProvider.ts` — Add helper method `_collectAllMovedSessionIds`

Add a private helper method that collects all session IDs affected by a move (including subtask IDs for epics). This eliminates the redundant DB query problem and ensures all `moveCards` send sites can easily include subtask IDs.

**Insert after `moveCardToColumn` (after line 4608):**

```ts
/**
 * Collect all session IDs affected by a column move — for epics, this includes
 * subtask session IDs (since moveCardToColumn cascades the move atomically).
 * Use this to build the `moveCards` delta so the webview updates ALL affected cards.
 */
private async _collectAllMovedSessionIds(workspaceRoot: string, sessionId: string): Promise<string[]> {
    const db = this._getKanbanDb(workspaceRoot);
    if (db && await db.ensureReady()) {
        const plan = await db.getPlanBySessionId(sessionId);
        if (plan && !!plan.isEpic) {
            const subtasks = await db.getSubtasksByEpicId(plan.planId);
            const subtaskSessionIds = subtasks.map(st => st.sessionId).filter(Boolean);
            return [sessionId, ...subtaskSessionIds];
        }
    }
    return [sessionId];
}
```

**Note on `isEpic` type**: `KanbanPlanRecord.isEpic` is `number | undefined` (0/1), not boolean. Using `!!plan.isEpic` ensures correct truthiness and matches the pattern at `KanbanProvider.ts:1231`.

### 1b. `src/services/KanbanProvider.ts` — Use helper in `moveCardBackwards` handler (line 5312-5324)

**Replace the `moveCardBackwards` handler:**

### 2. `src/webview/kanban.html` — Update `lastBoardSignature` after optimistic move in `handleDrop`

In the `handleDrop` function, after the optimistic DOM move loop (after line 5872), add:

```js
// Keep the render-guard signature in sync with the optimistic mutation.
// Without this, a subsequent updateBoard with stale DB data (card still in
// old column) matches the stale lastBoardSignature, silently replacing
// currentCards with old-column data while the card is visually in the new column.
lastBoardSignature = buildBoardSignature(currentCards);
```

This mirrors what `moveCardsOptimistically` does at line 4321. Place this **before** the `if (validIds.length > 0)` block at line 5876 — it should run whenever `currentCards` was mutated, even if the subsequent selection-clear block doesn't execute.

**Clarification**: The `lastBoardSignature` update should be placed inside the `validIds.forEach` block's closing, right after line 5872, so it only runs when cards were actually moved. If `validIds` is empty, no mutation occurred and the signature update is unnecessary (but harmless).
