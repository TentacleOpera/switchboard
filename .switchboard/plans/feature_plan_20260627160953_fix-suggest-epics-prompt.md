# Fix Suggest Epics Prompt: Hardcoded Backlog Count & Backlog Question Hijacking Approval Flow

## Goal

Fix two related bugs in the "Suggest Epics" clipboard prompt generated by `_buildSuggestEpicsPrompt()` in `KanbanProvider.ts`:

### Problem 1: Hardcoded backlog count is stale at agent execution time

The prompt bakes in the literal `backlogCount` number at copy-time (line 8779):

```
After the proposal, ask: "BACKLOG has 12 ungrouped plan(s). Analyse those too? (yes/no)"
```

This is wrong because:
- The prompt is copied to the clipboard and may be pasted minutes or hours later.
- Between copy-time and paste-time, plans may be added/removed from BACKLOG.
- The agent **already reads the board snapshot** in step 1 (`cat kanban-board.md`) — it can count backlog entries itself.
- Hardcoding a number that will almost certainly be wrong gives the agent a false premise, which may confuse downstream reasoning.

### Problem 2: Backlog question inside PROPOSE step hijacks the grouping approval flow

The "Analyse backlog too?" question is embedded **inside step 3 (PROPOSE)**, immediately after the agent presents its epic groupings. This creates a catastrophic flow bug:

1. Agent presents epic groupings and then asks "BACKLOG has N ungrouped plan(s). Analyse those too?"
2. User says "yes" — meaning "yes, approve the groupings"
3. Agent interprets "yes" as answering the backlog question → goes off to review the backlog
4. The grouping approval (step 4 CONFIRM) and execution (step 5 EXECUTE) are never reached

The backlog question acts as an **unintended redirect** that intercepts the user's approval. The agent cannot distinguish between "yes, approve my groupings" and "yes, analyse the backlog".

### Root Cause

Both issues stem from line 8779 in `_buildSuggestEpicsPrompt()` (method starts at line 8745):
```typescript
`   After the proposal, ask: "BACKLOG has ${backlogCount} ungrouped plan(s). Analyse those too? (yes/no)"`
```

This single line introduces a stale hardcoded value AND places an ambiguous yes/no question in the middle of an approval flow.

Between copy and paste, the actual BACKLOG may change. Removing the hardcoded number eliminates this race entirely — the agent will count BACKLOG entries from the live board snapshot at execution time instead.

## Adversarial Synthesis

Key risks: (1) the new step 6 references a board snapshot that may be stale by the time execution completes, (2) the prompt gives no guidance on what the agent should do if the user says "yes" to analysing BACKLOG plans, and (3) the plan's original line numbers had drifted ~114 lines from the recent epic-fix commit. Mitigations: step 6 now instructs the agent to re-read the board snapshot before checking BACKLOG, includes a "repeat steps 2-5 for BACKLOG plans" loop-back instruction, and all line numbers have been updated to current values.
