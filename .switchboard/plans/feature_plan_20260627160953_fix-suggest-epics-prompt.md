# Fix Suggest Epics Prompt: Hardcoded Backlog Count & Backlog Question Hijacking Approval Flow

## Goal

Fix two related bugs in the "Suggest Epics" clipboard prompt generated by `_buildSuggestEpicsPrompt()` in `KanbanProvider.ts`:

### Problem 1: Hardcoded backlog count is stale at agent execution time

The prompt bakes in the literal `backlogCount` number at copy-time (line 8649):

```
After the proposal, ask: "BACKLOG has 12 ungrouped plan(s). Analyse those too? (yes/no)"
```

This is wrong because:
- The prompt is copied to the clipboard and may be pasted minutes or hours later.
- Between copy-time and paste-time, plans may be added/removed from BACKLOG.
- The agent **already reads the board snapshot** in step 1 (`cat kanban-board.md`) — it can count backlog entries itself.
- Hardcoding a number that will almost certainly be wrong gives the agent a false premise, which may confuse downstream reasoning.

### Problem 2: Backlog question inside PROPOSE step hijacks the grouping approval flow

The "Analyse backlog too?" question is embedded **inside step 3 (PROPOSE)**, immediately after the agent presents its epic groupings. This creates a catastrophic flow bug:

1. Agent presents epic groupings and then asks "BACKLOG has N ungrouped plan(s). Analyse those too?"
2. User says "yes" — meaning "yes, approve the groupings"
3. Agent interprets "yes" as answering the backlog question → goes off to review the backlog
4. The grouping approval (step 4 CONFIRM) and execution (step 5 EXECUTE) are never reached

The backlog question acts as an **unintended redirect** that intercepts the user's approval. The agent cannot distinguish between "yes, approve my groupings" and "yes, analyse the backlog".

### Root Cause

Both issues stem from line 8649 in `_buildSuggestEpicsPrompt()`:
```typescript
`   After the proposal, ask: "BACKLOG has ${backlogCount} ungrouped plan(s). Analyse those too? (yes/no)"`
```

This single line introduces a stale hardcoded value AND places an ambiguous yes/no question in the middle of an approval flow.

## Edge-Case & Dependency Audit

| Concern | Assessment |
|---------|------------|
| **backlogCount parameter removal** | The `backlogCount` param is computed in the `suggestEpics` case handler (lines 7915-7919) and passed only to `_buildSuggestEpicsPrompt`. If we remove it from the prompt, we can also remove the computation and parameter. No other callers exist. |
| **Prompt already instructs agent to read the board** | Step 1 already tells the agent to `cat kanban-board.md`. The agent has full visibility into BACKLOG — it can count entries itself. Safe to remove the hardcoded number. |
| **Backlog question still useful** | The ability to optionally include BACKLOG plans is still valuable — it just needs to be asked at the right time (after execution, not during proposal) and without a hardcoded count. |
| **Agent host compatibility** | The prompt is clipboard text pasted into any agent chat. Restructuring the flow has no host-specific concerns. |
| **Existing tests** | No tests directly exercise `_buildSuggestEpicsPrompt` output content. The change is safe from test regressions. |

    private _buildSuggestEpicsPrompt(workspaceRoot: string): string {
        return [
            'You are grouping loose Switchboard plans into epics. Follow this flow exactly — do not create any epic before the user approves.',
            '',
            '1. SCAN',
            `   Read the board snapshot:`,
            `     cat ${workspaceRoot}/.switchboard/kanban-board.md`,
            '',
            '   Scope: CREATED, CONTEXT GATHERER, and PLAN REVIEWED columns only.',
            '   Ignore BACKLOG and all post-coding columns.',
            '   Each plan line ends with an HTML comment, e.g.:',
            '     - [.switchboard/plans/foo.md](...) — Foo <!-- planId:abc-123 -->',
            '     - [.switchboard/epics/epic-def.md](...) — Bar Epic <!-- planId:def-456 epic -->',
            '     - [.switchboard/plans/baz.md](...) — Baz <!-- planId:ghi-789 subtask-of:"Bar Epic" -->',
            '   Skip lines tagged `epic` (they are epics) or `subtask-of:...` (already assigned).',
            '   Use the `planId:` value from the comment — NOT the filename — when calling create-epic.js.',
            '   (A path under .switchboard/epics/ also indicates an epic, but subtask detection',

            '',
            '2. READ PLAN BODIES',
            '   For each candidate plan in scope, read the full plan file.',
            '   Extract: goal, problem summary, dependencies, tags.',
            '   Use this — not just titles — to determine groupings.',
            '   Read plans in parallel where possible. If >25 candidates, first-pass cluster by',
            '   title then deep-read within each cluster.',
            '',
            '3. PROPOSE (single message, all groups at once)',
            '   Group by underlying capability theme, not by surface keyword.',
            '   Cross-provider plans that address the same capability go into one epic.',
            '   Minimum 2 plans per epic. Single-plan "groups" go in the Standalone section.',
            '   Flag POSSIBLE OVERLAP / REDUNDANCY / GAP where detected.',
            '   For each proposed epic: epic name, member plans with planId and one-line summary.',

            '',

            '   Wait for user approval or edits. Do not touch the database until confirmed.',
            '',
            '5. EXECUTE',
            '   For each approved group:',

            `   node .agents/skills/kanban_operations/create-epic.js "<epic name>" '["planId1","planId2",...]' "${workspaceRoot}"`,
