# Fix: Epic tab subtask links in markdown doc preview do not work

## Goal

Fix non-functional subtask links inside the rendered markdown preview of epic documents in the Epics tab. The links generated by `_regenerateEpicFile` are rendered as `<a href>` tags by VS Code's markdown API but receive no click handling in the sandboxed webview, leaving them silent. The fix adds a delegated click listener on `epicsPreviewContent` that intercepts `.md` link clicks, resolves relative paths, and routes them through the same `fetchKanbanPlanPreview` flow used by the sidebar subtask accordion links.

### Problem
In the Epics tab of `project.html`, when an epic's markdown document is rendered in the preview pane (`#epics-preview-content`), the subtask links embedded in the markdown body are non-functional. Clicking them does nothing — the subtask plan is not loaded into the preview area.

The sidebar subtask links (the expandable accordion list under each epic card) work correctly: clicking one loads the subtask's plan file into the epic preview pane via `fetchKanbanPlanPreview`. The links **inside the rendered markdown body** should behave identically, but they don't.

### Root Cause
The epic markdown document is auto-generated by `_regenerateEpicFile` in `src/services/KanbanProvider.ts` (line 8579). The subtask link line is at line 8592:

```typescript
return `- [ ] [${topic}](../plans/${basename}) — **${column}**`;
```

This produces markdown links like:
```
- [ ] [Implement auth](../plans/feature_plan_20260101_xxx.md) — **CREATED**
```

When VS Code's built-in markdown renderer (`markdown.api.render`, called at `src/services/PlanningPanelProvider.ts` line 1254) converts this to HTML, it produces standard `<a href="../plans/feature_plan_20260101_xxx.md">` anchor tags. These are injected into `epicsPreviewContent.innerHTML` (`src/webview/project.js` line 388).

**No click handler intercepts these `<a>` tags.** In a VS Code webview (sandboxed iframe), clicking a relative `<a href>` does not navigate — it silently does nothing. The sidebar subtask links work because `renderEpicSubtasks` (`src/webview/project.js` lines 1759–1778) explicitly attaches `click` event listeners to each `.epic-subtask-link` span that call `fetchKanbanPlanPreview`. The markdown-rendered `<a>` tags get no such treatment.

The fix: add a delegated click listener on `epicsPreviewContent` that intercepts clicks on `<a>` tags pointing to `.md` plan files, resolves the relative `href` against the currently-displayed file's directory, and triggers the same `fetchKanbanPlanPreview` flow used by the sidebar links.

## Complexity Audit

### Routine
- Single-file change (`src/webview/project.js`) — one new delegated event listener.
- The pattern already exists in the same file: `renderEpicSubtasks` (lines 1759–1778) attaches per-link click handlers that call `fetchKanbanPlanPreview`. This plan mirrors that exact flow but uses event delegation on the container.
- No backend/extension changes, no DB changes, no new message types.
- Path resolution is simple string manipulation (split on `/`, process `..` and `.` segments).
- The response routing (`kanbanPlanPreviewReady` → project.js line 384) already works — same code path as sidebar links.

### Complex / Risky
- **`markdown.api.render` href format**: The fix assumes the rendered HTML preserves relative `href` attributes as-is (e.g., `<a href="../plans/foo.md">`). If VS Code's markdown engine rewrites or transforms hrefs, the `href.endsWith('.md')` guard would fail. See **## Uncertain Assumptions**.
- **Path resolution correctness**: The hand-rolled resolver assumes forward-slash paths. This is safe because `planFile` values in the DB always use forward slashes, but the assumption is undocumented in the original code.

## Edge-Case & Dependency Audit

1. **Relative path resolution**: The epic file lives at `.switchboard/epics/<slug>-<uuid>.md`. Subtask links use `../plans/<basename>`, which resolves to `.switchboard/plans/<basename>`. We must resolve relative to the **directory of the currently-displayed file** (`_epicPreviewFilePath`), not a hardcoded base. This way, if a subtask's markdown also contains relative links, they resolve correctly too.

2. **Non-plan links**: The epic markdown may contain external URLs (`https://...`) or other non-plan links. The handler must only intercept links whose `href` resolves to a local `.md` file. External links should be left alone (or opened externally via `vscode.postMessage`).

3. **Already-previewing a subtask**: When a subtask is being previewed (via sidebar click), `_epicPreviewFilePath` is the subtask's path, not the epic's. If the subtask markdown also contains relative `.md` links, they should resolve relative to the subtask's directory. Using `_epicPreviewFilePath` as the base handles this naturally.

4. **Edit mode**: When edit mode is active (`state.editMode.epics`), the preview content is hidden and the textarea is shown. The click listener on `epicsPreviewContent` won't fire because the content isn't visible. No special handling needed — but we should still call `exitEditMode('epics')` if active, matching the sidebar link behavior (line 1767).

5. **Edit button visibility**: The sidebar links hide the Edit button when previewing a subtask (line 1776–1777). The markdown link handler must do the same, so users can't try to edit a subtask they're only previewing.

6. **Returning to the epic doc**: After clicking a subtask link in the markdown, the user returns to the epic doc by clicking the epic card in the sidebar again (which calls `selectEpic`). This is the existing pattern for sidebar subtask links — no new "back" button is needed for consistency.

7. **`fetchKanbanPlanPreview` response routing**: The response handler (project.js line 384) checks `_epicPreviewFilePath === msg.filePath`. As long as we set `_epicPreviewFilePath` before sending the request, the response will render in the epic preview pane. This is the same mechanism the sidebar links use.

8. **Non-plan local `.md` links**: If the epic markdown links to a local `.md` file that is not a plan file (e.g., a README), the handler will intercept it and load it into the preview pane. The extension's `_handleFetchKanbanPlanPreview` (`src/services/PlanningPanelProvider.ts` line 1224) reads and renders any workspace `.md` file — it does not validate that the file is a plan. This is acceptable behavior: the user sees the markdown content rendered, same as any plan preview. No special handling needed.

9. **Race Conditions**: The delegated listener is attached once at initialization. There is no risk of duplicate listeners being attached on re-renders (unlike per-link listeners in `renderEpicSubtasks` which are re-attached each render). The `requestId` increment (`++_kanbanPreviewRequestId`) ensures stale responses are ignored, same as the sidebar link flow.

10. **Security**: The extension's `_handleFetchKanbanPlanPreview` has an `isAllowed` check (line 1227) that verifies the resolved path is within workspace roots. This prevents path traversal attacks from malicious markdown content. The webview-side resolver could produce a path outside `.switchboard/`, but the extension-side check is the security boundary.

11. **Side Effects**: Setting `_epicPreviewFilePath` to the subtask path changes the preview state. If the user subsequently clicks "Edit", they would edit the subtask file, not the epic. The plan mitigates this by hiding the Edit button (line 1776–1777 pattern), same as the sidebar links.

12. **Dependencies & Conflicts**: No existing click listener on `epicsPreviewContent` (verified — no `epicsPreviewContent.addEventListener` calls exist in `project.js`). No conflict with other handlers.

## Uncertain Assumptions

1. **`markdown.api.render` preserves relative `href` attributes as-is**: The fix assumes that `vscode.commands.executeCommand('markdown.api.render', content)` produces `<a href="../plans/foo.md">` with the raw relative path preserved. If the markdown engine rewrites relative links (e.g., to `vscode-webview://` URIs, absolute paths, or `data-href` attributes), the `href.endsWith('.md')` guard would fail and the fix would silently do nothing. The user was advised to run web research to confirm this before implementation.

## Adversarial Synthesis

Key risks: (1) the `markdown.api.render` API may transform relative hrefs, breaking the `endsWith('.md')` guard — this is the load-bearing assumption and must be verified; (2) the hand-rolled path resolver assumes forward-slash paths (safe given DB conventions but undocumented); (3) non-plan `.md` links are intercepted but this is acceptable behavior. Mitigations: flag the href assumption for user verification, add a comment documenting the forward-slash assumption, and rely on the extension-side `isAllowed` check as the security boundary for path traversal.

## Proposed Changes

### File: `src/webview/project.js`

Add a delegated click listener on `epicsPreviewContent` that intercepts clicks on `<a>` tags pointing to local `.md` plan files. Place it **once** after the `epicsPreviewContent` variable declaration (after line 227), in the same scope as other variable declarations and static listeners. Do NOT place it inside `renderEpicSubtasks` or any per-render function — it is a one-time delegated listener.

**Insert after line 227 (after `const epicsPreviewContent = document.getElementById('epics-preview-content');`):**

```javascript
// Intercept clicks on <a> tags inside the rendered epic markdown preview.
// Subtask links in the auto-generated "## Subtasks" section are rendered as
// <a href="../plans/basename.md"> by VS Code's markdown renderer. In the
// sandboxed webview these do nothing — we resolve them and load the subtask
// into the preview pane, mirroring the sidebar subtask-link behavior.
//
// NOTE: This listener is attached once (delegation). It is NOT re-attached on
// each render — innerHTML replacement preserves the container's event listeners.
// NOTE: Path resolution assumes forward-slash paths, which is guaranteed by the
// DB planFile format (.switchboard/epics/<file>).
if (epicsPreviewContent) {
    epicsPreviewContent.addEventListener('click', (e) => {
        const anchor = e.target.closest('a');
        if (!anchor) return;
        const href = anchor.getAttribute('href') || '';
        if (!href) return;

        // Only intercept links to local .md files (plan files).
        // External URLs (http://, https://, mailto:, #anchors) are left alone.
        if (!href.endsWith('.md')) return;
        if (/^(https?:|mailto:|tel:|#)/i.test(href)) return;

        e.preventDefault();

        // Resolve the relative href against the directory of the currently-
        // displayed file. The epic lives at .switchboard/epics/<file>, so
        // ../plans/foo.md resolves to .switchboard/plans/foo.md.
        const basePath = _epicPreviewFilePath || (_epicSelectedPlan && _epicSelectedPlan.planFile) || '';
        if (!basePath) {
            showToast('Cannot resolve subtask link — no file context.', 'error');
            return;
        }

        const baseDir = basePath.includes('/') ? basePath.substring(0, basePath.lastIndexOf('/')) : '';
        // Normalize the joined path: split on /, process .. and . segments.
        const segments = (baseDir + '/' + href).split('/');
        const resolved = [];
        for (const seg of segments) {
            if (seg === '..') resolved.pop();
            else if (seg !== '.' && seg !== '') resolved.push(seg);
        }
        const resolvedPath = resolved.join('/');

        if (state.editMode.epics) exitEditMode('epics');
        _epicPreviewFilePath = resolvedPath;
        epicsPreviewContent.innerHTML = '<div class="kanban-empty-state">Loading preview...</div>';
        vscode.postMessage({
            type: 'fetchKanbanPlanPreview',
            filePath: resolvedPath,
            requestId: ++_kanbanPreviewRequestId
        });
        // Hide the Edit button while a subtask is previewed (not the epic itself).
        const btnEdit = document.getElementById('btn-edit-epics');
        if (btnEdit) btnEdit.style.display = 'none';
    });
}
```

### No other files need changes

The extension-side `fetchKanbanPlanPreview` handler (`src/services/PlanningPanelProvider.ts` line 1224, method `_handleFetchKanbanPlanPreview`) and the message dispatch case (line 2962) and the response routing (`kanbanPlanPreviewReady` → `src/webview/project.js` line 384) already work correctly — they are the same code path used by the sidebar subtask links. No backend changes are required.
