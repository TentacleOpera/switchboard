# Comms Monitor: Slack Prompt Causes Claude Error Response

## Goal

The prompt generated by the Comms Monitor for Slack source checks causes Claude to reply with an error instead of checking Slack messages. The user sees an error response in the monitor terminal when Slack is configured as a polling source. The prompt needs to be fixed so Claude can successfully query the Slack MCP server and report unread messages.

### Problem Analysis & Root Cause

**Symptom:** When the Comms Monitor polls with Slack as a configured source, Claude replies with an error (the user's report was truncated, but the prompt clearly fails to produce the expected Slack check results).

**⚠ KEY UNCERTAINTY:** The actual Claude error was never captured — the user's report was truncated at "Claude replies with an error '". The root cause is **speculative**. The prompt rewrite in this plan is a best-effort fix targeting the most likely prompt-level causes, but it may not resolve the real error if the cause is tool-permission, environmental, or something else entirely. Step 1 (diagnostic — capture the full error) is a **required first action** before any code change.

**Root cause investigation:** The Slack prompt line is built by `_buildSlackPromptLine()` in `src/services/TaskViewerProvider.ts` (line 20612):

```ts
private _buildSlackPromptLine(cfg: McpMonitorConfig): string {
    const channels = (cfg.slackChannels || '').split(',').map(s => s.trim()).filter(Boolean);
    const scopeParts: string[] = [];
    if (!cfg.slackDmOnly) {
        if (channels.length > 0) {
            scopeParts.push(`messages in channels: ${channels.join(', ')}`);
        } else {
            scopeParts.push('messages in channels (all)');
        }
    }
    if (!cfg.slackChannelOnly) {
        scopeParts.push('direct messages (DMs)');
    }
    const scope = scopeParts.join(' and ');
    const boundary = this._sourceBoundary(cfg, 'slack');
    return `Slack: unread ${scope} and @-mentions ${boundary}. Clearly label each item as [DM] or [channel: #name].`;
}
```

This produces a line like:
```
Slack: unread messages in channels: general, random and direct messages (DMs) and @-mentions since [timestamp]. Clearly label each item as [DM] or [channel: #name].
```

The full prompt sent to Claude (via `_buildMcpMonitorPrompt`, line 20571) is:

```
Check the following for anything new that needs my attention since [timestamp]. Report only what is new and noteworthy as a short bullet list. If nothing needs attention, reply 'All clear'. This is read-only — do NOT take any actions, send any messages, or modify anything.

- Slack: unread messages in channels: general, random and direct messages (DMs) and @-mentions since [timestamp]. Clearly label each item as [DM] or [channel: #name].
```

**Potential causes of the Claude error:**

1. **Ambiguous channel references:** The prompt says "messages in channels: general, random" but doesn't use the `#` prefix that Slack MCP servers expect. Claude may try to call the MCP server with bare channel names and get a "channel not found" error.

2. **Scope phrasing is confusing:** "unread messages in channels: general, random and direct messages (DMs) and @-mentions" — the "and" joining makes it ambiguous whether "@-mentions" applies to channels, DMs, or both. Claude may misinterpret the request.

3. **The prompt doesn't explicitly instruct Claude to use MCP tools:** The prompt says "Check the following..." but doesn't say "Use your MCP tools to check Slack." Claude may try to respond from its training data or hallucinate instead of calling the Slack MCP server.

4. **Newline flattening:** `sendRobustText` flattens newlines for CLI agents (`src/services/terminalUtils.ts` line 153: `text.replace(/[\r\n]+/g, ' ')`), so the preamble and the Slack line become one long line. This is generally fine for Claude, but the resulting prompt may be harder to parse.

5. **User-configured startup command overriding the fallback:** The `mcp_monitor` fallback startup command at line 3846 uses `--allowedTools "mcp__*"` (a broad wildcard covering all MCP tools including Slack). **However**, if the user has configured a custom startup command for `mcp_monitor` via `GlobalIntegrationConfigService.getAgentStartupCommands()` (line 214/218 in `GlobalIntegrationConfigService.ts`), that custom command may not include `--allowedTools "mcp__*"` or any MCP tool pattern, which would cause Claude to error when it tries to call Slack MCP tools. The custom command takes precedence over the fallback (see `getAgentStartupCommand` at line 3831–3857).

## Edge-Case & Dependency Audit

### Race Conditions
- The `_mcpMonitorTickQueue` serializes monitor ticks, so concurrent prompt sends are already guarded. No new race conditions are introduced by this change.

### Security
- The prompt explicitly instructs Claude to use MCP tools in read-only mode ("do NOT take any actions, send any messages, or modify anything"). The proposed rewrite reinforces this by adding "Use your Slack MCP tools to check" — the word "check" is read-only. No security regression.
- Channel names from `cfg.slackChannels` are user-supplied and inserted into the prompt. The `#` prefix addition does not introduce injection risk since Claude interprets the prompt as natural language, not code.

### Side Effects
- The prompt rewrite changes the text sent to Claude for ALL Slack source checks, not just the failing case. If the current prompt works for some configurations (e.g., no channels specified), the rewrite must not break those. The proposed rewrite preserves the "all channels" fallback and adds the `#` prefix only when channel names are present, so existing behavior is preserved for the no-channels case.
- The preamble change (Step 4) affects ALL sources (Slack, Gmail, GCal, custom), not just Slack. This is low-risk since it only adds "Using your MCP server tools," which is universally applicable for the monitor.

### Dependencies & Conflicts
- The Slack MCP server must be configured and authenticated in the user's Claude environment. If it is not, no prompt fix will help — the error is environmental.
- The `mcp_monitor` role's startup command must include `--allowedTools` that cover the Slack MCP server's tools. **Verified:** The default fallback (line 3846) uses `--allowedTools "mcp__*"` which covers all MCP tools including `mcp__slack__*`. However, a user-configured custom startup command (stored in `GlobalIntegrationConfigService` agent config under `startupCommands.mcp_monitor`) may override this and omit MCP tool permissions.
- The `McpMonitorConfig` type (defined in `GlobalIntegrationConfigService.ts` line 64) has `targetRole` defaulting to `'mcp_monitor'`. The prompt builder and terminal launcher both use this role string consistently.

## Dependencies

- `sess_mcp_monitor_terminal`: The Comms Monitor terminal must be running and healthy for any prompt to be sent. If the terminal is not running, the prompt is never sent and no error occurs (the monitor silently skips).
- `sess_slack_mcp_server`: The Slack MCP server must be configured in the user's Claude environment (typically via `~/.claude/claude_desktop_config.json` or equivalent). If the server is not configured, Claude will error when attempting to call Slack tools regardless of prompt wording.
- No external dependencies beyond the existing Switchboard extension infrastructure.

## Adversarial Synthesis

This plan is built on sand. The actual error was never captured — we are rewriting a prompt based on vibes and inference. The most dangerous outcome is spending time on a prompt rewrite only to discover the real error is "MCP server not configured" or "OAuth token expired" or a custom startup command that strips `--allowedTools`. The prompt improvements (explicit MCP tool instruction, `#` prefix, empty-scope guard) are genuinely valuable regardless of whether they fix this specific error, but they should not be treated as the definitive fix. ### Step 2: Fix `_buildSlackPromptLine` (line 20612)

Reword the prompt to be explicit about using MCP tools and fix the channel name format:

```ts
private _buildSlackPromptLine(cfg: McpMonitorConfig): string {
    const channels = (cfg.slackChannels || '').split(',').map(s => s.trim()).filter(Boolean);
    const channelList = channels.length > 0
        ? channels.map(c => c.startsWith('#') ? c : `#${c}`).join(', ')
        : 'all channels';
    const scopeParts: string[] = [];
    if (!cfg.slackDmOnly) {
        scopeParts.push(`messages in ${channelList}`);
    }
    if (!cfg.slackChannelOnly) {
        scopeParts.push('direct messages (DMs)');
    }
    // Guard: if both filters exclude everything, default to all
    if (scopeParts.length === 0) {
        scopeParts.push('messages in all channels and direct messages (DMs)');
    }
    const scope = scopeParts.join(' and ');
    const boundary = this._sourceBoundary(cfg, 'slack');
    return `Slack: Use your Slack MCP tools to check for unread ${scope} and @-mentions ${boundary}. Clearly label each item as [DM] or [channel: #name].`;
}
```

Key changes:
- Added "Use your Slack MCP tools to" — explicitly instructs Claude to call the MCP server.
- Channel names are prefixed with `#` if not already.
- Guard for the both-filters-checked edge case (prevents empty scope).
- Removed the ambiguous "and @-mentions" phrasing — now "@-mentions" is part of the scope list.

### Step 3: Verify startup command includes Slack tools

**Verified finding (source code audit):** The `mcp_monitor` fallback startup command at `TaskViewerProvider.ts` line 3846 is:
```
claude --model claude-haiku-4-5 --allowedTools "mcp__*"
```
This uses the broad `mcp__*` wildcard, which **already covers all Slack MCP tools** (`mcp__slack__*` matches the pattern). Therefore, for users using the default startup command, tool permissions are not the issue.

**However**, the `getAgentStartupCommand` method (line 3831) reads the user's custom startup command first:
```ts
const config = await this.getStartupCommands(workspaceRoot);
let cmd = config[role] || '';
// ...fallback only applies when cmd is missing or blank
```
If the user has configured a custom startup command for `mcp_monitor` (via the agent config UI or `GlobalIntegrationConfigService`), that custom command overrides the fallback. A custom command that omits `--allowedTools "mcp__*"` or uses specific tool names that exclude Slack would cause Claude to error when attempting Slack tool calls.

**Action:** In the diagnostic step (Step 1), check whether the user has a custom startup command configured. If so, verify it includes `--allowedTools "mcp__*"` or `--allowedTools "mcp__slack__*"` or equivalent. If it does not, the fix is to update the custom startup command, not the prompt. No code change to `getAgentStartupCommand` is needed — the fallback is correct.

**Research-confirmed finding (web research):** The canonical `--allowedTools` wildcard for Slack MCP tools is `mcp__slack__*` (server name `slack` as namespace prefix). The broad `mcp__*` wildcard used in the fallback (line 3846) does cover Slack, but per research it also **bypasses permission prompts for ALL MCP servers** the user has configured — a security consideration. If a narrower scope is desired for the monitor role (which only needs Slack/Gmail/Calendar), the startup command could list `mcp__slack__* mcp__gmail__* mcp__google-calendar__*` explicitly rather than the blanket `mcp__*`. This is a optional hardening step, not a fix for the Claude error — the current `mcp__*` fallback is functional.
