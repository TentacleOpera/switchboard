# Fix kanban.db sql.js WASM Memory Exhaustion ("disk I/O error" across all DBs)

## Goal

Stop the extension host from dying with `Error: disk I/O error` thrown from `dist/sql-wasm.js` across **every** open `kanban.db` at once, by bounding the sql.js WASM heap: evict idle DB instances, coalesce the per-write full-DB serialization, and add **always-on GC** for the two append-only *telemetry* tables that dominate DB size (`plan_events`/`activity_log`). Completed `plans` rows are content, not bloat â€” they are deliberately left to grow indefinitely (see **Out of Scope**). Keep sql.js as the engine (bundling constraint); keep DuckDB strictly optional and additive.

### Core problem / background / root-cause analysis

**Symptom (observed 2026-07-08).** The extension began emitting a flood of `Error: disk I/O error` originating in `dist/sql-wasm.js` (`a.handleError` â†’ `a.tb`), hitting **every** workspace's `kanban.db` simultaneously â€” Gitlab, `be`, `viaapp`, `viaapp-web`, `fe`, `ai`, `funnel-sandbox`, `switchboard` (~9 resident DBs) â€” across `getConfig`, `hasPlan`, `getBoardFilteredByProject`, `getWorktrees`, `_writeKanbanStateBackup`, and `_writeLocalBoardMirror`. The final stack before the unhandled rejection was the **Copy Prompt** path (`getWorktrees` â†’ `_buildActiveWorktreePathMap` â†’ `_cardsToPromptPlans` â†’ `_generatePromptForDestinationRole` â†’ `_handleMessage`), which is why the user-visible trigger looked like "Copy Prompt produced the wrong prompt."

**What it is NOT (verified this session):**
- **Not disk-full** â€” 23 GiB free on the data volume; 244M free inodes.
- **Not file corruption** â€” `PRAGMA integrity_check` returns `ok` on the affected DBs; the relevant plan rows are intact and readable via the `sqlite3` CLI.

**Root cause.** sql.js runs SQLite compiled to WebAssembly over a **single linear memory (heap) shared by every DB opened in the extension host**. When that heap can't satisfy an allocation, SQLite's in-memory VFS `xRead`/`xWrite` fail and SQLite returns `SQLITE_IOERR`, which surfaces as "disk I/O error." Because the heap is shared, exhaustion manifests on **all** DBs at the same instant â€” exactly the observed all-at-once pattern. This is heap exhaustion, not physical I/O.

**Four contributing mechanisms (in `src/services/KanbanDatabase.ts`):**

1. **Full-DB serialize on every write.** `_persist()` (`:7139`) calls `this._db.export()` (`:7150`) â€” a fresh full-size `Uint8Array` copy of the *entire* DB â€” on **every** mutation (every card move, config upsert, etc.). It then also fires `_writeKanbanStateBackup()` (`:7179`), an **un-debounced** full JSON dump, on every persist. (Note: `_scheduleLocalMirror()` at `:7178` *is* already debounced at 500 ms â€” `:6967`/`:6948` â€” so the mirror is not the culprit; `export()` and the JSON backup are.) A burst of writes = a burst of whole-DB allocations + churn.

2. **No idle/LRU eviction of cached instances.** Instances are cached in static maps `_instances` / `_instancesByDbPath` (`:779`, set at `:936`â€“`:937`). They are dropped only on `dispose()` (`:1308`) or a stale-reload (`:965`) â€” never on idle. Background pollers (ClickUp automation, Antigravity rescan, `IntegrationAutoPull`) keep **all ~9 workspace DBs resident permanently** in the one WASM heap, including the 4.8 MB `switchboard` DB.

3. **Trigger: the refresh storm.** The known identical-distribution `refreshRunSheets` loop drives reads/writes in a tight cycle, churning the heap to exhaustion. (Tracked separately; cross-referenced here, not duplicated.)

4. **DB bloat â€” measured.** The `switchboard` board DB is **4.8 MB**. Per-table (via `dbstat`): `plan_events` = 6,030 rows â‰ˆ **2.1 MB with indexes (~44%)**; `activity_log` = 1,643 rows â‰ˆ **0.47 MB (~10%)**; `plans` = 1,638 rows â‰ˆ 1.6 MB (real content â€” stays). The two append-only, grows-forever, **forensic-value-decays** tables are **~54% of the DB**. Larger resident DBs = more WASM heap per instance = closer to the ceiling.

5. **Bonus contributor.** `_persist()` calls a **temporary** diagnostic `_diagFeatureSnapshot('persist')` (`:7141`, def `:7128`) on every write, which runs a `SELECT` and appends to `feature-clobber-diagnostic.txt` â€” observed at **10 MB** in one workspace. It is explicitly marked *"TEMPORARY â€¦ Remove with the other probes."* It adds per-write CPU + unbounded file growth and should be removed (or gated) as part of this work.

6. **Reload-swap leaks the previous DB image into the shared WASM heap (added 2026-07-09; corroborated by web research, verified in code).** `_reloadIfStale` (`:4994`) drains writes, saves `const previousDb = this._db` (`:5013`), then assigns `this._db = new SQL.Database(new Uint8Array(fileBuffer))` (`:5015`). On the **success** path `previousDb` simply falls out of scope â€” **`previousDb.close()` is never called**, and in fact `this._db.close()` appears *nowhere* in the file (only transient temp DBs at `:1119`/`:1144`/`:1239`â€“`:1240` are closed). In sql.js, dropping a `Database` without `close()` does **not** free its memory: the database's file buffer stays pinned forever in Emscripten's process-wide MEMFS registry (`FS.nodes`), invisible to V8's GC. So **every successful stale-reload leaks a full DB image** into the one shared heap. Under mechanism 3's refresh storm â€” which drives `_reloadIfStale` in a tight loop across all ~9 workspaces â€” this is a fast, unbounded native leak that climbs straight toward WASM's ~2 GB ceiling and then fails every DB's next MEMFS write as `SQLITE_IOERR`. This is likely a *dominant* contributor, not a minor one, and the fix is small and surgical: `previousDb?.close()` after a successful swap (and close the just-built instance on the rollback path at `:5025` before restoring `previousDb`). Workstream A owns this `close()` discipline.

**Why not just switch the DB engine?** The VSIX ships with **no `node_modules`** â€” webpack bundles everything into one `dist/extension.js`. Native SQLite (`better-sqlite3`) ships per-platform `.node` binaries that cannot be webpack-inlined; adopting it means per-OS/arch prebuilds, `electron-rebuild`/ABI matching, and abandoning the single-bundle model, across ~4,000 installs on many old versions. sql.js is pure WASM/JS and bundles cleanly. `node:sqlite` (built-in, Node 22.5+) is the natural *future* migration target but is gated on the Electron/Node version across the install base and still experimental â€” **out of scope here**. **DuckDB is not a board-engine candidate**: it is an OLAP/columnar engine (wrong shape for high-frequency single-row OLTP), its Node binding is a native module (same bundling blocker, larger binary), and today it is used only via a shelled-out `duckdb` CLI (`ArchiveManager.ts`, `execFile`) that **degrades gracefully when absent and is off by default**. Most installs lack the CLI, so DuckDB must **never** be load-bearing for board data.

## Complexity Audit

### Routine
- Removing the temporary `_diagFeatureSnapshot`/`appendFeatureClobberDiag` probe (dead diagnostic, explicitly marked for removal).
- Adding a `purgeOldPlanEvents(olderThanDays, minPerPlan)` method mirroring the existing `purgeOldTombstones(olderThanDays)` age-purge pattern (`:4367`).
- Wiring `cleanupActivityLog(...)` (already exists, `:7422`) into a periodic sweep (it currently has no scheduled caller).

### Complex / Risky
- **Idle-eviction with correctness**: must drain in-flight writes (`await existing._writeTail`, pattern at `:959`â€“`:960`) and run the synchronous final flush (`dispose()` â†’ `exportStateToFile()` â†’ `flushLocalBoardMirror()`, `:1308`/`:7098`) **before** `db.close()`, or risk losing the last write / emitting a truncated file. Must also survive a read arriving mid-eviction (lazy reopen must be race-safe against the eviction timer).
- **Persist coalescing without data loss**: debouncing `export()` means an in-memory write is not yet on disk; a crash or an external stale-reload in the debounce window must not lose or clobber it. Requires a dirty flag + forced flush on dispose, on eviction, and on the stale-reload path.
- **Retention migration on ~4,000 installs**: `plan_events`/`activity_log` shipped in released versions; pruning must age-out only, be idempotent, clamp bad inputs (as `purgeOldTombstones` clamps `olderThanDays` to â‰¥1), and never run as an unbounded `DELETE` that could nuke an audit trail. `VACUUM` to reclaim pages is itself a full-DB rewrite (transient memory spike) â€” must be throttled/opt-in, not per-prune.

## Edge-Case & Dependency Audit

- **Race Conditions:**
  - *Evict vs. in-flight write* â€” eviction must `await _writeTail` and force-flush before `close()`.
  - *Evict vs. incoming read* â€” a `getInstance()` arriving during eviction must either join the teardown then reopen, or be blocked until teardown completes; never operate on a half-closed `_db`.
  - *Debounced persist vs. stale-reload* (`_reloadIfStale`) â€” if a debounced write is pending and the file mtime changes, the pending in-memory state must win/flush before any reload, or be merged; do not silently discard.
  - *VACUUM vs. concurrent write* â€” VACUUM must hold the write chain (`_writeTail`) so no export races the rewrite.
- **Security:** none. No new external surface. Retention SQL uses parameterized queries (matches existing `_persistedUpdate`). Removing the diagnostic file-write *reduces* on-disk footprint of internal state.
- **Side Effects:**
  - Aged-out `plan_events`/`activity_log` rows become unqueryable for old forensics (acceptable by design; recent window preserved). Document in the retention setting's description.
  - Lazy reopen adds one-time latency (~init cost) to the first access of an evicted workspace's board.
  - Coalescing delays the on-disk mirror by up to the debounce window; the local board mirror / JSON backup / snapshot publisher fire off the *coalesced* persist, so their cadence drops too (a feature, not a regression).
- **Dependencies & Conflicts:**
  - Overlaps the **refresh-storm** fix (same hot path). This plan is independent and complementary â€” it bounds the heap regardless of trigger frequency â€” but landing both compounds the benefit. Do not duplicate the refresh-storm change here.
  - Migration rule (CLAUDE.md): shipped state migrates/preserves; retention must age-out, not wipe.

## Dependencies
- `sess_refresh_storm â€” identical-distribution refreshRunSheets loop` (the exhaustion *trigger*; tracked separately). This plan does not require it to land first, but the two are mutually reinforcing.

## Adversarial Synthesis

**Risk Summary:** The dangerous failure modes are all data-loss-on-teardown: evicting or debouncing without draining/flushing the last write, and a retention `DELETE` that ages out more than intended. Mitigations: reuse the existing `_writeTail` drain + `dispose()` final-flush before every `close()`, gate `export()` behind a dirty flag with forced flush on dispose/evict/stale-reload, and model the prune on the existing clamped, age-based `purgeOldTombstones` with a per-plan minimum-keep floor. Land the four workstreams independently (eviction â†’ coalescing â†’ retention â†’ probe removal), each behind its own verification, so a regression in one is isolated.

### `src/services/KanbanDatabase.ts` â€” Workstream A: idle-eviction of cached instances
- **Context:** static `_instances`/`_instancesByDbPath` (`:779`) never evict on idle; `getInstance` sets at `:936`â€“`:937`; drain-before-null pattern at `:959`â€“`:960`; `dispose()` at `:1308`.

## Uncertain Assumptions

Web research on sql.js / Emscripten memory internals was run on **2026-07-09**; it resolved the three original uncertainties (all now folded into the plan body above) and surfaced mechanism 6. Recorded here for auditability:

**Resolved / confirmed by research:**
1. **`db.close()` does not lower RSS.** WASM linear memory only grows; `close()` releases the DB's allocations back to the internal allocator for **reuse** and **unlinks its MEMFS file buffer** (this is what prevents mechanism-6's leak) but never shrinks `WebAssembly.Memory` or returns RAM to the OS. â†’ Workstream A's acceptance criterion reframed to "growth capped / leak reclaimed," not "RSS drops."
2. **`ALLOW_MEMORY_GROWTH=1` is on in standard sql.js builds; the ceiling is ~2 GB (32-bit WASM).** Exhaustion is a failed `memory.grow()`/`mallocâ†’0`, which the MEMFS write path surfaces as `SQLITE_IOERR`. â†’ Confirms the crash mechanism end-to-end.
3. **`export()` peaks ~2â€“3Ã— DB size** (active pages + MEMFS copy + serialized `Uint8Array` + Node `Buffer`). â†’ Coalescing (Workstream B) reduces the transient *peak*, not just churn. (Research also warned `export()` resets connection PRAGMAs â€” **verified a non-issue here**: this codebase sets no `journal_mode`/`synchronous`/`cache_size` PRAGMAs, only read-only `PRAGMA table_info`. No re-application needed.)

**Still genuinely open (version-specific â€” confirm before/during implementation, do not block planning):**
- The exact heap size at which `memory.grow()` fails in the Electron/Node build VS Code actually ships (host-resource- and version-dependent), which sets how aggressive the eviction TTL and debounce must be.
- Whether the pinned `sql-wasm` version's `export()` truly closes-and-reopens under the hood (the mechanism behind the PRAGMA-reset claim) â€” irrelevant to correctness here since no PRAGMAs are set, but confirm before relying on export() side effects.
- `node:sqlite` viability across the install base â€” research suggests VS Code 1.121+ (Electron 37 / Node 24) exposes it built-in with zero bundling, i.e. the "future" migration is closer than the plan's framing; still out of scope for this hotfix given ~4,000 installs on older versions, but worth re-checking before Phase 2 commits to keeping sql.js indefinitely.

## Out of Scope
- **Pruning completed/archived `plans` rows.** Plans are records, still rendered in the COMPLETED column, and their growth (~a few thousand rows / low-single-digit MB) is acceptable and **did not cause the crash**. Resident-heap pressure is solved by eviction (A) + coalescing (B) + telemetry GC (C) â€” never by deleting plan history. Bounding the *hot* DB at scale (so the per-write full-file serialize stops growing with lifetime plan count â€” noticeable around ~10k plans / ~10 MB) is handled **structurally** by the Phase-2 follow-up **`split_kanban_hot_cold_dbs.md`** (move dormant plans to a mandatory cold sql.js store, never delete). Out of scope for this hotfix.
- **Auto-archive behavior/defaults.** Whether to auto-move stale completed cards off the board is a **UX/board-tidiness** decision, separate from this reliability fix, and *not a memory lever regardless*: archiving is an `UPDATE` on the board row (the row **stays**) plus an optional DuckDB upsert â€” it reclaims nothing in `kanban.db`. Track as a separate UX plan if wanted; note AutoArchive ships **off by default** (`enabled: false`), so today most boards never archive at all.
- **DB engine swap** (`better-sqlite3` native prebuilds, `node:sqlite`, DuckDB-as-board-engine). Ruled out by the no-`node_modules` single-bundle constraint (see root cause); `node:sqlite` is a possible *future* migration once the Electron/Node baseline supports it across the install base.
