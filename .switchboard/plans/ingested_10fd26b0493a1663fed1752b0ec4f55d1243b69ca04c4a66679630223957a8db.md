# Fix ClickUp Sidebar Initial Load Bug

Fix the race condition where ClickUp tickets don't load on first sidebar load, requiring a manual refresh click. The issue occurs because the integration provider preference arrives after the initial state, preventing the hierarchy restore cascade from triggering.

## Root Cause

In `TaskViewerProvider.ts`, the `initialState` message (sent via `_sendInitialState()`) includes `clickupHierarchyState` but not the integration provider preference. The `integrationProviderPreference` is sent separately. In `implementation.html`, the hierarchy restore logic checks `if (clickUpSelectedSpaceId && lastIntegrationProvider === 'clickup')` before triggering the load cascade. If the provider preference message hasn't been processed yet, `lastIntegrationProvider` is still null, so the cascade doesn't trigger.

## Solution

Add the integration provider preference to the `initialState` message so the webview receives both pieces of information simultaneously, eliminating the race condition.
