# Fix Plan Delete Silent Failure and Zombie Plan Resurrection

Fix the plan delete button in the ticket view so it no longer silently fails or resurrects deleted plans from disk.

## Problem Summary

When clicking **Delete** in the review ticket view, the confirmation dialog appears but the plan remains alive after clicking Delete. The panel stays open and no error is shown. The specific plan that refuses to die is:

`/Users/patrickvuleta/Documents/GitHub/switchboard/.switchboard/plans/move-agent-prompt-config-to-kanban_move_agent_and_prompt_configuration_to_kanban_view.md`

## Root Cause Analysis

Three interacting bugs in the delete flow cause this behavior:

1. **Silent failure swallowing** — `ReviewProvider.ts` treats any `false` return from `_handleDeletePlan` as "user cancelled" and resets the UI to a success state, hiding the actual error from the user.
2. **Missing file deletion fallback** — `_handleDeletePlan` only deletes the local `.md` file if `sheet.planFile` is populated. If the runsheet has an empty/missing `planFile` (e.g., from a legacy or corrupted runsheet), the physical file is never removed.
3. **Zombie resurrection via sync** — After the failed deletion removes the DB runsheet/events but leaves the file on disk, the subsequent `_syncFilesAndRefreshRunSheets` call re-discovers the orphaned file and re-registers it as a brand-new plan.

## Proposed Changes

### 1. Fix ReviewProvider Silent Failure (`src/services/ReviewProvider.ts`)

Distinguish user cancellation from actual deletion failures. When `ok === false`, check if it was a cancellation (undefined/empty message) or an actual error. For actual errors, surface them in the UI instead of sending a success reset.

**Change:**
- Update the `deletePlan` case in `ReviewProvider.ts` message handler.
- If the backend command returns `false` AND no explicit cancellation was detected, post `ticketActionResult` with `ok: false` and an appropriate error message.
- Optionally: change `_handleDeletePlan` to throw with a descriptive message instead of returning `false` on actual failures, so the ReviewProvider catch block naturally surfaces it.

### 2. Add Fallback File Deletion in `_handleDeletePlan` (`src/services/TaskViewerProvider.ts`)

If `mirrorPath` cannot be resolved from `sheet.planFile`, attempt to locate and delete the actual plan file using:
- The `planFileAbsolute` from the `_currentPlan` context (passed in from ReviewProvider if available)
- A direct scan of `.switchboard/plans/` for a file matching the session's topic or a known naming convention

**Change:**
- If `sheet` exists but `sheet.planFile` is empty, log a warning and attempt to resolve the file path from a fallback.
- If no path can be resolved, show an explicit error to the user before returning `false`.

### 3. Prevent Zombie Resurrection (`src/services/TaskViewerProvider.ts`)

Add a short-lived "recently deleted" guard so `_syncFilesAndRefreshRunSheets` and `_reconcileOnDiskLocalPlanFiles` do not re-register a plan that was explicitly deleted seconds ago.

**Change:**
- Maintain an in-memory `_recentlyDeletedPaths` Set (with 10s TTL) in `TaskViewerProvider`.
- In `_handleDeletePlan`, add the deleted file path(s) to this set immediately before/after deletion.
- In `_reconcileOnDiskLocalPlanFiles`, skip files whose paths are in `_recentlyDeletedPaths`.
- Also skip calling `_syncFilesAndRefreshRunSheets` at the end of `_handleDeletePlan` if the file deletion failed, to avoid racing with the file watcher.

### 4. Add Diagnostic Logging to `_handleDeletePlan`

Add `console.log` statements at every decision point in `_handleDeletePlan` so the VS Code Developer Console reveals exactly where deletion fails for future debugging.

**Log points:**
- Resolved workspace root
- RunSheet found / not found
- `mirrorPath` resolved value
- `brainSourcePath` resolved value
- Confirmation dialog result
- Each file deletion attempt and result
- `deleteRunSheet` and `db.deletePlan` outcomes
- Final return value
