# Fix Plan Delete Silent Failure and Zombie Plan Resurrection

## Goal

Fix the plan delete button in the review ticket view so it no longer silently fails or resurrects deleted plans from disk.

## Metadata

**Tags:** bugfix, backend, workflow, reliability
**Complexity:** 7

## User Review Required

- Approve the change to throw errors from `_handleDeletePlan` (affects both Review and Kanban delete paths).
- Approve removing the outer catch-and-return-false safety net in `_handleDeletePlan` (lines 12267-12270) so thrown errors propagate to callers.

## Problem Summary

When clicking **Delete** in the review ticket view, the confirmation dialog appears but the plan remains alive after clicking Delete. The panel stays open and no error is shown. The specific plan that refuses to die is:

`/Users/patrickvuleta/Documents/GitHub/switchboard/.switchboard/plans/move-agent-prompt-config-to-kanban_move_agent_and_prompt_configuration_to_kanban_view.md`

## Root Cause Analysis

Three interacting bugs in the delete flow cause this behavior:

1. **Silent failure swallowing** — `ReviewProvider.ts` treats any `false` return from `_handleDeletePlan` as "user cancelled" and resets the UI to a success state, hiding the actual error from the user.
2. **Missing file deletion fallback** — `_handleDeletePlan` only deletes the local `.md` file if `sheet.planFile` is populated. If the runsheet has an empty/missing `planFile` (e.g., from a legacy or corrupted runsheet), the physical file is never removed.
3. **Zombie resurrection via sync** — After the failed deletion removes the DB runsheet/events but leaves the file on disk, the subsequent `_syncFilesAndRefreshRunSheets` call re-discovers the orphaned file and re-registers it as a brand-new plan. The root cause is `_reconcileOnDiskLocalPlanFiles` → `_reviveDeletedLocalPlanForPath` (line 9601), which calls `db.reviveDeletedPlans()` for ANY deleted local plan whose file still exists on disk.
4. **Critical fourth factor discovered during plan review:** The outer try-catch at lines 12267-12270 in `_handleDeletePlan` catches ALL thrown errors, shows a generic `showErrorMessage`, and returns `false`. This means any `throw new Error(...)` added at the inner error sites (lines 12193, 12202, 12214) will be swallowed by this outer catch and converted back to `return false` — which ReviewProvider will then treat as "user cancelled", recreating the exact same silent-failure bug through a different code path. The outer catch MUST be modified to re-throw, or the error contract change is futile.

### Complex / Risky
- **Error contract change (inner throws + outer catch re-throw):** Converting `return false` to `throw new Error` at the three inner error sites (lines 12193, 12202, 12214) AND modifying the outer catch (lines 12267-12270) to re-throw changes the method's contract from "returns `false` on any failure" to "returns `false` only for user cancellation, throws on all other failures". Both the ReviewProvider path (via `vscode.commands.executeCommand`) and the Kanban path (via webview message handler at line 7206) consume this method. The Kanban path is inside a try-catch at line 6438 that shows `vscode.window.showErrorMessage`. The ReviewProvider path has its own catch block at line 305. Both will correctly surface thrown errors, but this must be verified.
- **Race condition with file watcher:** `_syncFilesAndRefreshRunSheets` (line 12500) triggers `_reconcileOnDiskLocalPlanFiles` (line 9443), which in turn calls `_reviveDeletedLocalPlanForPath` (line 9546). Even if the file is successfully deleted, VS Code's native file watcher or a competing workspace process could recreate or re-scan the file within milliseconds. The `_recentlyDeletedPaths` guard must cover both the reconciliation loop AND any concurrent watcher callbacks.
- **Path resolution priority:** When `sheet.planFile` is empty, `_handleDeletePlan` must resolve the file path via the new `planFileAbsolute` parameter from ReviewProvider. If THAT is also missing (e.g., called from Kanban where the parameter won't be passed), the method must fail explicitly rather than falling back to fuzzy directory scanning.
- **Cross-workspace DB revival:** `_reviveDeletedLocalPlanForPath` queries by path and workspace. If a plan file exists in a shared location (brain directory) or multi-root workspace, the revival logic may activate the wrong workspace's record. The `_recentlyDeletedPaths` guard uses normalized absolute paths, which mitigates this for the local case but not for brain plans (those already have tombstones).
