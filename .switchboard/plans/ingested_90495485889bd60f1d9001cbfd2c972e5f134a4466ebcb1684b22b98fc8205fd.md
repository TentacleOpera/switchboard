# Add /clear-before-prompt option to kanban setup tab

Add a toggle in the kanban's **Setup** tab that, when enabled, sends `/clear` to a terminal 2 seconds before dispatching any prompt message. This applies to all terminal dispatches (batch, autoban, single-plan, and pair programming). Off by default, persisted globally.

---

## Goal

Add a global toggle in the kanban Setup tab that, when enabled, sends `/clear` to the target terminal 2 seconds before dispatching any prompt, giving CLI agents a clean context window. Off by default, persisted via VS Code global settings.

## Metadata
**Tags:** UI, workflow, frontend
**Complexity:** 4

## User Review Required

- Confirm that `/clear` (a CLI-agent slash command) is the intended clear mechanism, vs. the shell `clear` command. The current plan targets CLI agents (Claude Code, etc.) that accept slash commands — plain shell terminals will produce a "command not found" error.
- Confirm the 2-second fixed delay is acceptable, or whether it should be configurable.

### Complex / Risky
- **Persistence mechanism choice**: The plan originally specified `vscode.workspace.getConfiguration('switchboard')` (VS Code settings.json), but the existing `allowUnknownComplexityAutoMove` pattern uses `this._context.workspaceState` (internal workspace storage). Using `getConfiguration` is the correct choice here because: (a) the requirement says "persisted globally", and `workspaceState` is per-workspace; (b) `getConfiguration` with `true` target writes to global User settings, making the toggle survive workspace switches. **Clarification**: This is an intentional deviation from the `allowUnknownComplexityAutoMove` pattern, which uses workspace-scoped persistence. The global scope is appropriate because terminal clearing behavior is a user preference, not a per-workspace concern.
- **Analyst dispatch bypass**: `TaskViewerProvider.ts` line 13742 has a separate `sendRobustText` call in the analyst dispatch path that does NOT go through `_attemptDirectTerminalPush`. The analyst role is informational (not a coding agent), so clearing its terminal is likely undesirable. **Clarification**: The `/clear` injection should ONLY be in `_attemptDirectTerminalPush`, which handles all coding-agent dispatches. The analyst path is correctly excluded.
- **Terminal closure during delay**: If the terminal closes during the 2-second wait, the subsequent `sendRobustText` call will fail. This is an acceptable risk — `sendRobustText` already handles terminal errors gracefully.

## Edge-Case & Dependency Audit

### Race Conditions
- **Concurrent dispatches**: If two dispatches target the same terminal nearly simultaneously, both could send `/clear`, resulting in double-clearing and two 2-second delays (4s total). This is benign — the second `/clear` is a no-op, and the extra delay is minor. No mutex needed.
- **Delay vs. terminal state**: The 2-second delay is a heuristic. If a CLI agent processes `/clear` faster, the idle wait is wasted time. If it's slower, the prompt may arrive before the clear completes. 2 seconds is a reasonable default for known CLI agents (Claude Code, Gemini CLI).

### Security
- `terminal.sendText('/clear', true)` sends a literal string to the terminal shell. If a non-CLI-agent terminal is the target (e.g., a plain bash shell), `/clear` will produce a "command not found" error in the terminal. This is a cosmetic issue, not a security risk. The error appears in the terminal but does not break anything.

### Side Effects
- The 2-second delay adds latency to every dispatch when the toggle is on. For batch operations dispatching N plans, this adds 2N seconds. Users who enable this toggle are explicitly opting into this tradeoff.
- The `/clear` command erases the terminal's visible scrollback in most CLI agents. If a user was reviewing previous output, it will be lost. This is the intended behavior.

### Dependencies & Conflicts
- No cross-plan dependencies identified.
- No conflicts with existing features. The toggle is additive (off by default).
