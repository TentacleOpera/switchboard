# Plan: No Git Operations for Dispatched Agents

## Goal
Inject an unambiguous git-prohibition directive into every prompt generated by the canonical `buildKanbanBatchPrompt()` builder, and align all persona files so that no dispatched agent (lead, coder, intern) executes state-mutating git commands. The user/parent agent retains sole authority over repository state.

## Metadata
**Tags:** backend
**Complexity:** 3

## User Review Required
> [!NOTE]
> - This plan adds text to every dispatched agent prompt. Token cost increase is ~40 tokens per prompt — negligible relative to typical plan payloads (2k–8k tokens).
> - The `lead_coder.md` persona currently **encourages** git commits (line 17). This plan replaces that directive — verify the new wording matches your intent.
> - Read-only git commands (`git status`, `git log`, `git diff`) are **explicitly allowed** so agents can still inspect repo state for context.
> - Tool-level runtime enforcement (intercepting git commands at the shell layer) is **out of scope** — documented as a future enhancement.

So three of your five file changes (coder.md, intern.md, lead_coder.md) will be invisible to any agent dispatched through the standard Kanban flow. You're decorating a room nobody walks into.

2. **The original plan MISSED THE CANONICAL PROMPT BUILDER ENTIRELY.** It focused exclusively on persona files — the one injection point that's broken — and completely ignored `agentPromptBuilder.ts`, which is the *only* path that reliably reaches every dispatched agent. If I hadn't caught this, the "implemented" plan would have had zero actual effect. Spectacular.

3. **It's just text.** There is no enforcement. An agent running on Claude/GPT can simply… ignore it. "Please don't run git" is a suggestion, not a sandbox. The plan even marks tool-level enforcement as "optional." Optional! For a safety feature! The whole point is preventing repository corruption, and the mitigation is *asking nicely*.

4. **Token waste.** You're injecting ~40 tokens of prohibition text into EVERY prompt — planners, reviewers, coders, leads. Planners don't even execute code. Why are you telling the planner not to run git commands? That's like putting a "no swimming" sign in the desert.

5. **What about read-only git?** The prohibition says "no git operations" but `git log`, `git status`, and `git diff` are essential for agents to understand repo context. A blanket ban could cripple agent effectiveness. Did anyone think about this?

6. **The lead_coder.md actively ENCOURAGES git commits** right now (line 17: "you may group changes into logical commits"). So currently the system is schizophrenic — one file says "please commit" and the new plan says "never commit." Hope the agent doesn't read both.

### Balanced Response

Grumpy raises real issues. Here's how the improved plan addresses each:

1. **Persona injection bug — acknowledged, not blocked on.** The plan explicitly documents that persona file changes are defense-in-depth, not the primary mechanism. The `agentPromptBuilder.ts` change IS the primary enforcement and is NOT affected by the persona directory bug. The persona files are still worth updating because they're referenced directly in manual workflows and AGENTS.md, and they should be correct regardless.

2. **Prompt builder is now the centerpiece.** The improved plan adds the prohibition as a shared constant injected into every role branch of `buildKanbanBatchPrompt()`. This is the canonical path — every dispatch surface routes through it. This was the critical gap in the original plan.

3. **Text-only enforcement is the correct first step.** Tool-level interception requires runtime command parsing, a new extension API surface, and testing across platforms. That's a complexity-7+ plan on its own. Prompt-level prohibition is how every major AI system enforces behavioral constraints today. It's not perfect, but it's effective for well-behaved models and it can be shipped immediately. Tool-level enforcement is explicitly documented as a future enhancement (not "optional" — "deferred and tracked").

4. **Planner inclusion is intentional.** The planner generates plans that get dispatched to coders/leads. If the planner's prompt already contains the prohibition, it's more likely to generate plans that respect it (e.g., not writing "have the subagent commit the changes" into a plan). The 40-token cost is trivial — a single plan payload is 2k–8k tokens.

5. **Read-only git is explicitly allowed.** The improved directive text distinguishes between state-mutating commands (commit, push, reset, etc.) and read-only commands (status, log, diff). Agents can still inspect repo state.

6. **lead_coder.md contradiction is fixed.** Line 17 is replaced with an explicit prohibition, eliminating the conflicting directive.

### 5. Standalone Rule File (Reference Document)
#### [CREATE] `.agent/rules/no_git_for_agents.md`
- **Context:** Provides a single authoritative reference for the git prohibition policy. Can be linked from AGENTS.md, persona files, or future onboarding docs. Not wired into any code path — purely a reference document.
- **Logic:** Concise policy document (under 40 lines) covering: what's prohibited, what's allowed, rationale, and enforcement status.
- **Implementation:**

```markdown
# Rule: No Git Mutations for Dispatched Agents

## Policy
Agents dispatched via Switchboard (lead coder, coder, intern, and any
sub-agents they spawn) **MUST NOT** execute state-mutating git commands.

### Prohibited Commands
Any git command that changes repository state:
`commit`, `push`, `pull`, `fetch`, `merge`, `rebase`, `reset`,
`checkout` (branch switching), `branch` (create/delete), `stash`,
`cherry-pick`, `revert`, `tag` (create/delete), `am`, `format-patch`.

### Permitted Commands
Read-only git commands that inspect state without modifying it:
`status`, `log`, `diff`, `show`, `blame`, `shortlog`, `describe`.

### Rationale
1. **State coordination**: Only the parent agent or user manages repo state.
2. **Race conditions**: Concurrent git writes from multiple agents corrupt history.
3. **Rollback safety**: The parent agent must track all state changes for recovery.
4. **Atomicity**: Sub-agents cannot coordinate commits across parallel work.

### Enforcement
- **Prompt-level** (active): The `buildKanbanBatchPrompt()` function in
  `src/services/agentPromptBuilder.ts` injects a git prohibition directive
  into every dispatched prompt.
- **Persona-level** (active): All execution personas (lead_coder, coder,
  intern) include the prohibition in their behavioral rules.
- **Tool-level** (future): Runtime interception of git commands at the
  shell layer. Not yet implemented — tracked as a future enhancement.

### Exceptions
None. If an agent needs a git operation performed, it must return work
to the parent agent or user with an explicit request.
```

- **Edge Cases Handled:** Document is self-contained — no code dependencies. Under 40 lines as specified.

### 6. Tool-Level Enforcement (DEFERRED)
#### [DOCUMENT ONLY] — No file changes
- **Context:** Runtime interception of git commands (parsing shell invocations for `git ` prefix, rejecting or flagging them) would provide hard enforcement. This requires a new extension API surface, platform-specific shell integration, and extensive testing.
- **Status:** Explicitly **deferred**. Not in scope for this plan. Should be tracked as a separate complexity-7+ plan if desired.
- **Why deferred:** The prompt-level prohibition is the industry-standard approach for LLM behavioral constraints and is sufficient for well-behaved models. Tool-level enforcement adds defense-in-depth but is architecturally complex and warrants its own plan with proper design.
