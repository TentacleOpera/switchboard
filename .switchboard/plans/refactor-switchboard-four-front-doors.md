# Refactor Switchboard Entry Points to Four `switchboard-`Tagged Front Doors (Skillify Internal Workflows, No Host Drift)

## Goal

Reduce Switchboard's user-facing command surface to exactly **four self-named front doors** — `/switchboard`, `/switchboard-cloud`, `/switchboard-remote`, `/switchboard-memo` — and move every internal, extension-dispatched workflow (`improve-plan`, `improve-feature`, `accuracy`, `switchboard-orchestrator`) out of `.agents/workflows/` into stripped `.agents/skills/` bodies the extension reads by path. The result must be **identical in Antigravity and Claude Code** — no host drift — and must not break the planner automation on the ~4,000 installs that persist a workflow path in their DB.

### Problem & root-cause analysis

**What the user sees (the UAT failure).** In Antigravity the slash menu still shows `/switchboard-chat`, `/switchboard-index`, `/switchboard-manage`, `/switchboard-orchestrator`, `/switchboard-split` — the exact mess the prior consolidation was supposed to remove. The management console the user expected to reach via a single `/switchboard` is instead buried in a stripped skill the front door merely *points at*.

**Root cause 1 — `.agents/workflows/` is doing two incompatible jobs.** Antigravity (and Devin) natively expose **every** `.agents/workflows/*.md` as a typeable slash command, keyed purely by filename (confirmed by the user against the live Antigravity loader, and by `docs/imported_document_2026_07_09t00_31_11.md`). But the extension *also* uses that same folder as its internal prompt library — the planner reads `.agents/workflows/improve-plan.md`, the coder's accuracy add-on reads `.agents/workflows/accuracy.md`, the orchestrator persona is launched from `.agents/workflows/switchboard-orchestrator.md`. So internal machinery is structurally guaranteed to leak into the user's slash menu. There is **no frontmatter or manifest flag that hides a workflow file from Antigravity** — the only lever is *which folder the file is in*.

**Root cause 2 — host drift.** The prior fix operated on the wrong surface for the wrong host. `MIRROR_MANIFEST` (`src/services/ClaudeCodeMirrorService.ts`) renames `switchboard-index.md → /switchboard` and sets `invocation` gates — but that manifest is **Claude-Code-only**. Antigravity ignores it and keys on the raw filenames. So Claude Code got a clean-ish `/switchboard`, and Antigravity got `/switchboard-index` plus every leaked persona. The two hosts are driven by two different mechanisms that were never reconciled. Any fix must be verified against **both** surfaces or it will drift again.

**Root cause 3 — abandoned design.** The "single adaptive router that detects environment and routes" model (`switchboard-index.md`) added indirection users didn't want: locally it does almost nothing except forward to the management console, making the primary product a second-class citizen behind a hidden skill. The chosen model instead is **four explicitly-named doors** whose `description` frontmatter is self-explanatory; the user picks. Simpler, honest, and — because the doors are real files in `workflows/` — first-class in every host.

**Dead code.** The splitter agent was retired, so `switchboard-split.md` is dead code that should not exist. `switchboard-manage.md` (workflow) is a 27-line "thin router" whose entire body is *"go read the skill"* — a pure duplicate of the 524-line `switchboard-manage` skill.

### Grounded facts (verified in code, 2026-07-11)

- **Antigravity command surface = `.agents/workflows/*.md` filenames.** Skills are discovered from `.agents/skills/<name>/SKILL.md` **only when frontmatter is present**; a stripped (frontmatter-less) skill dir and flat `.md` files in `.agents/skills/` are ignored by discovery but remain readable via `view_file` (host doc `:10-40`).
- **Claude Code command surface = the mirror only.** There is no `.claude/commands/` dir; CC skills/commands come entirely from `.claude/skills/`, generated by `generateClaudeMirror` from `MIRROR_MANIFEST`. `invocation` gates: `default` = slash + model-auto; `no-model` = slash only; `no-user` = model-only (hidden from slash) (`buildSkillMd:306-323`).
- **The manifest reads the same `.agents/` bodies both hosts read.** `resolveSourceFile` joins `entry.source` to `.agents/`. So a single body file under `.agents/` is the shared source of truth; the manifest only decides how CC surfaces it. **This is the anti-drift lever: point the manifest at the same files Antigravity reads, with matching visibility.**
- **Extension → workflow-file coupling (must be repointed):**
  - Planner role: `DEFAULT_PLANNER_WORKFLOW = '.agents/workflows/improve-plan.md'` (`agentPromptBuilder.ts:743`) and a **DB-persisted** per-project `plannerWorkflowPath` (`:171, :912`; an `.agent→.agents` migration already exists — `planner-workflow-path-migration.test.js`).
  - Feature planner: `DEFAULT_FEATURE_PLANNER_WORKFLOW = '.agents/workflows/improve-feature.md'` (`:744`) — a code constant, not persisted.
  - Accuracy add-on: hardcoded `'.agents/workflows/accuracy.md'` strings appended to coder prompts (`:344, :1623`) — code constant, not persisted.
  - Orchestrator: `TaskViewerProvider.ts` builds the persona path `path.join(root, '.agents','workflows','switchboard-orchestrator.md')` (`:8094`) plus three prompt strings (`:8106, :8118, :9618`) — code, not persisted.
  - Management console dispatch: `TaskViewerProvider.ts:22197, :22278` read `.agents/skills/switchboard-manage/SKILL.md` — code, not persisted.
- **Dynamic-scan safety:** `generateClaudeMirror` auto-mirrors flat `switchboard-*.md` files in `.agents/skills/`. All existing flat files there use **underscores** (`web_research.md`, `clickup_api.md`, …) and none start with `switchboard-`, so relocating personas as skill **directories** (not flat `switchboard-*.md`) adds nothing to the CC surface.

## Design Decisions (stated, not deferred)

1. **Four front doors, all `switchboard-`tagged, each self-contained.** `workflows/` holds *only* these four; each carries a `description` that explains itself. The user picks — no environment-detecting router.
2. **The door body IS the persona** (no forwarding to a hidden skill). `/switchboard` contains the full management-console body inline; `/switchboard-cloud` the plan-brake; `/switchboard-remote` the remote-control playbook; `/switchboard-memo` the memo protocol.
3. **Internal, extension-dispatched workflows become stripped skills** under `.agents/skills/` — `improve-plan`, `improve-feature`, `accuracy`, `switchboard-orchestrator`. Invisible to Antigravity's menu, hidden from CC's slash menu (`no-user` where mirrored), read by the extension by their new path.
4. **`switchboard-manage` collapses into `/switchboard`.** The 524-line skill body moves into `workflows/switchboard.md`; the `switchboard-manage` skill and the `switchboard-manage.md` thin-router workflow are deleted. The engine's console dispatch repoints to `workflows/switchboard.md`.
5. **Delete dead code:** `switchboard-split.md` (splitter agent retired), `switchboard-index.md` (router model abandoned).
6. **`accuracy` is not a Switchboard door** — it's a coder prompt add-on — but it still must leave `workflows/`; it becomes a stripped skill the coder-prompt builder references by its new path.
7. **Parity is a first-class requirement.** Every logical door/skill has ONE `.agents/` body; the manifest mirrors the same files with matching visibility so Antigravity and Claude Code present an identical surface. Verification runs against both.
8. **Migrate shipped state.** The persisted `plannerWorkflowPath` is released user state; moving `improve-plan.md` requires a DB migration that rewrites the old default to the new skills path (preserving user-custom values).

## Host Surface Parity (Antigravity ↔ Claude Code) — the anti-drift contract

End state, asserted for **both** hosts:

| Logical entry | `.agents/` body (single source) | Antigravity sees | Claude Code manifest | CC sees |
| :-- | :-- | :-- | :-- | :-- |
| `/switchboard` | `workflows/switchboard.md` | slash cmd (file present) | `{source:'workflows/switchboard.md', name:'switchboard', invocation:'default'}` | slash cmd |
| `/switchboard-cloud` | `workflows/switchboard-cloud.md` | slash cmd | `…name:'switchboard-cloud', default` | slash cmd |
| `/switchboard-remote` | `workflows/switchboard-remote.md` | slash cmd | `…name:'switchboard-remote', default` | slash cmd |
| `/switchboard-memo` | `workflows/switchboard-memo.md` | slash cmd | `…name:'switchboard-memo', default` | slash cmd |
| improve-plan (internal) | `skills/improve-plan/SKILL.md` (stripped) | invisible, readable by path | `{source:'skills/improve-plan', name:'improve-plan', invocation:'no-user'}` | hidden from slash, model-loadable |
| improve-feature (internal) | `skills/improve-feature/SKILL.md` (stripped) | invisible | `…name:'improve-feature', no-user` | hidden |
| accuracy (internal) | `skills/accuracy/SKILL.md` (stripped) | invisible | `…name:'accuracy', no-user` | hidden |
| switchboard-orchestrator (engine-launched) | `skills/switchboard-orchestrator/SKILL.md` (stripped) | invisible | **not in manifest** (launched by path) | not a CC skill |

**Parity invariant:** the set of *user-visible commands* is `{switchboard, switchboard-cloud, switchboard-remote, switchboard-memo}` on **both** hosts, and every internal body is reached by path (never as a user command) on both. Because each row's Antigravity body and CC manifest `source` are the **same file**, content cannot drift.

## Proposed Changes

### A. `.agents/workflows/` — reduce to the four doors
- **Create/relocate the four door bodies:**
  - `switchboard.md` — new front door; **absorbs the full `switchboard-manage` console body** (verbatim), with a one-line `description` (e.g. "Local Switchboard management console — drive the board when the VS Code extension is running"). **Sequencing (feature-internal):** the sibling subtask *"Make the Switchboard Manage Console Speak Human and Offer the Next Step"* edits this same console body at its current location (`.agents/skills/switchboard-manage/SKILL.md`) and **must land first** — this refactor then absorbs the already-humanized body verbatim. Landing this refactor first would delete the sibling plan's target file and invalidate every line reference in it.
  - `switchboard-cloud.md` — from `switchboard-chat.md` body (cloud-VM plan-brake); `description` explains "plan first, don't auto-code in a remote VM".
  - `switchboard-remote.md` — from `sw-remote.md` body (remote-control via Notion/Linear MCP proxy); `description` explains the remote-control use.
  - `switchboard-memo.md` — from `memo.md` body (rename only); `description` unchanged in meaning.
- **Delete:** `switchboard-index.md`, `switchboard-manage.md`, `switchboard-split.md`.
- **Remove after relocation:** `switchboard-chat.md`, `sw-remote.md`, `memo.md`, `improve-plan.md`, `improve-feature.md`, `accuracy.md`, `switchboard-orchestrator.md` (their bodies now live in doors or skills).

## Migration & Compatibility

- **Stale workspace files removed (see §F) — the make-or-break migration.** Existing installs carry the old workflow files in their `.agents/workflows/`; the copy step never deletes source-removed files. Without adding the retired names to `cleanupLegacyAgentFiles` + the blocklist, Antigravity keeps showing the old commands post-update — the exact prior UAT failure, at install scale. This is mandatory, not cosmetic.
- **Released user state migrated:** the persisted `plannerWorkflowPath` (see §G). No-op for users on the default (fallback constant already updated) except where the default was explicitly persisted; rewrite is idempotent and match-gated.
- **Control-plane source files** (`.agents/`, `AGENTS.md`) are not user data — no archival; the mirror regenerates `.claude/` on rebuild.
- **No `MIGRATION_Vnn_SQL` body is edited**; the new migration is appended as the next version.
- **Backwards muscle-memory:** `/memo`, `/improve-plan`, `/sw-remote`, `/switchboard-split` stop existing as commands by design. The user manual documents the rename; `/switchboard-memo` replaces `/memo`.

## Non-Goals

- Touching background proxy skills (`clickup-*`, `linear-*`, `web-research`, …) — already stripped/invisible.
- Building any new MCP capability; Cowork remains served by the separate uploaded `switchboard-cowork` skill (not a door).
- Reworking `accuracy`'s behavior — only its location + the references to it.

## Complexity Audit
### Routine
- The file moves and the manifest edits are mechanical and pattern-reuse (frontmatter strip + `descriptionFallback` already established).

### Complex / Risky

- **Cross-file atomicity** — a half-applied change (file moved but constant not repointed) breaks planner/orchestrator dispatch at runtime.

## Adversarial Synthesis
Key risks: (1) **stale files on existing installs** — moving files in source while the old copies linger in ~4,000 workspaces means Antigravity keeps showing the old commands post-update; this, not the repo state, is the true UAT surface, mitigated by extending `cleanupLegacyAgentFiles` + the blocklist (§F) and verifying against a simulated dirty install; (2) host drift — fixing one surface and not the other, the prior in-repo failure, mitigated by the parity table + dual-host verification; (3) breaking the planner on shipped installs via the `plannerWorkflowPath` move, mitigated by a match-gated, idempotent DB migration plus the updated default constant; (4) partial application leaving a dispatch pointing at a moved file, mitigated by a post-change grep gate for any surviving `.agents/workflows/{improve-plan,improve-feature,accuracy,switchboard-orchestrator,switchboard-chat,sw-remote,memo,switchboard-index,switchboard-manage,switchboard-split}.md` reference in `src/`.
